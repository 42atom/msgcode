## Context

- heartbeat 现在已经有巡检主链，下一刀不是加新控制面，而是让它先按 wake 合同推进。

## Goal / Non-Goals

- Goal:
  - heartbeat 优先消费 pending wake record
  - 明确 `mode=now|next-heartbeat` 的消费差异
  - 固定轻路径 hint 与重路径 capsule 的进入条件
- Non-Goals:
  - 不新增第二任务状态系统
  - 不把 heartbeat 扩成大编排器
  - 不先做 vitals 背压

## Plan

- [x] 固定 heartbeat consume 顺序：`pending wake -> task facts -> generic巡检`
- [x] 明确 `now` 立刻消费、`next-heartbeat` 只投递到下一拍
- [x] 明确轻路径只读 hint，重路径必须命中 capsule
- [x] 补回归：wake consume 优先级与 mode 分流

## Acceptance Criteria

1. heartbeat 会优先消费 pending wake record，而不是先做泛泛巡检。
2. `mode=now` 与 `mode=next-heartbeat` 行为清楚且可测。
3. 轻路径不强制建 capsule；重路径无 capsule 不进入正式执行。
4. wake consume 不会把 heartbeat 变成第二任务状态系统。

## Notes

- 这张单是 consume 接线，不补 vitals。

## Findings

- heartbeat 先 wake、后 task 的执行顺序已经固定在：
  - [wake-tick.ts](/Users/admin/GitProjects/msgcode/src/runtime/wake-tick.ts)
- 实际消费逻辑已收在：
  - [wake-heartbeat.ts](/Users/admin/GitProjects/msgcode/src/runtime/wake-heartbeat.ts)
- `mode=now` 与 `mode=next-heartbeat` 的分流已由 schedule -> wake 主链锁住：
  - `now` 会在 record 落盘后尝试立刻接到 heartbeat
  - `next-heartbeat` 只投递到下一拍消费
- 轻路径 / 重路径边界已固定：
  - 无 `taskId` 只回 `hint`
  - 有 `taskId` 才进入 capsule 主链
- 这层没有新增第二状态系统：
  - heartbeat 只消费 wake 合同
  - 真相仍在 wake record / task / dispatch 文件面
