## Context

- 当前 `todo` CLI 直接把 workspace 待办落到 `.msgcode/todo.db`。
- 相比 `memory`，`todo` 的状态更简单，也更接近纯文件场景。
- 但当前还缺一个判断：`todo` 是否真的需要和 `issues/` 并存，还是只是历史遗留的轻待办壳。

## Goal / Non-Goals

- Goal:
  - 先判断 `todo` 是否应该独立存在
  - 若应保留，再把 `todo` 存储收口到 `.msgcode/todo.json`
  - 保持 `msgcode todo add|list|done` 合同稳定
  - 让 AI 能直接读写、grep、diff todo 状态
- Non-Goals:
  - 不顺手重做 `schedule`
  - 不引入新的 todo 服务层
  - 不为历史 `todo.db` 长期保留双写兼容

## Plan

- [x] 判断 `todo` 的独立价值：
  - 与 `issues/` 的边界是什么
  - 是否只是历史轻待办壳
  - 是否值得继续保留独立命令面
- [x] 若保留，定义 `.msgcode/todo.json` 最小结构：
  - version
  - updatedAt
  - items
- [x] 若保留，对齐现有 todo 语义：
  - `pending`
  - `done`
  - `createdAt`
  - `doneAt`
- [x] 若保留，把 `add / list / done` 改为直接读写 JSON
- [x] 若保留，使用原子写入，避免半写盘
- [x] 若保留，明确历史数据处理方式：
  - 若存在 `todo.db`，是否一次性导入
  - 导入后是否停止继续读写 `.db`
- [x] 若保留，补一组 file-first regression lock

## Acceptance Criteria

1. 仓库内有一个明确判断：`todo` 是保留还是退役。
2. 若保留：
   - `msgcode todo add|list|done` 不再依赖 `todo.db`
   - workspace 待办真相源变为 `<workspace>/.msgcode/todo.json`
3. 若保留，todo 文件满足：
   - AI 可直接读取
   - AI 可直接编辑
   - 可 `grep`
   - 可 `git diff`
4. 若保留且历史 `todo.db` 仍需处理，只允许一次性迁移；不长期保留双真相源。
5. 若退役，必须明确：
   - todo 能力由谁替代
   - 与 `issues/` 的迁移边界
   - 不留下半死不活的双入口

## Notes

- 这单比 `memory` 更直接，但仍要先过一次奥卡姆检查：它是否真的需要继续存在。
- 若 `todo` 保留，file-first 更自然。
- 真正要避免的是：
  - `todo.json` 和 `todo.db` 长期双写
  - 迁移后又发明新的中间包装层
- 当前判断：
  - `todo` 应保留，但边界必须收薄
  - 它不替代 `issues/`
  - 它只承载轻量、短期、无需完整任务文档的待办
- 本轮已补：
  - 主真相已切到 `<workspace>/.msgcode/todo.json`
  - `todo.db` 若存在，只做一次性导入
  - 导入完成后改名为 `todo.db.legacy.bak`
  - 不做长期双写
