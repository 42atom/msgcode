# 任务：强状态图腾 allowlist

## Parent Task

- `tk0361`

## Context

当前仓库已经有足够多的真相源：

- `issues/`：任务状态
- `.msgcode/dispatch/`、`.msgcode/subagents/`：执行连续性
- `.msgcode/status.log`：共享近况
- `.msgcode/STATUS`：只读快照

在这种前提下，强状态图腾最容易出问题：

- 一旦给得太多，就会变成第二状态系统
- 一旦语义不硬，就会和 `status.log`、`issues/`、运行时快照互相打架

## Goal / Non-Goals

### Goal

- 明确当前 allowlist 是否为空
- 固定强状态图腾准入条件
- 阻止普通状态偷渡成图腾文件

### Non-Goals

- 不写代码
- 不新增 flag 系统
- 不碰普通业务状态表达
- 不扩现有 runtime 主链

## 结论

当前 allowlist = 空。

也就是：

- 这轮不批准任何新的强状态图腾文件
- 未来若要新增，必须先拿真实失败证据证明现有真相源无法表达

## Admission Test

只有同时满足下面四条，才允许未来进入 allowlist：

1. 状态必须跨进程、跨重启继续生效
2. 状态不能从 `issues/`、dispatch、subagent、`status.log`、只读快照推导出来
3. 状态必须允许人工直接介入或解除
4. 状态错误会改变整条执行主链，而不只是某个局部业务分支

任一不满足，就不配升格成图腾文件。

## Explicitly Rejected

以下内容明确不允许做成强状态图腾：

- 普通任务状态
- 当前线程工作进度
- 最近一次错误
- 最近一次验证结果
- 普通阻塞原因
- UI 展示态
- 任意业务 feature flag

这些内容要么已有真相源，要么应继续留在普通文件或运行时上下文里。

## Reserved Candidates

即便未来允许，也只可能落在这一类极少数人工可介入状态：

- 全局暂停
- 人工接管
- 全局发送锁

但本轮没有一项拿到足够失败证据，所以仍然不入 allowlist。

## Guardrails

- 默认空列表
- 新状态入列必须先补失败证据，不接受“先预留着”
- 图腾文件一旦存在，也只能表达极少数强状态，不能扩散为普遍 flag 系统

## Plan

- [x] 盘点现有真相源是否已足够表达状态
- [x] 固定强状态准入四条件
- [x] 明确本轮 allowlist 为空

## Notes

- 这张单的核心结论就是“默认空”。
- 未来若真要入列，必须先补真实失败证据，再回来改 allowlist。

## Acceptance Criteria

1. 当前 allowlist 明确为“空”，不是语义含糊的候选池。
2. 强状态准入条件清楚，且要求真实失败证据先行。
3. 普通业务状态和共享近况不会被误升格成图腾文件。
