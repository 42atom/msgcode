## Context

- 用户已经明确确认：`session continuity` 当然要做，而且会认真做，但它必须建立在 `work continuity` 的地基之上。
- 当前仓库已有 heartbeat、task supervisor、subagent 等运行时积木，也已有 docs-first task truth，但还没有把“session 如何服务工作恢复”正式收口。
- 对比外部系统的结论很清楚：如果先追聊天自然度和 transcript 延续，而不先钉死任务真相源与恢复协议，最终得到的很可能只是一个健谈但不可靠的 agent。

## Goal / Non-Goals

- Goal:
  - 为 msgcode 定义正确的 session continuity 建造顺序
  - 明确 session 需要消费哪些工作真相源
  - 明确 compaction / memory / transcript / rebuild 如何服务工作连续
  - 明确 session 只是感官层缓存，而不是工作真相源
- Non-Goals:
  - 不让 session 反过来成为任务真相源
  - 不把聊天自然度优先级排在任务恢复之前
  - 不先实现大而全的记忆系统

## Plan

- [x] 写清楚 `session continuity` 的职责边界：
  - transcript continuity
  - memory injection
  - compaction / rebuild
  - conversational naturalness
- [x] 明确 session rebuild 时必须优先读取哪些工作真相源
- [x] 明确 session rebuild 优先“即时重构必要上下文”，而不是同步沉重 transcript
- [x] 冻结 rebuild 的输入 -> 处理 -> 输出：
  - 输入：`issues/*.md`、dispatch records、task checkpoint、subagent records、artifact / verify 证据、可选 transcript tail
  - 处理：先回答“还欠什么 / 做到哪 / 从哪续 / 证据在哪”，再选择恢复指针与活动子任务
  - 输出：最小 work capsule，供 session bootstrap / prompt injection 使用
- [x] 明确 `session continuity` 与 `truth continuity / work continuity / process continuity` 的依赖顺序
- [x] 规定后续任何 session 设计都不得覆盖 `issues/` 与 workspace dispatch truth
- [x] 规划最小实现挂点，而不是先做大规模会话平台改造

## Acceptance Criteria

1. 仓库内明确写死：session continuity 不是地基，work continuity 才是地基。
2. 存在清晰顺序：
   - `truth continuity`
   - `work continuity`
   - `process continuity`
   - `session continuity`
   - `persona continuity`
3. session rebuild 设计必须先回答：
   - 还欠什么
   - 做到哪
   - 从哪续
   - 证据在哪
4. transcript / memory / compaction 的职责是服务工作恢复，而不是替代工作真相源。
5. session continuity 的恢复不要求先完整同步 transcript；系统应能基于工作地基即时重构必要上下文。
6. “必要上下文” 至少包括：
   - 当前未完成父任务 / 子任务
   - 每个活跃任务的恢复指针
   - 关联 artifact / verify 证据
   - 活动 subagent / dispatch 绑定
7. rebuild 输出必须是一个最小 work capsule，而不是整份 transcript 回放。
8. 该单只能在 `tk0205` 之后正式推进实现。

## Notes

- `session continuity` 更像感官层缓存，不是真相源。
- 真正的信任建立在交付连续性上，而不是会话自然度上。
- 这单不是反 session；这单是给 session 找正确位置。
- `即时重构必要上下文` 的最小形态应像 work capsule：
  - open tasks
  - checkpoint / nextAction
  - verify / artifact refs
  - active dispatch / subagent refs
- 当前最短缺口已经明确：
  - `WorkCapsule.childTasks` 不能只从 active dispatch 推导
  - 父任务文档里已声明、但尚未派发的子任务，也必须在 rebuild 时可见
- 本轮已补第一刀：
  - `buildWorkRecoverySnapshot()` 现在会从父任务 `## Child Tasks` / `implicit.waiting_for` 收子任务
  - `snapshot.drift` 顶层字段已与 `workCapsule.drift` 对齐，避免调用方猜结构
- 本轮已补第二刀：
  - 当没有 runtime / dispatch checkpoint，但父任务下仍有 `pending` 子任务时，`workCapsule.nextAction.type` 不再退回 `manual`
  - 现在会明确返回 `dispatch`，并把下一张待派发子任务 id 放进 `nextAction.params.childTaskId`
- 本轮已补第三刀：
  - `TaskSupervisor.resumeTask()` 不再粗暴把已有恢复指针改写成“继续执行当前任务”
  - 现在若 blocked 任务已经有更具体的 `checkpoint.nextAction`，恢复时会原样保留
- 本轮已补第四刀：
  - `TaskSupervisor.resolveDispatchWorkItem()` 重建 work capsule 时，开始带上现有 runtime cache task
  - 这避免了 doc-first dispatch 恢复时，被较粗的 dispatch checkpoint 把更精确的 runtime 恢复指针冲掉
- 本轮已补第五刀：
  - `buildWorkRecoverySnapshot()` 选 dispatch checkpoint 时，不再错误取最老那条
  - 现在会优先取最新 dispatch 的 checkpoint，避免恢复时把旧派单摘要当成当前恢复锚点
- 本轮已补第六刀：
  - `WakeWorkCapsule.sourceStamp.issueStateNames` 不再错误写入 issue slug
  - 现在记录的是 issue 文件名列表，source stamp 才真正对齐文件真相源，能用于后续漂移校验
- 本轮已补第七刀：
  - `wake-heartbeat` 真实主链不再把 `runtimeTask` 丢成 `null`
  - 现在消费 wake 时，会先读取同 task 的 active runtime task，再把 checkpoint 一起带进 `WakeWorkCapsule`
  - 这样 wake 恢复拿到的就不是“只有文档和派单”的半套胶囊
- 当前收口判断：
  - 这张单到这里已经完成了“定顺序 + 冻协议 + 补最小挂点”的目标
  - transcript / memory / compaction 的更厚实现仍然是后续层收益，不属于这张单继续深挖的范围
