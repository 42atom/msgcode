## Context

- 仓库级原则已经收口为：能落成文件真相源的，先落文件；不能落成真相源的，只做薄 runtime。
- 当前 `memory` 原始数据已经在 `<workspace>/memory/*.md`，但检索索引默认落在 `~/.config/msgcode/memory/index.sqlite`。
- 现在真正要回答的，不是“要不要情绪化地反数据库”，而是：
  - `memory` 的真相层到底是什么
  - `index.sqlite` 是真相层、索引层，还是缓存层
  - 当前实现是否过黑、过重、过难迁移

## Goal / Non-Goals

- Goal:
  - 明确 `memory` 的 truth / index / cache 边界
  - 判断 `index.sqlite` 是否应降权、替换或保留
  - 给出 file-first 口径下的最小可行收口方案
- Non-Goals:
  - 不预设“一定彻底删除 SQLite”
  - 不直接重写整套记忆检索实现
  - 不打断当前 `tk0205` 主线实施

## Plan

- [x] 盘清 `memory` 当前真实结构：
  - 原始文件
  - chunk 生成
  - FTS 索引
  - 向量索引
  - 检索入口
- [x] 明确三层边界：
  - truth
  - index
  - cache
- [x] 判断 `index.sqlite` 当前是否越权：
  - 是否承载了只有它知道的事实
  - 离开它能否仍从文件真相源恢复
  - AI 是否必须通过黑箱接口才能使用记忆
- [x] 比较三种方案：
  - 保留 SQLite，但降权为显式、可导出、可调试的派生索引
  - 改为纯文件 chunk/index
  - 混合：文件 truth + 可丢弃索引
- [x] 给出建议路线与迁移成本评估

## Acceptance Criteria

1. 仓库内有一份明确判断，区分 `memory` 的：
   - truth layer
   - index layer
   - cache layer
2. `index.sqlite` 被明确归类为以下之一：
   - 可接受的派生索引
   - 需要降权的黑箱索引
   - 应替换的过重实现
3. 有一份具体建议，回答：
   - 原始真相源存哪
   - AI 默认读什么
   - 检索加速层是否可丢
   - 索引如何做到可见、可导出、可调试
   - 迁移后如何避免 SDK-only 黑箱
4. 不把这单写成“先宣判、后找证据”的运动式改造。

## Notes

- 下一张现役实现单已拆出并收口：
  - [tk0228.pss.memory.p1.append-log-truth-and-rebuild-contract.md](/Users/admin/GitProjects/msgcode/issues/tk0228.pss.memory.p1.append-log-truth-and-rebuild-contract.md)

- 当前直觉是：`memory` 的问题更像“索引层太黑”，不一定是“只要有 SQLite 就错”。
- 更稳的审判标准是：
  - 真相是不是文件
  - 索引丢了能不能恢复
  - AI 能不能直接 `cat / grep / diff` 自己的家底
- 更可能的方向不是“数据库清零”，而是：
  - 降权
  - 显式化
  - 可导出
  - 可调试
- 本轮补充一条硬原则：
  - `append` 才是记忆真相根
  - 压缩 / 摘要 / 索引 都只能是派生层
  - 这些派生层可以帮助读取、检索、注入，但不能反过来覆盖或替代 append 真相
- 对应工程口径应固定为：
  - 原始追加先落盘
  - 再做 chunk / index / summary
  - 任一派生层损坏或过时时，必须可从 append log 重建
- 这条原则的价值不在“多存一点”，而在：
  - 可追溯
  - 可重算
  - 可迁移
  - 不让摘要版本绑架未来理解
- 当前已确认的第一批事实：
  - truth 层仍然是 `<workspace>/memory/*.md`
  - index 层默认落在 `~/.config/msgcode/memory/index.sqlite`
  - `src/cli/memory.ts` 的 `index` 命令目前仍有两个明显未收口点：
    - `TODO: 检查是否需要重新索引（mtime/sha256）`
    - `TODO: 计算 sha256`
  - 当前实现里 `sha256` 实际被写成空字符串，这意味着 SQLite 里保存了文档记录，但还没有真正锁住内容指纹
  - 这进一步说明：当前 SQLite 更像“可丢的派生索引”，还不够资格升格为 truth layer
  - `listener.ts` 当前虽然会探测 `vectorAvailable`，但实际检索仍只调用 `store.search()` 这条 BM25/FTS 路径
  - 也就是说，当前正式可工作的主链是 `FTS-only`，不是“真正已接通的 hybrid retrieval”
- 当前第一判断：
  - `index.sqlite` 现在不该被当成记忆真相
  - 更合理的口径是：文件是 truth，SQLite 是可重建索引
  - 下一步应继续审计：离开 SQLite，AI 还能否直接基于 append/file truth 工作；哪些能力只是“查得更快”，哪些事实却被索引独占
- 当前第二批事实：
  - `listener.ts` 已改成对外诚实回报：当前 `memoryMode` 固定为 `fts-only`
  - 这次修改不是降级，而是把“能力探测”和“真实已接通主链”分开
  - 当前仓库里虽然存在 `searchVector()` / `searchHybrid()` 代码，但主入口：
    - `listener.ts`
    - `memory search`
    仍都只走 `store.search()`
  - 进一步看 `searchVector()` 的 SQL，还引用了 `fts.chunk_id_hash`，但当前 FTS 表结构并未创建这个字段
  - 这说明向量/混合检索更像“存在代码形状”，还不是已被主链消费、已被行为测试锁实的正式能力
- 当前第二判断：
  - 资源管理这条线上，必须区分：
    - 已正式可用的 `FTS-only` 索引能力
    - 尚未接通主链的 `vector/hybrid` 候选能力
  - 对外口径应先说真，再决定要不要把 hybrid 做成正式能力
- 当前三层边界判断：
  - truth layer：
    - `<workspace>/memory/*.md`
    - `memory add` 的 append-only 追加内容
  - index layer：
    - `~/.config/msgcode/memory/index.sqlite`
    - `documents / chunks / FTS` 这些可重建结构
  - cache layer：
    - 当前基本不存在正式 cache 层
    - `EmbeddingCacheEntry` 只有类型，没有真正落地主链实现
- 对 `index.sqlite` 的定性：
  - 当前应归类为“需要降权的派生索引”
  - 不是 truth layer
  - 但也没必要在这轮情绪化删除
- 三种方案比较后的建议：
  - 方案 A 保留 SQLite，但明确降权：
    - 成本最低
    - 最符合当前主链
    - 也是本轮建议路线
  - 方案 B 改为纯文件 chunk/index：
    - 更透明
    - 但会增加检索与更新成本
    - 当前没有足够收益支撑立即迁移
  - 方案 C 文件 truth + 可丢弃索引：
    - 本质上与方案 A 接近
    - 差别只在是否把索引导出/调试能力正式补齐
- 推荐路线：
  - 保留 SQLite
  - 明确降权为可丢弃索引
  - 先收三件事，不新开大重构：
    - `memory index` 真正补上 `sha256 + dirty detection`
    - 对外口径继续诚实：当前正式检索是 `FTS-only`
    - 为后续 `stats / debug` 增加更清楚的可见性，避免 SQLite 继续当黑箱
- 迁移成本判断：
  - 低成本项：
    - 补 `sha256`
    - 补 dirty detection
    - 收口混合检索口径
  - 高成本项：
    - 彻底移除 SQLite
    - 改纯文件 chunk 索引
    - 把 hybrid/vector 真接成正式主链
- 当前收口：
  - 这张单已经完成“判断和定性”
  - 下一步若继续，不该再写抽象宣言，而应拆成具体实现单
