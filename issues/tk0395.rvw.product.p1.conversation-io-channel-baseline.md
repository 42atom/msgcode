# 任务：Conversation IO Channel Baseline

## Context

现在要把语音输入输出能力收成 msgcode 自己的产品边界，而不是继续依赖外部项目形态。

目标不是兼容 `openclaw` 插件，也不是直接把 `clawpod` 吃进内核，而是先定义：

- 会话输入从哪里来
- 会话输出到哪里去
- 哪些是本机音频设备
- 哪些是远程桥接

当前最关键的物理事实是：

1. `system speaker / 外接音响 / 蓝牙耳机` 都属于本机音频设备族
2. `HomePod` 不属于本机音频设备；它本质上是 Siri/Shortcut/HTTP 桥接出来的远程语音入口
3. 产品层可以统一为“会话通道”，实现层不能把这两族混成一个设备抽象

## Goal / Non-Goals

## Goal

- 定义 `input_channel` 与 `output_channel` 的最小真相
- 区分本机音频设备与 HomePod bridge 两类通道
- 固定后续实现顺序：先本机音频，再 HomePod bridge pack
- 为 settings / CLI / runtime 提供统一口径

## Non-Goals

- 不写 ASR/TTS/音频路由代码
- 不写 HomePod HTTP server
- 不做 openclaw 兼容层
- 不设计大一统语音平台
- 不修改现有 UI 页面

## 基线结论

产品层统一抽象为：

- `input_channel`
- `output_channel`

实现层先分两族：

1. 本机音频设备族
   - `system-mic`
   - `system-default`
   - `audio-device:<device-id>`

2. 远程桥接族
   - `homepod-bridge`

其中：

- `homepod-bridge` 不是音频设备
- `homepod-bridge` 是可选 bridge pack，不是 core appliance 硬依赖

## UI Placement

- 首页 `main.html` 顶栏右侧只显示一个轻量状态入口：
  - `语音外放设备：...`
- 详细配置放在 `settings.html -> general`
- 首页不展开输入通道与播放策略，不把主窗口做成音频控制台

## 最小实现顺序

1. 先定义 settings/CLI/runtime 共用的 channel 真相
2. 先做本机音频主链
3. 再做 HomePod bridge pack

## Plan

1. 先收 `input_channel` / `output_channel` 的产品真相
2. 明确本机音频设备族与 `homepod-bridge` 的边界
3. 固定实现顺序，避免后续先做 HomePod 再回头补本机音频主链

## Acceptance Criteria

1. 文档明确存在 `input_channel` / `output_channel` 最小真相
2. 文档明确区分本机音频设备族与 `homepod-bridge`
3. 文档明确 `HomePod` 不是 core，而是可选 bridge pack
4. 文档固定后续顺序：先本机音频，再 HomePod bridge

## Notes

- 这张单只解决产品边界，不解决实现细节
- 不允许先设计“统一语音平台”再倒推需求
- 不允许把 `openclaw` 当 msgcode 的插件标准
