# 任务：Thread context token breakdown probe

## Context

当前上下文一旦变重，只能靠主观印象判断“系统提示/工具定义/项目文档/历史对话/当前问题”谁在吃 token。

需要一个很薄的 probe，快速给出类似：

- `[系统提示 10k tokens]`
- `[工具定义 20k tokens]`
- `[项目代码/文档 80k tokens]`
- `[历史对话 5k tokens]`
- `[新问题 100 tokens]`

## Goal / Non-Goals

## Goal

- 给单次 thread/context 输出一个 token 构成视图
- 明确每一段的 token 数和占比
- 复用现有 prompt/context/budget 真相，不新造第二套上下文装配逻辑

## Non-Goals

- 不新增 prompt 控制层
- 不新增自动 compact / 自动裁剪
- 不新增“最佳预算建议器”
- 不把 probe 变成新的上下文主链

## Plan

1. 基于现有 prompt 组装点取数：
   - `buildDialogSystemPrompt`
   - `buildConversationContextBlocks`
   - `buildDialogPromptWithContext`
   - 现有 budget token estimator
2. 第一刀只输出固定五段：
   - `system_prompt`
   - `tool_definitions`
   - `project_context`
   - `history_dialogue`
   - `current_input`
3. 输出两种视图：
   - `json`
   - `text`
4. 若某段当前无数据来源，直接显式标记 `unavailable`，不猜测补数

## Acceptance Criteria

- 输出结构固定，能快速看出五段 token 数
- probe 复用现有上下文真相，不新造拼接器
- 缺失段落直接暴露，不做兜底估算
- 第一刀不改变任何 prompt 行为

## Notes

- 这是一枚观察探针，不是控制器。
- 它服务于 review / 调优 / 事故定位，不服务于运行时自动决策。
