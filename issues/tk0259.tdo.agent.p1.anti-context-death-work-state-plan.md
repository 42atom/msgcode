# 任务：抗 context death 的工作态重建设计计划

## 背景

用户希望把“summary 不是连续思考本体，只是有损重建接口”收口成 msgcode 的正式设计计划。

仓库里已经有一条 `work continuity -> work capsule -> session continuity` 主线，但当前 runtime `work capsule`
是给系统恢复执行用的最小投影，不等于给模型跨窗口恢复工作态用的完整支架。

需要补一份计划，把这两层关系讲清楚：

- 哪些继续由现有 `work capsule` 负责
- 哪些应该由新的“工作态骨架”承担
- 什么时候触发
- 以什么文件形态存在

## 目标

- 明确 `summary`、`work capsule`、`working state scaffold`、`evidence` 的职责边界
- 给出适配 msgcode 当前 file-first 架构的最小落地方案
- 保持 Unix 风格，不新增大一统控制面

## 非目标

- 不实现 persistent KV / latent state
- 不把 runtime `work capsule` 直接改造成第二真相源
- 不立即改 heartbeat / wake / task runtime 代码

## 交付

- [ ] 一份正式 plan，说明：
  - 问题定义
  - 与现有 `work capsule` 主线的关系
  - 最小文件设计
  - 触发条件
  - 分阶段落地路径
- [ ] 一份 `WORKSTATE` 协议文档，固定职责边界、模板和触发规则
