# 任务：DMG Packaging Implementation Slice

## Context

当前主线已经有两段实现：

- `tk0374`：`project -> bundle-root`
- `tk0375`：`bundle-root -> .app`

但最后一刀还停在合同：

- `tk0373` 只定义 `app -> dmg -> signing -> notarization` 边界

仓库还没有一张现役实现单承载：

- `tk0375` 产出的 `.app` 如何最小收成 `.dmg`

## 为什么现有文件不能直接承载实现职责

`tk0373` 只回答分发链合同。

它说明：

- 输入只认 `.app`
- 顺序是 packaging -> signing -> notarization
- 签名和公证属于分发链，不回写 app/ artifact

但它不承载这次代码实现状态。

如果把实现直接塞回 `tk0373`，会把：

- 分发链合同
- packaging 代码推进
- 本次验证证据

重新揉成一张单。

`tk0375` 只承载 app host implementation。

它的边界到：

- `bundle-root -> .app`

不能继续替 `dmg` implementation 背状态，否则 `app` 和 `dmg` 的切口会重新变糊。

`pl0362` 是主线导航。

它负责固定：

- `artifact -> app -> dmg`

plan 不是实现单，不能代替当前 slice 的推进状态。

所以这里必须单开 1 张最薄实现单，只承载：

- `.app -> .dmg`

## Goal / Non-Goals

### Goal

- 只消费 `tk0375` 产出的 `.app`
- 产出可验证的 `.dmg`
- 显式停在 packaging implementation

### Non-Goals

- 不回写 `.app` 结构
- 不做 signing / notarization 实现
- 不碰授权、更新、App Store、跨平台安装器

## Implementation Slice

最小实现只做：

1. 校验输入是完整 `.app`
2. 用 macOS 原生 `hdiutil` 产出 `.dmg`
3. 对外暴露明确 CLI 入口
4. 在结果和输出文案中明确 signing / notarization 仍未实现

## Plan

- [x] 补 1 张最薄 dmg packaging implementation task
- [x] 增加只消费 `.app` 的 `.dmg` 打包入口
- [x] 增加最小测试，证明不会回退到 bundle-root / 源码树
- [x] 跑通最小验证链，只停在 packaging

## Acceptance Criteria

1. 输入只认 `.app`
2. 能稳定生成 `.dmg`
3. 签名 / 公证只保留明确占位边界，不写假实现
4. 不扩授权、更新、App Store

## Notes

- 这是 Desktop packaging 的第 3 张实现切片。
- 这张单只承载 dmg packaging implementation，不替 signing / notarization 占位。
