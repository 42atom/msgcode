# Context

用户明确要求将飞书升级为主通道，iMessage 退后。当前仓库真实使用重心已经偏向飞书，但默认启动与文档口径仍然是 iMessage-first：

- [src/config.ts](/Users/admin/GitProjects/msgcode/src/config.ts) 默认 transport 仍是 `imsg`
- [src/deps/manifest.json](/Users/admin/GitProjects/msgcode/src/deps/manifest.json) 仍把 `imsg/messages_db` 作为启动硬依赖
- [README.md](/Users/admin/GitProjects/msgcode/README.md) 首屏仍写 `iMessage-first`
- [docs/CHANGELOG.md](/Users/admin/GitProjects/msgcode/docs/CHANGELOG.md) 旧记录仍强调 iMessage 通道优先

这会导致主链和产品定位打架，也让 launchd/daemon 主线继续被 iMessage 权限问题污染。

# Goal / Non-Goals

## Goal

- 默认主通道改为飞书优先
- iMessage 改为显式可选通道，不再占据默认启动主链
- 启动依赖与 README/帮助口径同步收口

## Non-Goals

- 不删除现有 iMessage 实现
- 不重写 listener / router / security 大主链
- 不顺手处理历史 archive 文档

# Plan

- [x] 调整默认 transport 解析：有飞书配置时默认只启 `feishu`
- [x] 将 `imsg/messages_db` 从启动硬依赖退到可选依赖
- [x] 调整配置/探针逻辑，避免把未启用的 iMessage 继续当错误
- [x] 更新 README / CHANGELOG / issue 口径
- [x] 补最小回归测试

# Acceptance Criteria

- 无显式 `MSGCODE_TRANSPORTS` 时：
  - 有飞书配置 => 默认 `feishu`
  - 无飞书配置 => 回退 `imsg`
- `src/deps/manifest.json` 不再把 `imsg/messages_db` 视作全局 `requiredForStart`
- README 首屏和快速开始改成飞书主通道
- 测试覆盖默认 transport 与依赖口径变化
