# 任务：settings 写入口最小基线

## Context

当前已经有稳定只读面：

1. `appliance profile`
2. `appliance general`
3. `appliance capabilities`
4. `appliance doctor`

但这还不等于设置页可交付。

若现在直接让 UI 自己改：

- `.msgcode/config.json`
- `.msgcode/SOUL.md`
- `.msgcode/ORG.md`

就会立刻长出第二套写面和格式依赖。

## Goal / Non-Goals

### Goal

- 先定义 settings mutation 的单一主链
- UI 一律只能通过正式 CLI 写回
- 先诚实区分“可写”和“只读”

### Non-Goals

- 不一次性实现所有设置项写回
- 不做 RPC / Socket / 后台设置中心
- 不允许 Electron/前端直接改写 markdown/json 真相文件
- 不把没有硬真相源的项伪装成可保存

## Baseline

当前只允许一类写面先进入正式主链：

1. `我的资料`
   - 我的称呼
   - SOUL 内容
   - 组织名称
   - 位置城市

当前继续只读：

1. `通用`
   - 工作区根目录
   - 日志路径
   - 启动方式
   - 多语言
   - 账号连接
2. `智能体能力配置`
   - 只读当前能力位空/实状态
   - 不做 provider 编辑
3. `诊断`
   - 永远只读

UI 约束：

- 只有当前真正可写的分组才出现保存动作
- 只读分组不能伪装成可编辑输入态

## Plan

- [ ] 定义正式 mutation CLI 入口
- [ ] 明确 profile 写回的文件映射
- [ ] 明确 general/capabilities/doctor 当前为何继续只读
- [ ] 做一轮圆桌复核

## Acceptance Criteria

1. settings 页不会被默认为“所有 tab 都可写”。
2. profile 是当前唯一允许进入写主链的设置分组。
3. UI 不拥有任何直接改写真相文件的特权。
4. general / capabilities / doctor 继续只读，不强行补第二中心。
5. 只读 tab 不出现“能改但不能存”的假交互。

## Notes

- 这刀先定边界，不急着把所有 mutation 做完。
- 允许“只有部分 settings 可保存”。
- 少做比乱做更对。
