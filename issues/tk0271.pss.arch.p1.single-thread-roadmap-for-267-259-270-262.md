## Context

当前同时存在四条重量级计划：

- `pl0267` 确定性反馈基础设施
- `pl0259` 抗 context death 的工作态骨架
- `pl0270` 图驱动上下文路由
- `pl0262` 企业版自包含 / App/UI / 跨平台

问题不在于设计本身过厚，而在于并发过高。

需要一份明确的单线程执行顺序表，限制当前阶段的主脑带宽，避免多条线同时开洞。

## Goal / Non-Goals

- Goal: 明确这四条线的推荐顺序。
- Goal: 定义每条线进入实现前的前置条件。
- Goal: 定义本阶段暂缓项与切换条件。
- Goal: 限制并发，避免把认知主链和交付主链混做。
- Non-Goals: 不重写原计划。
- Non-Goals: 不为这四条线新增统一控制面。
- Non-Goals: 不直接改代码。

## Plan

- [ ] 产出一份正式顺序表，覆盖推荐顺序、WIP 上限、进入条件、暂停条件和依赖关系。

## Acceptance Criteria

1. 仓库内存在一份可执行的单线程顺序表。
2. 顺序表明确 `0267 -> 0259 -> 0270 -> 0262` 的推进关系与切换条件。

## Notes

- 问题核心不是设计过厚，而是并发过高。
- 目标是限制当前阶段的主脑带宽，而不是再造控制层。
