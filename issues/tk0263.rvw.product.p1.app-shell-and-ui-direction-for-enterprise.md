# 任务：企业版 App Shell 与 UI 方向

## Context

当前方向已经收得更具体：

- appliance 需要可见入口
- 主 UI 只做门厅
- 主题结果面走 sidecar 站点
- `msgcode` runtime 继续做主脑

所以这张单不再回答“是不是需要 App”，而是回答：

- App 壳最小应该做什么
- 为什么是 Electron 薄壳
- 第一版主门厅只允许哪些内容

## Goal / Non-Goals

### Goal

- 明确企业版是否需要 App shell
- 明确 UI 最小范围：
  - 配置
  - 运行状态
  - 日志/诊断
  - 工作区与任务视图
- 明确 Electron 与替代路线的取舍口径

### Non-Goals

- 不直接搭 Electron 工程
- 不直接做视觉设计稿
- 不替跨平台模型问题做结论

## Plan

- [x] 盘点 appliance 场景下 CLI 不足的具体点
- [x] 列出最小 App/UI 面，不先膨胀成全平台工作台
- [x] 对比 Electron 与替代路线的成本、风险、复用度

## Acceptance Criteria

1. 能说明为什么需要 App 壳，而不是继续只有 CLI。
2. 能给出最小 UI 面，而不是抽象“做个客户端”。
3. Electron 是薄壳，不会被写成第二主脑。

## Notes

- 这张单先做产品与架构方向判断，不直接进入前端实现。
- App shell 必须服务 appliance 交付，不为“看起来完整”硬做桌面端。
- 第一版机构卡片只允许 3 个字段：名称、交税地、统一社会信用代码。
- 当前已补最小 host contract 实现：
  - `msgcode appliance hall --workspace ... --json`
