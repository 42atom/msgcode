## Context

- 用户现象：`msgcode` 收到“帮我查看 163.com 的头条新闻”之类请求时，被路由到 `tool`，但日志显示 `toolCallCount=0`，最终返回 `MODEL_PROTOCOL_FAILED`。
- 浏览器底座并非未实现：`src/tools/bus.ts` 已支持 `browser`，`src/runners/browser-pinchtab.ts` 已接入真实执行链路。
- 真正断点在“执行核暴露给模型的工具集合”：
  - `src/agent-backend/types.ts` 的 `PI_ON_TOOLS` 不包含 `browser` / `web`
  - `src/agent-backend/tool-loop.ts` 的 `getToolsForLlm()` 在 `pi.enabled=true` 时直接返回 `PI_ON_TOOLS`
  - 该函数没有读取 `tooling.allow`，因此即使 workspace 允许 `browser`，模型仍看不到 `browser`
- 次级问题：`detectPreferredToolName()` 只识别 `read_file/write_file/edit_file/bash`，不会把“新闻/网页/站点读取”自然语言提升为 `browser/web` 的强约束重试。

## Goal / Non-Goals

- Goal: 明确 `browser` 调不起来的真实根因，并给出最小修复方向。
- Non-Goals: 本单不直接实现修复；本单不讨论 browser runner 自身能力是否完备。

## Plan

- [ ] 让 LLM 工具暴露层改为读取 workspace `tooling.allow`，而不是硬编码 `PI_ON_TOOLS`
- [ ] 为 `browser` 补 tool schema 与回归测试，验证其真实进入 `tools[]`
- [ ] 视需要补“网页/新闻/实时信息”到 `browser/web` 的 preferred tool 意图识别

## Acceptance Criteria

1. 能提供代码级证据证明：当前执行核发给模型的工具列表中没有 `browser`。
2. 能提供代码级证据证明：workspace `tooling.allow` 与执行核 `tools[]` 当前不一致。
3. 后续修复时，应有回归锁确保 `browser` 出现在执行核工具暴露层，而不是只存在于 Tool Bus/CLI。

### 修复记录（2026-03-06）

**问题根因**：
- 执行核发给模型的 `tools[]` 由 `PI_ON_TOOLS` 硬编码生成
- `PI_ON_TOOLS` 不包含 `browser`
- `getToolsForLlm()` 不读取 workspace `tooling.allow`
- 结果：workspace 允许 browser 但模型看不到

**修复方案**：
1. 新增 `src/tools/manifest.ts`：工具说明书注册表（TOOL_MANIFESTS）+ 暴露解析器（resolveLlmToolExposure）
2. 修改 `src/agent-backend/tool-loop.ts`：`getToolsForLlm()` 改为从 manifest 派生
3. 修改 `src/lmstudio.ts`：`getToolsForLlm()` 改为从 manifest 派生
4. 新增 `test/p5-7-r8c-llm-tool-manifest-single-source.test.ts`：回归锁

**验证结果**：
- ✅ browser 在 workspace tooling.allow 包含时，真实进入执行核 tools[]
- ✅ 不再存在独立硬编码白名单决定 LLM 工具暴露
- ✅ 测试覆盖 allowed/registered/exposed/missing 四类状态
- ✅ 三门通过：npx tsc --noEmit, npm test (1432 pass), npm run docs:check

**风险说明**：
- 保留 `PI_ON_TOOLS` 定义（向后兼容）
- `AGENT_TOOLS` 静态常量保持不变
- 实际工具暴露由 `getToolsForLlm()` 动态计算
- **P5.7-R8c 返工完成**：
  - ✅ browser manifest 改为真实 PinchTab 合同（operation: tabs.open/tabs.action/instances.launch/...）
  - ✅ 主链直接使用 manifest 的 toOpenAiToolSchemas()，删除旧 schema 生成逻辑
  - ✅ 删除无执行分支的 5 个工具暴露（list_directory/search_file/search_content/todo_read/todo_write）
  - ✅ 不存在 expose != execute 漂移
