## Context

Issue 0119 已完成“CLI 单工具参考 vs 当前 runtime 差距”审计，用户明确要求先收口两条最贴近参考文档且最能减少主链污染的缺口：

1. `read_file` 缺少二进制检测、大文件导航与下一步建议
2. bash/read_file 的预览文本在执行层与呈现层被多处二次裁剪，职责不清

当前仓库已经持续删除 tool-loop 里的裁判/恢复层；这轮实现必须继续遵守“做薄、服务 LLM、不抢执行权”的主线，而不是再加新的中间层。

## Goal / Non-Goals

### Goal

- 为 `read_file` 补齐二进制/大文件/路径错误的正式合同
- 让执行层直接产出稳定 `previewText`，tool-loop 只转发，不再给 bash/read_file 造第二套预览
- 为本轮运行时行为变化补齐 issue/plan/changelog/测试证据

### Non-Goals

- 不把工具面重构成单一 `run(command)`
- 不在本轮改造 `write_file/edit_file` 合同
- 不引入新的帮助系统或预算控制层

## Plan

- [x] 扩展 `ToolResult` / `read_file` data 合同，支持 `previewText`、大文件导航元数据
- [x] 在 Tool Bus 中为 `bash` 生成稳定预览，并为 `read_file` 增加二进制检测、大文件预览、明确下一步建议
- [x] 在 tool-loop 中优先转发执行层的 `previewText`，保留通用 fallback
- [x] 补齐 `tools.bus` 回归测试，并跑通类型检查与 docs 校验
- [x] 回写 Issue 0119、Plan 0119 与 `docs/CHANGELOG.md`

## Acceptance Criteria

1. `read_file` 遇到二进制文件时，不再硬按 UTF-8 读取；错误信息能明确指出文件类型与下一步建议
2. `read_file` 遇到大文件时，不再全量塞回主链，而是返回预览、字节数与导航 guidance
3. `bash` / `read_file` 的回灌预览优先来自执行层 `previewText`，tool-loop 不再为这两类结果重新裁一层
4. `bun test test/tools.bus.test.ts test/p5-7-r25-tool-result-context-clip.test.ts test/p5-7-r3h-tool-failure-diagnostics.test.ts`、`npx tsc --noEmit`、`npm run docs:check` 全部通过
