# 任务：主界面新建聊天边界

## Context

“新建聊天”按钮看起来很小，但如果边界不清，很容易演化成：

- UI 直接新建空 thread
- UI 提前写聊天气泡
- UI 拥有 thread 生命周期

这都不该发生。

## Goal / Non-Goals

### Goal

- 定义“新建聊天”按钮的最小职责
- 定义它和 web `chatId` 的关系
- 定义 runtime 何时真正创建 thread

### Non-Goals

- 不实现按钮交互
- 不改 inbox CLI
- 不做前端状态机设计

## Plan

- [ ] 定义“新建聊天”按钮只切上下文，不直接建 thread
- [ ] 定义第一条消息落盘前的 UI 本地状态
- [ ] 定义真正创建 thread 的时机

## Acceptance Criteria

1. “新建聊天”不会直接创建空 thread。
2. 它和 web `chatId` 的关系清楚。
3. thread 真正创建的时机清楚。

## Notes

- 第一条消息未落盘前，只存在 UI 本地待发送上下文。
- 真正的 thread 仍应由 runtime 在落盘时创建。
- “新建聊天”不是“新建空 thread”，而是“切到新的 web chatId 草稿上下文”。
