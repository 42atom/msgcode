## Context

现在仓库里已经有不少 file-first 真相源：

- `SOUL`
- `memory`
- `sessions`
- `issues`
- `dispatch`
- `reflection`
- `inbox`
- `AIDOCS`

但它们分散在 `README`、`protocol`、`issues` 和 plan 文档里，没有一张统一的“Agent 认知文件结构表”。

这会带来两个问题：

- 新人和后续自己很难一眼看懂：哪些文件在表达“Agent 认识你”的认知面，哪些只是运行时派生物
- 人类巡检面和机器协议面虽然在理念上已经分开了，但缺少一张总览表来稳定共识

## Goal / Non-Goals

- Goal:
  - 为 msgcode 补一张 Agent 认知文件结构表
  - 优先把这张表放进 README，作为统一入口视图
  - README 这一版只放面向人类的文件面，不把 `.json/.jsonl/.ndjson` 机器协议面混进来
  - 讲清楚灵魂、长期记忆、每日轨迹、会话原始记录、任务、派单、反思、资产分别落在哪
  - 标清每类文件的路径、作用、生命周期、是否真相源、谁可读写
  - 让这张表同时服务用户入口、维护者认知和模型执行
- Non-Goals:
  - 不为了“像别人一样完整”而先新增一堆文件
  - 不把机器协议面全改成人类巡检式文件名
  - 不新建控制面、中台或第二状态系统

## Plan

- [x] 给每类文件标注：路径、职责、真相级别、生命周期、人类可读性
- [x] 输出一张统一文件结构表
- [x] 明确现有缺口属于“文档缺口”还是“主链缺口”

## Acceptance Criteria

1. 仓库里存在一张明确的 Agent 认知文件结构表，且 README 是首个落点。
2. 至少覆盖：
   - `SOUL`
   - `memory/*.md`
   - `.msgcode/sessions/*`
   - `.msgcode/inbox/*`
   - `.msgcode/dispatch/*`
   - `.msgcode/reflection/*`
   - `issues/*`
   - `AIDOCS/*`
3. 每类文件都标清：
   - 路径
   - 作用
   - 是否真相源
   - 生命周期
   - 是否给人巡检
   - 是否给机器协议使用
4. 文档能回答一句话：Agent 到底通过哪些文件“认识你”、记住工作、继续推进任务。

## Notes

- 这张单的核心不是“文件越多越好”，而是“该显式的真相源要能一眼看懂”。
- 若现有主链已经足够表达，就优先补文档，不先加结构。
