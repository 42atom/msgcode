## Context

### 问题证据

当前 msgcode 的真实问题：
1. `route=no-tool` 会让执行型请求直接结束
2. `MODEL_PROTOCOL_FAILED` 会把本应继续循环的任务提前打死
3. 我们用 `looksLikeExecutionRequest()` 这类消息级意图识别在替 LLM 做主
4. 真实日志已证明：
   - 同一句自然语言，有时能进工具链，有时又掉回 no-tool
   - 这不是 LLM 不会，而是框架在拦

### OpenClaw 关键做法

1. 默认创建完整工具集（不是按当前消息内容猜）
2. 工具过滤来自配置/权限/策略，不来自消息语义分类
3. skills prompt 注入给模型，让模型自己决定如何用工具
4. 重点做参数归一化和 schema 兼容，不做 no-tool 裁判链

### 架构铁律

1. 系统只负责给工具、给 skill、给路径
2. 不得基于当前消息内容替 LLM 决定"该不该调用工具"
3. 不得再保留 `route=no-tool -> MODEL_PROTOCOL_FAILED` 这类收尾链
4. 只保留明确的配置/权限边界，不保留主观流程边界

## Goal / Non-Goals

### Goal
- 对齐 OpenClaw 的工具暴露主链
- 移除基于消息意图的工具拦截逻辑
- 收口 no-tool / MODEL_PROTOCOL_FAILED 这类前置或中途裁判
- 保留并收口工具过滤到配置/权限层
- 强化参数归一化与 schema 兼容

### Non-Goals
- 不重构 scheduler/browser/memory 功能本身
- 不做 prompt 分层实验
- 不顺手改业务技能内容
- 不扩到 quota/hard-cap
- 不新增 parser 去解析伪 `[TOOL_CALL]` 文本
- 不做新的"创建请求/删除请求/停止请求"关键词分类
