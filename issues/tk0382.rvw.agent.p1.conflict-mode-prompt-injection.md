# TK0382：Conflict Mode Prompt Injection

## Context

当前仓库已经明确两条铁律：

- 先改提示词/合同/说明书，再考虑改代码
- 系统服务 LLM，不替 LLM 做决定

用户已经明确希望把“本机冲突如何处理”先收成一档可热切换的 AI 处置模式，而不是先写 runtime 控制层。

因此这张单只负责把 mode 语义收成正式提示词合同：

- `full`
- `assisted`

## Goal / Non-Goals

### Goal

- 定义 `full` 与 `assisted` 的最小语义差异
- 明确这两档只通过提示词注入影响 AI 行为
- 明确模式必须支持热切换
- 明确后续实现应接在哪条 prompt/skill 主链上

### Non-Goals

- 不新增 conflict-policy runtime
- 不新增控制面或治理平台
- 不直接修改 browser/session/bash runner 的冲突处理代码
- 不定义更大的“电脑管理模式体系”

## Core Contract

### Mode: `full`

注入给 AI 的语义是：

- 你需要自行处理本机故障与运行冲突
- 不要把本可自行解决的问题抛给用户

### Mode: `assisted`

注入给 AI 的语义是：

- 如果遇到冲突，先判断是否可能影响用户正在使用的电脑或用户自己的服务
- 优先告知并确认，不直接处置

## Hot Switch

这两档模式必须支持热切换。

含义是：

- 用户切换后，下一轮 prompt 注入立即生效
- 不要求重启 app
- 不要求重建整套 runtime

## Plan

1. 先确认现有 prompt/skill 注入主链的真实落点
2. 补一张实现切片，只做 mode prompt 注入和热切换入口
3. 第一刀不碰任何自动修复代码

## Acceptance Criteria

1. `full` 与 `assisted` 的提示词差异清楚，且只围绕“本机冲突处置姿态”展开
2. 合同明确要求热切换
3. 合同明确第一刀只做 prompt 注入，不做 runtime 策略层
4. 后续实现者可直接据此开做而不再重新发明模式边界
