## Context

用户希望未来产品除了主 UI 外，还能有一组独立的本地网站，分别承接不同关注面。

这些站点不是主 UI 本体，而是：

- 按主题拆开的本地 sidecar 站点
- 由 skill 或能力包产出结果
- 再由本地网站做结果可视化

当前讨论到的方向包括：

- 公司财务/税务驾驶舱
- 员工组织信息与飞书人物识别参考
- 老板个人风险、投资、健康等高敏信息面
- 小红书运营日志、已发内容列表等场景化运营站点

用户已经明确：

- 这是一条独立规划线
- 不要影响当前开发主线
- 主 UI 最多只保留基础机构信息和入口

## Goal / Non-Goals

- Goal: 定义“主 UI 门厅 + 多个本地 sidecar 站点”的产品结构。
- Goal: 明确 skill、结果文件真相源、可视化站点三者关系。
- Goal: 定义哪些内容适合做基础包，哪些适合做扩展包。
- Goal: 固定“独立规划、不进入当前主线”的边界。
- Non-Goals: 不直接做 UI 设计稿。
- Non-Goals: 不直接实现任何本地网站。
- Non-Goals: 不把高敏个人数据默认并入主 UI。

## Plan

- [ ] 产出一份独立 plan，定义主 UI 与本地 sidecar 站点的边界。
- [ ] 说明 sidecar 站点的最小形态、分类方法和数据来源。
- [ ] 说明 skill 结果如何以文件真相源形式被网站读取和可视化。
- [ ] 明确这条线与当前开发主线的隔离关系。

## Acceptance Criteria

1. 仓库内存在一份独立 plan，能说明主 UI、sidecar 站点和 skill 结果可视化的关系。
2. 该 plan 明确写清不进入当前开发主线，不要求马上实现。

## Notes

- 这条线首先是产品与信息架构规划，不是 runtime 需求。
- 目标是把未来可扩展的网站形态先收口成稳定口径。
