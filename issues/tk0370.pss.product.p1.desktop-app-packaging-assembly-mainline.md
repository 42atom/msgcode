# 任务：Desktop App Packaging Assembly Mainline

## Context

现有真相源已经把几件事分别收口了：

- `tk0262` 只定义自包含交付基线，不承载 app 装配顺序
- `tk0263` 只定义 App shell / UI 方向，不承载 build artifact / dmg 装配
- `tk0265` 只定义去 `brew` / 去 `npm install` 的 bootstrap 主线，不承载桌面 app 产物
- `pl0274` 只是一份离线规划，讲 Electron 薄壳与 runtime 关系，不是现役执行单
- `rf0000` / `rf0001` 是参考约束，不是 task / plan 真相源

所以当前缺的不是“再讨论 Electron 要不要做”，而是一张更薄的执行单：

- 现有 CLI/runtime/UI surfaces 已经有哪些输入
- Desktop App 主线要装出什么产物
- `build artifact -> app -> dmg` 的顺序怎么走

## Goal / Non-Goals

### Goal

- 固定 Desktop App packaging assembly 的唯一主线
- 明确输入面只来自现有：
  - CLI/runtime
  - host/UI surfaces
  - bootstrap 产物
- 明确输出面只到：
  - 本地 app bundle
  - build artifact
  - dmg 分发物
- 明确每一步只做装配，不发明第二主脑

### Non-Goals

- 不写 Electron 代码
- 不起新的授权母线
- 不重开 UI 设计单
- 不重开 bootstrap 实现单
- 不讨论自动更新、商店分发、跨平台安装器

## Plan

- [ ] 证明现有文件为什么不能直接承载 packaging assembly 职责
- [ ] 固定 packaging assembly 的输入清单、输出清单与阶段顺序
- [ ] 明确 Desktop App 主线与 `tk0262 / tk0263 / tk0265 / pl0274` 的关系
- [ ] 给出最小验收口径：已有 surfaces 能被装成 app / build artifact / dmg 规划

## Acceptance Criteria

1. 能清楚说明为什么必须新开一张 packaging assembly task，而不是把职责塞回 `tk0262 / tk0263 / tk0265 / pl0274`。
2. 这张单只定义一条很薄的装配主线：`CLI/runtime/UI surfaces -> build artifact -> app -> dmg`。
3. 文档明确声明：Electron 只是宿主壳，`msgcode` 仍是主脑。
4. 文档不扩成大母线，不新增授权、更新、跨平台等旁路范围。

## Notes

- 这张单是现役执行单，不替代基线、方向和参考文档。
- 后续若进入实现，仍应按最小切片单独拆，不把 packaging assembly 膨胀成总控面。
