# 任务：workspace 身份与路径失效重定位基线

## Context

现在系统的工作区真相主要靠路径字符串：

- route store 存 `workspacePath`
- thread front matter 存 `workspacePath`
- workspace config 跟着 `<workspace>/.msgcode/config.json` 走

这条主链在目录改名、父目录改名、整体搬家时会直接变脆。

更糟的是，`loadRoutes()` 当前会在路径不存在时直接删掉 active route。

这会导致：

- 线程恢复时语境丢失
- 历史归档无法稳定回到原工作区
- 用户只改了目录名，系统却把绑定当成彻底消失

## Goal / Non-Goals

### Goal

- 给每个 workspace 一个稳定身份 `workspaceUid`
- route 继续主用绝对路径，但额外保存 `workspaceUid`
- thread front matter 额外保存 `workspaceUid`
- 路径失效时先按 `workspaceUid` 轻量重定位
- 找不到时明确报“失联”，而不是静默删除或乱猜

### Non-Goals

- 不用 git 跟踪目录身份
- 不做 inode / FSEvents / Spotlight 常驻追踪
- 不做多阶段恢复中心
- 不做全盘扫描

## Plan

- [x] 在 `<workspace>/.msgcode/workspace.json` 建立最小身份真相
- [x] `/bind` 时创建或读取 `workspaceUid`
- [x] `routes.json` 追加 `workspaceUid`
- [x] thread front matter 追加 `workspaceUid`
- [x] route 加载时由“路径失效直接删”改成“先尝试重定位”
- [x] 重定位只在 `WORKSPACE_ROOT` 下做轻量扫描
- [x] 找不到时保留 route 并回显失联状态

## Acceptance Criteria

1. workspace 有单一身份真相，不再只靠路径字符串。
2. 目录改名或搬家后，系统会先尝试按 `workspaceUid` 找回，而不是直接删 route。
3. 找不到时会明确报失联，不会静默改绑到别的目录。
4. 线程与 route 都能知道自己原来属于哪个 workspace 身份。

## Notes

- 这刀解决的是“路径失效后的最小恢复”，不是做目录追踪平台。
- 路径仍然是第一真相；`workspaceUid` 只在路径失效时出场。
- 轻量重定位只建议扫描 `WORKSPACE_ROOT` 下的 `.msgcode/workspace.json`，不做全盘搜索。
- 已实现代码：
  - `src/runtime/workspace-locator.ts`
  - `src/routes/store.ts`
  - `src/routes/cmd-bind.ts`
  - `src/runtime/thread-store.ts`
