## Context
- iMessage 账号/通道不稳定，限制了 msgcode 的可用性与可靠性。
- 需要引入飞书 Bot 作为更稳定的消息通道（优先长连接 WebSocket），并尽量复用现有 message -> handler -> tmux 的闭环。
- 当前未绑定 chat 会被丢弃（路由返回 null），用户希望“未绑定 chat 指向一个默认工作目录”以便开箱即用。

## Goal / Non-Goals
### Goals
- 增加飞书 Bot 作为 transport（WebSocket 长连接收消息 + API 回消息）。
- 未绑定 chat 时，自动路由到默认工作目录（配置项可控），同时 /where 能看到默认目录提示。
- 冒烟期允许飞书消息绕过 iMessage 白名单（显式开关，默认关闭）。

### Non-Goals
- 不在 MVP 中实现飞书文件/图片回传与资源上传（先 text 跑通闭环）。
- 不在 MVP 中实现精细化飞书身份鉴权（先 allow-all 开关，后续再收口）。
- 不强行移除 iMessage 相关能力（先并存，允许后续切换为 feishu-only）。

## Acceptance Criteria
- 飞书群里发送 `/where`，能返回“默认工作目录”提示（未绑定时）。
- 飞书群里发送 `/bind acme/ops`，能成功落盘 routes 并路由到对应 workspace。
- 飞书群里发送普通文本，能进入现有 handler/tmux 流程并回消息（text）。
- 未配置飞书时不影响现有 iMessage 流程。
- `npm test` 通过。
