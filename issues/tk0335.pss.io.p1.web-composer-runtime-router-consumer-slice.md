# 任务：Web composer one-shot consumer

## Context

现在已经有：

- `.msgcode/inbox/`
- `msgcode inbox add`
- `RuntimeRouterHandler.handle()` 这条真实处理主链

但还没有：

- 谁来消费 web inbox
- 如何把它接进现有 handler 主链
- 如何坚持 UI 只读真相源，不长第二状态层

## Goal / Non-Goals

### Goal

- 读取一条 web inbox 请求
- 直接复用 `RuntimeRouterHandler.handle()`
- 让输出继续落到现有 `.msgcode/sessions/*.jsonl` 与 `.msgcode/threads/*.md`

### Non-Goals

- 不起 daemon
- 不扩 `runtime.current_transport`
- 不复用 `listener.handleMessage()`
- 不做 UI 自己先画气泡

## Plan

- [x] 增加 one-shot web inbox consumer 入口
- [x] 读取单条 inbox 请求并还原最小入站消息
- [x] 直接调用 `RuntimeRouterHandler.handle()`
- [x] 保持输出继续落在现有 session/thread 真相源
- [x] 补最小回归锁，证明 UI 仍只读真相源

## Acceptance Criteria

1. 存在 one-shot 消费入口。
2. 能把单条 inbox 请求送入 RuntimeRouterHandler。
3. UI 继续只读真相源，不维护第二套聊天状态。

## Notes

- 这张单只收最薄消费入口，不碰 UI 样式。
- 核心原则：复用 handler，不复用 listener。
- 第一版只承接自然语言 web 输入，不扩 transport。
- 已实现 `msgcode inbox consume-web --workspace ... [--request-number rqNNNN] --json`。
- 读面仍然只认现有 truth：
  - `.msgcode/inbox/*.md`
  - `.msgcode/sessions/*.jsonl`
  - `.msgcode/threads/*.md`
- 没有新增 daemon、transport、optimistic bubble。
