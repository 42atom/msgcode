# 任务：nexu host and thin service selective adoption baseline

## Context

对 `nexu` 的探索已经确认三件事实：

1. `apps/desktop/main/runtime/daemon-supervisor.ts` 证明成熟桌面端会有正式宿主 runtime。
2. `apps/controller/src/services/openclaw-gateway-service.ts` 证明薄 service 是对的。
3. `apps/controller/static/runtime-plugins/openclaw-weixin/` 已经长成重量级 runtime plugin 树，不适合 msgcode 直接照抄。

当前缺的不是更多探索，而是把“学什么 / 不学什么 / 先后顺序”收成一张可执行薄单。

## Goal / Non-Goals

## Goal

把对 `nexu` 的参考收成 msgcode 自己的选择性吸收基线：

- 学正式宿主
- 学薄 service
- 不学重量级 plugin runtime

## Non-Goals

- 不兼容 `nexu` / `openclaw` 插件体系
- 不引入 controller 平台
- 不写 HomePod 或音频实现代码
- 不新增大母线

## Plan

1. 继续沿 `Electron runtime + host bridge + src UI` 主链推进，不回头重做宿主方向。
2. 所有外部能力接入默认先走薄 service，只允许健康检查、状态、启动/停止、最薄请求转发。
3. `wpkg` pack 允许带小边车代码，但禁止长成第二套 runtime/plugin host。
4. 先在文档里钉死这三条边界，再在后续任务里按需落最薄实现。

## Acceptance Criteria

- 明确写清：
  - `nexu` 哪两类东西值得吸收
  - 哪一类东西不照抄
  - msgcode 里的对应落点是什么
- 不新增实现代码
- 不改现有主线顺序
