## Context

- 仓库级原则已经收口为：能落成文件真相源的，先落文件；不能落成真相源的，只做薄 runtime。
- 当前 `memory` 原始数据已经在 `<workspace>/memory/*.md`，但检索索引默认落在 `~/.config/msgcode/memory/index.sqlite`。
- 现在真正要回答的，不是“要不要情绪化地反数据库”，而是：
  - `memory` 的真相层到底是什么
  - `index.sqlite` 是真相层、索引层，还是缓存层
  - 当前实现是否过黑、过重、过难迁移

## Goal / Non-Goals

- Goal:
  - 明确 `memory` 的 truth / index / cache 边界
  - 判断 `index.sqlite` 是否应降权、替换或保留
  - 给出 file-first 口径下的最小可行收口方案
- Non-Goals:
  - 不预设“一定彻底删除 SQLite”
  - 不直接重写整套记忆检索实现
  - 不打断当前 `tk0205` 主线实施

## Plan

- [ ] 盘清 `memory` 当前真实结构：
  - 原始文件
  - chunk 生成
  - FTS 索引
  - 向量索引
  - 检索入口
- [ ] 明确三层边界：
  - truth
  - index
  - cache
- [ ] 判断 `index.sqlite` 当前是否越权：
  - 是否承载了只有它知道的事实
  - 离开它能否仍从文件真相源恢复
  - AI 是否必须通过黑箱接口才能使用记忆
- [ ] 比较三种方案：
  - 保留 SQLite，但降权为显式、可导出、可调试的派生索引
  - 改为纯文件 chunk/index
  - 混合：文件 truth + 可丢弃索引
- [ ] 给出建议路线与迁移成本评估

## Acceptance Criteria

1. 仓库内有一份明确判断，区分 `memory` 的：
   - truth layer
   - index layer
   - cache layer
2. `index.sqlite` 被明确归类为以下之一：
   - 可接受的派生索引
   - 需要降权的黑箱索引
   - 应替换的过重实现
3. 有一份具体建议，回答：
   - 原始真相源存哪
   - AI 默认读什么
   - 检索加速层是否可丢
   - 索引如何做到可见、可导出、可调试
   - 迁移后如何避免 SDK-only 黑箱
4. 不把这单写成“先宣判、后找证据”的运动式改造。

## Notes

- 当前直觉是：`memory` 的问题更像“索引层太黑”，不一定是“只要有 SQLite 就错”。
- 更稳的审判标准是：
  - 真相是不是文件
  - 索引丢了能不能恢复
  - AI 能不能直接 `cat / grep / diff` 自己的家底
- 更可能的方向不是“数据库清零”，而是：
  - 降权
  - 显式化
  - 可导出
  - 可调试
- 这单优先产出判断，不急着产出大重构。
