## Context

- 当前默认 `tooling.allow` 不含 `vision`，真实图片工作区要额外手工打开。
- 用户要求每个工作空间工具默认打开，不要限制。
- 但 `mem` 虽然挂在 tool manifest 里，Tool Bus 当前没有执行分支；直接默认开放只会得到 `unsupported tool in P0: mem`，属于伪开放。

## Goal

- 让缺省 workspace 配置默认包含当前已实现正式工具
- 消除默认工具面里对 `vision` 的额外限制
- 保持“不开假工具”的口径

## Non-Goals

- 不把未实现的 `mem` 伪开放
- 不恢复 legacy `desktop`
- 不改显式 workspace 自定义 allow 的行为

## Goal / Non-Goals

- Goal:
  - 默认 workspace 工具面默认全开到当前已实现正式工具
- Non-Goals:
  - 不把未实现工具伪装成可用
  - 不动显式 workspace 的自定义 allow

## Plan

- [x] 默认 `tooling.allow` 加入 `vision`
- [x] 对齐默认工具面相关回归
- [x] 记录未实现 `mem` 不默认开放的边界

## Acceptance Criteria

1. `DEFAULT_WORKSPACE_CONFIG["tooling.allow"]` 包含 `vision`。
2. 缺省 workspace 的 `getToolsForLlm()` 默认包含 `vision`。
3. 相关回归通过。
4. `mem` 不被假开放。

## Result

- 默认 workspace 工具面现在按“当前已实现正式工具默认全开”执行。
- `vision` 进入默认 `tooling.allow`，新工作区不再需要额外手工开启。
- `mem` 仍不进默认开放面，因为它当前没有 Tool Bus 执行分支；保持真实边界。

## Notes

- 这次只修默认工具面真相，不改用户显式写在 workspace config 里的自定义工具面。
- 如果后续要把 `mem` 也变成真默认工具，必须先补执行主链，再谈开放。
