# 任务：工作区共享 status.log 基线

## Context

当前设计里，一个工作区下会同时出现：

- 网页线程
- 飞书线程
- 子线程

它们不能共用同一条原始会话，否则会串味。

但它们又需要共享少量近况，否则彼此完全失明。

用户认可的方向是：

- 每个工作区一份追加式小本本
- 原始聊天仍留在线程自己的 `jsonl`
- 共享层只放短结论和状态变化
- 真正进实现前，再补 append / tail 合同切片

## Goal / Non-Goals

- 定义 `<workspace>/.msgcode/status.log` 的角色
- 定义最小记录格式
- 定义最薄读写原则
- 定义主界面怎样把它展示成“工作状况”
- 结合圆桌审议收紧单行和 append-only 约束
- 补充单一 append 合同与非实时展示边界

### Non-Goals

- 不做线程互相读全量聊天
- 不做中间协调器
- 不做自动 hook / 定时器 / 关键词扫描
- 不做 UI “记一笔”按钮
- 不实现写盘逻辑
- 不把它做成第二份聊天摘要
- 不把它做成实时监控面

## Plan

- [ ] 定义 `status.log` 的单一职责
- [ ] 定义最小记录格式
- [ ] 定义线程读取规则
- [ ] 定义线程写入规则
- [ ] 定义 UI 消费口径
- [ ] 明确第一版不做项

## Acceptance Criteria

1. `status.log` 被清楚定义为工作区共享近况簿，不与 `summary`、`WORKSTATE`、原始会话混淆。
2. 记录格式足够短，能只靠追加文本表达已确认决定和状态变化，并带原文引用。
3. 读取与写入原则足够薄，不引入新控制层或轮询机制。

## Notes

- 第一版先把它收成追加式文本文件，不引入结构化数据库或中间控制层。
- 新的显示顺序由 UI 负责，文件本体保持自然追加顺序。
- 圆桌结论要求补上：单行约束、append-only、只记阶段性结论。
- 进一步要求补上：单一 append 合同、UI 不得暗示实时状态、轮转留到实现阶段。
