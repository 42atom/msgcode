# 任务：web thread 与新建聊天边界基线

## Context

当前原型已经明确：

- 用户层统一只看 thread
- UI 不该暴露 session/channel 双重心智
- thread 仍然坚持 file-first

但还有三条边界没钉死：

- web thread 的 `chatId`
- thread 标题到底谁写
- “新建聊天”按钮到底在做什么

如果这三条不收住，后面很容易滑向：

- UI 自己发明 thread
- UI 自己截断第一条消息拼标题
- web 每次发消息都裂成新 thread

## Goal / Non-Goals

### Goal

- 定义用户层 thread 的统一语义
- 定义 web thread 的稳定 `chatId`
- 定义 thread 标题必须回到 runtime
- 定义“新建聊天”按钮的边界
- 拆出后续可排期的小任务

### Non-Goals

- 不改 archive 主线
- 不改现有 sessions/threads 存量文件
- 不实现 UI
- 不进入 transport 平台化

## Plan

- [x] 收 thread 的用户层统一语义
- [x] 收 web thread 的 `chatId` 合同
- [x] 收 thread front matter 的最小标题合同
- [x] 收“新建聊天”按钮边界
- [x] 拆出 `0342 / 0343 / 0344`

## Acceptance Criteria

1. 用户层 thread 的统一语义被钉死。
2. web thread 为什么必须有稳定 `chatId` 被说明清楚。
3. 标题来源和“新建聊天”边界被说明清楚。
4. 这条线不与 archive 主线冲突。

## Notes

- 不推翻 thread。
- 先收 archive，再收 web thread 锚点。
- thread 碎先按活跃面噪音处理，不先推翻 thread 哲学。
- UI 不得获得 thread 创建权。
- web `chatId` 必须是一 thread 一 id，不是一消息一 id，也不是一浏览器永远一 id。
