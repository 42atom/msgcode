## Context

- review 指出 [`src/agent-backend/tool-loop.ts`](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts) 像“上帝对象”：模型管理、配额、Prompt 生产、provider 分支循环都塞在一起。
- 实查结果支持这个判断，但更准确地说它现在是高耦合 `God module`，不是单点 1566 行问题；文件当前实际结束于 1565 行。
- 现状至少混着六类职责：
  - backend/model 解析
  - quota 策略
  - system prompt/hint 拼装
  - tool schema 暴露
  - OpenAI/MiniMax provider 适配
  - 双 provider tool loop 执行与 journal 记录
- 其中最明显的维护风险是两条 provider loop 主体平行重复：解析参数、执行工具、写 `actionJournal`、检查 quota、回灌 `tool_result`、请求下一轮、收尾返回，都要双改双验。

## Goal / Non-Goals

- Goal:
  - 把 `tool-loop.ts` 收口为更薄的执行入口
  - 去掉 OpenAI / MiniMax 两条主循环骨架重复
  - 让 provider 只保留请求/响应适配差异
  - 继续减少热路径里的系统拼装和代决痕迹
- Non-Goals:
  - 不新增 manager/controller/supervisor
  - 不改变现有 tool loop 对外合同
  - 不重做 tool bus 或 prompt 全体系
  - 不引入新的中间控制面

## Plan

- [x] 盘清 `tool-loop.ts` 当前职责切面，冻结“必须留在主函数里的东西”和“可外提纯函数”
- [x] 抽出双 provider 共用的 loop 骨架：
  - [x] 参数解析
  - [x] tool 执行
  - [x] journal 记录
  - [x] quota 检查
  - [x] `tool_result` 回灌
  - [x] final metadata / result 收尾
- [ ] 把 provider 差异压到最小：
  - [x] 首轮请求
  - [x] 下一轮请求
  - [ ] tool call / response block 转换
  - [ ] finish reason / stop reason 映射
- [x] 视最小收益，再把 system prompt/hint 拼装外提为纯 builder
- [x] 补或更新回归测试，锁住：
  - OpenAI 路径行为不变
  - MiniMax 路径行为不变
  - quota / continuable / journal / final answer 合同不漂

## Acceptance Criteria

1. `tool-loop.ts` 不再同时内联两份 provider 主循环骨架。
2. provider 差异只剩协议适配，而不是整段执行逻辑复制。
3. `tool-loop.ts` 主入口职责明显收窄，不再同时堆积过厚 prompt 拼装和执行细节。
4. 现有外部合同保持不变：
   - `runAgentToolLoop(...)`
   - `actionJournal`
   - `quotaProfile / remaining* / continuable`
   - 工具失败与成功回灌语义
5. 至少有一组回归测试能防止双路径再次漂移。
