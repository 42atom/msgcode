# Goal

把自动验证闭环收成最薄主链：

- 任务可以声明验证命令
- 运行时或 LLM 可以显式执行验证
- 验证结果可落证据并回给 LLM

## Non-Goals

- 不做 verifier framework
- 不做任务类型到验证器的自动大映射
- 不新增独立 critic / reviewer agent
- 不把系统变成 finish supervisor

## Notes

- 这张单吸收的是 review 里的真缺口，不吸收 Big Harness 倾向。
- 原则仍然是：
  - AI 是主执行者
  - 系统只暴露验证能力和结果真相

## Progress

- 已落地两刀：
  - 任务文档可通过 `## Verify` / `## Verification` 段声明验证命令
  - heartbeat 创建 dispatch 时会提取这些命令
  - dispatch 会把 `verificationCommands` 透传给 subagent `taskCard.verification`
  - subagent 委派 prompt 会显式渲染 `verification` 段
  - 新增 `msgcode verify run`
  - `verify run` 会显式执行任务文档里的验证命令
  - `verify run --dispatch <id>` 会把验证证据回写到 `dispatch.result.evidence`

## Result

- 这张单当前已经成立：
  - 验证命令有统一声明位置
  - 验证命令能被 dispatch / subagent 透传
  - 验证命令能被显式执行
  - 验证结果能被标准化输出，并可回写 dispatch 证据
- 保持了薄主链：
  - 没有新增 verifier framework
  - 没有新增 critic / reviewer agent
  - 没有把系统变成自动裁判层

## Follow-up Consumed

- 这轮口径已被 heartbeat 主链正式消费：
  - 子代理 `DONE` 后，子任务先进入 `rvw`
  - heartbeat 扫描 `rvw` 子任务时，若存在 `## Verify / ## Verification`，就执行验证命令
  - 验证通过后才推进 `pss`
  - 第一次验证失败则停在 `rvw`，并把证据写到 `.msgcode/evidence/<taskId>.json`
  - 下一次 heartbeat 若仍看到同一失败证据，则推进到 `bkd`

## Residual Risk

- 当前仍不做自动重试，也不做智能裁决。
- 这是有意取舍，不是缺口：系统只跑命令、写证据，不替主脑做最后判断。
