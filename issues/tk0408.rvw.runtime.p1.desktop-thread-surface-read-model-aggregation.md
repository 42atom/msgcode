# 任务：Desktop Thread Surface Read Model Aggregation

## Context

当前桌面主链已经成立：

1. thread surface 可写
2. 写桥已隔离到 child process
3. thread 文件写入会 push UI 更新
4. Sketch 已成为 UI 真相源

但 renderer 还在自己拉：

- workspace-tree
- thread
- profile
- capabilities
- hall
- neighbor

再在前端内存里拼出 left/middle/right 三栏。

这违反了“UI 只做画笔”的方向。

## Goal / Non-Goals

### Goal

把桌面 thread surface 需要的读面聚合下沉到 runtime，renderer 只消费更完整的 read model。

### Non-Goals

1. 不把聚合搬进 main process
2. 不新增事件总线、控制面、状态平台
3. 不顺手做流式、中断、多渠道抽象
4. 不先处理 `pendingComposerThreads` 这类 UI 瞬态债

## Decision

正确收口是：

- runtime 负责聚合 read model
- bridge 继续只做桥
- renderer 继续只做渲染

不采用：

- main process 组装完整 JSON 树
- task-supervisor 接管普通桌面对话回包
- 把 detached child 换成新的平台层

## Plan

1. 盘点当前 UI 真正需要的 surface 字段
- 左栏
- 中栏
- 右栏

2. 在 runtime 侧定义更完整的 desktop thread surface read model
- 目标优先放在已有 surface 文件
- 先复用 `workspace-thread-surface.ts`
- 必要时再薄引一层聚合文件

3. 让 bridge 只传这个 read model
- 不增加额外业务判断
- 不让 preload/main 变业务层

4. renderer 改成纯消费
- 不再主动拼 profile/capabilities/hall/neighbor 多份 envelope
- 只做 section 切换、折叠、输入等 UI 层状态

## Acceptance Criteria

1. renderer 不再负责聚合多份 runtime surface
2. main/preload 仍只做桥
3. 读面字段能直接映射左中右三栏
4. 没有新增平台层或控制面

## Notes

### 对 Gemini 三点的裁决

1. `pendingComposerThreads`
- 是债，但不是这张单的主问题
- 先不动

2. detached child
- 当前方向不改
- 不引入 task 平台来接普通桌面对话

3. renderer 组装数据
- 判断对
- 收口点应在 runtime，不在 main

一句话：

这张单只解决“谁负责组装 UI 真相”，答案是 runtime，不是 renderer，也不是 main。
