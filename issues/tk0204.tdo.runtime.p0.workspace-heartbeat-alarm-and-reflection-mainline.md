## Context

- 用户已经明确提出：主脑不仅要会响应，还要会周期性回看工作目录、检查未完成任务、监督 subagent、在完成后写日记并回收记忆与 skill 改进。
- 现有仓库已经有 `scheduler`、`jobs`、`/task`、`TaskSupervisor`、`subagent` 等积木，但缺一条正式的”主脑节律”主链。
- 对标结果显示：
  - `pi-mono` 在文件唤醒上很薄
  - `openclaw` 在 heartbeat / cron / subagent 分工上最清楚
  - `hermes-agent` 在学习循环叙事上最完整
- msgcode 需要在不加厚控制面的前提下，把这三块收口成自己的主循环。

## Goal / Non-Goals

- Goal:
  - 为每个 workspace 建立正式 heartbeat 主循环
  - 提供精确定时 alarm/cron 唤醒机制
  - 把任务巡检、subagent 监督、反思回收接进同一条主链
  - 让所有状态继续优先落到文件真相源
- Non-Goals:
  - 不新建一个重型 autonomic platform
  - 不把 `issues/` 状态真相源替换成数据库状态
  - 不在本单先做大而全的学习系统

## Acceptance Criteria

### A. 主循环地基

1. 主脑存在一条明确 heartbeat 主循环，而不是只靠用户消息驱动。 ✅
2. heartbeat 与 alarm/cron 分工明确：
   - heartbeat = 周期 awareness / 巡检 / 推进
   - alarm/cron = 精确定时唤醒 (Phase 2)
3. heartbeat 默认读取 workspace 文件真相源，而不是另建隐藏状态层。 ✅
4. 没有需要动作时，heartbeat 能静默结束，不刷屏。 ✅
5. `HEARTBEAT.md` 被明确定义为主脑草稿/提示文件，不承载硬状态真相；与 `issues/` 或 `.msgcode/*` 合同冲突时，一律以后者为准。 ✅
6. 任务完成后，存在最小 reflection 流程，至少覆盖：
   - 日记 (Phase 2)
   - memory candidate (Phase 2)
   - skill candidate (Phase 2)
7. 存在一版 heartbeat 协议模板和一版 alarm 文件协议，能支撑后续实现接线。 (Phase 2)
8. alarm 文件协议明确写清 (Phase 2)
9. reflection candidate 不会自动进入正式 memory / skill；下一轮 heartbeat 必须存在最小审核动作决定其去向。 (Phase 2)

### B. 经理型主脑口径

10. 主脑默认优先派单，而不是自己直接下场执行：
    - 面对可委派任务时，优先走 `dispatch + subagent`
    - 只有明确的小任务才允许主脑亲做
11. `completed` 不能只靠子代理口头回报，必须绑定 artifact、evidence 或明确验收结果。
12. 主脑不能只写 `.msgcode/dispatch/*.json`，还必须推进对应 `issues/*.md` 状态；必要时继续推进父任务。
13. 主脑对子代理和用户都要忠诚：
    - 对子代理：会派单、巡检、验收、续推
    - 对用户：阻塞、失败、证据不足、需求变化时及时说真话
14. 至少存在一条测试或 real-test 证据，证明主脑不是“只会派一次”，而是会管理完整闭环：
    - 读任务
    - 选 persona
    - 发 dispatch
    - 子代理执行
    - 主脑验收
    - 回写文档
    - 继续推进下一单或如实汇报阻塞

### C. 当前 Phase 1 对齐点

15. 该主循环能驱动 `tk0202` 与 `tk0203`，使 persona 装配和 doc-first 派单真正转起来。 ✅
    - ✅ 传递 persona 给 subagent runtime

    - ✅ 推进子任务文档状态 (tdo -> pss)
    - ⚠️ 父任务推进需要更完整的多任务依赖检查 (Phase 2)

## Notes

- 这单是主线单。
- 如果没有这条主循环，前面的 persona 与 doc-first dispatch 只能算被动组件，不会自己转。
- 设计目标不是”比别人功能更多”，而是”比别人更薄、更真、更能长期跑”。
- 验收口径不再只是“会不会起 heartbeat”，而是“它是不是一个对用户忠诚、对子代理负责的经理型主脑”。
- Phase 1 已完成最小可跑主链：读任务 -> 选 persona -> 发 dispatch -> 结果可见
- Phase 2 继续补 alarm/cron 和 reflection
