# 任务：web thread chatId 合同

## Context

飞书天然有稳定 `chatId`。

web 没有稳定 `chatId` 时，就会出现：

- 每发一条消息
- 就可能新起一个 thread

这会直接把 thread 做碎。

## Goal / Non-Goals

### Goal

- 定义一个 web thread 对应一个稳定 `chatId`
- 定义 `chatId` 与“新建聊天”的关系
- 定义 UI 与 runtime 的最小职责边界

### Non-Goals

- 不实现 localStorage 细节
- 不改飞书 thread 主链
- 不做全局会话中心

## Plan

- [ ] 说明 web 为什么必须有稳定 `chatId`
- [ ] 说明一个 web thread 对应一个 `chatId`
- [ ] 说明“新建聊天”如何切到新 `chatId`
- [ ] 说明 runtime 何时真正创建 thread

## Acceptance Criteria

1. web thread 的 `chatId` 语义被钉死。
2. “一条消息一个 chatId”被明确禁止。
3. “一个浏览器全局只用一个 chatId”被明确禁止。
4. UI 与 runtime 在 `chatId` 上的职责边界清楚。

## Notes

- UI 可以准备新的 `chatId`。
- 没有第一条消息落盘前，不应有正式 thread 文件。
- `chatId` 是 thread 级别，不是 request 级别，也不是浏览器全局级别。
- 当前已落地最小实现：
  - `msgcode inbox add` 强制要求传 `chatId`
  - web inbox 消费后，同一 `chatId` 在 thread cache 清空后仍会续写同一 thread 文件
