## Context

- 当前飞书出站文本/图片/文件发送已经基本打通，但飞书入站非文本消息还没有进入 msgcode 既有附件主链。
- 代码证据显示 `src/feishu/transport.ts` 当前仅在 `msgType === "text"` 时解析正文，非文本消息不会生成 `attachments`。
- `src/listener.ts` 在 `!text && !hasAttachments` 时会直接 return，因此飞书语音、图片、文件即使收到了事件，也可能在“收到消息”主日志之前被静默跳过。
- 当前 `src/feishu/transport.ts` 同时承担事件接收、文本解析、资源上传、文件发送等职责，入站/出站/资源边界混在一起，不利于继续补齐附件能力。
- 用户希望为后续变化做准备：把“渠道接入”和“能力提供方”分层，统一 msgcode 内部 input/output 结构，并按阶段平滑迁移，而不是继续在现有 transport 上堆补丁。

## Goal / Non-Goals

### Goals

- 把飞书入站音频/图片/文件/PDF 接入现有 `attachments -> vault -> media pipeline` 主链。
- 自动处理策略收口为：仅图片走自动视觉，其余附件只作为附件交给模型决定后续处理。
- 明确 `channel` 与 `provider` 的职责边界，避免 transport 同时承担多种角色。
- 冻结一套 msgcode 内部统一入站消息/附件合同，减少后续渠道扩展时的状态分叉。
- 冻结目录约定：原始下载附件进入 `downloads/`，所有过程产物进入 `artifacts/`。
- 采用分阶段迁移：先加观测，再拆飞书边界，再补齐附件映射，最后考虑其他渠道迁移。

### Non-Goals

- 本轮不重写整个消息系统，不引入新的总线、编排器、控制面。
- 本轮不改动 ASR/OCR/TTS/LLM 的能力实现，只收口它们作为 `provider` 的边界。
- 本轮不承诺一次性迁移 iMessage；先以飞书为样板跑通。

## Acceptance Criteria

1. 飞书 `audio/image/file` 消息进入系统后，不再因 `!text && !hasAttachments` 被静默跳过。
2. `listener` 继续只处理统一 `InboundMessage` / `attachments` 结构，不感知飞书私有字段。
3. 飞书渠道代码的职责边界清晰：
   - 入站适配
   - 出站发送
   - 资源下载
4. `provider` 与 `channel` 的长期目录方案已文档化，后续扩展不需要重新发明边界。
5. 至少有一组飞书语音/图片/文件的回归测试或契约测试，能够锁住“进入附件主链”的行为。
6. 目录语义清晰：`downloads/` 只放原始附件，`artifacts/` 只放过程产物，不再混用。
