# Context

Review 指出 [Issue 0062](/Users/admin/GitProjects/msgcode/issues/0062-feishu-first-imsg-optional.md) 的一个主链问题：

- 静态 [src/deps/manifest.json](/Users/admin/GitProjects/msgcode/src/deps/manifest.json) 已把 `imsg/messages_db` 退到 `optional`
- 但默认 transport 仍会在“无飞书凭据”时回退到 `imsg`
- 结果 `msgcode preflight --json` 会出现 `0/0 启动必需`，无法反映 fallback-imsg 的真实启动前提

这会让 preflight 从“启动前检查”退化成假绿。

# Goal / Non-Goals

## Goal

- 让 preflight 与真实 transport 选择保持一致
- fallback-imsg 时，`imsg/messages_db` 应重新进入本轮 `requiredForStart`
- feishu-only 时，不把 iMessage 依赖误报为启动必需

## Non-Goals

- 不回退 0062 的 Feishu-first 静态口径
- 不重做依赖系统或 probe 体系
- 不顺手处理权限探针的 iMessage 条件化问题

# Plan

- [x] 在依赖清单加载阶段增加 transport-aware 启动依赖提升逻辑
- [x] 让提升逻辑能读取 `.env` 中的 transport / feishu 凭据，而不是只看当前进程环境
- [x] 补 preflight 回归测试，锁住 fallback-imsg 与 feishu-only 两条路径
- [x] 更新 issue / plan / changelog 留痕

# Acceptance Criteria

- 无飞书凭据时，`msgcode preflight --json` 不再显示 `0/0 启动必需`
- 有飞书凭据时，preflight 不把 `imsg/messages_db` 当作 `requiredForStart`
- 静态 `src/deps/manifest.json` 仍保持 Feishu-first 口径，不硬编码 iMessage 启动必需
