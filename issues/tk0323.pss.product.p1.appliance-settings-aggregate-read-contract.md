# 任务：core settings 聚合读面合同

## Context

现在 settings 页已经有四块只读面：

1. profile
2. general
3. capabilities
4. doctor

若 UI 每切一个 tab 都去起一次 CLI，启动开销会直接反映到设置页体验上。

但 `doctor` 本质上是 probe 面，不该拖慢整个 settings 首屏。

## Goal / Non-Goals

### Goal

- 定义一个单一 core settings 聚合读面
- 让 UI 一次取走 profile/general/capabilities
- 保留每块分组自己的 warning/error

### Non-Goals

- 不改成长连接/RPC
- 不删除现有细分 CLI
- 不把 settings 做成缓存中心
- 不把 doctor 硬塞进首屏聚合

## Baseline

建议增加：

`msgcode appliance settings --workspace <...> --json`

返回三段：

- profile
- general
- capabilities

`doctor` 继续独立：

`msgcode appliance doctor --workspace <...> --json`

每段都保留：

- `status`
- `warnings`
- `errors`
- `data`

顶层再给一个聚合状态。

UI 使用规则：

- 打开 settings 时一次读取
- 用户显式刷新时再读取
- 不按 tab 切换高频重跑
- `doctor` tab 按需单独读取

状态消费规则：

1. 先解析 JSON envelope
2. 再看顶层 `status`
3. 再看各段 `status`
4. `warning` 继续显示数据，但必须露出 warning
5. `error` 只阻断当前段
6. `warning` 即使带非零退出码，UI 也必须优先解析 stdout 中的 envelope

## Plan

- [ ] 定义聚合 envelope 结构
- [ ] 定义分段 status 合成规则
- [ ] 明确 core settings 初始化和 doctor 按需读取的边界
- [ ] 做一轮圆桌复核

## Acceptance Criteria

1. settings 页的 core settings 无需按 tab 高频起 CLI。
2. 单一聚合读面不替代现有细分 CLI，只做 UI 入口。
3. 每块 warning/error 不会在聚合层被吞掉。
4. `doctor` 不会拖慢整个 settings 首屏。
5. 不引入 RPC、Socket 或后台 settings 守护进程。
6. UI 不把聚合读面当作 tab 轮询接口。

## Notes

- 这刀只服务 settings 页初始化与显式刷新。
- `doctor` 保持独立，不为了整齐把 probe 面塞进 core settings。
