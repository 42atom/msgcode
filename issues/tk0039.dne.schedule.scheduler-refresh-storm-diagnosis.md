## Context

`/Users/admin/.config/msgcode/log/msgcode.log` 在 `2026-03-08 03:17:51` 到 `03:17:55` 之间连续出现大量 `[Scheduler] 已刷新` 与 `[Scheduler] 下次唤醒`。同一时间窗内还出现了多次 `[Scheduler] 已停止`、`[Scheduler] 已启动`、`[Scheduler] jobs.json 不存在，创建新文件`，说明存在重复 refresh 或多实例起停链路，但当前日志没有 refresh 来源，无法区分是正式运行时重复调用，还是测试/热加载污染到同一日志文件。

已知事实：

1. `live-cron` 创建/删除主链已经修通，不是本单阻塞点。
2. 风暴时间窗为 `2026-03-08 03:17:51` 到 `03:17:55`。
3. 现有 `JobScheduler.refresh()` 没有来源标签，`requestSchedulerRefresh()` 也不区分调用方。
4. 显式 refresh 调用点目前只有：
   - `JobScheduler.start()`
   - `commands.ts` 的 `SIGUSR2`
   - `schedule-sync.ts` 的 `refreshActiveJobScheduler()`

## Goal / Non-Goals

### Goal

1. 给 scheduler refresh 链路补最小来源证据。
2. 找出短时间重复 refresh 的真实来源。
3. 在拿到证据后做最小修复，收口回单一 mutation -> refresh 主链。
4. 补测试或日志锁，防止同类回归。

### Non-Goals

1. 不重构 scheduler 引擎。
2. 不修改 schedule 功能语义。
3. 不做额外日志降噪重构。
4. 不扩到 browser、memory、event-queue。

## Plan

- [x] 给 `refresh` / `armTimer` / refresh 入口补来源标签与关键上下文

- [x] 基于证据做最小修复并补回归测试
- [x] 更新 issue/plan/CHANGELOG（若涉及外部可见运行时行为变化）

## Acceptance Criteria

1. 能明确回答 refresh 风暴的真实来源，不依赖猜测。
2. 修复后 `schedule create/remove` 主链仍保持正常。
3. 正式 `msgcode.log` 不再出现短时间大量重复的 `已刷新 / 下次唤醒` 风暴。
4. 至少有一条测试或日志锁覆盖本次根因。
5. 完成后提交 commit，并重启 `msgcode`。
