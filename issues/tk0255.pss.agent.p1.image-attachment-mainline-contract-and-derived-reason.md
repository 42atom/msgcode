## Context

- 2026-03-19 family 真实线程里，图片入站后没有走 `vision`，而是先撞了 `read_file`，最后回成“我没有视觉能力”。
- 同一工作区的 `.msgcode/config.json` 没有显式 `tooling.allow`，所以真实 allowlist 回退到了默认配置；默认不含 `vision`。
- 自动图片派生已经有 `status=unavailable`，但 `listener` 只记 `status`，没有把 `reason` 进日志，也没有把它送进 agent-backend 本轮上下文。
- 现有 exec prompt 已经写了“图片 follow-up 复用事实”和“不要对图片本体 read_file”，但这条规则离现场太远，第一轮图片附件场景仍然会偏航。

## Structure Review

- 硬伤 1：图片事实主链分裂。
  - 自动图片摘要、正式 `vision`、skill fallback 都存在
  - 但失败 reason 没有进入单一执行上下文
  - 结果模型只能看到“有个图片附件”，看不到“为什么自动摘要没成”
- 硬伤 2：图片规则只放在系统 prompt 深处，没有贴住本轮图片消息。
  - 这不是缺新层
  - 是图片合同离执行现场太远
- 非硬伤：默认 allowlist 不含 `vision`
  - 这是设计取舍，不是本次要全局改掉的 bug
  - 真实图片群应按 workspace 显式 opt-in

## Goal

- 把图片附件主链规则前置到 agent-backend 本轮消息上下文
- 把 `derived.status=unavailable` 的 `reason/error` 一起进日志与上下文
- 让模型知道“这是图片附件”“不能对图片本体 read_file”“若无 `vision` 应转 `vision-index`”，而不是瞎撞

## Non-Goals

- 不把 `vision` 加进全局默认 allowlist
- 不新增图片状态层
- 不改 Tool Bus 策略
- 不替模型做图片理解

## Goal / Non-Goals

- Goal:
  - 图片附件场景把主链规则前置到本轮上下文
  - `derived unavailable reason` 进入日志与上下文
- Non-Goals:
  - 不改全局默认 `tooling.allow`
  - 不新增图片状态层或控制面

## Plan

- [x] 在 `listener` 提取 `derived unavailable reason`
- [x] 在 agent-backend 附件上下文里前置图片主链规则
- [x] 补专属回归并验证

## Acceptance Criteria

1. 图片附件场景的 agent-backend 输入会明确写出图片主链规则。
2. `derived.kind=vision` 且 `status!=ok` 时，`reason/error` 不再丢失。
3. `listener` 日志会记录 `unavailable` 的 `reason/error`。
4. 相关回归通过。

## Result

- `listener` 现在会：
  - 把图片附件规则前置进本轮消息上下文
  - 明确禁止对图片附件路径使用 `read_file`
  - 明确要求：有 `vision` 就走 `vision`；没暴露就按 `vision-index` 走
  - 明确禁止再说“我没有视觉能力”
- `derived.kind=vision` 的 `unavailable reason/error` 现在会：
  - 进 `listener` 日志
  - 进 agent-backend 本轮上下文
- 这刀只收口现场合同，没有新增任何控制层
