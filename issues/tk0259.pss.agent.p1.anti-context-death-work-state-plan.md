## Context

用户希望把“summary 不是连续思考本体，只是有损重建接口”收口成 msgcode 的正式设计计划。

仓库里已经有一条 `work continuity -> work capsule -> session continuity` 主线，但当前 runtime `work capsule`
是给系统恢复执行用的最小投影，不等于给模型跨窗口恢复工作态用的完整支架。

需要补一份计划，把这两层关系讲清楚：

- 哪些继续由现有 `work capsule` 负责
- 哪些应该由新的“工作态骨架”承担
- 什么时候触发
- 以什么文件形态存在

## Goal / Non-Goals

- Goal: 明确 `summary`、`work capsule`、`working state scaffold`、`evidence` 的职责边界。
- Goal: 给出适配 msgcode 当前 file-first 架构的最小落地方案。
- Goal: 保持 Unix 风格，不新增大一统控制面。
- Non-Goals: 不实现 persistent KV / latent state。
- Non-Goals: 不把 runtime `work capsule` 直接改造成第二真相源。
- Non-Goals: 不立即改 heartbeat / wake / task runtime 代码。

## Plan

- [x] 形成一份正式 plan，说明问题定义、与现有 `work capsule` 主线的关系、最小文件设计、触发条件与分阶段落地路径。
- [x] 形成一份 `WORKSTATE` 协议文档，固定职责边界、模板和触发规则。

## Acceptance Criteria

1. 存在一份正式 plan，能说明 `summary`、`work capsule`、`WORKSTATE` 与 `evidence` 的职责边界。
2. 存在一份 `WORKSTATE` 协议文档，能指导后续最小实现。

## Notes

- 当前方案默认不碰 runtime 主链代码，先以协议与文档收口。
- 现有相关主线：`work continuity -> work capsule -> session continuity`。
- 后续 `0276 / 0277` 已补出纯文本试跑、真实 compact 样本、读回主链与 handoff 回归锁。
