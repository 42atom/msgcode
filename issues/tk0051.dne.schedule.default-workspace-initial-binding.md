## Context

- 当前未显式绑定的新群会在消息入口回落到 `default` workspace。
- 但该 fallback 没有持久化到 `routes.json`，导致依赖显式 route 的链路仍认为“未绑定”。
- 2026-03-09 的真实日志已经出现该断裂：普通消息能在 `default` 中处理，但自然语言创建 cron 最终失败，报 `工作区未绑定到任何群组`。

## Goal / Non-Goals

- Goal: 让“新群默认落到 default”成为真实绑定事实，而不是临时路由假象。
- Goal: 让依赖 route store 的链路（尤其 schedule 创建）与默认落地主链一致。
- Non-Goals: 不新增 scheduler 专用 fallback。
- Non-Goals: 不重构 RouteStore 数据结构。

## Plan

- [x] 创建 Plan 文档，冻结最小修法与回滚口径
- [x] 在默认 fallback 主链上持久化 route 到 `routes.json`
- [x] 补默认路由持久化测试
- [x] 补 default workspace 下 `schedule add` 回归锁
- [x] 运行针对性测试并记录结果
- [x] 更新 CHANGELOG（运行时行为变更）

## Acceptance Criteria

1. 新群首次落到 `default` workspace 后，`routes.json` 中出现对应 route。
2. 未显式 `/bind` 的新群在 `default` workspace 内可成功创建可投递的 schedule。
3. 显式 `/bind` 的优先级保持不变。
4. 不为 `schedule` 单独新增兼容层或专用 fallback。
