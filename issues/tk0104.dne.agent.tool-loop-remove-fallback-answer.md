## Context

`tool-loop` 当前仍保留 `buildToolLoopFallbackAnswer()` 这条系统代答旁路：当模型在工具轮后返回空文本或残留协议片段时，系统会直接拼出“读取成功，内容预览如下”“命令执行完成”等用户答复。这会继续抢走模型执行权，使执行核从“结果回给模型”滑回“系统替模型交卷”。

## Goal / Non-Goals

### Goal

- 删除 `buildToolLoopFallbackAnswer()` 这条系统代答旁路
- 当模型没有给出合法最终答复时，继续要求同一个模型补一条真正的最终答复
- 保持现有 `tool_result -> 模型` 主链，不新增第二条规则层输出

### Non-Goals

- 本轮不处理 `finish supervisor` 的整体退役
- 本轮不修改 verify phase 语义
- 本轮不重写整个 tool-loop

## Plan

- [x] 新建 `0104` issue 与对应 plan，冻结本轮边界
- [x] 在 `tool-loop` 两条主路径中删除 `buildToolLoopFallbackAnswer()` 调用
- [x] 当模型工具轮后没有给出合法最终答复时，再发起一次“只要最终答复、禁止工具”的补答请求
- [x] 补行为锁：系统不再自己回“读取成功/命令执行完成”，而是等待模型给真正最终答复
- [x] 跑定向测试与类型检查，记录行为变化

## Acceptance Criteria

- `buildToolLoopFallbackAnswer()` 不再影响用户可见最终答复
- 当模型首轮结束文本为空或仍含协议残片时，系统会向模型补要最终答复，而不是系统代答
- 用户最终看到的仍是模型答复，不是规则化模板
