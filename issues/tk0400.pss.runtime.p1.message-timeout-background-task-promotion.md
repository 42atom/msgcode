# 任务：Message timeout background task promotion

## Context

这是一张止血单。

事故面很简单：

- 飞书前台消息 run 会占用 chat 级串行队列
- HomePod 这类长 bash 探测一旦拖住，后续消息只能入队，用户侧表现为“完全没回复”
- 原系统只有 `/task run` 的显式后台任务主链，没有 message 超时后的自动脱离

## Goal / Non-Goals

### Goal

- message 来源默认允许最多 3 分钟 wall clock
- 超时后结束当前前台消息 run，不再继续霸占 chat 队列
- 立刻创建后台 task 继续执行
- 立刻回执飞书：已转后台继续
- task 完成 / blocked / failed 后主动回原 chat
- 用户面只暴露三位任务号 `001-999`

### Non-Goals

- 不默认派 `subagent`
- 不新增 task alias 表、任务注册表或第二状态中心
- 不把 task runtime 变成新的多代理编排平台
- 不改动正式 UI

## Plan

### 主链

```text
message run
  -> 3分钟内正常完成：直接回帖
  -> 3分钟超时：返回 continuable
      -> listener.createTask(...)
      -> 立即回执 "已转后台继续"
      -> trigger heartbeat now
      -> task run 在后台继续
      -> 终态回原 chat
```

### 边界

- `message -> task` 只是后台化，不是换执行臂
- 后台 task 仍由主脑执行：`executeTaskTurn -> executeAgentTurn`
- `subagent` 仍是另一条独立 tmux 执行臂，只在明确委派时使用

### 任务号

- 内部继续保留 `taskId` UUID 作为真相
- 对人新增三位 `taskRef`
- `/task status|cancel|resume [任务号]` 支持三位任务号
- `/task status` 在没有活跃任务时回最近一条任务，便于手机侧查看是否已终态

## Acceptance Criteria

- 飞书前台不再因长工具回合静默卡死
- 超时后用户能立即收到后台转移回执
- heartbeat 能立即续跑，而不是等下一次周期 tick
- task 终态会主动回原 chat
- 用户面不再要求输入长 UUID

## Notes

- 这是运行时事故修复，不是产品新功能包装。
- 真相源仍是 `.msgcode/tasks` 单文件任务记录；三位任务号只是用户面投影。
