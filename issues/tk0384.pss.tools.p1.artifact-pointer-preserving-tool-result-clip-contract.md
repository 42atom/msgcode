# TK0384：Artifact Pointer Preserving Tool Result Clip Contract

## Context

当前仓库里，几类工具已经具备“正文太大时给模型一个继续入口”的基础能力：

- `bash`：`fullOutputPath`
- `browser tabs.text`：`textPath`
- `ghost_*`：`artifacts[]`

但 `tool-loop` 现在仍然对所有 `previewText` 统一 `512` 字符从头截断。

这会带来一个很具体的问题：

- 如果 `previewText` 前面放了大量正文/JSON
- 而关键 artifact 指针排在后面
- 模型最终就看不到真正该继续追读的路径

这和 `read_file` 的问题是同一家族：不是没有防爆，而是防爆后把继续主链裁断了。

## Goal / Non-Goals

### Goal

- 定义带 artifact 指针的 tool result 在裁剪后的硬保留字段
- 明确 `bash/browser/ghost` 第一刀的保留口径
- 确保模型无论结果多长，都能看到后续继续动作的路径

### Non-Goals

- 不重写所有 preview builder
- 不扩到 `read_file`（它由 `tk0383` 处理）
- 不扩到全部工具家族
- 不新增 preview controller

## Core Contract

### bash

如果存在 `fullOutputPath`，那么回灌模型时必须保留：

- `exitCode`
- `fullOutputPath`

正文尾部可以裁，但 `fullOutputPath` 不能被裁没。

### browser

如果存在 `textPath`，那么回灌模型时必须保留：

- `operation`
- `textPath`
- `textBytes`（若存在）

正文 preview 可以裁，但 `textPath` 不能被裁没。

### ghost

如果存在 `artifacts[]`，那么回灌模型时至少要保留：

- `statusSummary`
- artifact 路径列表中的前若干项

结构化正文/文本正文可以裁，但 artifact 路径不能被裁没。

## Plan

1. 先确认 `tool-loop` 当前对 `previewText` 的统一裁剪位置
2. 只补“关键元数据硬保留”合同
3. 第一刀按 `bash/browser/ghost` 三类覆盖
4. 如果这刀证明有效，再决定是否推广到更多工具

## Acceptance Criteria

1. `bash` 的 `fullOutputPath` 在裁剪后仍可见
2. `browser` 的 `textPath` 在裁剪后仍可见
3. `ghost` 的 artifact 路径在裁剪后仍可见
4. 这张单明确不扩到 `read_file`
5. 这张单明确不重做 preview 平台

## Notes

- `pl0128` 解决的是执行层 footer 统一，不是 `tool-loop` 最后一跳的硬保留问题
- `pl0131` 解决的是 browser 产出 text artifact，不是 artifact 指针在回灌时是否会被二次裁掉
- 所以这次仍需新 task，而不是回头重写旧单
