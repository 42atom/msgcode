## Context

用户已经明确：

- 当前产品更适合以 `DMG` 方式分发
- 将来要尽快拿出去见人，需要一个有产品感的客户端壳
- 真正的主脑仍然是 `msgcode`
- 壳主要承担：
  - bundled `msgcode` 二进制
  - 授权入口
  - 启动/停止本地 runtime
  - 打开本地 UI
- 用户也希望为未来 Linux / Windows 保留可能性

同时，基于对 `ChatWise.app` 的本地私有分析，已经确认一种可借鉴的形态：

- Electron 壳
- 本地 Web UI
- 主进程窄桥接
- 独立更新链

但 `msgcode` 不应把主脑搬进 Electron，而应继续让 runtime 保持中心地位。

## Goal / Non-Goals

- Goal: 定义 Electron 薄壳的职责边界。
- Goal: 定义 bundled `msgcode` runtime 与本地 Web UI 的关系。
- Goal: 定义授权、启动、状态、更新在壳中的最小职责。
- Goal: 保留未来 UI 层跨平台可能，但不误判为整机能力已跨平台。
- Non-Goals: 不直接实现 Electron 工程。
- Non-Goals: 不直接做视觉设计。
- Non-Goals: 不改当前 runtime 主线。

## Plan

- [ ] 形成一份独立 plan，定义薄壳、runtime、本地 Web UI 三层结构。
- [ ] 固定壳中应该做和不该做的事情。
- [ ] 定义最小交付链：授权、启动、打开 UI、更新。
- [ ] 明确这条线与当前开发主线隔离。

## Acceptance Criteria

1. 仓库内存在一份独立计划，能说明 Electron 薄壳的最小职责和边界。
2. 该计划明确写清：`msgcode` 仍是主脑，Electron 只是宿主壳。

- 当前目标是明确“该怎么做薄”，不是尽快起壳工程。
