# 任务：ui-protype 原型研究面恢复实现

## Context

`tk0379` 只负责把 `ui-protype` 的职责口径收回原型研究面，不该继续背代码改动。

这刀只收 `ui-protype`，不动 `src`，不碰正式宿主桥接。

## 为什么不能直接用 tk0379 承载代码改动

`tk0379` 的 scope 已经写死为“只收职责口径，不改代码，不改视觉，不改 `src`”。

如果把实现硬塞回去，会把边界文档和代码落点混成一张单，后续很难分清：

1. 哪些是职责边界
2. 哪些是实现状态
3. 哪些是历史收口

所以需要单独开一张最薄实现单，只管代码恢复。

## Goal / Non-Goals

### Goal

- 去掉 `window.msgcodeReadonlySurface.runCommand` 这类正式宿主桥接依赖
- 去掉只服务正式读面联通的错误/空状态逻辑
- 恢复成可静态阅读/演示的原型页面
- 保留既定一级入口和 UX 结构

### Non-Goals

- 不改 `src`
- 不做正式 UI
- 不做视觉大改
- 不改打包脚本
- 不做宿主桥接

## Plan

- [ ] 把 `main.js` 改成静态原型数据驱动
- [ ] 把 `main.html` 收回原型演示结构，去掉 formal error surface
- [ ] 只在必要时补最小样式，不重做视觉系统

## Acceptance Criteria

1. `ui-protype` 不再依赖正式宿主桥接。
2. `ui-protype` 回到原型研究面。
3. `settings` 等既定入口仍在。
4. `docs:check` 和 `node --check ui-protype/main.js` 可过。

## Notes

- 这张单只收 `ui-protype` 的静态恢复，不回写 `src`。
- 代码面只允许保留一级入口和原型结构，不保留正式宿主桥接。
- 验证优先按最小相关面走：`npm run docs:check`、`node --check ui-protype/main.js`。
