# Context

当前 graph 质量结论里，`dead-unresolved` 与 `dead-entry` 信号比传统意义上的 dead-leaf 更强，但现有总死码率把三者混成单一数字，容易误读成“仓库近一半都是垃圾代码”。

同时，`scripts/`、`features/`、外部参考与一次性入口也会进入统计，进一步抬高 dead 指标。需要先把度量口径分层，再决定哪些才是真重构目标。

# Goal / Non-Goals

## Goal

- 明确 `dead-unresolved`、`dead-entry`、`dead-leaf` 三类指标的解释口径
- 明确主产品统计与辅助目录统计的分层方式
- 给出后续 graph 健康汇报应如何展示的最小规范

## Non-Goals

- 不修改 runtime/product 业务代码
- 不在这张单里优化 codegraph 本体
- 不把 graph 做成新平台

# Plan

1. 盘点当前 graph 统计里哪些目录会污染产品死码率判断
2. 定义最小分层视图：
   - 产品代码
   - 辅助脚本/测试入口
   - 外部参考/归档
3. 定义三类 dead 指标的展示与解释口径
4. 给出后续健康报告的最小输出模板

# Acceptance Criteria

1. 有正式文档说明为什么不能只看单一 dead rate
2. 主产品代码与 scripts/features/AIDOCS 的统计边界明确
3. `dead-unresolved`、`dead-entry`、`dead-leaf` 三类指标各自含义清楚
4. 后续 graph 汇报可以按分层视图重复执行

# Notes

- 这是一张口径修复单，不是代码重构单
- 目标是先把数字说清楚，再决定是否值得动代码
