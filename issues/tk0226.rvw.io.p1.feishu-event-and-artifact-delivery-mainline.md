# Goal

把 I/O 系统的第一张现役主线单收成：

- Feishu 负责把世界输入变成事件
- Feishu 负责把系统结果送成可验证交付
- 调度和任务推进不留在 I/O 层

## Non-Goals

- 不做多通道平台
- 不做 mail / voice / browser / ghost 的统一抽象层
- 不在这张单里扩 heartbeat / dispatch / subagent
- 不重构整个 `src/feishu/transport.ts`

## Plan

- [x] 定一版 I/O 薄合同：
  - [x] 输入是事件
  - [x] 输出是交付
  - [x] I/O 不承载任务状态真相
- [x] 盘清 Feishu 当前主链：
  - [x] 入站事件最小字段
  - [x] 附件落盘边界
  - [x] 出站文本/文件/图片交付边界
  - [x] 真实回执与失败保真
- [x] 补最小回归或 live 证据：
  - [x] 成功发送时，结果会带真实 `receipt`
  - [x] 文件/图片发送失败时，错误能保真回到主链
- [ ] 明确后续拆相：
  - [ ] mail
  - [ ] voice
  - [ ] browser / ghost

## Acceptance

1. 仓库内存在一份明确的 `IO.md` 协议文档。  
2. I/O 被明确限制为“事件输入 + 交付输出”，不再和调度系统混写。  
3. Feishu 现役主线的最小边界被说清：
   - 入站事件
   - 附件落盘
   - 文本/文件/图片交付
   - 真实回执 / 真实失败
4. 后续 mail / voice / browser / ghost 不会直接平铺进主线，而是按同一合同另拆薄单。
