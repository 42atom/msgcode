# 任务：Desktop Writable Thread Surface Mainline

## Context

当前正式代码真相是：

1. Electron 主窗口合同仍是 readonly：
   - `/Users/admin/GitProjects/msgcode/src/ui/main-window/readonly-thread-surface.ts`
   - `blockedActions: ["archive", "new chat", "composer", "send"]`
2. host bridge 只暴露：
   - `workspace-tree`
   - `thread`
3. 这让桌面端只能看，不能写。

外部参考真相是：

1. `openclaw` 明确存在 `chat-compose` 输入面：
   - `/Users/admin/GitProjects/GithubDown/openclaw/ui/src/styles/components.css`
2. `codex` 也把 composer 当 thread 输入主链，而不是外围补丁：
   - `/Users/admin/GitProjects/GithubDown/codex/docs/tui-chat-composer.md`
   - `/Users/admin/GitProjects/GithubDown/codex/sdk/typescript/src/thread.ts`

所以这不是“要不要做一个输入框”，而是“桌面 thread 是否具备可写合同”。

## Goal / Non-Goals

1. 把主窗口从 readonly thread surface 演进到 writable thread surface。
2. 明确 thread 的可写语义，而不是按飞书/非飞书硬编码。
3. 让桌面端成为正式输入面，不再长期依赖外部渠道。

### Non-Goals

1. 不做多渠道编排平台。
2. 不做大而全消息中心。
3. 不先暴露温度、TTS、视觉模型等低频设置。
4. 不把 composer 做成新的控制面。

## 核心合同

桌面输入是否出现，只看 thread 合同：

- `thread.writable`

最小解释：

1. `writable=false`
   - 只读消息流
   - 不显示 composer
2. `writable=true`
   - 显示 composer
   - 允许桌面端提交输入

UI 不知道“往哪发”，只知道“能不能发”。

写回目标是 runtime 内部真相，不进入 renderer 合同。

## 写桥合同

bridge 只补一个最薄入口：

- `sendThreadInput({ workspacePath, threadId, text }): Promise<void>`

最小解释：

1. UI 只提交：
   - `workspacePath`
   - `threadId`
   - `text`
2. runtime 自己根据 thread 真相解析写回目标
3. 不把渠道类型、replyTarget、路由细节泄漏给 UI

## 最小主链

1. 左栏继续选 workspace / thread
2. 中栏继续读 thread
3. 当 thread 可写时，底部出现 composer
4. 用户输入走同一条主链：
   - composer -> `sendThreadInput()` -> runtime 解析 thread 真相 -> agent run -> thread append

禁止额外加：

- 渠道调度层
- 中间命令翻译层
- 双写缓存层

## P0 范围

### 必做

1. 把 `readonly-thread-surface` 合同升成可表达 `writable` 的主线合同
2. bridge 补出最小写回入口
3. 仅对可写 thread 显示 composer
4. composer 只做最小输入行为

### 非 P0

1. 多媒体输入
2. slash GUI 完整替代
3. 复杂草稿恢复
4. 线程内高级模型参数
5. 渠道路由细节外露给 UI

## Plan

1. 先定 writable thread 合同
2. 再定 bridge 写回合同
3. 再让正式 UI 只对可写 thread 显示 composer
4. 最后补最小验证

## 实现顺序

1. 先定 host bridge 写回签名
2. 再补 writable thread 合同
3. 再让正式 UI 只对可写 thread 显示 composer
4. 最后补最小测试

## Composer 最小行为

P0 只做：

1. 单文本输入
2. `Enter` 发送
3. 成功清空
4. 失败保留

P0 不做：

1. 草稿恢复
2. 多模态输入
3. slash GUI
4. 高级编辑器行为

## Acceptance Criteria

1. 仓库里明确存在 writable thread 合同文档
2. 文档明确：composer 不是页面装饰，而是 `thread.writable` 的表现
3. 文档明确：写桥签名是 `sendThreadInput({ workspacePath, threadId, text })`
4. 文档明确：写回目标由 runtime 解析，不进入 UI 合同
5. 文档明确：readonly thread 仍可存在，但不显示 composer
6. 没有引入新的渠道编排层

## Notes

一句话：

这张单解决的是“桌面 thread 可不可写”，不是“底部输入区好不好看”。

补充约束：

1. P0 不做流式输出，但后续桌面流式能力应落在 thread 读面的推送模型，不应把读桥永久锁死在 request-response。
2. P0 的 `sendThreadInput()` 仍返回 `Promise<void>`；后续若做中断，可再演进为可取消句柄，但不在本单提前实现。
