## Context

`routeByChatId()` 当前在未命中显式 route 和静态配置时，会回落到 `default` workspace，并且立即 `setRoute()` 写入 `routes.json`。这意味着系统会替用户和模型把一个未绑定 chat 永久路由到 default workspace。这个行为最早是为了解决 schedule 链和消息链断裂，但从当前“AI 主执行权 + 显式边界”主线看，它仍属于系统代路由。

## Goal / Non-Goals

### Goal

- 保留 default workspace 的临时 fallback，维持开箱即用
- 删除 default fallback 的自动持久化
- 让需要真实 route 的链路重新显式依赖 `/bind`

### Non-Goals

- 本轮不移除 default workspace fallback 本身
- 本轮不修改 `/where` 对默认工作目录的展示
- 本轮不重写 route store

## Plan

- [x] 新建 `0106` issue 与对应 plan，冻结本轮边界
- [x] 修改 `src/router.ts`：default fallback 不再 `setRoute()`
- [x] 更新回归锁：default fallback 仍可解析，但 route store 不应被写入；依赖显式 route 的 schedule add 重新失败闭合
- [x] 跑定向测试、类型检查和 docs:check

## Acceptance Criteria

- 未绑定 chat 仍可临时落到 default workspace
- `routes.json` 不再因为临时 fallback 被自动写入 default route
- `schedule add` 这类需要真实 route 的链路在未 `/bind` 时明确失败
