## Context

当前 skill 主线已经收口为：

- skill 默认是说明书（`SKILL.md`）
- wrapper 只保留给真桥接
- `msgcode` 二进制重新回到第一公民位置

因此，后续最关键的验证对象不只是“代码函数能不能跑”，而是：

- skill 文案能否把模型带到正确主链
- wrapper 退役后，模型是否还能正确找到工具/CLI
- 模型是否会嘴上说“完成”，但没有真实副作用
- 出问题时，日志是否足够帮助定位是 skill 误导、工具选错，还是运行时主链故障

这类问题无法只靠源码 grep 或纯单测判断，需要一套固定 prompt corpus + debug 观测口径。

## Goal / Non-Goals

- Goal: 定义一组固定 prompt 作为 skill live 验证案例
- Goal: 为每条案例明确目标 skill、预期工具路径、禁止路径与副作用
- Goal: 冻结 debug 模式下的标准观察点，便于快速定位误导点
- Goal: 让这套 corpus 同时服务于 skill 改造验证与大模型能力测试
- Non-Goals: 本轮不实现新的 benchmark 平台
- Non-Goals: 本轮不自动化所有 live 执行
- Non-Goals: 本轮不直接修改 tool-loop 或 CLI 主链逻辑

## Plan

- [x] 明确 prompt corpus 的定位：skill 改造后的真实验证合同
- [x] 定义 v1 的固定案例集与分组
- [x] 为每条案例写清：目标 skill、预期工具、禁止路径、预期副作用
- [x] 明确 debug 日志默认观察点
- [x] 把 corpus 落到 AIDOCS，供后续反复复用

## Acceptance Criteria

1. 仓库中存在正式 issue + plan 说明 `skill live prompt corpus v1` 的用途与边界。
2. 存在一份可直接复用的 prompt corpus 文档，含固定 case 列表。
3. 每条 case 至少包含：prompt、目标 skill、预期执行路径、禁止路径、预期证据。
4. 文档明确 debug 模式下要看哪些日志/产物字段。
5. 这套方法能同时服务 skill 改造验证与模型能力测试。
