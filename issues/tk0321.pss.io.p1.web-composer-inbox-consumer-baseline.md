# 任务：Web Composer Inbox 消费基线

## Context

现在已经有：

- `.msgcode/inbox/`
- `msgcode inbox add`

但还没有：

- 谁来消费这条输入
- 是否复用现有 `listener.handleMessage()` 主链
- Web composer 后面到底要不要有新 transport

## Goal / Non-Goals

### Goal

- 明确 Web composer 输入的最小消费主链
- 明确现有 `listener / route / workspace config` 哪些能复用
- 明确 UI 仍然只能被动渲染 runtime 回写

### Non-Goals

- 不直接实现 Web transport
- 不做 inbox daemon
- 不做 UI 第二状态层
- 不做跨渠道上下文统一大重构

## Plan

- [x] 盘点现有 `inbox -> listener` 可复用主链
- [x] 定义 Web composer 最小消费路径
- [x] 明确必须新增的最少代码点
- [x] 明确这轮绝不做的扩展路径

## Acceptance Criteria

1. 说清楚谁消费 inbox。
2. 说清楚是否复用现有 listener 主链。
3. 说清楚哪些实现会违反当前原则，必须禁止。

## Notes

- 这张单只收消费主链，不碰 UI 视觉。
- 仍然坚持：原始输入先落文件，UI 不能自管聊天真相。
- 结论：不要复用 listener 作为 Web 消费核心；应复用 `RuntimeRouterHandler.handle()` 这条真正的工作主链。
