# 任务：Neighbor / Coworker 可选模块基线

## Context

用户希望未来在局域网里部署多个 `msgcode`，让它们像同事一样自由沟通和互相帮助。

但这条线如果不先把边界讲清楚，很容易滑成：

- 中央编排平台
- 全局共享 workspace
- 第二任务系统
- 跨节点 RPC 总线

所以这张单先不做实现，先把前提和阶段路线钉住。

## Goal / Non-Goals

### Goal

- 明确这条线是 `msgcode` 的可选 neighbor/coworker 模块
- 明确第一阶段只做发现、联系人三态、异步邮箱、轻量 health 探针
- 明确完整 workspace 不共享，只共享显式 export surface
- 明确聊天记录式查看和协作线程属于后续阶段
- 形成一份能交给外部模型审阅的正式文档组合

### Non-Goals

- 不实现网络发现
- 不实现联系人系统
- 不实现聊天 UI
- 不接现有 dispatch 主链
- 不做多人群聊状态机
- 不做复杂编排

## Plan

- [x] 起正式 plan，定义 neighbor/coworker 模块边界
- [x] 起正式 task，作为这条线的真相源入口
- [x] 用“唱反调”视角做一轮结构审视
- [x] 把反方指出的真实风险收进 plan
- [ ] 等用户拿去给 Opus 和 Gemini 外审

## Acceptance Criteria

1. 第一阶段被明确收紧为：
   - 发现
   - `discovered / known / contact`
   - 异步邮箱
   - 轻量 `health`
2. 文档明确写死：
   - thread 不是任务真相源
   - export surface 只允许结果导出
   - 默认不共享完整 workspace
   - `health` 只代表最近可达性，不代表业务活跃度
3. 存在一份反方 review，指出主要风险和最小修正。
4. 主线索引能正确反映这张新单，不与当前仓库事实冲突。

## Notes

- 这条线现在更像“联系人和邮箱模块”，不是“多 Agent 平台”。
- 线程、patch、review、聊天记录视图、结果导出接入都被后移到第二阶段。
- 当前已先落一条只读文件主链切片：`0302`
- 如果后续方向被证明正确，再评估是否把它逐步收到主能力里。
