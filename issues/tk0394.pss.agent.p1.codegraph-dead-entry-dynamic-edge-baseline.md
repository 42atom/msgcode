# Context

当前 graph 统计里存在一批 `dead-entry`。这类符号不是普通死叶子，而更像：

- CLI 注册入口
- Electron main/preload/renderer 启动入口
- 事件/注册/调度回调
- 运行时动态分发点

它们没有被静态图连上，不代表真的无用。需要先收一份“动态边基线”，把假死入口与真死入口分开。

# Goal / Non-Goals

## Goal

- 定义 dead-entry 的最小识别口径
- 盘点当前最重要的动态入口类别
- 给出后续应优先补图的入口族顺序

## Non-Goals

- 不在这张单里重写 codegraph
- 不一次性解决所有动态边
- 不把动态分发拉成新的控制层

# Plan

1. 盘点当前 dead-entry 集中在哪些入口族
2. 区分：
   - 假死入口（真实会触发，但图没连上）
   - 真死入口（没有注册也没有消费）
3. 给出最小补图优先级
4. 选一组最薄入口族作为后续实现候选

# Acceptance Criteria

1. 有正式文档说明 dead-entry 不等于死代码
2. 至少列出当前仓库最主要的动态入口族
3. 给出先补哪一组动态边的顺序
4. 后续实现可以基于这张基线开薄单，而不是凭感觉修

# Notes

- 这张单只做入口基线，不直接写代码
- 优先帮助后续图质量判断，不抢主产品开发节奏
