## Context

`0028` 已确认当前系统的核心问题不是 LLM 没能力，而是框架仍在多处替 LLM 规定流程、设次数上限、按预想工具顺序裁判。当前主链已经证明 `read_file + bash` 和 schedule create/trigger/remove 能真实执行，但剩余控制逻辑仍在 `prompt.ts` 和 `tool-loop.ts` 中生效，随时可能在别的任务上继续阻断。

当前代码证据：

1. `src/agent-backend/prompt.ts`
   - `整个问题最多调用工具 3 次`
   - `第一轮必须优先产出 tool_calls`
   - `没有 tool_calls 前禁止直接回答`
2. `src/agent-backend/tool-loop.ts`
   - `preferredToolName`
   - `hasDisallowedPreferredToolMismatch`
   - `findUnexpectedToolNames`
   - 多处 `你必须调用工具 ...`
   - `HARD_CAP_TOOL_CALLS = 20`
   - `HARD_CAP_TOOL_STEPS = 64`
3. 日志证据：
   - `/Users/admin/.config/msgcode/log/msgcode.log`
   - `2026-03-08 00:00:08.643`：`MODEL_PROTOCOL_FAILED`

## Goal / Non-Goals

### Goal

1. 删除或降级 `prompt.ts` 中仍在生效的流程型硬命令。
2. 移除 `tool-loop.ts` 中的 preferred-tool / mismatch / 强制 tool_calls 裁判器。
3. 让默认 quota/hard cap 不再成为阻断主链的原因，默认口径对齐 99+ 次调用。
4. 对齐 `lmstudio.ts` 与 `tool-loop.ts` 的剩余工具暴露语义。
5. 补测试并做真实 smoke。

### Non-Goals

1. 不重构 scheduler / browser / memory。
2. 不做 prompt 分层实验。
3. 不新增 supervisor / parser / fallback / review loop。
4. 不新增新的控制层来替代旧控制层。

## Acceptance Criteria

1. `prompt.ts` 不再包含“最多 3 次”“第一轮必须 tool_calls”“没有 tool_calls 禁止回答”这类硬命令。
2. `tool-loop.ts` 中 preferred-tool / mismatch 裁判器被删除或失效，不再作为主链阻断。
3. 默认 quota/hard cap 不再轻易截断主链，默认口径符合 99+ 次调用。
4. `lmstudio.ts` / `tool-loop.ts` 的工具暴露语义一致。
5. 至少补上 prompt、no-tool、hard-cap 回归测试，并保留 read skill / bash / schedule 主链回归。
6. 完成后提交 commit，并更新 `0028` 状态。
