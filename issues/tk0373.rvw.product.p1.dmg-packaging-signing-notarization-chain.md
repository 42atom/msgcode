# 任务：DMG Packaging Signing Notarization Chain

## Context

`app bundle` 也不是最终交付。

对当前产品来说，正式外发面还需要收成：

- dmg packaging
- signing
- notarization

这一步如果不单独立 task，很容易把：

- app 宿主装配
- 分发介质打包
- Apple 交付链

混成同一层，最后又让母线膨胀。

## Goal / Non-Goals

### Goal

- 定义 dmg 链的输入只来自 app bundle
- 定义 packaging、signing、notarization 的最小顺序
- 定义 dmg 分发物的输出边界
- 定义这条链的最小验收口径

### Non-Goals

- 不设计 build artifact
- 不设计 app bundle host assembly
- 不扩自动更新、App Store、跨平台安装器
- 不写签名或公证实现代码

## Plan

- [x] 明确 dmg 链只承认 app bundle 为输入
- [x] 明确 packaging、signing、notarization 的先后顺序
- [x] 明确 dmg 分发物完成后的最小验收
- [x] 明确为什么这一步只是交付链，不是新的产品母线

## 输入面

`tk0373` 只承认一个上游输入：

- `tk0372` 产出的本地 `.app` bundle

也就是说，这一步只消费：

- 已经完成宿主装配的 Desktop App

这里不再重新读取：

- `src/**`
- `bundle-root` 的组装细节
- Electron host 源码树
- 任意开发态 `npm install` / `tsx` / 本地打包脚本链

一句话：

- dmg 链只认 `.app`，不回退到 artifact 或源码安装链

## 分发容器定位

`dmg` 在这一步只有一个职责：

- 作为 macOS 外发分发容器，承接已经成形的 `.app`

它不是：

1. 新的宿主层
2. 新的 runtime 打包层
3. 新的产品主线

所以这里的判断很硬：

1. `.app` 先独立成立
2. `dmg` 只是把这份 `.app` 收成正式对外分发物

## 最小顺序

这条链的顺序固定为 3 步：

1. packaging
   - 把 `tk0372` 产出的 `.app` 收进 `dmg` 分发容器
2. signing
   - 对正式分发物执行签名，保证外发身份与系统校验边界成立
3. notarization
   - 对已经完成签名的正式分发物走 Apple 公证链，形成可分发交付面

这里的关键点不是实现命令，而是顺序边界：

- 先有 `.app`
- 再有 `dmg`
- 再进入签名 / 公证交付链

不允许把这些动作混回：

- app host assembly
- build artifact assembly
- runtime 构建

## 签名与公证链边界

这一步只回答交付边界，不回答实现细节。

它只收口 3 个问题：

1. 签名与 notarization 属于分发链，不属于 app host assembly
2. 签名与 notarization 只作用在已经完成装配的正式分发物
3. 这条链的目标是形成可对外发布的 macOS 交付物，不是反向定义产品结构

所以这里明确不做：

1. 重定义 `.app` 内部结构
2. 回写 artifact 合同
3. 承担授权、更新、App Store 交付策略
4. 写具体签名、公证实现代码

## 最小输出面

这一步完成后，只要求形成一份可外发的 macOS 分发物。

最小输出可以收口为：

1. 一个承接 `.app` 的 `dmg`
2. 一条清楚的 signing / notarization 交付顺序
3. 一个明确的边界：`dmg` 只做分发，不再承担宿主装配

验收重点不是“桌面 app 能不能启动”。

那是 `tk0372` 的验收。

这里的验收是：

- `.app` 已经是既成输入
- `dmg` 只是正式外发容器
- signing / notarization 只是对外交付链，不新增产品职责

## 与 app bundle 的切口

`tk0372` 和 `tk0373` 的切口也必须很硬：

- `tk0372` 回答“怎么得到可启动的 `.app`”
- `tk0373` 回答“怎么把这份 `.app` 收成可外发的 `dmg` 交付链”

所以 `tk0373` 不再决定：

- 宿主壳要不要存在
- runtime 如何放进 `.app`
- UI 如何接上 bundled runtime

这些都以前一步完成的 `.app` 为准。

## 排除项

`tk0373` 明确不进入：

1. build artifact 结构重定义
2. app bundle host assembly
3. 自动更新链
4. App Store 分发
5. 跨平台安装器
6. 具体签名或 notarization 实现代码

## Acceptance Criteria

1. 能明确说明 dmg 链的输入只来自 `tk0372` 所定义的 `.app` bundle。
2. 能明确说明 `dmg` 只是分发容器，packaging、signing、notarization 是一条交付顺序，不新增产品职责。
3. 能明确说明签名 / 公证属于分发链边界，不回写 app host 或 artifact 合同。
4. 能明确排除自动更新、App Store、跨平台安装器范围。
5. 文档能说明这张单为什么必须独立存在，而不是塞回 app bundle 或母线。

## Notes

- 这张单是 `tk0370 / pl0362` 的第 3 个子产物切片。
- 它只收口正式分发链，不替代 app host，也不替代产品策略。
- 这里的 `dmg` 是容器，不是新的宿主层。
