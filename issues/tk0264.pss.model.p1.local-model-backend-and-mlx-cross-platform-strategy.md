# 任务：本地模型后端与 MLX 跨平台策略

## Context

当前系统里，“本地模型”其实混着两种东西：

1. `LM Studio / OMLX` 这类本地 HTTP 后端
2. `mlx-whisper` 这类 Apple Silicon 媒体模型

如果不先拆开，后面会继续把：

- cross-platform
- local model
- MLX
- appliance core

混成一团。

如果企业版要进入：

- 自包含交付
- App 形态
- 跨平台兼容

那这条边界必须先说清。

## Goal / Non-Goals

### Goal

- 盘点当前本地模型依赖哪些 macOS / MLX 条件
- 区分：
  - mac-only 能力
  - 可替换能力
  - 必须抽象的能力边界
- 给出跨平台本地模型路线建议

### Non-Goals

- 不立即改 chat/provider 主链
- 不直接接入新的模型平台
- 不把问题扩成完整模型控制面

## Plan

- [x] 盘点当前本地模型与视觉路径的系统依赖
- [x] 明确 MLX/mac 与跨平台路线的冲突点
- [x] 输出 appliance 建议能力矩阵

## Acceptance Criteria

1. 能说清当前哪些能力被 macOS / MLX 锁死。
2. 能区分 core appliance、local model pack、apple media pack 三类。
3. 不用“以后再看”回避跨平台边界。

## Notes

- 这张单的目标是把模型后端边界说清，不是立刻平台化 provider。
- 当前正式口径应先允许 cloud/API 作为 core appliance 基线。
- `mlx-whisper` 单列为 mac-only optional pack，不把它伪装成“本地模型通用能力”。
