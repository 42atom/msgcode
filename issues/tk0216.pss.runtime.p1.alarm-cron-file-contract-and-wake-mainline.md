## Context

- `tk0204` 的 Phase 1 已完成，当前未做部分主要是 `alarm/cron` 与 `reflection`。
- `schedule-v2` 与 wake 主链已经存在，但还没有被正式收成“heartbeat Phase 2 的精确定时唤醒面”。
- 这单的目标不是重做调度平台，而是把：
  - 文件协议
  - wake 消费
  - heartbeat 接线
  收成同一条最小主链。

## Goal / Non-Goals

- Goal:
  - 明确 `alarm/cron` 的文件协议边界
  - 让 schedule/wake 成为 heartbeat 的正式精确定时入口
  - 给 Phase 2 留下一条最小、可回归、可 real-test 的主链
- Non-Goals:
  - 不做新的 orchestrator
  - 不重写 schedule 存储
  - 不在本单处理 reflection

## Plan

- [x] 明确 `schedule-v2` 与 `alarm/cron` 的正式映射口径
- [x] 明确 workspace 侧文件真相源：
  - `schedules/*.json`
  - `.msgcode/wakeups/*`
  - `issues/`
- [x] 收最小主链：
  - schedule 命中
  - 写 wake 记录
  - wake 消费
  - heartbeat 推进任务
- [x] 补一组最小回归锁
- [x] 补一轮 real-test 或等价 smoke 证据

## Acceptance Criteria

1. 仓库内存在明确判断：`alarm/cron` 的正式入口就是 `schedule-v2 -> wake -> heartbeat`。
2. 不新增第二套精确定时控制面。
3. 至少有一条测试或 real-test 证据，证明定时命中后确实能推动 heartbeat 主链。
4. 与 `tk0204` 的边界明确：
   - `tk0204` 负责 heartbeat Phase 1 主循环
   - 本单负责 Phase 2 的精确定时唤醒

## Notes

- 这单本质上是把现有积木收成正式合同，不是新增层。
- 若现有 `schedule-v2` 足以表达，就不再创造新协议。
- 当前正式映射已经收成：
  - `schedules/*.json` = 什么时候该叫醒
  - `.msgcode/wakeups/records/*.json` = 已到点的唤醒真相
  - `heartbeat` = 真正推进 `issues/` 的执行面
- 本轮已补正式接线：
  - `commands.ts` 启动 task runtime 时，会为 `schedule-wake` 装默认 `triggerNow` hook
  - `mode=now` 不再只是留接口，而是直接走 `workspace heartbeat`
  - 这条链仍然是单一主链：`schedule-v2 -> wake record -> heartbeat`
- 本轮已补回归：
  - `test/p7-pl0210-schedule-wake-idempotent.test.ts`
  - 新增用例锁住 `mode=now` 会即时触发 `workspace heartbeat`，并推进任务文档到 `pss`
- 本轮收口判断：
  - `alarm/cron` 的最小主链已经成立
  - 后续若继续，重点应转向：
    - real-test
    - `next-heartbeat` 在正式 runtime 下的消费链
    - 更细的 runner 接线
