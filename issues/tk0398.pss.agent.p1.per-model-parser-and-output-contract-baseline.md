# 任务：Per-model parser and output contract baseline

## Context

当前不同模型已经暴露出真实差异：

- 输出长度不稳
- tool 协议形状不稳
- 某些模型更适合 JSONL/structured parser
- 某些模型只能靠更严格的 prompt/stop/max-output 收口

继续假设“所有模型都走同一 parser / 同一协议约束”会让运行行为越来越不可控。

这里要明确分两层：

- request contract：约束请求怎么发，负责输出长度、stop、reasoning effort
- parse contract：约束结果怎么读，负责 tool call 形状、完成判定、文本抽取

不能把“输出长度不受控”误归因给 parser 本身。

## Goal / Non-Goals

## Goal

- 定义一份按模型配置的 parse contract 基线
- 定义一份按模型配置的 request contract 基线
- 明确哪些模型需要独立 max-output/stop/tool-protocol/completion-signal 约束

## Non-Goals

- 不重做 provider 层
- 不引入新控制平台
- 不做模型自动探测魔法
- 不在本单里实现所有模型适配代码

## Plan

1. 盘点当前实际模型族与现有 parser：
   - `AssistantParser`
   - `CodexParser`
2. 拆出最小配置面，分 request / parse 两组：
   - request contract
     - `model_selector`
     - `tool_protocol`
     - `request_max_output_tokens`
     - `request_stop_sequences`
     - `reasoning_effort_policy`
   - parse contract
     - `parser_kind`
     - `completion_signal`
     - `tool_call_shape`
3. 只先支持显式按模型配置，不做自动猜测。
4. parser 只允许解析 provider/native/日志中已存在的结构化输出，不允许从自由文本里猜 tool call。
5. 第一刀先覆盖当前已经有真实差异的模型族。

第一刀覆盖范围固定为四族：

- `tmux:codex`
- `tmux:claude-code`
- `agent:openai-compat`
- `agent:minimax`

## Acceptance Criteria

- 明确哪些模型走哪种 parser
- 明确哪些模型需要独立 request contract
- 明确哪些模型需要独立 completion/tool-call 解析约束
- 第一刀保持显式配置，不引入自动推断
- parser 失败时直接暴露错误，不做 repair/fallback
- 不改现有业务逻辑

## Notes

- 这张单解决的是“模型层协议差异”，不是宿主层问题。
- 宿主、plugin、sidecar 不负责收口模型输出行为。
- parser 不是“从自由文本猜工具调用”的解释器；它只读已经存在的结构化输出或确定性日志格式。
- 若模型未绑定 contract，应该直接报错，而不是落回默认 parser。
- `tmux` 族与 `agent` 族必须分开看：前者多是 parse contract，后者同时涉及 request contract。
