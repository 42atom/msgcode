## Context

当前仓库已经有一批运行时零件，但它们还没有形成统一的“重资源并发冲击防护”主链：

- `src/runtime/run-store.ts`
- `src/runtime/task-supervisor.ts`
- `src/runtime/model-service-lease.ts`
- `src/tools/bus.ts` 中的 `DesktopSessionPool`
- `src/runners/tts/auto-lane.ts`

这些模块分别解决了 run 记账、显式 task、模型 idle release、desktop 单飞、自动 TTS latest-wins，但系统仍缺少一层非常薄的统一 admission 语义：

- 多个 chat 同时打本地模型时，当前更多是“慢、卡、抖”，而不是明确的 `busy`
- desktop/browser 类重资源已有局部 single-flight，但它只在各自模块内部生效，跨 workspace 时没有一致的上层语义
- 当前用户对目标口径已经收敛：不做新的 runtime control plane，只要一个最小的队列机制或明确的“服务器繁忙”返回

已有草稿：

- [AIDOCS/design/plan-260312-thin-agent-runtime.md](/Users/admin/GitProjects/msgcode/AIDOCS/design/plan-260312-thin-agent-runtime.md)

这份草稿可作为参考，但不作为本次执行真相源，因为它中段已经长出了 `policy.json / queue.jsonl / unified lease` 等更重的控制面方向，不符合当前“做薄、删层、单主链”的项目原则。

## Goal / Non-Goals

### Goal

- 为本地重资源建立统一、明确、可测试的 admission 语义
- 在多并发冲击下，让系统返回稳定的 `queued` 或 `busy`，而不是随机卡住或隐式互撞
- 复用现有 run / task / lease 主链，不新增第二套 runtime 控制面

### Non-Goals

- 本轮不实现完整的 thin runtime / agent OS
- 本轮不新增 `runtime/policy.json`、`queue.jsonl`、`leases.json`
- 本轮不把 `probe` 变成 admission 调度输入
- 本轮不统一所有资源类型；先只覆盖最危险的重资源
- 本轮不处理 tmux / 轻量 chat 的全局编排

## Plan

- [x] 冻结并发冲击防护的术语和边界：`run / heavy resource / queued / busy / latest-wins`
- [x] 设计并记录一个薄的 `heavy admission` 主链，只做 `运行 / 排队 / 拒绝繁忙`
- [x] 先覆盖 `local-model` 与 `desktop` 两类重资源，明确它们的容量、队列和 overflow 行为
- [x] 明确不同来源的默认策略：交互请求优先 `busy`，可恢复任务才进入有界队列
- [x] 明确现有 `AutoTtsLane` 等 latest-wins 场景不并入通用队列，只保留“过时即丢弃”
- [x] 定义最小观测与测试口径，保证后续实现可回归

## Acceptance Criteria

1. 有一份正式 plan 文档明确：本次只做“重资源 admission MVP”，不做新的 runtime 控制面。
2. 计划明确区分：
   - 什么请求应该直接 `busy`
   - 什么请求可以进入有界队列
   - 什么 latest-wins 场景应直接丢弃旧任务
3. 计划明确 Phase 1 只覆盖 `local-model` 与 `desktop`，并解释为什么不先做全资源统一。
4. 计划明确不新增 `policy.json / queue.jsonl / leases.json` 这类全局状态文件。
5. 计划给出后续实现所需的最小测试与观测口径。

## Notes

- 现有代码证据：
  - `RunStore` 只记录最小 lifecycle，不承担调度控制面
  - `TaskSupervisor` 有 `maxConcurrentTasks` 配置，但当前不是系统级重资源 admission
  - `ModelServiceLeaseManager` 负责 idle release，不负责 admission / backpressure
  - `DesktopSessionPool` 只做本模块内部单飞，错误语义仍是局部的 `Session busy`
  - `AutoTtsLane` 已经证明：对于“过时即无意义”的场景，`latest-wins` 比排队更合理
- 本次工作产物以 issue + plan 为主，不直接动运行时代码。
