## Context

- 当前仓库已经有 `subagent` 主链，但子代理的“能力”和“人格”还没有被正式拆开。
- 用户最初想法是给不同 agent 装不同技能包；现在已经收口为更薄的方案：skills 共池安装，人格通过文档装配。
- 这样可以避免复制安装面，也能让 `codex / claude-code / local` 在同一套能力边界里按任务切换人格。

## Goal / Non-Goals

- Goal:
  - 明确公共技能池是默认能力底座
  - 为主脑与子代理建立 persona 文档协议
  - 让子代理启动时支持 `client + persona + task card` 装配
- Non-Goals:
  - 不为每种 agent 复制一套技能目录
  - 不新增新的 agent 控制面或多层 supervisor
  - 不在本单直接引入外部最佳实践平台

## Plan

- [x] 盘点当前 runtime skills / optional skills，定义公共技能池边界
- [x] 新增 persona 文档协议，明确放置路径、命名与字段
- [x] 规定 persona 只表达角色、方法、质量门槛、禁区与回传格式，不表达安装面
- [x] 让主脑与 subagent 启动口径支持显式 persona 选择
- [x] README / prompt / subagent skill 对齐“skills 共池，persona 装配”的正式口径

## Acceptance Criteria

1. 仓库中存在正式 persona 文档协议，而不是零散 prompt 片段。
2. persona 与 skill 的边界明确：
   - skill = 会什么
   - persona = 怎么做
3. `codex / claude-code / local` 不再依赖各自独立技能包，默认复用同一公共技能池。
4. 子代理主链能表达至少这三要素：
   - client
   - persona
   - task card
5. 文档明确写出：后续可以参考外部优秀实践，但当前先以仓库内最小主链落地。

## Notes

- 这单解决的是“谁来做、按什么人格做”。
- 人格文件优先采用说明书方式，不优先做代码控制层。
- 若后续参考外部实践，应在这条主链上迭代，而不是另起一套 agent 框架。
- 当前正式 persona 路径已冻结为：`docs/protocol/personas/<persona-id>.md`
- 当前正式 persona 已收口为：
  - `frontend-builder`
  - `code-reviewer`
  - `api-builder`
- `subagent runtime` 已正式接上：
  - `client`
  - `persona`
  - `task card`
- `subagent` skill 已与 runtime 口径对齐，不再引用不存在的 persona 路径

## Reflection

- 这轮真正的问题不是“有没有 persona 概念”，而是合同漂移：
  - heartbeat 已经在传 `persona + taskCard`
  - subagent runtime 却没正式字段
  - skill 还指向一个仓库里并不存在的 `docs/protocol/personas/`
- 现在已按最薄方式收口：
  - 建唯一 persona 路径
  - runtime 真读取 persona 文档
  - task card 和 persona 一起注入委派 prompt
  - 不新增控制层，不做第二套 registry
