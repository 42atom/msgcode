# 任务：Appliance 预安装软件 doctor 合同

## Context

现在已经有一份独立的“预安装软件必须清单”参考：

- [rf0002.rvw.product.appliance-required-preinstall-software.md](/Users/admin/GitProjects/msgcode/docs/plan/rf0002.rvw.product.appliance-required-preinstall-software.md)

但它还只是参考，不是 install / doctor 的正式执行合同。

这会继续留下一个灰区：

- 文档说必须预装
- 现场却没有前置检查
- 结果运行时才暴露成 shell / browser / desktop 等物理边界错误

## Goal / Non-Goals

### Goal

- 让 appliance install / doctor 有单一的预安装软件检查口径
- 让 Core Appliance 与 Optional Pack 的检查边界分开
- 让缺失项在 install / doctor 阶段 fail-closed

### Non-Goals

- 不直接实现 DMG / PKG / App 打包
- 不在这张单里决定哪些依赖改为随包
- 不把 Optional Pack 的完整安装器一起做掉

## Scope

这张单只回答三件事：

1. install / doctor 如何引用 `rf0002`
2. Core Appliance 基础项如何检查
3. Optional Pack 触发式依赖如何声明和检查

## Acceptance Criteria

1. 预安装软件清单有单一引用面，不再散落在多个脚本和文档里。
2. Core Appliance 缺失托管 Bash / `zstd` 这类基础项时，install / doctor 能前置报错。
3. Optional Pack 的依赖检查按“启用对应能力包才检查”执行，不污染 Core Appliance 默认路径。
4. 运行时不再继续把这类缺失改写成“AI 无法访问本地设备”之类的伪能力问题。

## Plan

- [x] 明确 install / doctor 对 `rf0002` 的单一引用点
- [x] 明确 Core Appliance 基础项的前置检查口径
- [x] 第一刀只落 bash / `zstd` 两项前置检查
- [x] 明确缺失项的失败文案与 fail-closed 边界

## Notes

- `rf0002` 是参考真相，不是实现本身。
- 这张单的目标是把参考收成 install / doctor 合同，不另造第二套依赖系统。
- 当前第一刀只检查：
  - 托管 Bash
  - `zstd`
- 当前实现面：
  - `install-appliance.sh` 在复制 runtime 前先跑 preinstall 检查
  - `doctor-appliance.sh` 在验安装根前先跑同一份 preinstall 检查
  - 两边都通过 `lib-appliance-preinstall.sh` 共享同一口径
