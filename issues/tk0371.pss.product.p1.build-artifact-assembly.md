# 任务：Build Artifact Assembly

## Context

当前母线已经明确：

- Electron 只是宿主壳
- `msgcode` runtime 仍是主脑
- 正式交付不能回退到 `brew` / `npm install`

但这还不等于 build artifact 已经被单独说清。

如果不先把 artifact 收成独立输入输出合同，后面的 app bundle 和 dmg 很容易把：

- runtime 运行物
- bootstrap 运行物
- host/UI surfaces

揉成一坨，最后又回到“源码态安装链”。

## Goal / Non-Goals

### Goal

- 定义 build artifact 的唯一输入面
- 定义 build artifact 的最小目录/产物边界
- 定义 artifact 与源码态、bootstrap、app bundle 的切口
- 定义 artifact 完成后的最小验收口径

### Non-Goals

- 不设计 app bundle 宿主结构
- 不设计 dmg 打包、签名、公证
- 不扩授权、自动更新、跨平台
- 不写任何打包实现代码

## Plan

- [x] 盘点 artifact 只承认哪些现有输入
- [x] 收口 artifact 最小结构与排除项
- [x] 明确 artifact 与 `tk0265` bootstrap 产物的关系
- [x] 明确为什么 app bundle 必须消费 artifact，而不是直接消费源码树

## 输入面

这一步只承认两类现有输入，不新增第三类中间层：

1. runtime 交付面
   - 已编译的 runtime 入口与依赖树
   - 现有 launcher 入口 `bin/msgcode`
   - 依据 `src/runtime/runtime-entry.ts`，正式入口优先消费 compiled/runtime，而不是源码 `src/*.ts`
2. appliance bootstrap 交付面
   - 只承认 `tk0265` 已定义的 appliance 脚本与 manifest 合同
   - 最小范围是：
     - `bootstrap/install-appliance.sh`
     - `bootstrap/doctor-appliance.sh`
     - `bootstrap/first-run-init.sh`
     - `bootstrap/upgrade-appliance.sh`
     - `bootstrap/rollback-appliance.sh`
     - `bootstrap/lib-appliance-preinstall.sh`

这里先不把 app host/UI 源码树并进来。

原因很简单：

- 现在仓库里还没有现役 Electron host 源码主线
- `tk0372` 的职责才是把宿主壳如何消费 artifact 说清楚
- `tk0371` 只负责收干净“可被宿主消费的运行物”

## 最小产物结构

build artifact 完成后，只需要形成一份可复用的 `bundle-root`：

- `bundle-root/runtime/`
  - 编译后的 `msgcode` runtime
  - runtime 运行所需的随包依赖
- `bundle-root/bin/msgcode`
  - 对外稳定 launcher
- `bundle-root/appliance.manifest`
  - 安装、doctor、upgrade 共同消费的最小真相
- `bundle-root/bootstrap/`
  - 只保留 appliance 主线需要的脚本

这份产物的目标只有一个：

- 让 `install-appliance.sh` 与后续 app bundle 宿主都消费同一份 `bundle-root`

## 排除项

artifact assembly 明确不收这些内容：

1. 源码树
   - `src/**`
   - `test/**`
   - `features/**`
2. 开发态安装链
   - `npm install`
   - `npx tsx`
   - `bootstrap/bootstrap.sh`
   - `bootstrap/bootstrap-agent-pack.sh`
   - `bootstrap/Brewfile`
   - `bootstrap/Brewfile.agent`
3. 宿主壳与分发职责
   - Electron host 结构
   - app bundle 外壳
   - dmg、签名、公证
4. 仓库运行噪声
   - `artifacts/desktop/**`
   - 本地日志、临时截图、调试产物
   - docs 与 issue 本身

## 与源码树切口

这一步的硬切口只有一句：

- build artifact 可以从源码树“生成”，但 app bundle 不能再直接“消费”源码树

具体边界：

1. artifact assembly 之前
   - 允许从仓库里读取编译产物与 appliance 脚本真相
2. artifact assembly 完成后
   - 下游只认 `bundle-root`
   - `tk0372` 只能消费 artifact，不再回读 `src/`、`bootstrap/Brewfile`、`package.json` 或开发者安装链

这条切口是为了避免下游重新滑回：

- 源码态启动
- `npm install`
- `brew` 补依赖
- 宿主壳直接替 runtime 做决定

## 与 `tk0265` 的关系

`tk0265` 已经定义了 appliance 安装主线只认：

- `bundle-root`
- `install-root`
- workspace

所以 `tk0371` 的职责不是重写 bootstrap，而是保证上游真的能稳定产出这份 `bundle-root`。

也就是说：

- `tk0265` 定义“怎么安装”
- `tk0371` 定义“装什么进去”

## 验收补充

完成 `tk0371` 时，至少要能回答：

1. artifact 只从哪些已存在 surface 收输入。
2. 哪些内容必须进入 `bundle-root`。
3. 哪些内容必须明确排除，不能混进 artifact。
4. 为什么 `tk0372` 必须消费 artifact，而不是重新读取源码树。

## Acceptance Criteria

1. 能明确列出 build artifact 只从哪些现有 surfaces 收输入。
2. 能明确说明 artifact 产出后，app bundle 只能消费 artifact，不直接回退到源码安装链。
3. 文档明确声明：artifact 不承担 app 宿主职责，也不承担 dmg 分发职责。
4. 文档不扩到授权、更新、跨平台。
