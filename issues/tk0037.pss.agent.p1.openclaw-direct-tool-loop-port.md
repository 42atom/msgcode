## Context

### 问题证据

最新真实日志（2026-03-08 00:00:00）：
- 用户发：`定一个每分钟发送的任务 发：live cron`
- 随后：
  - `agent-first tool-loop started`
  - `toolCallCount=0`
  - `route=no-tool`
  - 最终 `MODEL_PROTOCOL_FAILED`

**结论**：LLM 没被支持继续尝试，而是被 `allowNoTool`/收尾链直接判死

### OpenClaw 对照

OpenClaw 的做法：
- 默认直接给完整工具集
- 只按配置/权限过滤
- 不按当前消息内容做工具裁判
- 不走 `route=no-tool -> MODEL_PROTOCOL_FAILED`

### 架构铁律

1. 系统只给工具、给 skill、给路径
2. 不得基于消息内容替 LLM 判断"该不该调用工具"
3. 不得再保留 `allowNoTool` 作为默认收尾出口
4. 不得让 `toolCallCount=0` 成为执行型请求的终点
5. LLM 没调出工具时，继续循环，不直接判死

## Goal / Non-Goals

### Goal
- 去掉 `allowNoTool` 这条收尾主链
- 去掉消息级裁判逻辑
- 对齐 OpenClaw 的完整工具面 + 配置/权限过滤
- 统一 tool-loop.ts 与 lmstudio.ts 的工具暴露语义

### Non-Goals
- 不新增任何意图识别函数
- 不新增 parser 去解释 `[TOOL_CALL]` 文本
- 不重构 scheduler/browser/memory 功能本身
- 不做 prompt 分层实验
- 不碰 quota/hard cap
