## Context

- 用户希望主脑像当前 Codex 协作方式一样，能创建任务、拆任务、派给不同执行臂，并持续巡检直到交付。
- 用户不关心中间 tmux 噪音、调度细节或子代理内部过程；用户只看任务文档状态和最终交付是否解决问题。
- 当前仓库已经有 `/task`、`TaskSupervisor`、`subagent` 主链，但还没有把“派单状态”正式落到 `issues/` 文档协议里。

## Goal / Non-Goals

- Goal:
  - 把任务文档变成主脑派单与交付监督的正式控制面
  - 让父任务、子任务、subagent 执行记录互相绑定
  - 让用户通过任务文档和任务看板获取状态，而不是看运行时过程
- Non-Goals:
  - 不新建一套独立任务平台或 orchestration UI
  - 不让用户承担子任务级验收噪音
  - 不让运行时状态覆盖文件名状态真相源

## Plan

- [ ] 定义父任务 / 子任务文档关系与最小字段协议
- [ ] 规定派给 `codex / claude-code / local` 的子任务如何在文档中表达 `assignee / reviewer / links / artifacts`
- [ ] 为 `/task` 与 `subagent` 补“派单记录绑定”口径
- [ ] 让主脑 heartbeat / 巡检逻辑优先回写子任务文档与父任务摘要
- [ ] 对齐任务看板视图，让用户能直接看“未完成 / 已完成 / 最终交付”

## Acceptance Criteria

1. 用户目标至少对应一个父任务文档，状态真相源仍然只看文件名状态槽位。
2. 每个被委派出去的执行单至少对应一个子任务文档，而不是只存在于 runtime 内存或 tmux。
3. 子任务文档能明确表达：
   - `assignee: codex|claude-code|local|...`
   - 验收标准
   - 预期产物路径
   - 关联父任务与 subagent taskId
4. 主脑验收子任务后，先改子任务状态；必要子任务完成后，再推进父任务状态。
5. 用户默认只需要看：
   - 父任务状态
   - 关键交付物链接
   - 最终验收结果
6. 任务看板 HTML 不需要新协议即可展示这套父子任务流转。

## Notes

- 这单解决的是“做了什么、做到哪、用户怎么看”。
- 父任务默认面向用户；子任务默认面向主脑内部派单与验收。
- 推荐口径：
  - 子任务 `reviewer` 默认 `agent`
  - 父任务 `reviewer` 默认 `user`
- 后续若要参考外部优秀实践，也应先保护这套 doc-first 真相源。
- 当前已落地的最小接线：
  - heartbeat 创建 dispatch 时，会从子任务正文 `## Parent Task` 解析真实 `parentTaskId`
  - heartbeat 调度 runnable 子任务时，会尊重子任务 `implicit.waiting_for`
  - 子任务完成后，若父任务 `## Child Tasks` 或 `implicit.waiting_for` 中列出的子任务都已进入 `pss/dne`，父任务会自动推进到 `rvw`
  - 子任务处于 running 时，heartbeat 会消费正文 `## Follow-up / ## Supervisor Message`，对同一条补充说明只继续发送一次
  - 子代理委派 prompt 不再只含裸 goal；现在会补父任务摘要与最近失败证据摘要
- 当前仍未做的部分：
  - 更复杂的多子任务依赖、关键/非关键子任务分层、父任务摘要回写
