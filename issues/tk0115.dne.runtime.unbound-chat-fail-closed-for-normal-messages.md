## Context

`default workspace` 已经不再自动持久化为真实 route，但普通消息主链仍会通过 `routeByChatId()` 临时落到 `default`。这意味着系统仍在替未绑定 chat 偷偷做路由决定，用户与模型都以为这是一条真实工作链路。

## Goal / Non-Goals

### Goal

- 未绑定 chat 的普通消息不再自动落到 `default workspace`
- 未绑定时给出节流的 `/bind` 提示，避免消息黑洞
- 保留命令链路对 `default workspace` 的显式 fallback 能力

### Non-Goals

- 本轮不删除 `resolveCommandRoute()` 的 default fallback
- 本轮不改 `/bind`、`/where` 等显式控制面体验
- 本轮不改安全/预算边界

## Plan

- [x] 新建 `0115` issue 与对应 plan，冻结边界
- [x] 为 `routeByChatId()` 增加显式 `allowDefaultFallback` 选项
- [x] 让 `listener` 主聊天链路和 `/status` 快车道在未绑定时 fail-closed
- [x] 补充“未绑定普通消息提示绑定”和“主聊天链路不再默认落 route”的测试
- [x] 更新 `docs/CHANGELOG.md`
- [x] 跑定向测试、类型检查和 docs:check

## Acceptance Criteria

- 未绑定 chat 的普通消息不会再自动使用 `default workspace`
- 未绑定普通消息会收到一次节流的 `/bind` 提示
- `/bind`、`/where`、命令 resolver 仍然可在未绑定时工作
