# TK0383：Read File Pagination And Binary Handle Contract

## Context

当前 `msgcode` 的 `read_file` 还有三个硬问题：

- 大文本超过阈值后仍是 `head + tail` 拼盘，不是连续正文
- `tool-loop` 回灌模型前又再压一轮 preview，信息继续损失
- 二进制和超大非文本还没有正式的 handle/blob 语义

这导致系统虽然防住了上下文爆炸，但没有把“下一步怎么继续读”收成正式合同。

最新老师口径已经很清楚：

- `pi-mono`：分页读取是第一公民，明确 `offset/limit` 与继续提示
- `openclaw`：在分页合同之上，再做上下文预算内的连续多页聚合

这次只修主链，不补平台。

## Goal / Non-Goals

### Goal

- 定义小文本的全文返回边界
- 定义大文本的连续分页合同
- 定义二进制/超大非文本的 handle/blob 语义
- 明确 `tool-loop` 对已经分页的 `read_file` 结果不再二次压扁

### Non-Goals

- 不扩到 `grep / log / exec` 家族
- 不做 `openclaw` 式自适应多页聚合
- 不做新的 preview controller
- 不为了这次任务新增中间控制层

## Core Contract

### 文本文件

- 小文本：直接全文返回
- 大文本：返回一段连续正文，不再使用 `head + tail`
- 分页结果至少要有：
  - `offset`
  - `limit`
  - `has_more`
  - `next_offset`
  - `total_bytes` 或 `total_lines`

### 二进制 / 超大非文本

- 不直接回正文
- 返回元信息与后续可消费引用
- 第一刀可用 `blob`/`handle` 风格字段表达，但不要求引入独立 blob 平台

### tool-loop

- 对已经分页且仍在预算内的 `read_file` 结果，不再二次压成极短 preview
- `tool-loop` 继续保留通用预算边界，但不再破坏 `read_file` 的分页正文语义

## Plan

1. 先确认现有 `read_file` 输入输出合同与 `tool-loop` 二次裁剪落点
2. 新增实现切片，只改 `read_file` 与 `tool-loop` 的最小必要部分
3. 第一刀先做文本分页与二进制 handle/blob 语义
4. 等这刀稳定后，再决定是否把同样合同推广到 `grep/log/exec`

## Acceptance Criteria

1. 大文本 `read_file` 改为连续分页，不再返回 `head + tail`
2. 结果明确包含继续读取指针
3. 二进制/超大非文本不再走正文读取，而是走元信息/handle/blob 语义
4. `tool-loop` 对 `read_file` 不再做破坏分页语义的二次压扁
5. 这张单明确不扩到其它基础工具
