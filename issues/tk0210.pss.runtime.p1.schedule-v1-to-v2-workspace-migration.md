## Context

- `tk0204` 的时间主链已经落到了 `schedule -> wake job -> wake record -> heartbeat consume`。
- 但老 workspace 里的 `schedules/*.json` 仍可能是旧 `v1` 格式。
- 现有新主链只认 `version = 2` 的 schedule；旧文件不会自动进入新 wake 主链。
- 我们不想现在做全仓一键重写，也不想先批量铺 `HEARTBEAT.md / wakeups / reflection` 目录。

## Goal / Non-Goals

- Goal:
  - 定义 `schedule v1 -> v2` 的最小迁移规则
  - 提供一条安全、可回滚、只改 `schedules/*.json` 的迁移路径
  - 让目标 workspace 迁移后可直接被 `src/runtime/schedule-wake.ts` 读取
- Non-Goals:
  - 不预创建 `HEARTBEAT.md`
  - 不预创建 `wakeups/jobs/*.json` 或 `wakeups/records/*.json`
  - 不在本单切 `tk0203`
  - 不做全仓批量自动迁移

## Plan

- [x] 盘清旧 `v1` schedule 字段与新 `v2` 字段映射
- [x] 定义不可丢失字段：
  - `scheduleId`
  - 时间语义
  - 用户原始 message / hint
  - enable/disable 状态
- [x] 定义 `v1 -> v2` 最小转换规则：
  - `cron/tz/message` 如何映射到 `schedule + wake`
  - `delivery` 缺失时如何降级
  - `wake.mode` 默认值
  - `latePolicy` 默认值
- [x] 定义回滚规则：
  - 迁移前备份
  - 单 workspace 可逆恢复
- [x] 提供最小迁移脚本或命令入口
- [x] 补测试：
  - 旧 `v1` 文件可转 `v2`
  - 转换后能被 `schedule-wake` 主链读取
  - 不合法旧文件不会写坏现有 schedule

## Acceptance Criteria

1. 存在一份明确的 `v1 -> v2` 字段映射规则。
2. 迁移只改 `<workspace>/.msgcode/schedules/*.json`，不批量改其他运行时文件。
3. 迁移后的 schedule 能被新 `schedule-wake` 主链读取。
4. 保留用户原始 `scheduleId` 与核心时间意图，不擅自改名。
5. 存在最小回滚方式，避免迁移失败后 workspace 失效。
6. 该迁移单不抢 `tk0203` 当前主线，只作为后续接入新时间主链的配套能力。

## Notes

- 这单是配套迁移单，不是当前主线单。
- 先让新主链稳定，再迁旧 workspace，比先全量迁再补主链更稳。
- 默认态度是：
  - 只迁需要接新主线的 workspace
  - 不做无差别全仓升级
