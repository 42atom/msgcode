## Context

- 当前仓库已经有 `TaskSupervisor`、`subagent`、`heartbeat`、`issues/` 文档协议，但真正的工作连续性还没有被钉成正式地基。
- 用户已经明确提出：会话连续性要做，但必须建立在工作连续性之上；先保证“还欠什么、做到哪、挂了从哪续”，再谈聊天自然和 session continuity。
- 现有 `openclaw` 对比也进一步说明：session continuity 可以很强，但如果 work continuity 不硬，系统重启后仍然容易只剩“聊过什么”，而不是“还要做什么”。

## Goal / Non-Goals

- Goal:
  - 明确 msgcode 的 `truth continuity` 与 `work continuity` 正式地基
  - 为父任务 / 子任务 / subagent / checkpoint / restart 续跑建立统一恢复协议
  - 让 session continuity 未来只能建立在这层地基之上
- Non-Goals:
  - 不先实现聊天历史压缩、记忆注入或人格连续性优化
  - 不引入第二任务状态系统
  - 不用隐藏数据库覆盖 `issues/` 与 workspace 文件真相源

## Plan

- [ ] 定义 `truth continuity` 的最小真相源集合：
  - `issues/*.md`
  - `<workspace>/.msgcode/dispatch/*.json`
  - artifact 路径
  - verify / log / run 证据
- [ ] 明确 `work continuity` 的最小恢复协议：
  - 显式恢复指针（主锚点对齐现有 `TaskRecord.checkpoint.nextAction`）
  - checkpoint
  - blocked / resume
  - restart 后继续点
  - subagent / parent task 绑定
- [ ] 定义恢复指针数据结构：
  - 主存放位置：runtime task record 的 `checkpoint`
  - 委派任务镜像：`<workspace>/.msgcode/dispatch/*.json` 内最小 `checkpoint` 摘要
  - 任务文档不承载易变运行指针，只承载身份、状态、验收与链接
  - 更新时机：每次 durable checkpoint 写盘，以及 blocked / resume / completed / failed 边界
- [ ] 定义 `work capsule` 最小结构：
  - `taskId`
  - `phase`
  - `checkpoint`
  - `activeDispatch`
  - `nextAction`
  - 可选 `vitalsSnapshot`
- [ ] 规定父任务、子任务、dispatch record 各自承载什么，不允许重复承载状态
- [ ] 定义工程约束：
  - `single-writer`
  - `atomic write`
  - `drift detection`
- [ ] 让 heartbeat / task supervisor 恢复逻辑优先读这套真相源，而不是依赖会话上下文
- [ ] 为后续 session continuity 写清楚依赖关系：session 只能消费这层地基，不能反客为主
- [ ] 输出一份结构化 work recovery protocol，供 `tk0206 / tk0207` 直接消费

## Acceptance Criteria

1. 仓库内存在一份明确口径，说明哪些文件是 `truth continuity` 真相源。
2. 仓库内存在一份明确口径，说明哪些机制属于 `work continuity` 恢复协议，并写清恢复指针的存放位置、字段形状与更新时机。
3. dispatch record、task document、subagent record 各自边界清楚，不出现重复状态真相源。
4. 任一 session 丢失或重启后，系统仍能仅依赖文件真相源判断：
   - 还欠什么
   - 做到哪
   - 下一步从哪续
   - 恢复后首个执行 action 是什么
5. 本单必须输出一份结构化 `work recovery protocol`，至少定义：
   - `checkpoint` 最小字段
   - dispatch record 最小字段
   - `work capsule` 最小字段
   - task / dispatch / subagent record 的职责边界
6. 工程约束被明确写死：
   - 同一 workspace truth source 一次只允许一个写入者
   - runtime record 必须原子写入
   - 恢复前必须做 drift 检测
7. 后续 `tk0206 / tk0207` 设计必须以本单为前置，而不是平行另起。

## Notes

- 这单是地基单。
- 不是“让 AI 更会聊天”，而是“让 AI 不会丢活”。
- 恢复协议必须有显式恢复指针，不能要求主脑靠翻 transcript 或日志去猜“刚才做到哪了”。
- 恢复指针默认对齐现有 runtime 的 `TaskCheckpoint`：
  - `currentPhase`
  - `summary`
  - `nextAction`
  - `updatedAt`
- 最小收口优先复用现有 runtime 落点：
  - `TaskRecord.checkpoint`
  - `<workspace>/.msgcode/dispatch/*.json`
  - `<workspace>/.msgcode/subagents/*.json`
- `single-writer` 的最小定义：
  - 同一 workspace 真相文件只能有一个活动写入者
  - 如果当前部署仍是单 session，这条天然成立；一旦进入多 session，就必须显式持有写入权
- `drift detection` 的最小作用：
  - 检测 task document、dispatch、subagent、runtime checkpoint 是否互相打架
  - 一旦打架，不假装恢复成功
- 正确顺序必须固定：
  - `truth continuity`
  - `work continuity`
  - `process continuity`
  - `session continuity`
  - `persona continuity`
