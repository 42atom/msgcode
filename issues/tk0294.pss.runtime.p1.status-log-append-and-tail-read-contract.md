# 任务：status.log 追加与 tail 读取合同

## Context

`0292` 已经把 status.log 收成：

- 一工作区一本共享小本本
- append-only
- 只记已确认决定和状态变化

但如果没有实现前合同，后面会马上掉进两个坑：

1. 模型直接把散文写进文件
2. UI 为了展示去扫全文件或自己做状态聚合

## Goal / Non-Goals

### Goal

- 明确 status.log 的最小 append 合同
- 明确单行格式化和 tail 读取规则
- 明确 tail 读取规则
- 明确 UI 只读最近几条，不把它当监控流

### Non-Goals

- 不实现自动 hook
- 不实现定时写入
- 不实现实时监控

## Plan

- [ ] 固定 append 输入字段
- [ ] 固定单行落盘格式与字符清洗规则
- [ ] 固定 UI tail 读取与展示边界

## Acceptance Criteria

1. 写入合同不是“模型直接 echo 文件”。
2. 读取合同不是“UI 扫全文件并自己总结”。
3. 第一版明确只做 append + tail，不做自动协调器。

## Notes

- 这张单是 `0292` 的实施前切片，不推翻 `status.log` 的基线定义。
- 第一版只关心 append 和 tail，不关心 hook、定时器、实时监控，也不替日志做去重判断。
- 仓库现有 append-only 模式已经对照，后续实现应优先复用现有运行时习惯。
- 运行时若没有稳定 append 合同，UI 就不应直接接这份文件。
