## Context

本轮已经拿到了一个可复现、可观察、非模拟的真实飞书验证闭环：

1. 真实用户在飞书群发送消息
2. `msgcode` 通过 Feishu WS 收到入站事件并写入 workspace
3. bot 通过真实飞书出站接口回消息
4. 通过最近消息查询与本地日志交叉验证结果

这套方法比单元测试和纯 mock 更接近生产，也比直接操控 Electron 客户端更薄。它已经满足两个目的：

- 大重构后的真实冒烟测试
- 大模型能力测试的真实通道基座

因此需要把这套方法冻结成仓库级真相源，后续优先沿用，而不是每次重新发明验证方式。

## Goal / Non-Goals

- Goal: 固定 `Feishu live verification loop` 作为默认真实测试方法
- Goal: 明确这套方法的最小闭环证据与执行顺序
- Goal: 明确它既服务于 runtime smoke，也服务于大模型能力测试
- Non-Goals: 本轮不新增测试控制面
- Non-Goals: 本轮不做 Feishu Electron 桌面自动化
- Non-Goals: 本轮不把 API-only 发送伪装成完整入站验证

## Acceptance Criteria

1. 仓库中有正式 issue + plan 说明什么叫 `Feishu live verification loop`。
2. 文档明确：完整闭环至少包含真实入站、workspace 落盘、真实出站三段证据。
3. 文档明确：优先 Feishu Web/真实用户消息，不优先 Electron UI 自动化。
4. 文档明确：这套方法可以复用于模型能力测试，不只是 transport smoke。
5. 后续讨论真实测试方案时，有统一可引用的真相源路径。
