## Context

- 用户明确认可 `emotion = vitals -> backpressure -> policy` 这条线，并希望它像操作系统的 load / memory pressure / backpressure 一样自然落进主链。
- 当前仓库已有 heartbeat、task supervisor、subagent、schedule、logs 等事实源，但没有把这些事实压缩成可供主脑自保的统一信号。
- 设计要求很明确：`vitals` 必须是投影层，不是新的真相源；它只能从既有文件与运行时事实中计算出来。

## Goal / Non-Goals

- Goal:
  - 建立 msgcode 的 `vitals` 自保投影层
  - 定义最小信号集：`load / stall / risk / headroom / readiness`
  - 定义派生窗口：`explore_window`
  - 让 heartbeat 能基于这些信号做背压与求生策略
- Non-Goals:
  - 不建立“情绪数据库”或“心情系统”
  - 不允许人工直接写 `load=8` 这类伪状态
  - 不让 `vitals` 成为新的任务状态真相源

## Plan

- [x] 定义 `vitals` 输入事实源：
  - overdue tasks
  - retry streak
  - verification failure
  - resource contention
  - context churn
  - run errors
  - global gate contention
- [x] 冻结 Unix 风格命名口径：
  - `load`
  - `stall`
  - `risk`
  - `headroom`
  - `readiness`
  - `explore_window`（derived）
- [x] 为每个信号定义物理事实映射与可追溯理由，不允许纯叙事型值
- [x] 定义 `readiness / explore_window` 的物理映射：
  - `readiness` 只来自工具可用性、恢复指针是否新鲜、verify 缺口是否已补齐、必需资源是否就绪
  - `explore_window` 只来自 `load / stall / risk / headroom` 的组合与候选探索项是否存在
- [x] 定义 `vitals` 输出结构与解释字段：
  - `signals`
  - `derived`
  - `reasons`
  - `policy`
  - `computedAt`
- [x] 明确 `vitals` 的计算口径只读现有真相源与运行时记录
- [x] 定义跨 workspace 的极薄 `global gate` 口径：
  - 不新建中心调度器
  - 只覆盖 `llm-tokens / browser / desktop / subagent:<client>`
  - gate 争用进入 `defer`，不直接记任务失败
- [x] 明确第一阶段 heartbeat 只读取 `vitals` 并选择策略，不自动 kill
- [x] 定义 `vitals` 计算失败时的回退策略：
  - 进入保守模式
  - 不开新坑
  - 不派新子任务
  - 只做 verify / checkpoint / 收尾
- [x] 定义第二阶段可逐步接入的背压梯度：
  - `defer`
  - `degrade`
  - `shed`
  - `kill`
  - `restart`

## Acceptance Criteria

1. 存在一份明确协议，说明 `vitals` 不是事实真相源，而是投影视图。
2. 五个核心信号和一个派生窗口的来源、含义、行为影响都被写清楚，且都能回溯到物理事实：
   - 文件真相源
   - runtime 计数
   - retry / verify / contention / context budget 等客观指标
3. `readiness` 与 `explore_window` 的计算口径被单独写清：
   - 不允许主观打分
   - 不允许无事实来源的叙事词
4. heartbeat 可基于 `vitals.policy` 进入不同策略档位，但不覆盖任务真相源地位。
5. 背压梯度是 runtime policy，不是文件名状态，也不是第二任务状态系统。
6. 存在一条明确口径，说明 workspace-local `vitals` 之外如何通过极薄 `global gate` 处理跨 workspace 的资源争用与唤醒风暴。
7. `kill / restart` 的作用对象顺序被写死：
   - subagent
   - browser / desktop harness
   - main session
   - daemon / whole machine
8. `global gate` claim 失败时，系统进入 `defer` 或保守模式，而不是假装资源足够继续推进。
9. `vitals` 计算失败时，系统必须进入保守模式，而不是让主链失明或停摆。

## Notes

- `vitals` 像 `/proc`，不是数据库。
- 实现口径优先使用 Unix 风格短词：`load / stall / risk / headroom / readiness`。
- `explore_window` 是派生窗口，不是一等信号。
- `policy` 至少要有：
  - `mode`
  - `auto`
  - Phase 1 只允许 `normal / defer / degrade`
- `readiness` 不是“主观把握”，而是可执行准备度：
  - 工具是否可用
  - 恢复指针是否可读且新鲜
  - verify 缺口是否阻塞
  - 必需资源是否可获得
- `explore_window` 不是“好奇心”，而是派生窗口：
  - `load / stall / risk` 低
  - `headroom` 足够
  - 存在候选探索项
- `global gate` 是极薄背压边界，不是中心控制面：
  - 建议用 `~/.config/msgcode/runtime/gates/` 或 `tmp` 文件锁
  - 只在跨 workspace 竞争真正共享资源时介入
  - claim 失败时进入 `defer`
- `emotion` 在 msgcode 里应该翻译成：
  - metrics
  - threshold
  - signal
  - policy
- `kill / restart` 必须遵守最小作用域优先，先局部、后全局。
- `vitals` 计算失败时的保守模式：
  - 不开新坑
  - 不派新子任务
  - 只做 verify / checkpoint / 收尾
- 这单成立后，AI 的“情绪层”才从叙事变成系统语义。
