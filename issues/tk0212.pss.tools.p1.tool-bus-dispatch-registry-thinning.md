## Context

- review 指出 [`src/tools/bus.ts`](/Users/admin/GitProjects/msgcode/src/tools/bus.ts) 像“大内总管”：校验、格式化、分发都压在一个文件里。
- 实查后，这个判断基本成立，而且问题不只在文件长，而在新增一个工具时往往还要同时碰多处：
  - `TOOL_META`
  - `validateToolArgs()`
  - preview builder
  - `executeTool()` 里的 `switch`
- 当前 `bus.ts` 虽然已经把 desktop / feishu / file 等执行细节外移到 runner，但 bus 本体仍同时承担：
  - policy gate
  - sideEffect metadata
  - per-tool 参数校验
  - per-tool preview 格式化
  - per-tool dispatch
  - telemetry 记录与统一 finalize
- 这意味着“统一执行入口”还在，但“薄网关”还没真正成立。

## Goal / Non-Goals

- Goal:
  - 保留单一 Tool Bus 入口
  - 把 per-tool 校验/格式化/分发从 bus 主文件继续外提
  - 让 bus 只保留最小公共骨架：gate -> execute -> log -> finalize
  - 降低新增工具时修改 bus.ts 多处 case 的概率
- Non-Goals:
  - 不做插件平台化
  - 不引入新的控制面或复杂注册中心
  - 不改变现有工具能力边界
  - 不要求本轮把所有 preview 都删掉

## Plan

- [x] 盘清 bus 当前哪些逻辑属于真正的公共骨架，哪些属于 per-tool 适配
- [x] 第一刀先把 per-tool 定义收口到单处：
  - sideEffect/meta
  - 参数校验
- [x] 第二刀再继续外提：
  - [x] execute handler（文件类 + Feishu 类）
  - [x] preview formatter（文件类 + Feishu 类）
- [x] 先用读写类验证新骨架
- [x] 再挑 Feishu 或 browser 类继续验证
- [x] 确认新工具接入路径不再要求在 bus.ts 本体多点同步修改
- [x] 补回归测试，锁住：
  - tool policy / telemetry / finalize 行为
  - 现有工具 data / preview / error 合同不漂

## Acceptance Criteria

1. `bus.ts` 不再同时维护大段 per-tool `switch` 分发、校验和 preview 拼装。
2. Tool Bus 的公共骨架清楚且稳定：
   - policy gate
   - 统一错误包装
   - telemetry
   - preview footer / finalize
3. 新增一个工具时，不再默认需要同时改 `bus.ts` 的多处位置。
4. 现有工具对外合同保持不变：
   - `executeTool(...)`
   - `ToolResult`
   - 主要错误码
   - telemetry 记录
5. 至少一组测试能防止 bus 再回流成“大内总管”。
