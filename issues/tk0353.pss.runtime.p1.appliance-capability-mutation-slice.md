# 任务：appliance capability mutation 切片

## Context

当前能力配置读面已经成立，但它只诚实显示：

- 当前活跃 lane
- 当前有真相源的 brain / vision / tts / asr

它没有 provider 中心，也不该长成 provider 中心。

但当前仓库已经有足够硬的写面锚点：

- `model.<lane>.text`
- `model.<lane>.vision`
- `model.<lane>.tts`

所以第一刀可以只开放：

- 当前活跃 lane
- `brain / vision / tts`

其余能力位继续只读。

## Goal / Non-Goals

### Goal

- 增加正式 `appliance set-capability` CLI
- 只写当前活跃 lane 的 `brain / vision / tts`
- 支持设置与 clear
- 成功后返回当前能力读面

### Non-Goals

- 不做 provider registry
- 不做 API key / endpoint 写回
- 不做非活跃 lane 写回
- 不做 `asr / image / video / music / search` 写回

## Plan

- [x] 定义 `brain / vision / tts` 到 config key 的映射
- [x] 增加 `appliance set-capability` CLI
- [x] 对只读能力位返回正式错误
- [x] 复用现有能力读面回显结果

## Acceptance Criteria

1. UI 不需要直接改 `.msgcode/config.json`。
2. 只有 `brain / vision / tts` 可写。
3. 写回只影响当前活跃 lane。
4. 其余能力位继续诚实报只读或无真相源。

## Notes

- 这刀只补最小 mutation，不回头推翻 `0322` 的“少做”原则。
- 若后续要支持非活跃 lane，也必须另开切片，不在这一刀里偷塞。
- CLI 只开放：
  - `brain`
  - `vision`
  - `tts`
- `--model` 和 `--clear` 二选一。
- 成功后直接回读当前能力位读面，不让 UI 自己猜结果。
- 失败时仍然返回正式 JSON Envelope，不把 Node 原生错误栈甩给 UI。

## Review Notes

- 圆桌结论：这刀可以过，因为它仍然守住“当前活跃 lane + 单一 config 真相源”。
- 保留点：
  - 不做 provider 编辑
  - 不做非活跃 lane 写回
  - 不给 `image / video / music / search / asr` 造假写面
- 后续如果要做多 lane 编辑，必须另开切片，不在本单扩写。
