# 透明执行治理插件研究

> 归类说明：
> 本文首先是“重任务透明执行治理插件”的设计草案；
> 其中关于 Web 应用、三层视图、治理面板的部分，是这套插件能力未来的控制台投影，不是插件本体。
> 归类：Agent Core 增强插件研究，不进入当前实施主线。

## 1. 文档目标

本文档用于统一描述一套“透明执行治理插件”的能力边界，并记录它未来可能投影出的控制台形态，供其他 agent 审阅、补充和挑战。

但从能力分层上看，本文真正要定义的不是页面本身，而是：

**一套给智能体增加复杂项目治理、透明汇报、人机协同与中断恢复能力的插件协议。**

该系统的目标**不是**单纯做任务管理，也**不是**单纯做 agent orchestration，而是构建一个：

**面向本地 Agent 执行过程的透明执行治理插件（transparent execution plugin）**

它应当同时满足：

- 能承接高层自然语言需求
- 能拆解成项目、任务、决策节点与执行动作
- 能绑定工作目录 / workspace / worktree
- 能可视化当前进度、健康度、阻塞点与下一步
- 能支持用户随时介入、叫停、讨论、裁决与恢复
- 能在进展不顺时，主动向上级（用户）上报问题与候选方案

---

## 2. 核心设计结论

### 2.0 先冻结一句话

这套设计先是一个：

**transparent execution plugin**

然后才可能长成一个：

**transparent execution console**

也就是说，控制台只是它的未来交付形式，不是能力本体。

### 2.1 系统首要目标不是“自动化”，而是“透明可干预”

系统的关键价值不在于 agent 是否能自动完成任务，而在于：

- 用户能否看懂系统现在在做什么
- 用户能否理解系统为什么这样做
- 用户能否在关键节点及时介入
- 系统能否在项目不顺时主动上报，而不是闷头继续

因此，系统设计优先级应当是：

1. 透明度
2. 可中断性
3. 可介入性
4. 决策确认机制
5. 进度可视化
6. 执行编排

### 2.2 这不是传统 issue tracker，而是“治理面板”

更准确地说，这个系统是一个：

**human-in-the-loop 的本地 agent 执行治理面板**

它需要覆盖三条并行流：

- **执行流**：需求如何被拆解、执行、验收
- **观察流**：用户如何持续理解当前状态、风险和下一步
- **介入流**：用户如何提出意见、叫停、转讨论、确认选型、恢复执行

---

## 3. 典型使用场景

### 3.1 高层需求输入示例

> 帮我做个类似网易云音乐的本地 Web 应用，你做好了告诉我怎么用；我的音乐文件在 `xx` 地方。

这类需求不是单步 coding task，而是一个项目流，至少包含：

- 需求理解
- 需求拆解
- 技术选型
- 项目骨架生成
- 若干子任务执行
- 中途选型确认
- 用户反馈吸收
- 项目中断 / 重规划
- 最终交付说明生成

因此系统必须支持：

- 项目级对象
- 任务树
- 决策卡
- 工作区绑定
- 中断与恢复
- 异常上报
- 最终 Delivery Report

---

## 4. 系统定位

建议将系统定义为：

> 一个给人看得懂、给 agent 跑得稳、并允许上级随时介入的本地项目控制台。

其职责不是代替 agent，而是把 agent 的：

- 理解
- 计划
- 决策
- 执行
- 阻塞
- 风险
- 结果

全部以可理解的方式摊开，并允许用户在关键节点显式改写执行路径。

---

## 5. 顶层设计原则

## 5.1 决策透明优先于过程完整

系统不应默认展示全部日志、prompt 或全量链路细节，而应优先展示：

- 当前正在做什么
- 接下来要做什么
- 为什么这样做
- 哪些点会改变后续路径
- 当前需要用户做什么

因此，展示层目标不是“过程完整透明”，而是：

**足以支撑用户做判断的决策透明**

## 5.2 默认自主执行，关键节点必须可确认

系统应在小决策上自动推进，在大决策上显式停下等待确认。

建议遵循：

- 小事不停
- 大事必须问
- 有风险必须报
- 偏离原需求必须确认

## 5.3 中断不是异常，而是正式能力

任何项目都要考虑中断可能。中断不应被视作失败，而应是一等公民。

系统必须支持：

- 用户主动暂停
- 系统主动停下请求确认
- 出现问题后升级讨论
- 讨论完成后恢复或终止

## 5.4 日志不是透明度，摘要才是透明度

当系统向用户汇报时，默认不应抛出原始工程噪音，而应生成面向管理/决策的摘要。

## 6. 系统的关键能力边界

系统至少需要具备以下能力：

### 6.1 工作目录 / Workspace 跟踪

需要明确记录：

- 任务/项目对应哪个 repo
- 对应哪个 worktree / workspace
- 当前分支是什么
- 当前执行者是谁
- 最近改动了哪些文件
- 当前测试状态如何

### 6.2 状态跟踪

需要区分：

- 项目状态
- 任务状态
- 一次执行 run 的状态
- 健康度状态
- 是否正在等待用户确认

### 6.3 决策可视化

要清楚展示：

- 当前有哪些待确认节点
- 推荐方案是什么
- 为什么推荐
- 替代方案有哪些
- 用户目前可以执行什么操作

### 6.4 用户介入

支持用户在中途：

- 修改需求
- 提出技术建议
- 叫停
- 发起讨论
- 批准/拒绝选型
- 要求重规划

### 6.5 异常上报

项目不顺时，系统需要主动向用户上报，而不是持续盲跑。

### 6.6 恢复与复盘

暂停或升级讨论之后，要能以正式动作恢复，并保留完整决策记录。

## 7. 核心对象模型

建议至少定义以下对象。

## 7.1 Project

项目级对象，用于承接高层需求。

```json
{
  "id": "PROJ-001",
  "title": "本地音乐 Web 应用",
  "goal": "构建一个类似网易云音乐的本地 Web 应用",
  "status": "IN_PROGRESS",
  "health": "YELLOW"
}
```

## 7.2 Task

任务级对象，支持父子层级。

```json
{
  "id": "TASK-004",
  "project_id": "PROJ-001",
  "title": "实现音乐目录扫描",
  "status": "RUNNING",
  "parent_id": "TASK-001"
}
```

## 7.3 Spec

系统对需求的理解与拆解结果。

```json
{
  "project_id": "PROJ-001",
  "goal": "...",
  "assumptions": [],
  "constraints": [],
  "acceptance_criteria": [],
  "subtasks": []
}
```

## 7.4 Workspace

工作目录注册信息。

```json
{
  "project_id": "PROJ-001",
  "repo_path": "/repos/music-app",
  "worktree_path": "/repos/music-app/.worktrees/PROJ-001",
  "branch_name": "project/PROJ-001",
  "owner_agent": "codex"
}
```

## 7.5 Run

一次执行记录。

```json
{
  "id": "RUN-0091",
  "task_id": "TASK-004",
  "executor": "codex",
  "status": "FAILED",
  "started_at": "...",
  "ended_at": "...",
  "summary": "实现了目录扫描，但元数据解析测试失败"
}
```

## 7.6 Artifact

执行产物。

```json
{
  "id": "ART-22",
  "task_id": "TASK-004",
  "type": "test_report",
  "path": "artifacts/TASK-004/test-report.json"
}
```

## 7.7 Event

时间线事件。

```json
{
  "id": "EVT-39",
  "project_id": "PROJ-001",
  "type": "decision_requested",
  "actor": "system",
  "message": "请求确认本地存储方案"
}
```

## 7.8 Feedback

用户反馈对象，必须结构化，而不是只留聊天文本。

```json
{
  "id": "FDBK-101",
  "project_id": "PROJ-001",
  "task_id": "TASK-004",
  "author": "user",
  "type": "architecture_override",
  "content": "不要使用 PostgreSQL，改为 SQLite",
  "status": "pending_applied"
}
```

## 7.9 Decision

待确认的节点选型对象。

```json
{
  "id": "DEC-008",
  "project_id": "PROJ-001",
  "title": "是否使用 SQLite 作为本地索引存储",
  "recommended": "sqlite",
  "options": ["sqlite", "json-index", "memory-only"],
  "status": "awaiting_user"
}
```

## 7.10 ReviewSession

用于统一建模“停下来讨论下一步”的场景。

```json
{
  "id": "REV-003",
  "project_id": "PROJ-001",
  "trigger_reason": "连续两轮修复无进展",
  "severity": "high",
  "summary": "播放器与扫描索引集成持续失败",
  "recommended_option": "缩小范围，仅交付本地浏览与播放",
  "status": "awaiting_user_decision"
}
```

## 8. 状态机设计

### 8.1 项目/任务层状态

建议至少支持：

- `NEW`
- `TRIAGED`
- `PLANNED`
- `READY`
- `IN_PROGRESS`
- `REVIEW`
- `BLOCKED`
- `DONE`

额外增加与用户介入相关的正式状态：

- `NEEDS_USER_INPUT`
- `NEEDS_USER_APPROVAL`
- `PAUSED_BY_USER`
- `AWAITING_DIRECTION`
- `PLAN_REVISED`
- `ESCALATED`

### 8.2 Run 层状态

- `QUEUED`
- `RUNNING`
- `SUCCEEDED`
- `FAILED`
- `CANCELLED`

### 8.3 中断/失败状态区分

建议显式区分：

#### 中断类
- `PAUSED_BY_USER`
- `AWAITING_REVIEW`
- `AWAITING_DIRECTION`
- `BLOCKED_NEEDS_DECISION`
- `BLOCKED_NEEDS_INPUT`

#### 失败类
- `FAILED_EXECUTION`
- `FAILED_ENVIRONMENT`
- `FAILED_VALIDATION`

关键原则：

**暂停、待确认、升级，不要混成失败。**

---

## 9. 健康度模型

进度百分比无法准确反映 agent 系统是否推进健康，因此建议引入独立的健康度信号：

- `GREEN`：按计划推进
- `YELLOW`：可推进，但存在明显不确定性或偏差
- `RED`：明显卡住，需要用户介入

### 9.1 Green 判定示例

- 最近有状态推进
- 无高风险待决事项
- 验收趋势正常

### 9.2 Yellow 判定示例

- 存在待确认决策
- 出现一次失败但可恢复
- 范围开始膨胀
- 当前计划与现实有轻微偏差

### 9.3 Red 判定示例

- 连续失败
- 高风险未决
- 长时间无进展
- 需要改方向
- 已经不适合继续自动执行

---

## 10. 展示层设计原则

## 10.1 不展示“全部过程”，只展示“足以影响决策的过程”

最值得展示的不是全量日志，而是：

1. 它现在在做什么
2. 它接下来要做什么
3. 它为什么这么做
4. 哪些点会影响后续路径
5. 你现在可以做什么

## 10.2 展示层采用三层视图

### 第一层：概览层

回答两个问题：

- 它现在在哪一步？
- 有没有需要我介入？

应展示：

- 当前阶段
- 总体进度
- 当前阻塞点
- 当前待确认事项数
- 最近一次关键更新
- 下一步动作
- 项目健康度

### 第二层：计划层

用于展示系统“打算怎么做”。

建议包含：

#### 需求理解卡
展示：

- 当前目标
- 系统假设
- 范围
- 非目标
- 风险
- 待确认项

#### 计划树
展示：

- 父任务/子任务
- 依赖关系
- 当前优先级
- 哪些任务因为用户反馈被改写

#### 决策时间线
展示：

- 提议了什么
- 谁提议的
- 是否已确认
- 因该决策导致了哪些计划变化

### 第三层：执行层

仅在需要深挖时查看：

- 当前 workspace
- 最近改动文件
- 最近 diff 摘要
- 测试结果
- 错误摘要
- 构建日志
- 生成产物

默认不应展开全部执行噪音。

---

## 11. 介入方式设计

系统不应只提供聊天框，而应提供结构化介入方式。

## 11.1 三类结构化介入

### 需求修订

适用于：

- 改目标
- 缩范围
- 加功能
- 删除功能
- 改交付形式

触发效果：

- 需求版本递增
- 计划重算
- 未开始任务重新评估

### 方案建议

适用于：

- 技术栈建议
- 架构调整建议
- 优先级建议
- 实现路径建议

系统必须明确对建议的处理结果：

- 采纳
- 部分采纳
- 拒绝，并说明原因

### 节点确认

适用于：

- 技术栈根选型
- 数据存储方案
- 项目范围变化
- 高风险行为
- 偏离原始需求的扩展

---

## 12. 决策卡设计

“决策卡”应成为主交互单元，而不是辅助元素。

一个完整的决策卡应至少包括：

1. **决策标题**
2. **为什么现在要决定**
3. **候选方案**
4. **系统推荐**
5. **推荐理由**
6. **用户可执行动作**

示例：

### 决策卡：选择前端框架

- 当前建议：Vue + Vite
- 备选方案：React / Svelte
- 推荐理由：
  - 组件复杂度适中
  - 本地应用体量不大
  - UI 开发效率较高
- 当前影响：
  - 将决定项目骨架和后续播放器 UI 组织方式
- 用户动作：
  - 接受推荐
  - 选另一个方案
  - 让我补充理由
  - 稍后再说
  - 改成自由输入建议

---

## 13. 决策节点分级

并非所有节点都应打断用户。建议分成三级。

### 13.1 级别 A：自动决策

适用场景：

- 代码风格
- 普通目录命名
- 小型依赖替换
- 非关键组件拆分
- 小型测试补充

策略：

- 自动执行
- 仅记录，不主动提示

### 13.2 级别 B：提示型决策

适用场景：

- 中等规模模块划分
- 常见技术栈选择
- 普通依赖新增
- UI 风格近似选择

策略：

- 系统可先采用
- 界面显示“已自动采用，可修改”

### 13.3 级别 C：确认型决策

适用场景：

- 技术栈根选型
- 数据存储方案
- 项目范围变化
- 新建服务端
- 破坏性修改
- 偏离初始需求
- 安全/隐私/性能关键决策

策略：

- 必须进入 `Decision Queue`
- 必须显式等待用户处理

---

## 14. 叫停、讨论、恢复机制

## 14.1 控制动作

系统至少要支持以下动作：

- `Pause`：立即暂停
- `Stop after current step`：当前步骤完成后暂停
- `Escalate for discussion`：进入讨论模式
- `Replan`：保留现状，重做计划
- `Abort`：彻底终止并归档

## 14.2 “开会模式”/讨论模式

当用户选择“先开会研究下一步”时，系统应切换到正式模式，而不是只是停下来。

建议进入：

- `MEETING_MODE`
- `DECISION_REVIEW`
- `REPLAN_SESSION`

此时主界面焦点从“执行”切换到“决策”。

系统应自动生成一个**会议摘要包**，至少包含：

- 当前目标
- 当前阶段
- 已完成内容
- 当前阻塞
- 最近失败原因
- 已尝试方案
- 候选下一步
- 推荐方案
- 等待用户决定的问题

## 14.3 恢复动作

暂停/讨论后，恢复不应只是“继续跑”，而应支持：

- `Resume as-is`
- `Resume with revised plan`
- `Resume with narrower scope`
- `Fork alternative plan`
- `Close without completion`

---

## 15. 向上级（用户）反馈的机制

当进展不顺时，系统不能靠主观感觉决定是否要上报，而应具备正式的升级策略：

**Escalation Policy**

## 15.1 上报触发器

### 时间型触发器

例如：

- 超过设定时间仍停留在同一阶段
- 连续若干次 run 没有推进状态
- 长时间处于 blocked 无人处理

### 质量型触发器

例如：

- 连续两轮验收失败
- 同一问题反复出现
- 改动范围持续扩大
- 测试通过率下降
- 新增风险持续增加

### 决策型触发器

例如：

- 技术栈要根改
- 项目范围需要缩减
- 需要跳过部分需求
- 发现需求理解与最初目标冲突
- 出现多个成本差异较大的方向

---

## 16. 升级卡设计

系统向用户上报问题时，不应直接给原始日志，而应生成面向管理的升级卡。

升级卡至少应包含：

1. 当前任务是什么
2. 卡在哪里
3. 已做过什么
4. 为什么系统认为需要你介入
5. 推荐选项
6. 你可以执行的动作

示例结构：

```json
{
  "task_id": "TASK-006",
  "summary": "播放器与索引 API 集成连续失败",
  "stuck_point": "前端播放器拿到的数据结构与后端索引输出不一致",
  "attempted_actions": [
    "重试适配接口",
    "补充测试",
    "重构索引输出字段"
  ],
  "why_escalated": "连续两轮修复无显著进展，且后续路径依赖该接口稳定",
  "recommended_options": [
    "缩小范围，先交付本地浏览与播放",
    "冻结 UI，优先重做索引 API",
    "进入讨论模式"
  ],
  "allowed_user_actions": [
    "同意推荐方案",
    "改选其他方案",
    "要求重规划",
    "暂停项目"
  ]
}
```

## 17. 上报渠道设计

建议区分三条渠道：

### 17.1 叫停通道

用于用户主动控制执行节奏：

- 立即暂停
- 当前步骤后暂停
- 转入讨论
- 终止任务

### 17.2 讨论通道

用于用户参与判断：

- 质疑选型
- 调整实现路线
- 缩范围
- 修改优先级
- 要求重新拆解

### 17.3 上报通道

用于系统主动通知用户：

- 卡住了
- 不确定了
- 偏离了
- 风险升高了
- 需要更高层裁决

这三条渠道在 UI 中应明确区分，避免全部退化成聊天。

---

## 18. 报告机制

不建议纯定时汇报，更适合做**事件驱动式报告**。

### 18.1 进展摘要

在关键推进节点自动生成：

- 已完成内容
- 下一步
- 当前是否存在阻塞

### 18.2 异常摘要

在出现问题时自动生成：

- 问题是什么
- 影响是什么
- 已尝试什么
- 需要用户决定什么

## 19. 页面信息架构建议

建议采用以下布局思路，而不是纯聊天界面。

## 19.1 顶部：全局状态条

显示：

- 当前阶段
- 当前执行者
- 当前节点
- 进度
- 健康度
- 待确认事项数
- 最近更新时间

## 19.2 左侧：计划与结构

显示：

- 需求理解卡
- 任务树
- 依赖关系
- 当前阶段流转
- 已受反馈影响的任务

## 19.3 中间：当前焦点

主内容区根据上下文显示：

- 当前决策卡
- 当前任务详情
- 当前升级卡
- 当前会议摘要
- 当前交付预览

## 19.4 右侧：介入面板

提供动作：

- 提建议
- 修订需求
- 接受/拒绝方案
- 暂停
- 转入讨论
- 查看历史意见

## 19.5 底部/折叠区：执行证据

显示：

- diff 摘要
- 测试结果
- 日志
- artifacts

## 20. 当前焦点原则

主界面不应同时抛给用户大量待处理问题。

应遵循：

- 阻塞性问题优先
- 高影响问题优先
- 同类问题聚合
- 非紧急问题延后

也就是：

**系统每次只暴露一个最值得用户决定的当前焦点问题。**

其他问题进入队列，不应同时轰炸用户。

---

## 21. 用户反馈的系统化处理

用户说一句：

> 不要用 Electron，先做纯 Web

系统内部不应只是新增一条消息，而应触发：

1. 识别为“架构建议”
2. 作用域识别为项目级
3. 更新相关决策卡状态
4. 标记受影响任务为“需重规划”
5. 重算计划树
6. 在时间线记录“因用户反馈变更方案”

这意味着系统要把“聊天输入”转成“状态变更”。

---

## 22. 建议的最小可行产品（MVP）

第一阶段建议优先做以下能力，而不是先做复杂 orchestration。

### 22.1 需求理解卡可审阅
用户能看、能改、能确认。

### 22.2 决策卡
系统可以提出方案，用户可以选择、修改、延后。

### 22.3 计划树
用户能看到当前任务结构及其变化。

### 22.4 Decision Queue
集中展示所有待确认节点。

### 22.5 Timeline
显示关键事件和用户意见如何改变系统行为。

### 22.6 中断能力
支持：

- 一键暂停
- 步骤后暂停
- 转入讨论模式

### 22.7 升级卡
项目不顺时自动生成摘要给用户。

### 22.8 Review Session
支持“停下来开会研究下一步”。

### 22.9 恢复动作
会议结束后支持：

- 原计划继续
- 调整后继续
- 缩范围继续
- 终止

### 22.10 Workspace 跟踪
最少要能看到：

- repo path
- worktree path
- branch
- 最近改动文件
- 当前执行者

---

## 23. 不建议优先做的方向

以下内容不应作为第一优先级：

- 复杂多 agent 自治循环
- 让不同 agent 彼此自由讨论
- 全量思维链或 prompt 直接暴露给用户
- 过度复杂的看板动画或可视化特效
- 过早追求自动 merge、自动长期自治

原因是当前系统的核心问题不在于“agent 能不能继续自动跑”，而在于：

- 用户能不能看懂
- 用户能不能及时叫停
- 系统能不能在失控前上报
- 重大节点能不能显式确认

---

## 24. 建议的产品原则（可直接作为系统原则）

可将系统原则总结为以下几条：

### 原则 1
**默认隐藏执行噪音，默认暴露决策理由。**

### 原则 2
**默认允许用户提出建议，关键节点必须显式确认。**

### 原则 3
**任何任务都必须可暂停、可上报、可复盘。**

### 原则 4
**任何重大偏差都必须先形成摘要，再请求上级裁决。**

### 原则 5
**任何会让后续大量工作路径发生变化的决策，都应可见且可干预。**

---

## 25. 供其他 agent 审阅的重点问题

建议其他 agent 重点审阅以下问题：

1. 当前对象模型是否足以支撑“透明 + 中断 + 上报 + 恢复”
2. `Decision` / `Feedback` / `ReviewSession` 是否是一等公民，是否需要进一步细分
3. 当前状态机是否足以区分“暂停、待确认、失败、升级”
4. 升级卡与决策卡是否能覆盖大多数高价值人机交互
5. 展示层三层结构是否合理，是否还有更优信息架构
6. “每次只暴露一个当前焦点问题”的原则是否可实施
7. 健康度模型与进度模型是否需要联动
8. 哪些节点应自动化，哪些节点必须强制确认
9. Workspace 跟踪维度是否还应增加
10. 对后续接入具体 agent（如本地模型、Codex、Claude Code）时，哪些对象应保持稳定，哪些应做 adapter 层

---

## 26. 总结

这个系统的本质不是：

- 让 agent 更自由地自动工作
- 或做一个普通任务管理器

它真正要解决的是：

> 当 agent 在本地真实做项目时，如何让用户始终知道它在干什么、为什么这么做、什么时候该介入、出了问题如何上报、讨论之后如何恢复。

因此它应当被设计成一个：

**以透明度为核心、以中断与决策为关键能力、以用户治理为上层目标的本地 agent 执行控制台。**
