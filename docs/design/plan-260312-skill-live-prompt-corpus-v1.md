# Skill Live Prompt Corpus v1 计划

Issue: 0099

## Problem

skill 改造后的主风险已经从“代码函数错没错”转移到了“模型会不会走对主链”：

- `SKILL.md` 是否足够清楚
- wrapper 退役后模型是否还能找到正确入口
- 模型是否真的完成动作，而不是只给出看起来合理的回答
- 出错时能否从日志里快速看出：是 skill 误导、工具选择错误，还是运行时执行失败

这类风险需要一组固定 prompt 作为真实案例集，而不是继续临时想测试话术。

## Occam Check

1. 不加它，系统具体坏在哪？
   - 每次做 skill 验证都临时发消息、临时想 prompt、临时解释结果。这样无法比较“改造前后是否更稳”，也无法沉淀为复用方法。
2. 用更少的层能不能解决？
   - 能。直接冻结一组固定 prompt + 证据口径，不增加新的 benchmark 平台或控制面。
3. 这个改动让主链数量变多了还是变少了？
   - 变少了。把零散的人工测试收口成一套固定案例和统一观测方式。

## Decision

采用 **“固定 prompt corpus + debug 观测口径”** 的最小方案。

核心原则：

1. 每条 case 都对应一个明确的 skill/能力边界
2. 每条 case 都要求真实副作用或真实外部证据
3. 默认用 `Feishu live verification loop` 作为执行底座
4. 默认打开 debug 日志做观测，不只看最终回复

v1 不做平台化：

- 不做自动评分系统
- 不做复杂 benchmark harness
- 不做多模型榜单

只做一份可直接复用的案例集文档。

## Alternatives

### 方案 A：继续临时人工想 prompt

- 优点：零成本
- 缺点：不可比较，不可复用，容易遗漏关键路径

### 方案 B：直接做自动 benchmark 平台

- 优点：更炫
- 缺点：过重；当前问题是固定验证口径，不是缺平台

### 方案 C：固定 prompt corpus + debug 观测（推荐）

- 优点：薄、可复用、足以暴露 skill 路径偏差
- 缺点：前期仍以人工/live 执行为主

## Corpus 设计原则

每条 case 至少包含：

- `id`
- `目标`
- `prompt`
- `目标 skill`
- `预期工具/CLI 路径`
- `禁止路径`
- `预期副作用/证据`

v1 重点不在数量，而在覆盖最容易跑偏的 skill 边界。

## Case 分组

### A. 说明书能力发现

验证模型能否通过 `SKILL.md` 找到正确入口，而不是幻想旧 wrapper。

### B. 真实动作执行

验证模型是否真的完成副作用，而不是只给出解释。

### C. 抗误导 / fail-closed

验证模型在缺信息或旧入口失效时，是否能走对主链或正确报缺，而不是乱猜。

## Debug 观测口径

固定看三类证据：

1. **Feishu 通道证据**
   - 群里真实回复/文件
   - 最近消息查询命中
2. **Workspace 证据**
   - `.msgcode/config.json`
   - sessions / threads / schedules / 产物文件
3. **运行时日志**
   - `msgcode.log`
   - 重点字段：
     - `listener 收到消息`
     - `run lifecycle`
     - `agent request started/completed`
     - 工具调用成功/失败日志
     - `消息处理完成`

## Risks

### 风险 1：case 太多，维护成本高

回滚/降级：
- v1 只保留 10 条高价值 case

### 风险 2：只看最终回答，不看真实副作用

回滚/降级：
- 每条 case 强制要求至少一类外部证据

### 风险 3：日志过多，观察口径继续漂

回滚/降级：
- 文档中只固定最关键的 5 类日志点
