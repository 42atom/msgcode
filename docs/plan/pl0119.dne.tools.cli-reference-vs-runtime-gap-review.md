# CLI 单工具参考与当前 runtime 主链差距审计

## Problem

仓库已经明确要求“做薄、少加层、服务 LLM”，而新迁入的参考文档进一步把目标讲得更尖锐：尽量把主链收口成 `模型 -> CLI/工具 -> 真实结果 -> 模型`，让帮助、错误、输出格式本身成为模型的导航系统。

对照当前 `msgcode`，可以看到我们已经做对了一部分：

1. 已经保留 `bash` 作为通用执行通道
2. 已经把工具失败先回灌模型，而不是一上来就系统代答
3. 已经在持续删除 `finish supervisor`、`verify`、forced final state 这类热路径裁判层

但同时，当前实现仍有几处关键设计不足：

1. 工具面仍然是“多工具 + suppression + 提示词映射”，不是单一可发现主链
2. 执行层与呈现层的职责分界还不干净，输出裁剪/落盘/回灌分散在多个位置
3. 文件读取缺少统一的二进制/大文件合同，错误消息也不够“指下一步”
4. 帮助系统虽然存在，但没有作为 LLM 可渐进探索的正式主链
5. 预算边界仍然以内联控制逻辑和系统话术形式留在 tool-loop 热路径

## Occam Check

- 不加这次审计，系统具体坏在哪？
  - 后续很容易继续靠“再补一点提示词、再补一点 hint、再 suppress 一个工具”推进，结果表面更稳，主链却更散。
- 用更少的层能不能解决？
  - 能。优先做收口而不是加层：收口工具面、收口输出层、收口文件合同，再把 help 与预算边界往统一接口或外层机制移动。
- 这个改动让主链数量变多了还是变少了？
  - 变少了。本轮不加任何运行时层，只把现状的分叉和漂移点定位清楚，为后续删层做准备。

## Decision

选定方案：将该参考文档作为**方向性参照**而不是教条模板，先做 evidence-first 差距审计，不直接大改架构。

推荐优先级：

1. 第一优先：收口文件/输出合同
   - 先解决 `read_file` 的二进制/大文件/错误引导问题
   - 再解决 `bash-runner` 与 tool-loop 的双重裁剪分层问题
2. 第二优先：收口工具面
   - 减少“工具存在但默认 suppress，再靠提示词叫模型改走 bash”的认知折返
3. 第三优先：把帮助与预算边界从热路径里继续外推
   - help 改成模型可主动发现
   - 预算边界尽量减少直接生成用户可见系统文案

关键理由：

1. 当前最真实的痛点不是“没有单工具 run”，而是主链上仍有几段职责混合
2. 直接照搬 one-tool 模型，会误伤 `browser`、`feishu_send_file` 这类已经有价值的原生工具
3. 先把合同和分层收口，后续再决定是否进一步朝单工具面收缩，风险更低

## Alternatives

### 方案 A：全文照搬参考，只保留 `run(command)`

- 优点：方向最纯，认知模型最统一
- 缺点：会直接冲击现有 `browser`、`feishu_send_file` 等原生工具价值，改动面过大

### 方案 B：维持现状，只继续补提示词和 skill

- 优点：最省事
- 缺点：会继续堆 prompt/hint，无法解决输出层分裂、文件合同薄弱、预算边界内联等根问题

### 方案 C：先做差距审计，再选第一批收口点

- 优点：符合仓库“先改说明书/合同，再动代码”的口径
- 缺点：本轮只能得到结论，不直接产出运行时收益

推荐：方案 C

## Gap Review

### 1. 工具面仍然偏碎，且存在 suppression + 映射折返

证据：

- [src/tools/manifest.ts](/Users/admin/GitProjects/msgcode/src/tools/manifest.ts#L63)
- [src/tools/manifest.ts](/Users/admin/GitProjects/msgcode/src/tools/manifest.ts#L591)
- [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1202)

现状：

- 仓库同时维护 `browser / bash / read_file / write_file / edit_file / vision / mem / feishu_*`
- 但 `write_file / edit_file / vision / mem` 又被默认 suppress
- tool index 还显式告诉模型“写改文件优先用 bash”

问题本质：

- 这不是单一主链，而是“工具注册一套、默认暴露一套、提示词再教一套映射”
- 模型需要做额外的工具策略切换，而不是只专注完成任务

最小收口方向：

- 重新定义“默认正式工具面”
- 尽量减少“工具存在但默认不让用，再提示你绕另一条路”的状态

### 2. 执行层和呈现层还没有严格分开

证据：

- [src/runners/bash-runner.ts](/Users/admin/GitProjects/msgcode/src/runners/bash-runner.ts#L160)
- [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L405)
- [src/runtime/context-policy.ts](/Users/admin/GitProjects/msgcode/src/runtime/context-policy.ts#L112)

现状：

- `bash-runner` 已经在 runner 层做输出截断和完整输出落盘
- `tool-loop` 回灌模型前又做一次 `clipToolPreviewText`
- context policy 里还有第三套 preview clip helper

问题本质：

- “执行结果长什么样”和“给模型看什么样”被分散到了 runner、Tool Bus、tool-loop、context policy 多处
- 这会造成不同工具行为不一致，也会让后续扩展更容易继续补丁化

最小收口方向：

- 执行层只保真结果和必要 artifact
- 呈现层统一决定预览、截断、导航提示和元数据格式

### 3. `read_file` 缺少统一的二进制/大文件/导航合同

证据：

- [src/tools/bus.ts](/Users/admin/GitProjects/msgcode/src/tools/bus.ts#L753)
- [src/tools/bus.ts](/Users/admin/GitProjects/msgcode/src/tools/bus.ts#L1267)

现状：

- `read_file` 直接按 UTF-8 全量读文件
- 没有二进制检测
- 没有大文件阈值与 artifact 导航
- 失败时大多走通用 catch，直接把原始错误字符串返回

问题本质：

- 这与参考文档强调的“binary guard + 指路错误消息 + 大输出导航”差距最大
- 一旦模型误读图片、PDF、二进制或超大文本，当前合同要么报生硬错误，要么直接把大量内容塞回主链

最小收口方向：

- 为 `read_file` 增加：
  - 二进制检测
  - 大文件 preview + full path
  - 明确的下一步提示，如改用 `browser` / `vision` / `bash`

### 4. 帮助系统存在，但不在模型的主探索路径里

证据：

- [src/cli/help.ts](/Users/admin/GitProjects/msgcode/src/cli/help.ts#L87)
- [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1208)

现状：

- CLI 侧已经有 `help-docs --json`
- 但模型主链里主要靠：
  - tool schema
  - tool index 文本注入
  - skill index JSON + `read_file` 读 `SKILL.md`

问题本质：

- 帮助系统和 LLM 工具主链仍然是两套入口
- 参考文档强调的“按需探索、逐层发现”在当前实现里主要靠提示词，而不是统一可调用帮助入口

最小收口方向：

- 要么把帮助显式作为模型可发现的正式能力面
- 要么继续缩减工具面，让 schema 足够短、足够自解释

### 5. 预算边界仍以内联控制逻辑和系统文案形式留在热路径

证据：

- [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1340)
- [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1513)

现状：

- `tool-loop` 里仍有 per-turn call/step quotas
- 达到限制后直接返回系统生成的中文说明，并要求“下一轮 heartbeat 继续执行”

问题本质：

- 预算边界当然是允许存在的，但现在它还深度嵌在 tool-loop 热路径里，并直接塑造用户可见交付
- 相比参考文档“外部机制保障”的口径，我们离“预算是边界，不是系统语义层”还有距离

最小收口方向：

- 预算边界继续保留，但逐步把用户可见文案和热路径控制耦合拆开
- 让 continuation 更像 runtime 事实，而不是系统代答模板
