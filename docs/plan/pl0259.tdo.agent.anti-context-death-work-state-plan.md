# PL0259：抗 Context Death 的工作态重建设计计划

## 结论

`summary` 仍然要保留，但不能继续假装它就是 memory 本体。

msgcode 下一步更合理的做法，不是扩写“更长 summary”，而是增加一层更接近工作中的状态骨架：

- 保留现有 `issue` 作为任务真相源
- 保留现有 runtime `work capsule` 作为恢复执行的最小投影
- 新增一层轻量 `working state scaffold`，专门服务“跨窗口重建工作态”
- 原始日志、测试输出、trace 继续作为 `evidence` 保存

关键点是分层，而不是堆层。

## 问题定义

当前系统跨窗口时，通常只能依赖：

- summary
- recent window
- issue / checkpoint
- log / diff / task history

这些东西能恢复“发生了什么”，但不一定能恢复“当前脑子正在沿哪条路径收敛”。

因此真正的问题不是“有没有摘要”，而是：

`下一窗口能否以足够小的信息损失，恢复上一窗口的工作态。`

## 现有地基

仓库里已经有三条重要主线：

1. `truth continuity`
   - 任务、状态、dispatch、artifact 的文件真相源

2. `work continuity`
   - runtime 可从真相源重建最小 `work capsule`

3. `session continuity`
   - 只能建立在 `work continuity` 上层

现有 `work capsule` 已经明确是：

- 最小恢复投影
- 给 runtime / wake / resume 用
- 不持久化为第二真相源
- 不回放 transcript

这条主线是对的，不应推翻。

## 本计划的判断

我们要补的，不是替代现有 `work capsule`，而是补一层“给模型跨窗口恢复工作态”的骨架。

更准确地说：

- `runtime work capsule` 解决：系统怎样知道下一步该继续执行什么
- `working state scaffold` 解决：模型怎样更快恢复到上一次工作的姿态

这两者相关，但不应混为一个对象。

## 设计原则

### 1. 真相、恢复投影、工作态骨架、摘要必须分层

建议固定为四层：

1. 真相层
   - `issues/*.md`
   - dispatch / checkpoint / artifact refs

2. 恢复投影层
   - 现有 runtime `work capsule`

3. 工作态骨架层
   - 新增 `working state scaffold`

4. 摘要层
   - summary / compact prose

其中：

- 真相层回答“事实是什么”
- 恢复投影层回答“系统该怎么续跑”
- 工作态骨架层回答“模型该以什么姿态接上”
- 摘要层回答“给快速回读看的最短说明”

### 2. 不把 runtime capsule 越做越胖

现有 `work capsule` 已是最小运行时协议。

不建议把下面这些都塞回 runtime capsule：

- rejected hypotheses
- touched symbols
- impact radius
- failing traces
- patch intent

原因很简单：

- runtime 不需要这么多细节也能恢复执行
- 塞进去会让 capsule 从“恢复协议”膨胀成“万能记忆对象”

### 3. 工作态骨架必须是 checkpoint，不是聊天日志

它只在工作态即将断裂或显著变化时更新。

它不是每轮一写，更不是 transcript。

### 4. 仍坚持 file-first

先落成文件协议，再决定是否要自动化生成。

不要先做 manager / database / service。

## 目标对象：Working State Scaffold

### 作用

`working state scaffold` 不是任务真相源，也不是 runtime capsule。

它是给“下一窗口的模型”看的最小工作骨架。

它回答的问题是：

- 当前目标是什么
- 当前主假设是什么
- 哪些方向已经排除
- 当前 patch 想达成什么
- 当前最值得继续看的文件 / symbol / 测试 / trace 是什么
- 下一步从哪接

### 建议文件位置

建议先用最薄的文件落点：

- `<workspace>/.msgcode/workstates/<taskId>.md`

理由：

- 不污染 `issues/` 的状态真相
- 不和 runtime capsule 混在一起
- 明确它是工作态投影，不是主任务文档

### 建议最小字段

```md
---
task: tkNNNN
updated_at: <iso>
source_stamp: <hash>
truth_refs:
  - <issue path>
  - <dispatch/evidence path>
---

# Goal
一句话目标

# Current Intent
当前 patch / 当前推进意图

# Active Hypothesis
当前最主要的判断

# Rejected Hypotheses
- 已排除路径

# Touched Files
- file / symbol

# Impact Radius
- 受影响模块 / 调用链 / 风险面

# Key Evidence
- failing test
- logs
- trace

# Open Edges
- 未决问题

# Next Resume Point
- 下一窗口从哪接

# Short Handoff
给下一窗口的最短恢复说明
```

这里最重要的不是 prose，而是这些槽位本身。

## 与现有 runtime work capsule 的分工

### 现有 runtime capsule 保持不变

继续负责：

- taskId
- phase
- checkpoint
- nextAction
- activeDispatch
- source stamp

它服务：

- wake
- resume
- heartbeat
- task supervisor

### 新工作态骨架单独负责

负责：

- active hypothesis
- rejected hypotheses
- touched files / symbols
- impact radius
- key evidence
- current intent
- next resume point

它服务：

- 模型跨窗口恢复
- 人工 handoff
- context rebuild

## 触发规则

工作态骨架不应该持续生成，只在以下场景触发。

### 必触发

1. 即将 compact / 切窗
   - `contextUsagePct` 接近阈值时

2. 长任务 durable checkpoint 写盘时
   - heartbeat / task checkpoint / wake consume 前后

3. 从 `doi -> rvw` 前
   - 因为这时需要把工作态交给 review

### 条件触发

4. 主假设变化
   - 根因判断推翻
   - 方案从 A 切到 B

5. 关键证据新增
   - 失败测试、trace、核心日志、关键 diff

### 不触发

1. 每条消息后
2. 没有新信息时
3. 纯闲聊 / 轻问题

## 谁来写

按现有架构，建议分两段。

### Phase A：先人工/半自动

先只定义协议和调用位置，不强制 runtime 自动生成。

可由：

- agent 在关键边界主动写
- task supervisor / heartbeat 先留接口

### Phase B：再自动化

等协议稳定后，再考虑：

- checkpoint 时自动更新骨架
- compact 前自动刷新
- review 前自动落盘

先协议，后自动化。

## 与 graph/context layer 的关系

`working state scaffold` 不负责保存全部结构细节。

它只保存重建入口。

真正容易过时、适合动态获取的内容，应由 graph/context query 补：

- touched symbols 的依赖链
- impact radius 的扩展面
- 关键测试绑定
- 相关文件排序
- 风险最高的调用路径

所以关系应是：

- `workstate file` 保存恢复支架
- `graph/context query` 负责按需补细节

不要反过来把所有图信息都写死进文件。

## 最小实施路径

### Phase 1：协议落地

产物：

- `docs/protocol/WORKSTATE.md`

内容：

- 定义 `working state scaffold` 的职责、字段、触发规则
- 写清与 `issue`、`work capsule`、`summary` 的边界

### Phase 2：手工/代理写入闭环

产物：

- agent 在关键边界可生成 `<workspace>/.msgcode/workstates/<taskId>.md`

先不要求 runtime 强制依赖它。

### Phase 3：上下文装配优先读取

在 context rebuild 时，若存在 `workstate` 文件，则优先把它作为恢复骨架注入。

但它不能覆盖：

- issue 状态真相
- checkpoint 真相
- dispatch 真相

### Phase 4：图查询辅助恢复

把 `workstate` 文件中的：

- touched files
- symbols
- next resume point

作为 graph/context query 的入口键，而不是把所有结构一次性写死。

## 非目标

本计划明确不做：

1. persistent KV continuation
2. latent memory persistence
3. transcript full replay
4. 新建 memory platform
5. 把 runtime capsule 膨胀成万能对象

## 风险

### 风险 1：和现有 work capsule 重叠

规避：

- 明确一个给 runtime，一个给模型恢复

### 风险 2：又长出第二份 summary

规避：

- 固定槽位，不写散文
- 只在断裂边界更新

### 风险 3：系统层被迫理解太多语义

规避：

- 先协议，后自动化
- 先 agent 写，后 runtime 接

## 设计判断

如果这套方案和现有架构不能兼容，最先该删的不是 `issue`、不是 `work capsule`，而是这层 `working state scaffold`。

因为：

- 真相源不能动
- runtime 恢复协议不能胖
- 新层必须证明它真的提高了跨窗口重建质量

所以它必须从一份最小、可删、可观测的文件协议开始。

## 建议后续拆单

1. `WORKSTATE protocol`
2. `workstate file write triggers`
3. `context rebuild reads workstate before prose summary`
4. `graph query as workstate reconstruction sidecar`
