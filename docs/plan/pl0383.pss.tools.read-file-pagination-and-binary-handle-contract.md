# PL0383：Read File Pagination And Binary Handle Contract

## 结论

这条线先做一件事：

把 `read_file` 从“超限后 head+tail 拼盘”收正为“连续分页正文合同”。

二进制再补最薄的 handle/blob 语义。

## 为什么现有文件不能直接承载

- `pl0120` 解决的是 preview layering，不是连续分页正文
- `pl0154` 解决的是去掉教学文案，不是继续读取指针
- 这次问题已经变成一条新的基础工具合同，需要单独计划

## 实现顺序

### Step 1

收 `read_file` 文本合同。

目标：

- 小文本全文返回
- 大文本连续分页
- 明确 `offset/limit/has_more/next_offset`

### Step 2

收 `tool-loop` 的 read_file 回灌边界。

目标：

- 已分页且在预算内的正文，不再被二次压扁
- 只改 `read_file` 这一个工具，不扩大到全局

### Step 3

补二进制 / 超大非文本语义。

目标：

- 不直接回正文
- 回元信息与后续可消费引用
- 第一刀不做独立 blob 平台

### Step 4

补最小验证。

目标：

- 小文本全文
- 大文本分页
- 二进制 handle/blob
- tool-loop 不再破坏 read_file 分页正文

## 不做什么

- 不做 openclaw 式自适应多页聚合
- 不扩到 `grep / log / exec`
- 不做新的 preview controller
- 不新增中间控制层

## 一句话

先把 `read_file` 读对，再谈其它基础工具跟进。
