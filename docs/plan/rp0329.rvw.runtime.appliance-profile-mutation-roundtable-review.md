# RP0329：appliance profile mutation 圆桌复核

Decision:
- Recommendation: go
- Confidence: medium
- Scope: 接受 `set-profile` 进入提交，但继续把 mutation 严格限制在 profile 分组内

Why:
- `SystemsArchitect`：当前只有一个正式写入口，还没顺手把 general/capabilities 拉进来；主链仍然短。
- `SkepticalOperator`：入口仍是四个显式字段，不是 blob 保存器；写后回读也避免了 UI 自己发明成功态。
- `UserAdvocate`：最关键的“部分成功”已被诚实回出来，不再把多文件写入伪装成原子设置对象。
- `ExecutionLead`：顺序仍然克制，先读面、再 profile mutation，没有主次倒置。

Risks:
1. 写逻辑继续堆在 `appliance.ts` 里，后面容易长成 settings 写中心。
2. `ORG.md` 仍是补丁式改写，后续若出现冲突字段，需要单独处理。
3. UI 若不展示 `changedFiles`，用户仍可能误解“保存成功”的实际含义。

Guardrails:
- `set-profile` 只准服务 profile 四个字段，不准演变成通用 `set-setting`。
- 其他 tab 继续只读，不给保存按钮假象。
- UI 必须显示 `changedFiles` 或其对应的人话，不得只弹“保存成功”。
- 若后面再加 mutation，把写逻辑抽出 `appliance.ts`，不要继续堆路由文件。

Next Actions:
1. 提交 `0328` 代码与回归锁。
2. 下一步若继续 settings 主线，只能考虑把 profile 写逻辑抽到独立 runtime 模块。
3. general/capabilities 继续只读，除非先有新的真相源与合同。

Evidence:
- Docs: `tk0322`, `tk0328`, `rp0328`
- Code: `src/cli/appliance.ts`
- Tests: `bun test test/p7-tk0322-appliance-set-profile.test.ts`
