# PL0271：四条主线的单线程执行顺序表

## 结论

当前四条线不应并发推进。

推荐顺序固定为：

1. `pl0267` 确定性反馈基础设施
2. `pl0259` 工作态骨架纯文本试跑
3. `pl0270` 图驱动上下文路由 Sidecar
4. `pl0262` 企业版自包含与交付形态

一句话：

- 先把主链闭环做硬
- 再验证工作态骨架是否真的有用
- 再做外置上下文路由器
- 最后单开交付宇宙

## 为什么现在会乱

不是这几份 plan 写得厚，而是它们分属四个不同深度层：

1. `pl0267`
   - coding lane 的确定性反馈地基

2. `pl0259`
   - 跨窗口工作态重建

3. `pl0270`
   - 代码图驱动的上下文选择 Sidecar

4. `pl0262`
   - 企业交付、App/UI、跨平台与安装链

如果这四条线同时进入实现，系统会同时消耗：

- runtime 脑带宽
- agent prompt / protocol 脑带宽
- shell/tooling 脑带宽
- 产品交付脑带宽

结果一定是主线失焦。

## 调度原则

### 原则 1：同一时刻只允许一条“主实施线”

允许：

- 1 条主实施线
- 1 条轻量研究/文档线

不允许：

- 2 条以上重量级实现线并发

### 原则 2：认知主链优先于交付主链

在当前阶段：

- `agent feedback`
- `workstate`
- `context router`

优先级高于：

- 企业安装链
- App shell
- 跨平台打包

### 原则 3：没有闭环地基，不扩上下文智能

如果 `pl0267` 没做实：

- `pl0259` 会变成多一份摘要
- `pl0270` 会变成更贵的猜文件工具

所以 `pl0267` 必须先行。

### 原则 4：企业版是平行宇宙，不进当前认知主链节奏

`pl0262` 与前三条线不是一个版本节奏。

它应该被视作：

- 独立交付战场
- 独立里程碑
- 独立脑带宽预算

## 推荐顺序

## 第一顺位：PL0267

### 为什么第一

这是当前 ROI 最高的地基。

它直接决定：

- agent 改完代码能否快速看到真实错误
- 修复链是否能形成“改 -> 验 -> 再修”的最小闭环
- 后续 graph/router/workstate 是否有稳定反馈面可挂

### 当前阶段只做什么

- `tk0267`
- `tk0268`
- `tk0269`

即：

- 快反馈合同
- verify pack normalization
- verification evidence writeback

### 不做什么

- 不扩 graph
- 不碰企业交付
- 不自动无限修复

### 切换到下一条线的条件

满足以下条件后，才允许进入 `pl0259`：

1. coding lane 已有正式 verify pack
2. 最后一次验证结果有统一证据回写面
3. 至少一条真实代码修复链能稳定跑完“改 -> 验 -> 再修”

## 第二顺位：PL0259

### 为什么第二

它是低成本试错线。

最大价值不是自动化，而是验证一个问题：

`WORKSTATE` 是否真的比 prose summary 更能帮助跨窗口续接。

### 当前阶段只做什么

- 按 [WORKSTATE.md](/Users/admin/GitProjects/msgcode/docs/protocol/WORKSTATE.md) 跑纯文本协议
- 不写 runtime 自动化
- 不改现有 `work capsule`

### 建议推进方式

- 只靠 Agent prompt / skill / 协议要求
- 在两个边界试跑：
  - compact 前
  - `doi -> rvw` 前

### 不做什么

- 不新增 manager
- 不自动定时写盘
- 不把它变成第二份 summary 系统

### 切换到下一条线的条件

满足以下条件后，才允许进入 `pl0270`：

1. 有真实任务证明 `WORKSTATE` 能提高跨窗口续接质量
2. `WORKSTATE` 没和现有 runtime `work capsule` 打架
3. 团队确认它不是噪音文件

## 第三顺位：PL0270

### 为什么第三

它最容易把系统做脏。

一旦过早接进主链，就会长出：

- selector
- router policy
- cache
- graph manager
- 主链 fallback 编排

所以必须晚于：

- deterministic feedback 地基
- workstate 试跑验证

### 当前阶段只做什么

只允许做成：

**独立 Unix 小工具 / Sidecar CLI**

形态建议：

- `ts-graph-cli`
- 或同级外置 CLI

输入：

- task / diff / stack trace

输出：

- 最该看的文件集合
- 推荐测试集合
- 选择理由

### 不做什么

- 不接进 listener
- 不接进 context-policy
- 不作为硬依赖
- 不做仓库内大平台

### 切换到下一条线的条件

满足以下条件后，才允许讨论是否进入 `pl0262`：

1. Sidecar CLI 独立可用
2. 至少能稳定输出“最该看的 3-10 个文件”
3. 在脱离主链耦合的情况下已证明有价值

## 第四顺位：PL0262

### 为什么最后

这是交付线，不是认知线。

它和前三条线属于不同问题域：

- 前三条是在增强 agent 的工作能力
- 这条线是在增强产品的交付形态

混做的结果只会是：

- 运行时和打包链互相打断
- App/UI 需求吞掉 agent 主线脑带宽

### 当前阶段只做什么

- 保留为独立战略线
- 只做任务拆分和调研，不进入主实现节奏

### 不做什么

- 不和 `pl0267/pl0259/pl0270` 混在一个版本里推进
- 不在当前主分支上同时开打包、UI、模型后端重构

## 并发纪律

### 允许的并发

当前阶段最多：

- 1 条主实施线
- 1 条轻量文档/研究线

示例：

- 可接受：
  - `pl0267` 主实施
  - `pl0262` 只做文档调研

- 不可接受：
  - `pl0267` + `pl0259` + `pl0270` 同时进实现

### 暂缓纪律

当上一条线没有满足切换条件时：

- 下一条线只能停留在 `tdo`
- 不拆实现子单进入 `doi`

## 当前推荐动作

如果现在开始真正施工，建议只做这一条：

1. 执行 `pl0267`

同时只保留一个低成本旁路：

2. `pl0259` 仅做文本协议试跑，不写 runtime 自动化

其余：

3. `pl0270` 保持 plan 态
4. `pl0262` 保持战略调研态

## 判断口径

如果后面感觉系统又开始乱，先问这三个问题：

1. 当前是不是同时开了两条以上重量级实现线
2. 当前做的事情是不是在增强主链闭环，而不是扩新能力面
3. 当前要做的线，是否真的依赖上一条线已跑通

任一答不上来，就先停，不继续并发。
