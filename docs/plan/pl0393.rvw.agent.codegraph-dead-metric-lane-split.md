# Plan

## Decision

`dead rate` 不再单独作为仓库健康结论。后续默认同时展示：

1. `dead-unresolved`
2. `dead-entry`
3. `dead-leaf`

并且至少分两层：

1. `src/**` 主产品代码
2. `scripts/`、`features/`、`AIDOCS/` 等辅助/参考目录

## Current Evidence

当前 codegraph 统计已经说明单一 `dead rate` 会失真：

- `dead-entry`: 43
- `dead-leaf`: 3044
- `dead-unresolved`: 3396

其中 `dead-leaf` 与 `dead-unresolved` 的前排样本明显被辅助目录污染：

- `features/step_definitions/**`
- `features/support/**`
- `scripts/**`
- `ui-protype/**`

这意味着：

1. `dead-entry` 更适合作为主产品入口图缺口信号
2. `dead-leaf` 先看是否只是终端节点，不默认等于死码
3. `dead-unresolved` 先看建图闭合度和目录噪声，不直接给产品健康结论

## Reporting Shape

后续汇报默认拆三栏：

1. `src/**` 的 `dead-entry`
2. `src/**` 的 `dead-leaf` / `dead-unresolved`
3. `scripts/`、`features/`、`ui-protype/`、`AIDOCS/` 的辅助噪声区

解释口径固定为：

1. `dead-entry` = 潜在动态入口缺边
2. `dead-leaf` = 终端节点，需二次判断是否真死码
3. `dead-unresolved` = 图未闭合或解析噪声

## Deliverables

1. 一份最小的 graph 健康口径说明
2. 一份后续汇报模板
3. 一份目录分层建议
4. 一条可重复执行的报表入口：`node --import tsx scripts/codegraph-health-report.ts --workspace . --format text`

## Verify

- `npm run docs:check`
- `node --import tsx scripts/codegraph-health-report.ts --workspace . --format text`

# Notes

- 这张 plan 只负责统计口径，不负责实现 codegraph 新能力
- 后续落地入口已收口在 `scripts/codegraph-health-report.ts`，只调用现有 `codegraph` CLI，不新增 runtime 控制面
