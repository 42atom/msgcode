# PL0290：WPKG 可安装能力包基线

## 结论

第一版 `wpkg` 不需要是复杂新格式。

先直接收成：

- 一个带 `manifest.json` 的 zip 包

它的职责也只做三件事：

1. 给主界面增加 pack / site 入口
2. 给 agent 增加 skill 发现面
3. 给未来授权与商业留下稳定字段

一句话：

- 先做可安装能力包
- 不先做市场

## 物理形态

第一版 `wpkg` 默认就是 zip 包，只是后缀和语义是：

- `*.wpkg`

解压后最小目录结构建议为：

```text
company-finance.wpkg
├── manifest.json
├── web/
├── skills/
└── assets/
```

当前不强制要求：

1. 自定义二进制格式
2. postinstall 脚本
3. 远程依赖下载
4. 本地常驻服务

## Manifest 最小字段

第一版只定义这些字段：

1. `id`
2. `name`
3. `version`
4. `author`
5. `sites`
6. `skills`
7. `requires`
8. `commercial`
9. `licenseType`

其中：

- `commercial`
  - 只是给未来商业留位
- `licenseType`
  - 只是给未来授权留位

当前不进入任何支付实现。

## 安装后真相源

安装 `wpkg` 后，不直接让主界面扫包目录。

统一写两份注册表：

1. `<workspace>/.msgcode/packs.json`
2. `<workspace>/.msgcode/sites.json`

### packs.json

主界面用来展示：

- 内置包
- 用户安装包
- 包版本
- 包作者
- 是否启用

### sites.json

主界面用来展示：

- 可打开的基座 / sidecar 站点入口

这意味着：

- 主界面只读注册表
- 不直接解析每个包的内部目录

## Agent 发现口径

第一版也不要让 agent 直接扫包目录。

安装时应把 skill 发现面显式写入注册表或索引：

1. pack 内的 skill 路径
2. pack 的说明信息
3. pack 的能力范围

主链应是：

`wpkg -> 解压 -> 注册 packs/sites/skills -> 主界面 / agent 读取`

而不是：

`主界面或 agent 运行时自己去猜目录`

## 与主界面的关系

主界面只做三件事：

1. 显示已安装 packs
2. 显示 packs 带来的站点入口
3. 显示 pack 的基本状态

主界面不负责：

1. 承载包的业务建模
2. 执行包内 skill
3. 解释包内部资源结构

## 与基座业务的关系

`wpkg` 提供的是业务能力单元，不是 UI 主体。

所以必须坚持：

- 包的业务设计与主界面解耦
- 主界面只做安装、入口和可见性
- 业务内容仍在 pack 自带的 Web / skill / assets 里

## 第一版明确不支持

1. 包市场
2. 支付
3. 分成
4. 云端包下载
5. 任意安装脚本
6. 任意二进制注入

## 推荐下一步

如果以后进入实现，顺序应该是：

1. `0295`：先冻结 `packs.json` 与主界面 pack 可见面
2. `WPKG` 协议继续保持极薄
3. 先补 `hall.packs` 和 `sites`
4. 再做安装器

不要反过来先写安装器。
