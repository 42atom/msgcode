# RP0291：Neighbor / Coworker 模块反方审视

## 结论

`pl0291` 的方向是对的，但第一稿确实有长成“第二协作平台”的风险。反方审视后，当前最关键的收紧已经吸收进 plan。

## 反方指出的主要问题

### 1. 第一阶段 scope 过大

第一稿把发现、联系人、异步消息、协作线程、owner、artifact、聊天记录视图、export surface 都塞进了第一阶段。这样第一步就会从“邻居模块”滑成“轻量协作系统”。

已吸收的修正：

- 第一阶段收缩成：
  - 发现
  - 联系人三态
  - 异步邮箱
- 线程、owner、artifact、聊天记录视图、export surface 接入全部下放到第二阶段

### 2. `thread` 和 `owner` 会长成第二任务系统

第一稿里已经出现了 `claim / handoff / review / owner` 这类词，但没有把它和 `issues/` 主线彻底切开。这样后续实现时很容易把线程误做成轻量任务系统。

已吸收的修正：

- 明确 thread 只是消息容器，不是任务真相源
- thread 不能直接推进 `issues/` 状态
- 后续如果引入 owner，它也只是线程当前响应方，不是 `issues/` owner

### 3. `export surface` 太虚

第一稿虽然说“不共享完整 workspace”，但没有写死 export surface 的最小合同。那后面就很容易滑成任意远程查询、任意远程问答、任意远程工具调用。

已吸收的修正：

- 第一版把 export surface 钉成结果导出面
- 只允许：
  - 摘要
  - 报表
  - 截图
  - 文件
  - 结构化数据块
- 明确禁止：
  - 任意远程问答
  - 任意远程查询
  - 任意远程工具调用

### 4. 发现、识别、联系人混在一起

第一稿里“发现、握手、升级为联系人”说得太快，实际上把 reachability、identity、trust 三件事揉成了一个状态。

已吸收的修正：

- 补了三态：
  - `discovered`
  - `known`
  - `contact`

这样联系人不再是垃圾桶概念。

### 5. 主线索引有口径自撞

`pl0211` 还写着“issues 下已经没有 tdo 或 rvw”，但当前其实已经重新有一批 `tdo` 主线。这个不是架构问题，但会影响外部审阅的可信度。

已吸收的修正：

- 改成“历史旧主线已基本收口，当前重新进入新一批 `tdo` 定义阶段”

## 当前有效计划

当前 `pl0291` 已收成一条更薄的路线：

第一阶段只做邻居模块最小地基：

- 模块开关
- 节点发现
- 联系人三态
- 异步邮箱
- 离线容忍

第二阶段再进入协作面：

- 线程
- patch/review/handoff
- artifact 引用
- 聊天记录式查看
- export surface 的结果导出接入

第三阶段才评估是否进一步收进主能力。

## 判断

现在这条线更像一个可删的、局域网内的联系人与异步邮箱模块，而不是协作平台。这个收紧方向是对的，已经适合拿去给 Opus 和 Gemini 做外部审阅。
