# Three Pillar Next Requirements Map

## 定位

这份文档只回答一件事：

- 当前这轮主线收口后，下一阶段需求该怎么排

这份文档不做三件事：

- 不创造任务
- 不承载状态真相
- 不替代 `issues/`

## 当前阶段结论

当前主线已经收成：

- `tk0204(+tk0213)`：heartbeat/doc-first Phase 1 闭环已成立，并进入 `rvw`
- `tk0207`：session continuity 地基已进入 `pss`
- `tk0208`：memory truth/index 边界已进入 `rvw`

一句话：

- 这一轮已经把“先把主链做真”走到该停的地方了

所以下一阶段不该继续散打补丁，而该按三大板块重新开需求。

## 一、资源管理

### 目标

- 让主脑稳定拿到：
  - `LLM`
  - 文件系统
  - tools
  - memory

### 下一阶段最该做

1. 把 memory 正式收成：
- `append` 是真相
- chunk / summary / sqlite 只是派生层

2. 把当前审计结果改成最小实现单：
- `sha256`
- dirty detection
- debug/stats 可见性

3. 收口工具面：
- 清掉不必要 wrapper
- 明确哪些工具是正式主链，哪些只是候选能力

### 暂不做

- 不先重写整套 memory
- 不先把 vector / hybrid 包装成“已正式可用”
- 不先做大而全资源控制面

### 当前入口

- `tk0228`
- `tk0229`
- `tk0230`
- `tk0231`

## 二、调度系统

### 目标

- 让主脑稳定完成：
  - `cron / wake`
  - heartbeat
  - 分步执行
  - `dispatch / subagent`

### 下一阶段最该做

1. 把 `tk0204` 的后续工作正式拆相
- Phase 1 停在 `rvw`
- `alarm/cron`、`reflection` 另起新单

2. 把 `tk0207` 停在地基层
- 后续若继续，必须拆成新单
- 不再把 transcript / memory / compaction 一锅端进来

3. 让调度主链继续保持单一：
- 文件真相
- 薄 runtime
- 不新加中间裁判层

4. 吸收最值钱的 review 缺口：
- 补自动验证闭环
- 但只补验证命令和结果合同
- 不做 verifier 平台

### 暂不做

- 不先平台化 session
- 不先引入更重 orchestrator
- 不把恢复系统扩成第二套任务系统

### 当前入口

- `tk0204`
- `tk0213`
- `tk0207`
- `tk0232`

## 三、I/O 系统

### 目标

- 让系统自然接收世界输入，也能把结果自然送出去

### 下一阶段最该做

1. 把当前已有出口重新排优先级：
- Feishu
- browser / desktop
- mail
- voice

2. 定一个很薄的 I/O 原则：
- 只管输入输出契约
- 不在 I/O 层偷做任务编排

3. 让多模态需求回到统一口径：
- 输入是事件
- 输出是可验证交付

### 暂不做

- 不先做大一统消息中台
- 不先做“所有通道统一抽象层”
- 不为了完整感补厚 I/O 编排

### 当前入口

- `tk0226`
- `tk0214`
- `tk0215`
- `tk0218`

## 排序原则

下一阶段默认按这条顺序推进：

1. 先收当前 `rvw/tdo` 主链，不让旧债挂着
2. 再把新需求归到三大板块
3. 每次只开最薄一刀，不并发开大面

硬标准：

- 能落 issue 就落 issue
- 能靠现有主链解决，就不加层
- 能说清“为什么现在做”，才开新单

## 一句话

下一阶段不再按“哪里热修哪里”推进。

下一阶段按三大板块推进：

- 资源管理先收 memory 和工具边界
- 调度系统先正式收掉 `tk0204`
- I/O 系统已开第一张现役主线：先收 Feishu 的事件输入与交付输出边界
