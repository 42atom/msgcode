# Three Pillar Next Requirements Map

## 定位

这份文档只回答一件事：

- 当前这轮主线收口后，下一阶段需求该怎么排

这份文档不做三件事：

- 不创造任务
- 不承载状态真相
- 不替代 `issues/`

## 当前阶段结论

当前主线已经收成：

- 调度主线：`wake / heartbeat / work capsule / vitals / session continuity` 已有 `pss` 基线
- 资源主线：`memory append/index/summary/share/sqlite` 已有 `pss` 基线
- I/O 主线：`Feishu event / live receipt / inbox thin surface` 已有 `pss` 基线

一句话：

- 这一轮已经把三大板块的第一层主链都做真了

所以下一阶段不该回头补历史单，而该只按新事实开薄单。

## 一、资源管理

### 目标

- 让主脑稳定拿到：
  - `LLM`
  - 文件系统
  - tools
  - memory

### 下一阶段最该做

1. 只在出现新失败时继续开单：
- memory 读写一致性退化
- 工具暴露面再次漂移
- workspace 资源边界出现新物理失败

2. 新单默认做薄：
- 直接围绕文件真相源
- 不再重开 memory 平台或资源控制面

### 暂不做

- 不先重写整套 memory
- 不先把 vector / hybrid 包装成“已正式可用”
- 不先做大而全资源控制面

### 当前入口

- 暂无现成 `tdo`
- 若再开，优先从真实故障反推新 issue

## 二、调度系统

### 目标

- 让主脑稳定完成：
  - `cron / wake`
  - heartbeat
  - 分步执行
  - `dispatch / subagent`

### 下一阶段最该做

1. 让调度主链继续保持单一：
- 文件真相
- 薄 runtime
- 不新加中间裁判层

2. 只在新场景出现时继续开薄单：
- 新的 wake 来源
- 新的 heartbeat 物理边界
- 真实 session continuity 漂移

### 暂不做

- 不先平台化 session
- 不先引入更重 orchestrator
- 不把恢复系统扩成第二套任务系统

### 当前入口

- 暂无现成 `tdo`
- 若再开，必须直接指向可复现 runtime 事实
- 最近样例：
  - `tk0247`：真实 schedule job 因历史 `agentPrompt` payload 漂移失败，按日志事实补了读路径兼容
  - `tk0248`：真实 provider 400 `tool call and result not match`，按 replay 配对事实补了薄修复
  - `tk0249`：真实 heartbeat `require is not defined`，按单点 ESM 漂移补了薄修复
  - `tk0251`：真实 provider 400 `context window exceeds limit`，按 tool_result replay 前的大 previewText 放大量补了硬截断
  - `tk0252`：真实 Feishu smoke 里 large `read_file` 只给头部预览，按 head+tail 合同补了薄修复
  - `tk0253`：`tk0252` 后续 smoke 里尾行题仍会 `no-tool/只回 token`，按 `lastNonEmptyLine + 先读后答` 合同继续收口
  - `tk0254`：真实 family 图片后续追问漂移，按“复用最近图片事实 + 禁止对图片本体 read_file”合同补了薄修复
  - `tk0255`：真实 family 图片首轮偏航时 `derived unavailable reason` 丢失，按“图片规则前置到本轮上下文 + reason 进日志与上下文”补了薄修复

## 三、I/O 系统

### 目标

- 让系统自然接收世界输入，也能把结果自然送出去

### 下一阶段最该做

1. 把当前已有出口按真实需求排优先级：
- Feishu
- browser / desktop
- mail
- voice

2. 定一个很薄的 I/O 原则：
- 只管输入输出契约
- 不在 I/O 层偷做任务编排

3. 让多模态需求回到统一口径：
- 输入是事件
- 输出是可验证交付

### 暂不做

- 不先做大一统消息中台
- 不先做“所有通道统一抽象层”
- 不为了完整感补厚 I/O 编排

### 当前入口

- 暂无现成 `tdo`
- 若再开，优先来自真实通道需求，而不是为完整感补渠道

## 排序原则

下一阶段默认按这条顺序推进：

1. 先确认是新事实，不是历史计划残影
2. 再把需求归到三大板块
3. 每次只开最薄一刀，不并发开大面

硬标准：

- 能落 issue 就落 issue
- 能靠现有主链解决，就不加层
- 能说清“为什么现在做”，才开新单

## 一句话

下一阶段不再按“哪里热修哪里”推进。

下一阶段按三大板块推进：

- 资源管理只在出现新故障时再开薄单
- 调度系统只在现主链失真时再开薄单
- I/O 系统只在真实通道需求出现时再开薄单
