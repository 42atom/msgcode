# RF0002：Appliance Required Preinstall Software

## 结论

对 appliance / App 而言，基础运行依赖只能有两种归宿：

1. 随包携带
2. 预安装软件必须清单

不存在第三种“运行时再碰运气”。

## 适用边界

这份清单服务：

- appliance 安装器
- doctor
- 交付清单
- 企业现场部署

它不服务：

- 开发者 bootstrap
- 本地开发增强工具
- 尚未收口的实验性能力

## 纳入规则

只有同时满足下面两条，才允许进入这份清单：

1. 不随包携带
2. 缺失后会直接打断正式主链

如果某项依赖可以改为随包，就优先随包；不要偷懒把一切都塞进预安装清单。

## 执行规则

这份清单一旦引用到某条安装主链，就必须同时满足：

1. install / doctor 前置检查
2. 缺失时 fail-closed
3. 运行时不再把缺失伪装成 AI 能力问题

一句话：

- 缺失必须在安装面暴露，不准拖到执行面才发现。

## 当前已纳入的项

### Core Appliance 基础项

1. 托管 Bash
   - 条件：App 未随包提供 Bash runtime
   - 当前正式合同路径：
     - `/opt/homebrew/bin/bash`
     - `/usr/local/bin/bash`

2. `zstd`
   - 条件：App 继续沿用当前 `.wpkg -> tar.zst` 安装主链，且未随包提供解压 runtime

### Optional Pack 基础项

1. Browser pack
   - `Google Chrome / Chromium`

2. Desktop pack
   - `ghost-os`

3. tmux coding pack
   - `tmux`

这些项不属于 Core Appliance 默认基础项；只有用户启用对应能力包时，才进入那台机器的检查范围。

## 当前明确不纳入的项

这些要么应该随包，要么只是开发者增强，不进入“必须预装”：

1. `Node.js`
2. `tsx`
3. `npm`
4. `uv`
5. `bun`
6. `rg`
7. `fd`
8. `jq`
9. `fzf`
10. `bat`
11. `eza`

## 暂缓冻结的项

下面这些依赖还没完成产品边界收口，现在不该抢先写死成“必须预装”：

1. `LM Studio / OMLX`
2. `mlx-whisper`
3. 其他本地模型服务

原因不是它们不重要，而是它们未来到底属于：

1. Core Appliance
2. Optional Pack
3. 不承诺交付

当前还没有定。

## 使用方式

如果在写：

- install / doctor
- appliance 交付文档
- 企业部署清单

优先引用这份参考，不要再各自维护一份名单。

如果未来某项依赖改成随包，应该从这份清单删除，而不是保留成历史残影。
