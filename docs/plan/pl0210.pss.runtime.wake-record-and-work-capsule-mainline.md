# Wake Record And Work Capsule Mainline

## 问题

我们已经确认了一件事：

- `openclaw` 很会把 AI 准时叫醒
- 但它给未来自己的主合同，仍然主要是一段文本提醒

这对提醒系统够用，对长期工作主脑不够。

当前 msgcode 真正要解决的是：

- 定时器到点后，不能只给模型一句“现在 8:40 了，去总结新闻”
- 系统必须把“这是谁的活、做到哪了、下一步是什么、缺什么证据”一起带回来

所以这一段主线不是“抄 openclaw 的 cron”，而是：

- 学它的唤醒分层
- 用我们的文件真相源和 work capsule 把唤醒做硬

## 结论

下一段实现按这条主链落：

1. `wake job`
2. `wake record`
3. `work capsule`
4. `heartbeat consume`
5. `startup catch-up`
6. `vitals policy`

一句话：

- `alarm` 负责到点
- `wake record` 负责留合同
- `work capsule` 负责恢复工作面
- `heartbeat` 负责真正推进

## 向 openclaw 学什么

## 1. 学 heartbeat / cron 分层

这一刀是对的，直接借。

- `heartbeat` 负责周期巡检
- `cron/alarm` 负责精确唤醒
- 二者不要混成一个“大自动化循环”

这部分已经和 [pl0204.pss.runtime.heartbeat-alarm-and-reflection-mainline.md](/Users/admin/GitProjects/msgcode/docs/plan/pl0204.pss.runtime.heartbeat-alarm-and-reflection-mainline.md) 一致。

## 2. 学 wakeMode

保留最小二分：

- `now`
- `next-heartbeat`

语义：

- `now`: 到点后立刻尝试消费
- `next-heartbeat`: 只投递唤醒，等下一次周期 heartbeat 处理

不继续加第三第四种 mode。

## 3. 学 session target 的边界意识

不照搬命名，但借它的边界判断。

对 msgcode 来说，先收成两类：

- `run path`
- `task path`

必要时内部再映射到：

- `main`
- `isolated`
- `session:<id>`

但用户口径不暴露一堆 target 术语。

## 4. 学 light context

所有定时唤醒都不该默认背全量 bootstrap。

最小原则：

- 轻路径只读最少提示
- 重路径优先读 `work capsule`
- 不让 heartbeat 因为上下文太胖而变成新的负担

## 5. 学 startup catch-up

这点必须借，而且要更硬。

系统启动后至少要做：

- 清 stale running marker
- 扫 overdue wake jobs
- 扫已到点但未消费的 wake record
- 扫 unfinished dispatch / checkpoint

## 不向 openclaw 学什么

## 1. 不学“文本提醒即合同”

我们不能把未来工作的主合同，简化成一段 reminder text。

原因很直接：

- 文本只能提醒
- 不能描述恢复点
- 不能稳定承载证据要求
- 不能稳定承载 artifact 集

所以文本只能做：

- `hint`

不能做：

- `truth source`
- `recovery contract`

## 2. 不学主会话 pending event 纯内存

只要 pending wake 还是纯内存，一次意外重启就可能丢：

- 已经到点
- 但还没被 heartbeat 消费

这条在长期工作系统里不能接受。

所以 msgcode 的 wake 必须文件化。

## 3. 不学 session 作为恢复地基

session 可以帮助自然续聊，但不能承担：

- 还欠什么
- 做到哪
- 从哪续

这些必须由 `issues + dispatch + checkpoint + work capsule` 回答。

## msgcode 的最小设计

## 1. Wake Job

`wake job` 只回答：

- 什么时候该叫醒

建议文件落点：

- `<workspace>/.msgcode/wakeups/jobs/*.json`

最小字段：

- `id`
- `kind`
- `schedule`
- `mode`
- `taskId`
- `enabled`
- `createdAt`
- `updatedAt`

说明：

- `kind` 区分 `once` / `recurring`
- `mode` 只允许 `now` / `next-heartbeat`
- `taskId` 可为空，允许轻路径提醒

## 2. Wake Record

`wake record` 只回答：

- 这次已经到点的唤醒，具体是为了什么

建议文件落点：

- `<workspace>/.msgcode/wakeups/records/<wakeId>.json`

最小字段：

- `id`
- `jobId`
- `status`
- `path`
- `taskId`
- `hint`
- `scheduledAt`
- `claimedAt`
- `completedAt`
- `failedAt`
- `latePolicy`

状态建议：

- `pending`
- `claimed`
- `done`
- `failed`
- `expired`

这层是真相，不再靠内存事件队列补脑。

### Wake Record 生命周期

`wake record` 必须可恢复，但不能无限堆积。

最小策略：

- `pending / claimed`：永不自动删，直到进入终态
- `done / failed / expired`：进入可回收集合
- 定期回收时机：
  - startup catch-up 之后
  - 每日一次 housekeeping heartbeat
- 默认保留窗口：
  - 终态记录保留 `7d`
  - 若存在失败证据或人工追查标记，可延长到 `30d`

最小实现不做数据库，不做集中式 GC。

建议做法：

- 原地保留终态 JSON 到保留期结束
- 保留期后直接删
- 若后续确实需要长期审计，再单独加归档目录，不在第一版先做

一句话：

- `wake record` 是短期工作真相，不是永久日志仓库

### Wake Poison 与 Incident 升级

`wake record` 不能在 `claimed -> stale -> reclaim -> 再试` 里静默循环。

最小补充字段：

- `reclaimCount`
- `lastFailureCode`
- `lastFailureAt`
- `lastFailureSummary`

最小规则：

- 每次 stale reclaim 或执行失败，都要推进失败记忆
- 连续 reclaim / failed 超过阈值（默认建议 `3`）后，不再无限自动重试
- 该 record 直接进入 `failed`
- 同时落一份显式报告到 `AIDOCS/reports/incidents/`

这不是事故平台，只是把“死 record”从静默循环升级成可见事实。

## 3. Work Capsule

`work capsule` 只回答：

- 醒来以后，这份工作现在做到哪了

它不是人工维护文件，而是 runtime 现组装的恢复胶囊。

建议最小结构：

- `taskId`
- `phase`
- `checkpoint`
  - `summary`
  - `nextAction`
  - `artifactRefs`
  - `evidenceRefs`
- `activeDispatch`
  - `subtaskIds`
  - `blockedBy`
  - `subagentRefs`
- `nextAction`
  - `type`
  - `params`
  - `evidenceRequired`
- `wake`
  - `id`
  - `hint`
  - `scheduledAt`
- `vitalsSnapshot` (optional)
  - `load`
  - `risk`
  - `policyMode`

输入来源：

- `issues/*.md`
- `.msgcode/dispatch/*.json`
- task runtime `checkpoint`
- artifact / verify evidence
- 当前 `wake record`

### Work Capsule 一致性

`work capsule` 不是持久化真相，只是一次恢复投影。

所以它必须带 source stamp，防止“真相已变、胶囊未变”。

最小字段补充：

- `sourceStamp`
  - `taskCheckpointUpdatedAt`
  - `dispatchUpdatedAt[]`
  - `wakeRecordUpdatedAt`
  - `issueStateNames[]`

规则：

- `work capsule` 只在真正消费前现组装
- 若生成后到底层执行前发生明显延迟，先重校验 `sourceStamp`
- 若 source stamp 不一致，丢弃旧 capsule，重新组装

这条是硬约束：

- 不允许把过期 capsule 当真相继续执行
- 不允许用旧 `nextAction` 覆盖新的 task / dispatch 真相

### Revision Lock 裁决

`sourceStamp` 不只是提示字段，它要有最终裁决口径。

最小规则：

- 执行前：必须重校验 `sourceStamp`
- 若执行前已漂移：直接丢弃旧 capsule，重新组装
- 执行中若尚未产生副作用：允许中断并回到重组
- 执行中若已产生副作用：不强行继续写，不采用“最后写入者胜”
- 已产生副作用且发现漂移时：进入 `blocked / conservative`，把冲突写回 checkpoint，等待下一轮显式处理

一句话：

- 执行前强一致
- 执行中保守降级

## 4. Heartbeat Consume

heartbeat 不是自己去“猜今天该做什么”。

它的消费顺序应该固定：

1. 先看 pending wake record
2. 再看当前 workspace 未完成任务
3. 若命中 `taskId`，runtime 先组装 `work capsule`
4. 再把 `wake record + work capsule` 交给模型
5. 执行结果回写 dispatch / checkpoint / evidence

无事时：

- `HEARTBEAT_OK`

有事时：

- 做真实推进，不生成虚假回报

### Claim 原子性

`wake record` 如果没有 claim 机制，就会出现双消费者抢同一张单。

最小原则：

- 同一条 wake record，一次只能有一个活动消费者
- claim 必须是原子的
- claim 失败方只能重试或跳过，不能并行消费

最小实现建议：

- 在 `<workspace>/.msgcode/wakeups/claims/` 下为每条 wake 建一个 claim 文件
- claim 动作用原子创建：
  - Node `open(..., \"wx\")`
  - 或原子 `mkdir`
- claim 成功后，才能把 record 从 `pending` 写成 `claimed`
- claim 文件内容至少带：
  - `wakeId`
  - `owner`
  - `claimedAt`
  - `leaseUntil`
  - `safetyMarginSec`

恢复规则：

- 发现 `claimed` 但 `now > leaseUntil + safetyMargin`，才可进入 reclaim
- reclaim 前必须重读 record 状态与 source stamp
- reclaim 只针对 stale claim，不抢活跃 claim

默认建议：

- `safetyMarginSec = 10`

理由：

- 吸收 Node/runtime 调度抖动
- 吸收极小的系统时钟边界误差
- 防止 lease 边缘时刻误抢

这条和 `tk0205` 的 `single-writer` 一致：

- 第一版默认仍以单 workspace 单活动写入者为主
- claim 文件只是局部防撞，不引入中心调度器

## 5. Startup Catch-Up

启动恢复要做成正式步骤，不做 best effort 碰运气。

最小步骤：

1. 清 stale running marker
2. 扫 `wakeups/records` 里所有 `pending/claimed`
3. 找出 overdue 但未完成的 wake
4. 若 `latePolicy = run_if_missed`，进入待消费队列
5. 若 `latePolicy = skip_if_missed`，标记 `expired`
6. 对仍未完结的 task / dispatch 重建 `work capsule`
7. 回收超过保留期的终态 wake record
8. 检查 stale claim，必要时 reclaim
9. 若发现 poison wake，升级为 incident 并停止静默重试

## 6. Vitals Policy

`vitals` 不创造唤醒，只影响怎么消费已有 wake。

最小规则：

- `load` 高 -> 不开新坑，只收口
- `stall` 高 -> 先排障，不派新子任务
- `risk` 高 -> 先 verify / confirm
- `headroom` 低 -> 只做 checkpoint / reflect

如果 `vitals` 计算失败：

- 进入保守模式
- 不派新子任务
- 不开启探索型 wake

## 文件布局建议

```text
<workspace>/.msgcode/
  HEARTBEAT.md
  wakeups/
    jobs/
      *.json
    records/
      *.json
  dispatch/
    *.json
  subagents/
    *.json
  runtime/
    vitals.json
```

原则：

- job 是计划
- record 是本次触发事实
- capsule 是运行时投影

## 轻路径 / 重路径接线

## 轻路径

适用：

- 提醒
- 单次检查
- 问完即走

策略：

- 可没有 `taskId`
- 只有 `wake record + hint`
- 不强制建 `work capsule`

## 重路径

适用：

- 多轮推进
- 跨 session 续跑
- 需要交付

策略：

- 必须绑定 `taskId`
- 必须可组装 `work capsule`
- 无 capsule 不进入正式执行

## 实现顺序

## 第一刀

落 `wake job / wake record` 文件协议。

验收：

- 定时唤醒不再只存在内存
- 重启后能知道“哪些 wake 已到点但未消费”

## 第二刀

把 `wake -> work capsule` 接进 `tk0205` 的恢复协议。

验收：

- 有 `taskId` 的 wake 可以直接恢复到下一步动作
- 不需要模型自己翻所有文件考古

## 第三刀

把 `heartbeat` 改成优先消费 wake record，而不是泛泛巡检。

验收：

- heartbeat 从“定期看看”升级成“按合同推进”

## 第四刀

接 `vitals policy`。

验收：

- 高压下会 defer / degrade
- 但不改文件真相源

## 验收口径

1. 仓库内明确存在 `wake job / wake record / work capsule` 三层语义。
2. 到点唤醒不再只依赖一句文本提醒或纯内存事件队列。
3. 重路径任务在 wake 后可直接恢复到显式 `nextAction`。
4. 系统重启后能识别 overdue wake，并按 `latePolicy` 处理。
5. 同一条 wake record 不会被两个消费者同时 claim 成功。
6. `work capsule` 带 source stamp，底层真相变化时必须重建，不允许执行过期胶囊。
7. 终态 wake record 有明确回收窗口，不形成文件型内存泄漏。
8. `heartbeat` 负责消费 wake 合同，不直接变成第二任务状态系统。
9. `vitals` 只影响消费策略，不抢真相源。

## 一句话收口

我们不是要把 AI “叫醒一下”。

我们要做的是：

- **把未来的自己叫醒时，连同这份工作做到哪一起带回来。**
