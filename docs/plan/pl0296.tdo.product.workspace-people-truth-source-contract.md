# PL0296：工作区人物真相源合同

## 结论

第一版只做工作区级人物簿。

不要：

- 全局人物中心
- 自动匹配推荐
- 多工作区人物复用

先只把这三样钉住：

- 人物表
- 待关联渠道人物
- 主界面人物数量

## 人物表

建议文件：

- `<workspace>/.msgcode/people.json`

只收最小字段：

1. `id`
2. `title`
3. `tags`
4. `note`
5. `channels`

其中：

- `title`
  - 对应 UI 里的统一称谓
- `tags`
  - 自由标签
- `note`
  - 备注
- `channels`
  - 渠道身份列表

`channels[]` 第一版固定为：

- `channel`
- `channelId`
- `username`
- `displayName`

第一版不做：

- 真实姓名
- 组织树
- 自动关系推断

## 待关联渠道人物

建议文件：

- `<workspace>/.msgcode/people-pending.json`

只收最小字段：

1. `channel`
2. `channelId`
3. `username`
4. `displayName`
5. `seenAt`

这就够支撑：

- 待关联列表
- 选择真实人物
- 新建人物
- 忽略

并且固定：

- `channel + channelId` 是第一版唯一关联键
- `people-pending.json` 由导入侧维护，UI 只读

## 主界面人物入口

主界面只读：

- `people.json`

人数规则：

- 文件存在：显示条目数
- 文件不存在：不显示数字

不要同时去读 `pending`。

## people 页面读取

左侧通讯录：

- 读 `people.json`

右侧待关联：

- 读 `people-pending.json`

空态规则：

- 两个文件都不存在：显示空态 + 获取渠道人物入口
- 只有 `pending`：左侧空，右侧展示待关联
- 只有 `people`：右侧空，左侧展示通讯录

## 明确留后

这张单不解决：

1. 飞书成员拉取
2. 自动匹配
3. 全局人物底册
4. 跨工作区共享人物

如果以后继续拆，下一刀应该是：

- `people pending ingestion from feishu`

不是先把人物系统做重。
