# RP0307：status.log CLI 与 prompt 切片

## 结论

这轮只补了两条主链入口：

- `msgcode status-log add`
- `msgcode status-log tail`

以及一条 prompt 合同：

- 先读最近几条共享工作状况
- 命中条件才追加一条
- 不手写原始日志行

## 本轮实现

新增：

- `src/cli/status-log.ts`
- `test/p7-tk0307-status-log-cli-and-prompt.test.ts`

更新：

- `src/cli.ts`
- `src/cli/help.ts`
- `prompts/agents-prompt.md`

## CLI 入口

这轮新增的命令是：

- `msgcode status-log add --workspace <workspace> --thread <线程名> --kind <decision|state> --summary <一句话> --ref-path <路径> --ref-line <行号> --json`
- `msgcode status-log tail --workspace <workspace> --json`

保持的边界：

- 直接复用 `src/runtime/status-log.ts`
- 不新起数据库
- 不新起 coordinator
- 不让模型自己 `echo >> status.log`

## prompt 接线

`agents-prompt.md` 现在明确了：

- 若当前工作区有 `status.log`，先读最近几条共享工作状况
- 只把它当共享近况，不当原始聊天历史
- 若本轮形成已确认决定或状态变化，才通过 `msgcode status-log add` 追加一条
- 禁止手写原始日志行

## 验证

本轮验证通过：

- `PATH="$HOME/.bun/bin:$PATH" bun test test/p7-tk0294-status-log-contract.test.ts test/p7-tk0307-status-log-cli-and-prompt.test.ts`
- `./node_modules/.bin/tsc --noEmit`
- `npm run docs:check`

## 刻意不做

这轮故意没做：

1. 自动 hook
2. 定时写入
3. 语义去重
4. UI 直接写状态
5. `status.log` 实时监控化

这条线仍然保持：

- 文件追加
- CLI 包一层
- prompt 决定何时写
