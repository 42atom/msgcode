# RP0299：workspace people 读取主链实现切片

## 结论

`0296` 第一版不新起 `people.json`。

直接复用现有：

- `.msgcode/character-identity/*.csv`
- `.msgcode/people-pending.json`

并通过：

- `msgcode appliance people --workspace <...> --json`

把这两份真相源只读暴露给 UI。

## 为什么改

原 `0296` 文档把人物真相源写成了 `people.json`。

但仓库现有更硬的证据是：

- `src/skills/runtime/character-identity/SKILL.md`

它已经明确规定：

- workspace-local CSV
- `channel + chatId + senderId` 这一层身份事实

所以第一版必须沿现有 skill 走，不能为 UI 再造一套人物中心。

## 本轮实现

新增：

- `src/runtime/workspace-people.ts`
- `test/p7-tk0296-appliance-people-contract.test.ts`

扩展：

- `src/cli/appliance.ts`
  - 新增 `appliance people`

当前输出：

- `people`
  - 原始身份记录数组
- `pending`
  - 待关联身份数组
- `counts.people`
- `counts.pending`

## 圆桌吸收

本轮圆桌最关键的意见是：

- runtime 不该把 CSV 再包装成 UI 人物模型

所以当前实现已经继续收薄：

- 不在 runtime 里生成 `title/tags/channels` 卡片结构
- 只解析并输出原始身份记录
- UI 以后自己映射

同时补了一条硬约束：

- `people-pending.json` 读面已经收成 `channel + chatId + senderId`
- 旧 pending 键不再接受

## 明确不做

这轮继续不做：

1. 写 CSV
2. 飞书拉取器
3. 自动匹配
4. 全局人物中心
5. 跨 CSV 合并成同一真实人物

## 结果

`0296` 现在已经有一条能站住的最小主链：

- 现有 skill CSV 是真相源
- pending JSON 是导入侧补充面
- UI 只读 `appliance people`

这符合当前原则：

- 克制
- Everything is a file
- 不为 UI 另起控制面
