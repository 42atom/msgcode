# RP0282：Appliance Delivery Manifest v1

## 结论

第一版 appliance 交付物必须只包含三层：

1. 随包携带的核心运行物
2. 首次启动时生成的本地真相源
3. 默认不装、后挂的能力包

一句话：

- 包里带主脑
- 首启生成身份和工作区骨架
- 外围能力以后再挂

## 1. 随包携带内容

第一版安装包内必须直接携带：

### A. 运行核心

1. 已构建的 `msgcode` 可执行入口
2. Node 运行时
3. `msgcode` 核心 JS 产物
4. 核心 npm 依赖产物

这里的目标是替代当前这条开发链：

- [bin/msgcode](/Users/admin/GitProjects/msgcode/bin/msgcode) 里对 `tsx` 和源码目录的依赖
- `npm install`
- `src/*.ts`

### B. 核心协议与模板

1. 默认 `SOUL` 模板
2. 默认 `ORG` 模板
3. 默认 workspace 初始化模板
4. 核心协议文档或内置模板资源

理由：

- 首次启动必须能直接初始化
- 不能依赖现场再去拷 README 样板

### C. 核心数据能力

1. memory 核心依赖
2. `sqlite-vec` 对应平台产物
3. 日志/证据写回所需核心依赖

理由：

- memory 不属于可选能力
- 它是 appliance 核心认知主链的一部分

### D. 最小诊断工具

1. `start`
2. `status`
3. `doctor`
4. memory index/search
5. 日志定位入口

理由：

- 没有最小诊断面，就没有可运维的 appliance

## 2. 首次启动生成内容

第一版首次启动时，只生成最少真相源。

### A. 全局配置根

位置：

- `~/.config/msgcode/`

首启至少生成：

1. `~/.config/msgcode/.env`
2. `~/.config/msgcode/log/`
3. `~/.config/msgcode/souls/default/SOUL.md`
4. `~/.config/msgcode/souls/active.json`
5. `~/.config/msgcode/memory/` 的索引目录骨架

### B. 工作区骨架

位置：

- `<workspace>/`

首启至少生成：

1. `<workspace>/.msgcode/SOUL.md`
2. `<workspace>/.msgcode/ORG.md`
3. `<workspace>/.msgcode/config.json`
4. `<workspace>/.msgcode/providers.json`
5. `<workspace>/memory/`
6. `<workspace>/AIDOCS/reports/`
7. `<workspace>/issues/`（若按交付策略由 appliance 自建默认工作区）

### C. 最小机构卡片

位置固定为：

- `<workspace>/.msgcode/ORG.md`

第一版只允许：

```md
# 机构信息

- 名称：<机构名称>
- 交税地：<税务归属地>
- 统一社会信用代码：<18位代码>
```

不允许首启默认生成：

1. 员工名册
2. 组织树
3. 财务口径
4. 风险建议
5. 投资建议
6. 健康信息

## 3. 默认不随包携带内容

这些内容第一版不进 Core Appliance 默认安装包：

### A. Feishu Pack

1. Feishu transport 启用态
2. Feishu live smoke
3. Feishu 凭据预置

原因：

- 它是外部运行依赖，不是核心自包含条件

### B. Browser Pack

1. Patchright
2. Chrome/Chromium
3. `connectOverCDP` 相关运行物

原因：

- 包体重
- 平台复杂
- 不该阻塞第一版 appliance 落地

### C. Desktop Pack

1. `ghost-os`
2. `ghost_*` 主链依赖

原因：

- 权限和平台问题重
- 不是第一版核心能力

### D. Tmux Coding Pack

1. `tmux`
2. `uv`
3. `bun`
4. `rg/fd/jq/fzf/bat/eza`

原因：

- 这是重任务/开发者能力包
- 不是每台 appliance 都该默认背上的依赖

## 4. 第一版安装后必须成立的事实

安装完成后，即使不装任何 pack，也必须能做到：

1. `msgcode` 能启动
2. 能读写 `SOUL.md`
3. 能读写 `ORG.md`
4. 能建立 memory 索引
5. 能写日志和证据
6. 能在本地推进 task/heartbeat 主链

如果这 6 件事做不到，就不算第一版 Core Appliance 成立。

## 5. 第一版明确不承诺的事实

安装完成后，不默认承诺：

1. Feishu 一定可用
2. 浏览器自动化一定可用
3. 桌面自动化一定可用
4. tmux coding lane 一定可用
5. 完全内网一定可用
6. 本地模型一定内置

这些都应通过 pack 或后续交付档位解决。

## 一句话结尾

第一版 appliance 的交付标准，不是“看起来像一个 App”，而是：

- 包里已经带齐主脑
- 首启能生成最小机构身份和工作区骨架
- 不靠现场下载也能稳定成立
