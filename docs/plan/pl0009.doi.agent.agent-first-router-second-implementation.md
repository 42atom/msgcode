# Agent-First / Router-Second 最小改造实施计划

Issue: 0009
Parent Plan: `docs/plan/pl0032.doi.agent.unix-mainline-refactor.md`

## 定位

这是 `0009` 的第一阶段子计划，只覆盖 `agent-first / router-second` 这一刀。
`0009` 的当前总 plan 以 `docs/plan/pl0032.doi.agent.unix-mainline-refactor.md` 为准。

## 目标

去掉前置主裁判式 route classifier，改为单一智能体先看到工具，再自己决定是否调用。

## 核心问题

当前主链流程：
1. 用户消息 → `classifyRouteModelFirst()` 前置分类
2. 分类结果为 "no-tool" → 模型根本没看到工具上下文
3. 只能靠 fake tool marker recover 到 tool-loop

这和 Codex 式"单一智能体先看到工具，再自己决定是否调用"不一致。

## 改造方案

### 1. routed-chat.ts 改造

**当前逻辑**（第 184-226 行）：
```typescript
// 先分类
const classifier = await classifyRouteModelFirst({...});
const route = classifier.classification.route;

// 再分流
if (route === "no-tool" || selectedLevel === "LEVEL_2") {
    // 直接调用 runLmStudioChat，模型没看到工具
} else if (route === "complex-tool") {
    // plan/act/report
} else {
    // tool loop
}
```

**改成**：
```typescript
// 默认不前置分类，直接进入统一 agent 主链
// 让模型在首轮就看到工具，自己决定是否调用

// 例外情况：degrade 模式强制 no-tool
if (selectedLevel === "LEVEL_2") {
    // 保留 degrade 模式的强制 no-tool
}

// 例外情况：complex-tool 保留 plan/act/report 流程
// 但需要用户提供显式标记或配置

// 默认：统一调用 runAgentToolLoop，让模型看到工具
const toolLoopResult = await runAgentToolLoop({
    prompt: options.prompt,
    tools: activeToolSchemas,
    toolChoice: "auto", // 关键：让模型自己决定
    ...
});

// 如果模型返回无 tool_calls，视为 no-tool 结果
if (toolLoopResult.toolCall === undefined) {
    return {
        answer: toolLoopResult.answer,
        route: "no-tool", // 最终执行结果，不是前置分类
        decisionSource: "model", // 新增字段
        ...
    };
}
```

**保留功能**：
- degrade 模式的强制 no-tool（LEVEL_2）
- complex-tool 的 plan/act/report 流程
- 日志字段（但调整语义）

**删除/降级**：
- `classifyRouteModelFirst()` 不再作为默认主入口
- `ROUTE_CLASSIFIER_SYSTEM_PROMPT` 不再作为默认主入口
- fake tool marker recover 逻辑（不再需要）

### 2. tool-loop.ts 改造

**当前状态**：
- 第 675 行已经支持 `toolChoice: preferredToolChoice ?? "auto"`
- 已经可以让模型自己决定是否调用工具

**需要调整**：
- 确保在无 tool_calls 时能正确返回 no-tool 结果
- 明确区分"模型决定不调用"和"调用失败"

**最小改动**：
- 在首轮无 tool_calls 时，直接返回文本结果
- 不再进入 "required 模式重试" 逻辑（第 685 行）
- 添加 `decisionSource: "model"` 字段

### 3. types.ts 改造

**修改 `AgentRoutedChatResult` 接口**：

```typescript
export interface AgentRoutedChatResult {
    answer: string;
    route: "no-tool" | "tool" | "complex-tool"; // 改为"最终执行结果语义"
    decisionSource: "model" | "router" | "degrade"; // 新增：决策来源
    temperature: number;
    toolCall?: { name: string; args: Record<string, unknown>; result: unknown };
    actionJournal: ActionJournalEntry[];
    verifyResult?: VerifyResult;
    // 配额与续跑信息（保持不变）
    continuable?: boolean;
    quotaProfile?: "conservative" | "balanced" | "aggressive";
    perTurnToolCallLimit?: number;
    perTurnToolStepLimit?: number;
    remainingToolCalls?: number;
    remainingSteps?: number;
    continuationReason?: string;
}
```

**语义变更**：
- `route` 不再表示"router 替模型预判出来的路线"
- 而表示"本轮最终走出的执行形态"
- `decisionSource` 表示决策来源：
  - `"model"`: 模型自己决定（agent-first）
  - `"router"`: router 强制决策（degrade/fallback）
  - `"degrade"`: 降级模式强制 no-tool

### 4. handlers.ts 改造

**当前状态**：
- 通过 `runAgentRoutedChat()` 走 pre-route 逻辑

**改成**：
- 仍然调用 `runAgentRoutedChat()` 这个名字
- 但其内部语义变成"统一 agent 主链"
- `handlers` 不需要感知 route classifier

**无需修改**：
- handlers 层不需要改动
- 只需要调整内部语义

### Step 3: routed-chat.ts - 去掉默认 pre-classifier

```typescript
// 1. 删除默认的 classifyRouteModelFirst() 调用
// const classifier = await classifyRouteModelFirst({...}); // 删除

// 2. 保留 degrade 模式的强制 no-tool
if (selectedLevel === "LEVEL_2") {
    // degrade 模式
    const answer = await runLmStudioChat({...});
    return {
        answer,
        route: "no-tool",
        decisionSource: "degrade",
        actionJournal: [],
    };

// 3. 默认：统一调用 runAgentToolLoop
const toolLoopResult = await runAgentToolLoop({
    prompt: options.prompt,
    tools: activeToolSchemas,
    toolChoice: "auto", // 关键
    ...
});

// 4. 根据 toolLoopResult 决定最终 route
if (toolLoopResult.toolCall === undefined) {
    // 模型决定不调用工具
    return {
        answer: toolLoopResult.answer,
        route: "no-tool",
        decisionSource: "model",
        ...
    };
} else {
    // 模型决定调用工具
    return {
        ...toolLoopResult,
        route: "tool",
        decisionSource: "model",
    };
}

// 5. 保留 complex-tool 流程（但需要显式标记）
if (options.forceComplexTool) {
    // plan/act/report
}
```

## 验收标准

### 必过行为

1. ✓ 用户发自然语言请求时，默认先进入"看得见工具的主智能体"
2. ✓ 模型可自行决定不调用工具，直接回答
3. ✓ 模型也可自行决定调用 browser/bash/read_file/...
4. ✓ 不再存在"先被判成 no-tool，再靠 fake tool marker recover"的主流程

### 日志口径

日志能明确区分：
1. `decisionSource=model|router|degrade`
2. `route=no-tool|tool|complex-tool`（最终执行结果）
3. `toolCallCount`
4. `degradeApplied`

### 测试覆盖

- ✓ 两个 smoke 测试（纯问答、网页任务）
- ✓ 无回归：npm test 全量通过

## 冻结边界

**这轮只做**：
1. 去掉前置主裁判式 route classifier
2. 收口成单智能体自决策主链
3. 保持已有：manifest、tool-loop、verify gate、task-supervisor

**这轮不做**：
1. 不改 browser 工具合同
2. 不改配额默认值
3. 不改多任务/多代理
4. 不顺手重写 prompt 全套
5. 不加新的自然语言强绑定规则

## 风险说明

1. **低风险**：tool-loop 已经支持 auto 模式，只是 routed-chat 的前置分类器阻断了使用
2. **保持兼容**：degrade 模式、complex-tool 流程保持不变
3. **可回滚**：如果出现问题，可以快速恢复 pre-classifier 主路径

## 一句话总结

别再让 router 先替模型做主判断；默认主链必须是"单一智能体先看到工具，再自己决定直答还是 tool-calls"。
