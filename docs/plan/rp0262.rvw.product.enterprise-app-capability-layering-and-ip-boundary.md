# RP0262：企业版 App 能力分层与 IP 边界分类

## 结论

当前 msgcode 可以走企业版与 App 化，但不该把一切都塞进二进制。

更合理的第一版分层是四层：

1. 二进制核心
2. 文本合同层
3. 客户配置层
4. 可选受控服务层

其中真正应该优先放进二进制的，不是所有 skill，而是当前仓库里最有价值、最不适合明文交付的主链实现：

- 上下文装配核心
- tool loop / replay / budget / clip 核心
- schedule / route / jobs 真相自愈
- memory 索引与检索核心
- 企业版授权与能力开关

## 研究范围

本报告只回答三件事：

1. 当前仓库哪些能力最像“产品核心”
2. 这些能力应该落在哪一层
3. 企业版离线/App 路线下，先做什么最稳

不回答：

- 现在就选 Electron 还是别的壳
- 现在就决定 MLX 跨平台最终方案
- 现在就把所有 skill 私有化

## 仓库事实

### 当前交付仍偏开发态

来自 [README.md](/Users/admin/GitProjects/msgcode/README.md) 与 [package.json](/Users/admin/GitProjects/msgcode/package.json)：

- 安装默认依赖 `brew bundle --file bootstrap/Brewfile`
- 运行和开发依赖源码态 `tsx`
- 正式 shell 合同绑定 Homebrew Bash
- 浏览器主链依赖 `patchright`
- memory 依赖 `sqlite-vec`
- 飞书通道依赖 `@larksuiteoapi/node-sdk`
- 桌面能力依赖外部 `ghost-os`

所以“企业现场不下载”目前还不是现成事实，只是可规划方向。

### 当前真正值钱的能力不在单个 skill 文本里

来自以下主链文件：

- [context-policy.ts](/Users/admin/GitProjects/msgcode/src/runtime/context-policy.ts)
- [tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts)
- [store.ts](/Users/admin/GitProjects/msgcode/src/memory/store.ts)
- [browser-patchright.ts](/Users/admin/GitProjects/msgcode/src/runners/browser-patchright.ts)
- [workspace.ts](/Users/admin/GitProjects/msgcode/src/config/workspace.ts)

当前产品核心更像这些“如何装配、如何执行、如何自愈、如何约束”的实现，而不是一份 `SKILL.md` 文本本身。

## 分类

### A. 应优先放进二进制的能力

这类能力要么直接体现护城河，要么明文交付后很容易被照搬。

#### 1. 上下文装配核心

对应：

- [context-policy.ts](/Users/admin/GitProjects/msgcode/src/runtime/context-policy.ts)

理由：

- 它负责 `summary / recent window / checkpoint / soul` 的统一装配
- 这是“模型最终看到什么”的主链
- 这是效果差异的关键来源，不适合只靠明文 prompt 公开

建议：

- 放进本地核心二进制或受控本地模块
- 对外只暴露合同，不暴露全部装配细节

#### 2. Tool loop 核心

对应：

- [tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts)

理由：

- 这里承载工具暴露、配对回放、quota、clip、连续执行
- 最近 `tk0248 / tk0251` 这类真实修复都落在这里
- 这是实际成功率的硬核主链

建议：

- 作为企业版本地核心实现
- 不把整套 replay / clip / quota 细节写进公开 skill

#### 3. 调度真相自愈核心

对应：

- [commands.ts](/Users/admin/GitProjects/msgcode/src/commands.ts)
- [schedule-sync.ts](/Users/admin/GitProjects/msgcode/src/jobs/schedule-sync.ts)
- [store.ts](/Users/admin/GitProjects/msgcode/src/routes/store.ts)
- [store.ts](/Users/admin/GitProjects/msgcode/src/jobs/store.ts)

理由：

- `tk0256 / tk0259 / tk0260 / tk0261` 这批刚证明，schedule/routes/jobs 的真相自愈是企业稳定性的关键
- 这是运行基础设施能力，不是说明书能力

建议：

- 作为二进制核心的一部分
- 不需要用户可编辑

#### 4. Memory 核心

对应：

- [store.ts](/Users/admin/GitProjects/msgcode/src/memory/store.ts)

理由：

- 这里包含 SQLite、FTS5、`sqlite-vec`、索引 schema、重建逻辑
- 它是“文件真相 -> 可检索记忆”的实际桥梁
- 一旦做企业离线版，这是核心卖点之一

建议：

- 放进本地二进制核心
- 文本层只保留“如何使用 memory”，不保留底层索引策略细节

#### 5. 企业授权与能力开关

当前仓库还没正式做，但企业版必须有。

建议：

- 授权校验
- 版本/能力包开关
- 企业版专属模块开关

都进二进制，不进公开 skill。

### B. 应保留为文本合同的能力

这类内容适合透明、可审计、可快速迭代。

#### 1. Skill 说明书

包括：

- `SKILL.md`
- `prompts/*.md`
- `docs/protocol/*`
- `README.md`

理由：

- 这些内容的职责是帮助模型理解“何时用、怎么用”
- 它们是说明书，不是护城河本体
- 频繁调整时，文本比编译后更新更便宜

建议：

- 继续保留文本化
- 但不要把高价值配方全文放进 skill

#### 2. 面向用户的帮助和验收文档

包括：

- `/help`
- 测试口径
- 协议文档
- 企业安装说明

理由：

- 企业客户反而需要更透明
- 这部分藏起来没有收益

### C. 应保留在配置层的能力

这类内容随客户、workspace、环境而变化，不适合编进二进制。

对应：

- [workspace.ts](/Users/admin/GitProjects/msgcode/src/config/workspace.ts)

当前典型项：

- `tooling.allow`
- `tooling.fs_scope`
- `agent.provider`
- `runtime.kind`
- `model.local.*`
- `model.api.*`
- `policy.mode`
- Feishu 凭据与工作区上下文

建议：

- 二进制只负责读取、校验、应用
- 客户差异留在配置文件
- 不把客户特定策略硬编码到产品核心

### D. 更适合做受控服务的能力

这不是第一版企业离线包的必需项，但如果某项价值特别高、更新频率特别快，就不该直接交付到客户机器。

建议候选：

- 私有知识库增强检索
- 高价值行业模板库
- 企业统一授权/计量
- 复杂多模型编排策略

前提：

- 客户允许访问企业内服务或你方受控服务

如果目标是“完全内网”，这一层应做成企业内私有部署，而不是公网服务。

## 建议的第一版 App 分层

### 1. App 壳层

职责：

- 工作区切换
- 任务/会话视图
- 日志查看
- 设置页
- 手动诊断和运维入口

特点：

- 可以换壳
- 不应承载核心业务逻辑

### 2. 本地核心层

职责：

- daemon
- listener
- context assembly
- tool loop
- schedule/routes/jobs 真相维护
- memory 索引与检索
- enterprise 授权与能力开关

特点：

- 这是最应该产品化、最值得保护的部分
- App 只是它的壳，不是它本体

### 3. 开放说明层

职责：

- skill
- prompt
- protocol
- help docs

特点：

- 可读
- 可审计
- 可快速更新

### 4. 客户配置层

职责：

- 模型地址
- workspace 默认能力
- 安全/外联策略
- 企业品牌与组织配置

## 对“做 App 就把能力塞进二进制”的判断

不该这么粗暴。

更准确的说法是：

- 进二进制的是核心实现
- 留文本的是说明书和合同
- 留配置的是客户差异
- 受控服务承接高价值、重更新的附加能力

如果把所有 prompt、skill、策略都塞进二进制，代价会很高：

- 调整慢
- 审计难
- 解释性差
- 企业定制成本高

## 当前最大交付阻塞

按影响大小排序：

1. 安装链仍依赖 `brew` / `npm install`
2. shell 合同仍绑定 Homebrew Bash
3. browser 依赖 `patchright` 与外部 Chrome
4. desktop 依赖外部 `ghost-os`
5. 本地模型路线当前明显偏 macOS + MLX

这说明：

- 第一版企业包不要一开始就承诺“所有能力完全跨平台”
- 应先定义“核心离线包”和“可选扩展包”

## 建议的企业能力包拆法

### 核心离线包

建议包含：

- daemon / listener / CLI
- context assembly
- tool loop
- schedule/routes/jobs
- memory
- 文件工具
- 基础日志和诊断

目标：

- 不依赖 `brew`
- 不依赖 `npm install`
- 可离线安装

### 扩展包 A：模型后端包

建议拆开：

- macOS + MLX
- 通用 OpenAI-compatible 内网后端

目标：

- 不把模型后端和主包绑死

### 扩展包 B：Browser 包

建议包含：

- Patchright
- 受控 Chrome/Chromium 方案

目标：

- 避免所有企业客户都被浏览器依赖绑架

### 扩展包 C：Desktop 包

建议包含：

- `ghost-os`
- 桌面自动化桥接

目标：

- 把最重、最依赖系统权限的能力单独收敛

## 建议的下一步决策顺序

1. 先做 `tk0262`
   - 把“核心离线包 / 扩展包”正式定稿
2. 再做 `tk0264`
   - 把 MLX/mac 与跨平台模型边界定稿
3. 再做 `tk0263`
   - 确定 App 只是壳，还是要承载更多本地运维面
4. 最后做 `tk0265`
   - 反推安装链和交付形态

## 一句话判断

当前最该保护的不是 `SKILL.md`，而是 msgcode 主链里“如何装配上下文、如何执行工具、如何维护真相、如何自愈”的核心实现。

企业版第一步也不是先选 Electron，而是先把“哪些是核心离线包，哪些是扩展包”定清楚。
