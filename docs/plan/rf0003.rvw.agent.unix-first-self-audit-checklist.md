# RF0003：Unix-first Self Audit Checklist

## 结论

这不是宣言。

这是每次设计、实现、review 前后都要拿出来问自己的对照表。

如果答案开始变虚、变绕、变厚，说明系统正在偏航。

## 初心

我们不是在做：

- fleet control plane
- 多层编排器
- 万能控制中心
- 替模型做决定的裁判系统

我们在做：

- 一台可安装、可分发、可长期使用的 AI appliance
- 一个服务 LLM 的薄宿主
- 一条 file-first、可验证、可审计的执行主链

## 设计前自问

1. 这是主链问题，还是我在为一个边缘问题发明新层？
2. 这层是不是在替主决策者做决定？
3. 这层是不是在替另一层的缺陷擦屁股？
4. 拿掉它以后，系统会不会更接近单一主链？
5. 我是在解决真实失败，还是在提前防御想象中的失败？

任一答案偏向“替系统做主”，默认不加。

## 结构自检

1. 能做成一个执行口，就不要做 manager
2. 能做成一个 pack，就不要做 plugin runtime
3. 能做成一个 probe，就不要做控制器
4. 能做成一个 contract，就不要做中间平台
5. 能把真相留在文件里，就不要再造第二状态源

## Prompt / Skill / Code 顺序

1. 先问：提示词、skill、contract 能不能先解决
2. 再问：代码是不是只是在暴露更真实的能力边界
3. 最后才问：是否真的需要新增模块

如果还没试过更锋利的说明书，就不要先长控制层。

## Fail Fast Checklist

1. 我是不是在加 fallback？
2. 我是不是在加 repair？
3. 我是不是在用 try/catch 掩盖真实错误？
4. 我是不是在把应该报错的情况偷偷转成“默认值”？
5. 我是不是在用硬编码掩盖自己没理解？

任一为“是”，先停。

## 精度前置检查

1. 参数归属写清楚了吗？
2. 代码落点写清楚了吗？
3. 错误语义写清楚了吗？
4. 第一刀切口写清楚了吗？
5. review checklist 写清楚了吗？

没有精度的设计，不算设计，只算延期的复杂度。

## 原型 / 实现 / 宿主分层检查

1. 我是不是把原型当正式实现改了？
2. 我是不是让正式实现反写了原型？
3. 我是不是让 renderer 直接吃 runtime 内部实现？
4. 我是不是让宿主长成第二主脑？

任一为“是”，说明层级又混了。

## Parser / Probe / WPKG 警戒线

### Parser

- parser 只负责读输出
- request contract 负责控制生成参数
- capability contract 负责声明模型会不会
- parser 不猜自由文本，不做 repair middleware

### Probe

- probe 只负责观察
- probe 不给系统代做控制决策
- probe 不偷偷长成预算控制器

### WPKG

- pack 可以带边车代码
- pack 不能带第二套 runtime
- pack 不能反过来定义 core

## Review 前后必问

### Review 前

1. 有没有用注释掩盖胖函数？
2. 有没有把 request 问题塞给 parser？
3. 有没有把边界问题塞给宿主？
4. 有没有顺手平台化？

### Review 后

1. 这次改动让主链更短了吗？
2. 这次改动让错误更早暴露了吗？
3. 这次改动让真相更集中了吗？
4. 这次改动是不是只是“看起来更完整”？

如果最后一题答“是”，就该回退重想。

## 一句话

系统的职责是暴露真实能力、保留真实错误、维护最薄真相源。

不要替模型做决定，不要替未来做平台，不要替复杂性找借口。
