## 关键数据流

当前主链大致为：

```text
LLM
→ tool-loop 选择 tools/schema + system hints
→ executeTool()
→ 具体 runner / file op / feishu tool
→ ToolResult
→ tool-loop 序列化 tool_result 回灌模型
→ 模型继续 act 或输出最终答案
```

和参考文档相比，当前实现最大的偏差不在“有没有 bash”，而在：

1. 工具面并不单一
2. 输出处理层次没有完全收口
3. 文件与错误合同不够统一

## 不足点

### 1. 工具面仍然是多工具 + suppression + prompt 映射

Evidence

- Code: [src/tools/manifest.ts](/Users/admin/GitProjects/msgcode/src/tools/manifest.ts#L63)
- Code: [src/tools/manifest.ts](/Users/admin/GitProjects/msgcode/src/tools/manifest.ts#L591)
- Code: [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1202)

观察：

- 默认 suppress：`write_file / edit_file / vision / mem`
- 但 manifest 仍然完整注册这些工具
- `renderLlmToolIndex()` 还要补一层文本映射：“写改文件优先用 bash”
- tool-loop 继续注入 workspace hint、native tool hint、browser hint、skill index JSON

判断：

- 这说明当前模型并不是在一个稳定、统一、可自解释的能力面上工作，而是在“真实工具面 + 默认隐藏 + 额外提示解释映射”之间切换。

风险：

- 提示词越补越多
- 模型需要学“策略映射”，而不是直接学工具合同
- 工具存在但默认不用，会产生真相源漂移

### 2. 输出分层仍然散，执行层和呈现层没有严格断开

Evidence

- Code: [src/runners/bash-runner.ts](/Users/admin/GitProjects/msgcode/src/runners/bash-runner.ts#L160)
- Code: [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L405)
- Code: [src/runtime/context-policy.ts](/Users/admin/GitProjects/msgcode/src/runtime/context-policy.ts#L112)

观察：

- `bash-runner` 已经负责输出截断、tail 窗口、完整输出落盘
- `tool-loop` 回灌前再次统一裁 4000 chars
- `context-policy` 再有一套 `clipToolPreviewText`

判断：

- Layer 1/Layer 2 的职责还没有形成唯一边界
- 同一个输出在 runner、Tool Bus、tool-loop、context-policy 四处被加工

风险：

- 工具行为不一致
- 后续每新增一种工具都可能重复补裁剪逻辑
- 模型拿到的 preview 形式不稳定

### 3. `read_file` 的文件合同明显弱于参考文档

Evidence

- Code: [src/tools/bus.ts](/Users/admin/GitProjects/msgcode/src/tools/bus.ts#L753)
- Code: [src/tools/bus.ts](/Users/admin/GitProjects/msgcode/src/tools/bus.ts#L1267)

观察：

- `read_file` 直接 `readFile(filePath, "utf-8")`
- 只对 `SOUL.md` 做了一个特判 alias remap
- 没有二进制检测
- 没有大文件 preview/full output path
- catch 里直接透传 raw error message

判断：

- 这条链最接近参考文档里“PNG 乱码/大文件污染/错误不指路”的反例

风险：

- 模型误读图片/PDF/二进制时没有稳定引导
- 大文件容易把大量内容直接推进主链
- 错误对下一步动作帮助不足

### 4. 帮助系统存在，但没有成为模型可渐进探索的统一入口

Evidence

- Code: [src/cli/help.ts](/Users/admin/GitProjects/msgcode/src/cli/help.ts#L87)
- Code: [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1208)

观察：

- CLI 层面有 `help-docs`，而且支持 `--json`
- 但 LLM 主链里并没有把它作为正式工具暴露
- 当前主要靠 tool schema + tool index + skill 文本注入让模型知道怎么做

判断：

- 当前的帮助是“存在”，但不是“可发现主链”
- 这和参考文档里的逐层 help 发现机制仍有差距

风险：

- 模型不容易按需探索合同
- 很多帮助压力仍然落在 system prompt 和 skill 文案上

### 5. 预算边界还嵌在 tool-loop 热路径里，并直接生成用户文案

Evidence

- Code: [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1340)
- Code: [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts#L1513)

观察：

- 工具调用/步骤数超限时，tool-loop 直接返回系统生成的说明文本
- 文案里直接写“下一轮 heartbeat 继续执行”

判断：

- 预算边界当然合理，但它现在还带着明显的“控制软件”语气，且在热路径内直接塑造交付

风险：

- 用户看见的是系统节流语义，而不是单纯 runtime 事实
- 预算与交付层仍有耦合

## 风险点

- 若继续只补 prompt/hint，主链会继续变厚
- 若直接教条式改成 one-tool，可能误伤已有原生工具价值
- 若不先收口 `read_file` 和输出层，再谈“做薄”，容易停留在概念层

## 结论

参考文档对 `msgcode` 不是“完全不适用”，恰恰相反，它和仓库当前方向高度同向。但和参考相比，`msgcode` 仍欠三刀最关键的收口：

1. 收口文件/输出合同
2. 收口默认正式工具面
3. 收口帮助与预算边界的热路径语义

如果只让我选一个第一批动作，我建议先做 `read_file + 输出层`。这条链离参考文档讲的核心问题最近，而且能直接减少主链污染。
