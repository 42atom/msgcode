# RP0294：status.log 与现有 append-only 模式对照

## 结论

`status.log` 不需要新平台。

第一版只要沿用仓库现有 append-only 习惯：

- 单一写入口
- 统一格式化
- UI 只读最近几条

真正要避免的是两件事：

- 模型直接写原始文本
- UI 扫全文件后再自己二次总结

## 现有 append-only 习惯

### thread-store

文件：

- `/Users/admin/GitProjects/msgcode/src/runtime/thread-store.ts`

特点：

- 一线程一文件
- 追加 turn
- front matter + 结构化正文

启发：

- `status.log` 不该抢线程正文职责
- 原始会话仍留在线程文件

### run-store

文件：

- `/Users/admin/GitProjects/msgcode/src/runtime/run-store.ts`

特点：

- 统一 `append`
- 追加 JSONL
- 不做查询控制面

启发：

- `status.log` 也应只有一个最薄追加入口

### run-events

文件：

- `/Users/admin/GitProjects/msgcode/src/runtime/run-events.ts`

特点：

- 单一 event append
- append-only
- 失败只记日志，不长额外协调器

启发：

- `status.log` 第一版也该坚持：
  - 单一 writer
  - 失败不自动长补救层

### jobs/store

文件：

- `/Users/admin/GitProjects/msgcode/src/jobs/store.ts`

特点：

- `appendRun()` 只负责追加
- 原子写用于整体文件
- append 用于运行记录

启发：

- `status.log` 是运行记录面，不是配置面
- 不需要为它先引入大而全原子写平台

## 对 status.log 的直接约束

### 写入

第一版必须是：

- runtime 单一 writer
- 模型只提交结构化意图
- writer 统一清洗并格式化

不能是：

- 模型直接 `echo >> status.log`

### 格式

继续保持最小文本行：

- `时间 | 线程 | 类型 | 摘要 | 引用`

原因：

- 可读
- 可 tail
- 不重复发明第二套 JSONL 事件流

### 读取

UI 第一版只做：

- tail 最近 10-20 条

不做：

- 全文件扫描
- 实时监控流
- 语义聚合

## 并发与锁

这个问题真实存在，但第一版先不放大。

因为当前主链仍是：

- 单工作区
- 单 runtime 主写者

所以第一版先以：

- 单一 writer

解决，而不是先引入锁层。

如果未来出现真正多进程并写，再补锁，不在 `0294` 第一版里预支复杂度。

## 去重

第一版不做写入侧去重。

原因很简单：

- `status.log` 是追加日志
- writer 只该负责清洗、格式化、追加
- 不能替日志判断“这条还该不该记”

如果未来出现展示噪音，也优先在消费侧解决，而不是先让 writer 吞记录。

## 轮转

这是后手，不是第一版实现前置。

当前只在合同里承认：

- `status.log` 未来需要 rotate/archive

但不展开策略。

## 结论口径

`0294` 已经可以进入评审。

第一版实现时只需补：

- append writer
- tail reader

别长更多层。
