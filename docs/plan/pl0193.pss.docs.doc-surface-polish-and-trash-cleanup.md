# Plan: Doc Surface Polish And Trash Cleanup

## Problem

正式协议已经收口，但仓库表层还不够干净：

1. `.trash/` 里残留历史 md，继续制造噪声
2. `docs/tasks/` 仍是独立旧体系，和现行 `tk/pl/rs/...` 主链之间关系不够清楚
3. `AIDOCS/` 作为辅助材料区，文档数量大、命名杂、目录分层不稳
4. 历史正文与 archive 仍有不少旧时代文本残影

如果不处理这层，项目虽然“能用”，但长期仍不好搜、不好看、不好交接。

## Decision

本单把“表层整理”合并处理，但仍坚持分层：

- 正式真相源：`issues/`、`docs/plan/`、`docs/tasks/`、`docs/archive/`
- 辅助材料区：`AIDOCS/`
- 临时/废弃：`.trash/`
- 第三方：`node_modules/`、`venv/`、`vendor/` 不纳入本单

## Workstreams

### 1. `.trash/` 清理

- 分类：
  - 可直接删除
  - 需迁入 `docs/archive/`
  - 需迁入 `AIDOCS/`
  - 需提升为正式文档

目标：`.trash/` 最终只保留极少量待定内容，最好清空。

### 2. `docs/tasks/` 决策

需要拍板的不是“改不改几个文件”，而是长期口径：

- 方案 A：继续保留时间线命名，不并入 `pl/rp`
- 方案 B：逐步迁入更统一的 `rp/pl` 体系

默认建议：保留时间线职责，但把 README、正文引用和目录定位写清楚。

### 3. `AIDOCS/` 收口

建议分层：

- `AIDOCS/reports/`
- `AIDOCS/reviews/`
- `AIDOCS/notes/`
- `AIDOCS/refs/`
- `AIDOCS/design/`
- `AIDOCS/prompts/`
- `AIDOCS/artifacts/`

同时处理：

- 中英混排/空格/括号文件名
- `task_plan*`、旧 `plan-*`、自由命名文档
- `active/archive` 口径不清的子目录

### 4. 历史 md 表层净化

只做“值当的清理”：

- 入口文档和高频被读文档优先
- archive 历史资料不追求全量重写
- 目标是降噪，不是把历史抹平

## Deliverables

1. `.trash/` 清理清单
2. `docs/tasks/` 长期协议决策
3. `AIDOCS/` 目录与命名整顿方案
4. 一轮高价值历史正文清洗

## Acceptance

完成后应达到：

- 搜索结果更接近正式真相源
- 目录职责更容易一眼看懂
- `.trash/` 不再像第二文档仓库
- `AIDOCS/` 不再像散装收纳箱
