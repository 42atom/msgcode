# PL0403：Desktop Writable Thread Surface Mainline

## 结论

P0 不是“把原型里的 composer 画出来”，而是把 thread 合同从 readonly 升成 writable。

## 最小设计

新增的不是页面层，而是 thread 读面里的一个字段：

- `writable`

UI 只遵守一个规则：

- `writable=true` 才显示 composer

runtime 内部可以持有写回目标，但不进入 UI 合同。

## 写桥签名

最薄入口只要一条：

- `sendThreadInput({ workspacePath, threadId, text }): Promise<void>`

这条入口的职责是：

1. UI 提交 thread 身份和文本
2. runtime 自己查 thread 真相
3. runtime 自己解析写回目标

UI 不接触 `replyTarget`

## 最小演进

1. 复用现有三栏壳，不重做主窗口
2. 保留 readonly thread 的存在
3. bridge 在现有读桥旁边补一条最薄写回入口
4. 中栏 composer 只负责提交输入，不承担渠道策略

## 非范围

1. 不做多渠道统一收件箱
2. 不做复杂 slash GUI
3. 不做高级模型参数控制台
4. 不做音频/视觉/多模态输入

## 顺序

1. 先定 bridge 写回合同
2. 再定 writable thread 合同
3. 再改正式 UI composer 显隐
4. 最后补验证

## Composer 最小行为

1. 单文本输入
2. `Enter` 发送
3. 成功清空
4. 失败保留

## 一句话

桌面端能不能交互，关键不在 composer，而在 thread 有没有可写真相。 

## 补充注记

1. P0 不做流式，但桌面流式能力后续应落在 thread 读面的推送模型，而不是把读桥永久锁死在 request-response。
2. P0 的 `sendThreadInput()` 先保持 `Promise<void>`；后续若要支持中断，再单开切片演进为可取消句柄。
