# PL0332：people-pending 键合同

## 结论

`people-pending.json` 要想进入真实写链，最小键必须补齐成：

- `channel`
- `chatId`
- `senderId`

不能继续只停在：

- `channel`
- 缺 `chatId`
- 缺 `senderId`

因为已知人物 CSV 的唯一键已经是：

- `channel + chatId + senderId`

两边不一致，就不能安全做“已认领自动隐藏”。

## 最小字段

建议收成：

- `channel`
- `chatId`
- `senderId`
- `username`
- `displayName`
- `seenAt`

其中：

- `username/displayName` 只是展示信息
- 真正参与唯一键的只有前三个

## 读面过滤前置条件

只有在 pending 项同时具备：

- `channel`
- `chatId`
- `senderId`

时，读面才允许把它与 CSV 做差集，隐藏已认领项。

否则：

- 一律继续展示 pending
- 不允许 runtime 猜测它是否已经入 CSV

## 本轮实现

当前读面已经改成：

- 缺 `channel / chatId / senderId` 任一字段就 warning
- 该条 pending 不进入输出
- 不再接受旧 pending 键

## 明确不做

1. 兼容多套人物键
2. 读面按 `displayName` 或 `username` 做猜测匹配
3. UI 自己删除 pending 条目
4. 用旧键继续长期代替 `senderId`
