# RP0327：appliance core settings 圆桌复核

Decision:
- Recommendation: go
- Confidence: medium
- Scope: 接受这刀进入提交，但仍然把它严格限制为 core settings fan-in

Why:
- `SystemsArchitect`：当前实现是进程内聚合读，没有 shell-out、缓存、RPC，也没有把 `doctor` 硬塞进主链。
- `SkepticalOperator`：主风险仍在于后续有人把它做成 settings 专用读模型；当前这刀还没越线。
- `UserAdvocate`：把 `doctor` 拆出去更诚实；顶层也补上了分段 warning 摘要，不再只有空的 warning 状态。
- `ExecutionLead`：这刀顺序合理，仍然是“先读主链，再谈写面”，没有主次倒置。

Risks:
1. 聚合层后续继续手写空态，可能和单段 CLI 漂移。
2. UI 若只看顶层 `status`，仍可能把分段 warning 用差了。
3. 这条命令很容易被顺手扩成 mutation 或 settings 中心。

Guardrails:
- `appliance settings` 只做 fan-in，不准加缓存、默认值填充器、后台守护进程。
- `doctor` 继续独立按需读取，不为了整齐合并回来。
- UI 必须继续消费分段 `status/warnings/errors/data`，不能只看顶层。
- 下一刀若要做保存，只能走 `0322`，且只先开放 profile。

Next Actions:
1. 提交 `0326` 代码与回归锁。
2. 下一步若继续 settings 主线，只能先碰 `0322` 的 profile mutation，不能顺手给 `appliance settings` 加写入口。

- Docs: `tk0323`, `tk0326`, `rp0326`
- Code: `src/cli/appliance.ts`
