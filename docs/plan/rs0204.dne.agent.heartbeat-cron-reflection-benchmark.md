# Heartbeat Cron Reflection Benchmark

## 结论先行

三个参考项目里，没有一个可以原样照搬。

我的判断是：

- **第一性拆分最清楚**：`openclaw`
- **工程化水平最高**：`openclaw`
- **最像 Unix / 最薄**：`pi-mono`
- **产品野心和学习叙事最强**：`hermes-agent`

但如果目标是做出更好的 `msgcode` 主脑节律，我们不该复制任何一个。

我们应该吸收三者各自最强的一刀：

- 从 `pi-mono` 学 **文件驱动唤醒 + 静默无事**
- 从 `openclaw` 学 **heartbeat / cron / subagent 三者分工**
- 从 `hermes-agent` 学 **把记忆 / 日记 / skill 改进写进主脑行为**

然后用 `msgcode` 自己的优势把它们收口成更薄的主链：

- **文档真相源**
- **工作区真相源**
- **主脑唯一执行权**
- **文件优先，不搞第二控制面**

## 参考对象与证据

### `pi-mono`

关键证据：

- [README.md](/Users/admin/GitProjects/GithubDown/pi-mono/packages/mom/README.md)
- [events.md](/Users/admin/GitProjects/GithubDown/pi-mono/packages/mom/docs/events.md)
- [events.ts](/Users/admin/GitProjects/GithubDown/pi-mono/packages/mom/src/events.ts)

核心做法：

- 用 `workspace/events/*.json` 做事件真相源
- 事件类型固定为：
  - immediate
  - one-shot
  - periodic
- harness 监视目录，按文件类型设置 `setTimeout` 或 `cron`
- 周期任务可以返回 `[SILENT]`，避免刷屏
- memory 也是文件：全局 `MEMORY.md` + channel `MEMORY.md`

一句话总结：

- 它把“唤醒”做成了非常 Unix 的文件协议。

### `openclaw`

关键证据：

- [cron-vs-heartbeat.md](/Users/admin/GitProjects/GithubDown/openclaw/docs/automation/cron-vs-heartbeat.md)
- [heartbeat.md](/Users/admin/GitProjects/GithubDown/openclaw/docs/gateway/heartbeat.md)
- [cron-jobs.md](/Users/admin/GitProjects/GithubDown/openclaw/docs/automation/cron-jobs.md)
- [subagents.md](/Users/admin/GitProjects/GithubDown/openclaw/docs/tools/subagents.md)
- [heartbeat-policy.ts](/Users/admin/GitProjects/GithubDown/openclaw/src/cron/heartbeat-policy.ts)

核心做法：

- 明确区分：
  - `heartbeat`：主会话的周期 awareness
  - `cron`：精确定时和隔离执行
  - `subagent`：独立 session 的执行臂
- heartbeat 默认在主 session 里定期醒来
- cron 可以跑 main / isolated / current / custom session
- subagent 自带 announce 回传链
- 对 `HEARTBEAT_OK` 这种空结果有专门抑制逻辑，避免主会话被垃圾 summary 污染

一句话总结：

- 它是三者分工最清楚、控制粒度最完整的实现。

### `hermes-agent`

关键证据：

- [README.md](/Users/admin/GitProjects/GithubDown/hermes-agent/README.md)
- [cron.md](/Users/admin/GitProjects/GithubDown/hermes-agent/website/docs/user-guide/features/cron.md)
- [memory.md](/Users/admin/GitProjects/GithubDown/hermes-agent/website/docs/user-guide/features/memory.md)
- [prompt_builder.py](/Users/admin/GitProjects/GithubDown/hermes-agent/agent/prompt_builder.py)

核心做法：

- 把自己定义成 “built-in learning loop”
- 用 prompt 直接要求 agent：
  - 主动存 memory
  - 像 diary 一样记事
  - 复杂任务后沉淀 skill
- 用 cron 跑自动化任务
- memory 用受限字符数的小文件管理
- session_search 负责跨会话召回

一句话总结：

- 它最强的是“自我改进”的产品叙事和行为约束。

## 谁把第一性抓得最好

我投 `openclaw`。

原因不是它最薄，而是它最清楚地回答了三个问题：

1. 周期巡检到底是什么  
   = heartbeat
2. 精确定时到底是什么  
   = cron
3. 并行执行到底是什么  
   = subagent

这三个概念一旦混在一起，系统就会糊。  
`openclaw` 至少在概念边界上是清醒的。

## 谁工程化水平最高

也是 `openclaw`。

原因：

- 配置、文档、UI、测试、运行时策略是成套的
- 对空回执、delivery、summary 污染、session 隔离这些脏问题考虑得比较全
- heartbeat / cron / subagent 都有明确的运行时合同，而不是只停在 prompt

它的弱点也同样明显：

- 重
- 控制面多
- 配置层厚
- 很容易把主脑行为平台化

## 谁最像 Unix

我投 `pi-mono`。

原因：

- 事件就是文件
- 目录就是协议
- 监听文件变化就是唤醒机制
- 没事就 `[SILENT]`

这套东西很朴素，也很有劲。

它的短板是：

- 任务/派单/验收主链不够强
- 更像“会自己动的 bot”，不是“任务管理明确的主脑”

## 谁的产品野心最强

`hermes-agent`。

它明确把：

- 记忆
- diary
- skill 自我改进
- 周期 nudges

组合成“闭环学习”叙事，这个方向是对的。

但它的问题也很明显：

- 很多东西更像 prompt 文化，而不是状态真相源文化
- 学习循环的落点偏隐式
- 更像“好 agent 应该这样做”，而不是“系统把这条主链钉死”

## 对 msgcode 最有价值的三刀

### 1. 学 `pi-mono`：用文件唤醒，不要隐藏调度层

要点：

- 定时器应该绑定文件真相源
- 工作区应该能自己表达“什么时候醒来”
- 无事发生要支持静默结束

对 msgcode 的启发：

- `workspace/.msgcode/schedules/*.json`
- `workspace/.msgcode/HEARTBEAT.md`
- heartbeat / alarm 都优先走文件协议

### 2. 学 `openclaw`：把 heartbeat / cron / subagent 三分开

要点：

- heartbeat = 主脑周期 awareness
- cron = 精确唤醒
- subagent = 执行臂

对 msgcode 的启发：

- 绝不能把“定时器”“任务监督”“子代理执行”揉成一锅
- 否则后面一切都会变成补丁

### 3. 学 `hermes-agent`：把反思行为写成正式步骤

要点：

- 记忆不是事后人工整理
- skill 改进不是灵感驱动
- diary 不是可有可无

对 msgcode 的启发：

- 任务完成后应固定检查：
  - 是否写日报/日记
  - 是否写 memory candidate
  - 是否写 skill candidate

## msgcode 应该怎么超越他们

关键不是“功能更多”，而是“主链更短更真”。

我建议 msgcode 用下面这条主链超过他们：

`alarm/cron -> heartbeat -> doc-first task review -> subagent supervision -> reflection`

并且每一段都落到文件真相源：

- `issues/*.md`：父子任务状态真相源
- `.msgcode/schedules/*.json`：唤醒真相源
- `.msgcode/dispatch/*.json`：派单绑定真相源
- `AIDOCS/reports/daily/*.md`：日记/反思落点

## msgcode 应该超越的点

### 超越 `pi-mono`

我们要比它强在：

- 不只是“会醒”
- 还要“知道自己在推进哪些正式任务”

也就是：

- 事件驱动保持简单
- 但任务状态必须接进 `issues/`

### 超越 `openclaw`

我们要比它强在：

- 不把系统做成厚控制面
- 不让配置层和运行时状态压过文档真相源

也就是：

- 保留它的边界清晰
- 去掉它的控制面膨胀

### 超越 `hermes-agent`

我们要比它强在：

- 不只靠 prompt 说“你应该记日记/记记忆/改 skill”
- 而是把这些动作挂进 heartbeat 和任务完成流程

也就是：

- 学习行为落到文件与任务流程
- 不是只落到人格宣言

## 建议的 msgcode 主循环

### 1. Heartbeat

周期醒来，读：

- `HEARTBEAT.md`
- `issues/*.tdo|doi|rvw`
- 当前 workspace 的派单记录
- subagent 状态

作用：

- 看什么没做完
- 看谁卡住了
- 看是否该推进状态
- 看是否该协助子代理

### 2. Alarm / Cron

精确定时：

- 某个子任务 30 分钟后复查
- 明早 9 点生成日报
- 下周一做一次项目巡检

作用：

- 只负责唤醒
- 不负责思考

### 3. Reflection

在任务完成或每日收口时执行：

- 写今日简记
- 提炼 memory candidate
- 提炼 skill candidate
- 记录失败复盘

作用：

- 让主脑真的“变聪明”
- 但不搞隐式魔法

## 最终判断

如果只看参考实现：

- `openclaw` 最成熟
- `pi-mono` 最干净
- `hermes-agent` 最有野心

如果要做最适合 `msgcode` 的版本：

- 我们不能选其中一个当模板
- 我们应该做一个 **文档真相源 + 文件唤醒 + 主脑节律 + 任务回收** 的更薄系统

一句话：

- **他们做的是“会自动跑的 agent”**
- **我们要做的是“会按任务文档长期推进、会自己回收经验的主脑”**
