# PL0341：web thread 与新建聊天边界基线

## 结论

用户层统一只看 thread。

不再混用：

- session
- channel 会话
- 内部调查线

当前收口是：

- thread = 用户可见的对话单位
- channel = thread 的来源属性
- 内部调查线如果以后存在，也不叫 thread

## 为什么现在会觉得 thread 碎

根因有两层：

1. 工作区列表还没有真正落 archive
2. web 端还缺稳定 `chatId`

第一层由 [pl0337.rvw.product.workspace-and-thread-archive-baseline.md](/Users/admin/GitProjects/msgcode/docs/plan/pl0337.rvw.product.workspace-and-thread-archive-baseline.md) 解决。

第二层由本计划解决。

也就是说：

- 不改 thread 哲学
- 先收 archive，把测试/冒烟 workspace 和旧 thread 从活跃面移走
- 再给 web thread 一个稳定锚点

## 统一语义

### 用户层

主界面只展示：

- 工作区
- 线程

例如：

- `family`
  - `飞书线程`
  - `网页线程`
  - `历史线程`

### 内部层

保留现有文件面，不急着重命名：

- `threads/*.md`
- `sessions/*.jsonl`

但用户心智不再直接暴露 `session`。

## web thread 的 `chatId`

### 必须有

飞书天然有 `chatId`。

web 没有稳定 `chatId` 时，后果就是：

- 每发一条消息
- 就可能新起一个 thread
- 左边列表越来越碎

所以 web thread 必须有稳定 `chatId`。

### 最薄规则

每个 web thread 自己有一个稳定 `chatId`。

不是整个浏览器只有一个 `chatId`。

这是为了保证：

- 一个工作区下可以有多个 web thread
- 点“新建聊天”时能开新 thread
- 旧 thread 仍能继续续话
- 不会把所有 web 对话硬挤进同一条线上

明确禁止：

- 一条消息一个 `chatId`
- 一个浏览器全局只维护一个长期 `chatId`
- UI 用临时 request id 顶替 `chatId`

### 谁维护

第一版允许 UI 生成 web `chatId`，但只作为输入参数。

前提是：

- UI 只是在本地持有“当前准备发送到哪条 web thread”
- 真正的 thread 仍然由 runtime 在第一条消息落盘后创建

也就是说：

- UI 可以准备一个新的 `chatId`
- 但没有第一条消息，就没有 thread 文件

这能避免孤儿 thread。

## thread 标题

### 规则

thread 标题必须由 runtime 落进 thread front matter。

UI 不允许：

- 截断第一条消息拼标题
- 自己维护标题缓存
- 另存一份 thread 名称表

### 原则

用户看到的 thread 标题，必须来自 thread 文件本身。

第一版最小字段至少包括：

- `threadId`
- `chatId`
- `transport`
- `title`
- `createdAt`
- `updatedAt`
- `archived`

## “新建聊天”边界

### 它做什么

主界面的“新建聊天”只做一件事：

- 切换到一个新的 web `chatId` 上下文
- 打开一个尚未落盘的待发送草稿上下文

### 它不做什么

第一版明确不做：

- 不直接创建空 thread 文件
- 不抢先写 UI 消息气泡
- 不发明一条前端 thread 队列
- 不让 UI 获得 thread 生命周期主导权

真正的 thread 创建时机仍然是：

- 用户发出第一条消息
- runtime 接收并落盘

这符合 file-first。

## 与 archive 的关系

当前 thread 太碎，不靠改 thread 哲学解决。

优先还是：

- 测试/冒烟工作区归档
- 不重要 thread 归档

活跃面只留真正常用的工作区和 thread。

所以主界面上的“碎”，先当 archive 问题处理，再处理 web `chatId` 问题。
