# PL0291：Neighbor / Coworker Optional Module

## 定义

这条线不是：

- 多 Agent 编排平台
- 中央总控系统
- 全局共享工作空间
- 局域网共享脑子

这条线是：

- `msgcode` 的一个可选模块
- 开关默认可关
- 打开后可在局域网里发现其他开放节点
- 节点之间异步通信、建立联系人、交换结果
- 用户可观察聊天记录、产物和协作线程

一句话：

> 先做“邻居和同事”，不做“分布式总控平台”。

## 前提界定

第一版必须先承认几个前提。

第一，邻居能力是可选模块，不是主链硬依赖。不开时，当前 `msgcode` 的单机主链完全不受影响。

第二，联网发现先于任务协作。也就是说，先解决“看见谁、认识谁、能否发消息”，再谈“怎么派任务、怎么协作”。

第三，第一版默认是异步协作。对方可能在线，也可能离线；消息和联系人要能接受下线、重连、重复投递这些事实。

第四，第一版不做真正的多人对等群聊状态机，也不做任务流转协议。多人并发 claim、多人共享推进状态、投票式群聊，这些都先不做。

第五，第一版允许一个极薄的 `health` 探针，但它只表达“最近是否可达”，不表达内部任务活跃度，也不替代业务 heartbeat。

## 目标

这条线第一版只解决四件事：

1. 节点发现
2. 联系人建立
3. 异步消息投递
4. 最近可达性探针（health）

如果这四件事没站稳，就不要往上长协作线程和任务平台。

## 非目标

第一版明确不做：

- 中央调度器
- 全局任务数据库
- 多节点共享同一 workspace
- 自动共识投票
- 多人同时改同一任务状态
- 节点间远程接管对方执行环境
- 把 LAN 邻居模块做成新的主链控制面

## 工作空间边界

工作空间不能做成全局共享。

每个节点的本地 workspace 仍然是它自己的真相源和私有执行面。邻居模块默认不暴露完整 workspace，不允许别的节点直接把它当共享磁盘来读写。

跨节点协作时，应该共享的是对方显式开放的能力和 export surface，而不是整个工作空间。

例如运营同事节点可以声明：

- 我可以提供运营年报参考
- 我可以提供某段时间的 KPI 摘要
- 我可以提供渠道投放汇总 artifact

你向它请求后，它在自己的边界内计算和整理，最后回给你一份摘要、报表、截图或数据文件。你拿到的是结果，不是它整个 workspace。

一句话：

> 私有 workspace 不共享；只共享显式 export surface。

## export surface 边界

`export surface` 第一版必须写死成结果导出面，不是远程工具总线。

第一版只允许这类结果型导出：

- 摘要
- 报表
- 截图
- 文件
- 结构化数据块

第一版明确禁止：

- 任意远程问答
- 任意远程查询
- 任意远程工具调用
- 把邻居节点当成通用 RPC 端点

一句话：

> 第一版 export surface 只导出结果，不导出执行权。

## 交互模型

产品表现可以像“看同事微信聊天记录”，但底层不要做成即时通信平台。

底层真相应该仍然是线程文件和 artifact 引用。产品层只是把它渲染成聊天记录式视图。

但第一阶段已经允许一个极薄的查看面：

1. 左侧节点列表
2. 右侧邮箱记录 / 审计记录
3. 我在右，对方在左，系统事件中性

前提：

- 模块已开启
- 进入页面时做一次轻量可达探测

这里表达的是：

- 最近可达性
- 邮箱式往来记录

不是：

- 即时聊天在线状态
- 任务活跃度
- 群聊线程系统

最终产品成熟后，用户应看到三种东西：

- 联系人
- 线程
- 产物

其中线程是时间线，产物是结果面。聊天记录里可以直接挂 artifact，但 artifact 本身不塞进消息正文。

## 节点关系

每个 msgcode 节点都是平等节点，不区分 master 和 worker。

但这条 owner 约束属于第二阶段。第一阶段只做邻居和邮箱，不做线程 owner。

## 发现与联系人

邻居模块打开后，节点才会在局域网里声明自己的存在，并监听同类声明。

这里的设计重点不是“扫描局域网所有端口”，而是“发现那些已经主动暴露自己的人”。也就是说，应该基于一个很薄的发现协议和固定入口做握手，而不是像漏洞扫描器一样到处探测。

发现、识别、联系人必须拆开。

第一版最小三态：

- `discovered`
- `known`
- `contact`

含义分别是：

- `discovered`：我看见了一个会说同协议的节点
- `known`：我已识别它的身份和能力声明
- `contact`：我愿意长期保留它，接受与它的异步往来

发现成功后，节点先进入 `discovered`。握手和能力交换成功后，进入 `known`。明确建立长期关系后，才升级为 `contact`。

联系人必须是持久状态。对方下线、换 IP、暂时不可达，不应该让历史线程和关系消失。系统应把这些视为联系人暂时离线，而不是彻底失联。

## health 边界

第一版允许一个很薄的 `health`：

- 用途：判断节点最近是否可达
- 输出：成功 / 失败、最近探测时间、可选延迟
- 语义：网络可达性，不是业务活跃度

第一版明确不做：

- 心跳广播系统
- 节点负载上报
- 任务繁忙度
- 自动故障转移
- 用 `health` 推断“这个同事正在工作”

一句话：

> `health` 只回答“现在能不能碰到它”，不回答“它在忙什么”。

## 第一阶段消息

第一阶段只需要邮箱级消息，不需要完整协作线程。

最小集合：

- `hello`
- `handshake`
- `message`
- `delivery`

配合一个很薄的 `health` 探针，只要能支持发现、联系人建立、异步投递和最近可达判断，就够了。

## 第二阶段消息与线程

等第一阶段站稳后，再进入协作线程。

第二阶段最小集合：

- `ask`
- `claim`
- `patch`
- `review`
- `reject`
- `handoff`

线程是消息容器，不是任务真相源。它不能直接推进 `issues/` 状态，也不替代当前 `issues/` 主线。后续如果引入 owner，owner 也只是线程当前响应方，不是 `issues/` 的任务 owner。

## 结果交换

节点之间最应该交换的是：

- 结果摘要
- artifact
- patch
- review
- evidence

而不是远程控制命令。

这条线更像邮件列表里的补丁流，不像远程操纵别人桌面的系统。

## 可见性

这条线必须默认可审计。

用户应该能：

- 看到联系人列表
- 看到某个线程完整聊天记录
- 看到某条消息关联的 artifact
- 看到谁是 owner
- 看到某次 handoff 和 review 是怎么发生的

如果用户只能看到最后结果，看不到协作过程，这条线很快会丧失可信度。

第一阶段 UI 上允许的最小字段：

1. 当前节点
2. 最近探测时间 / 可选延迟
3. 联系人状态：`discovered / known / contact`
4. 邮箱记录

第一阶段明确不写：

- 最近活跃
- 正在工作中
- 正在处理什么

## 第一版骨架

第一版只做以下骨架：

### A. 节点侧

- 邻居模块开关
- 身份声明
- 能力声明
- 联系人持久化
- `health` 入口

### B. 通信侧

- 邻居发现
- 握手
- 异步消息投递
- 最近一次可达探针
- 简单重试和离线容忍

第一阶段到此为止。

## 第二版骨架

第二阶段才进入：

- 线程文件真相源
- `ask/patch/review/handoff`
- 单线程单响应方
- artifact 引用
- 聊天记录式查看
- export surface 的结果导出接入

第二阶段仍然不做真正群聊状态机，不做复杂编排。

## 第一版假设

为了控制复杂度，第二阶段再按这个假设走：

- 联系人可多，但单次协作线程默认只有一个当前响应方
- 其他联系人可以围观或评论，但不共享状态推进权
- 可以有类似“抄送”的参与者，但不做真正多人对等群聊协议

如果后续这条路线被证明有效，再评估是否扩到更复杂的群聊与多人协作。

## 与主链的关系

这条线是可选模块，不接管现有 dispatch / heartbeat / subagent 主链。

它更像：

- 一条新的 I/O / 协作能力面
- 一个“向别的 msgcode 求助”和“被别的 msgcode 求助”的能力边界

后续如果路线正确，再考虑把它逐步收到主能力里。但第一版先按可删模块设计，不先绑死到主链。

## 阶段路线

第一阶段先只做：

- 邻居模块开关
- 发现与联系人三态
- 异步邮箱
- 简单重试和离线容忍

第二阶段再考虑：

- 协作线程
- patch/review/handoff 合同
- artifact 引用
- 聊天记录式查看
- export surface 的结果导出接入

第三阶段才评估：

- 更复杂的多人协作
- 更丰富的联系人能力市场
- 是否从可选模块收进主能力面

## 验收口径

这条 plan 成立的最小验收不是“多智能”。

最小验收是：

- 节点能被发现
- `discovered / known / contact` 三态能成立
- 联系人能建立
- 对方下线不会丢邮箱与联系人关系
- 能异步投递和补投消息
- 默认不共享完整 workspace，只共享显式 export surface
