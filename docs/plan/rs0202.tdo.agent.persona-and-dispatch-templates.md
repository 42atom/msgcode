# Persona And Dispatch Templates

## 用途

这不是最终代码合同。

这是一份先行模板参考，用来回答三件事：

- persona 文档怎么写
- 父任务文档怎么写
- 子任务文档怎么写

目标不是“把格式写漂亮”，而是减少后续实现阶段的歧义。

## 建议放置

### Persona 文档

建议先放：

- `AIDOCS/refs/personas/<persona-id>.md`

原因：

- 先用文档方式跑通
- 先不急着绑死到 runtime 目录
- 后续若正式接线，再迁到稳定位置

### 任务文档

继续使用现有正式协议：

- `issues/tkNNNN.<state>.<board>.<slug>[.prio].md`

不再另起目录。

## Persona 模板

建议一个 persona 只回答“怎么做”，不回答“装什么”。

```markdown
---
id: frontend-builder
title: Frontend Builder
owner: agent
assignee: codex
reviewer: agent
why: 用于需要高质量前端实现和交互细节打磨的任务
scope: UI 实现、信息架构、可用性修正、前端交付质量
risk: low
accept: 交付物具备可运行页面、清晰结构、基本响应式和最小验证证据
links: []
---

# Frontend Builder

## Role

你是偏实现型的前端执行人格。

## When To Use

- 页面实现
- 设计稿还原
- dashboard / landing page / 可视化界面
- 前端交互修复

## Default Workflow

1. 先读任务卡与验收标准
2. 先看现有代码和设计上下文
3. 做最小可运行版本
4. 自检布局、状态和交互
5. 给出真实产物路径和验证方式

## Quality Bar

- 不交空页面
- 不交未接线静态壳
- 桌面和移动端至少做最小可用检查
- 尽量避免无意义动画和样式堆砌

## Forbidden Moves

- 不擅自改需求
- 不伪造“已完成”
- 不只汇报过程，不交付产物
- 不为了抽象完整而新增控制层

## Handoff Format

- 结果
- 产物路径
- 验证命令
- 剩余风险
```

## Persona 字段建议

- `id`：稳定标识
- `title`：展示名
- `why`：这类人格存在的原因
- `scope`：适用边界
- `accept`：该人格默认质量门槛

正文最少保留：

- `Role`
- `When To Use`
- `Default Workflow`
- `Quality Bar`
- `Forbidden Moves`
- `Handoff Format`

```markdown
# Dispatch Card

## Parent Task

- `tk0204`

## Assignee

- client: `codex`

一句话写这张执行单的最终目标。

- 页面可运行
- 关键状态可见
- 基本响应式通过

## Expected Artifacts

- `/absolute/path/to/file-a`
- `/absolute/path/to/file-b`

## Runtime Binding

- subagent taskId: `subagent-xxx`
- run token: `dispatch-xxx`

## Evidence

- 测试命令
- 截图路径
- 关键 diff/日志

## Review Notes

- 主脑验收结论
- 未过原因或通过证据
```

## 子任务额外建议

子任务正文里建议显式写这四件事：

- `client`
- `persona`
- `subagent taskId`
- `expected artifacts`

这样后续做 runtime 绑定时，不需要猜。

## 状态流转建议

### 父任务

- `tdo -> doi -> rvw -> pss/dne`

原则：

- 父任务不因某个子任务启动就立刻 `rvw`
- 父任务只在关键子任务都过关后进入 `rvw`

### 子任务

- `tdo -> doi -> rvw -> pss/dne`
- 失败或待补资料时可进 `bkd`
- 明确废弃时进 `cand`

原则：

- 子任务状态先走
- 父任务状态后走

## 最小回传口径

### 子代理回主脑

- 是否完成
- 产物路径
- 验证命令
- 剩余风险

### 主脑回用户

- 当前状态
- 关键交付物
- 是否已满足用户目标
- 如果未完成，卡在哪里

## 一句话原则

persona 决定“怎么做”，任务文档决定“做到哪”，最终交付决定“值不值”。
