# Plan: Doc Filename Protocol Migration

## Problem

当前文档命名是三套协议并存：

1. `issues/NNNN-<slug>.md`
2. `docs/plan/plNNNN.<state>.<board>.<slug>.md`
3. `docs/plan/rsYYMMDD-<topic>.md`

这套协议可用，但文件名本身不呈现状态；而仓库目标规范要求文件名成为状态真相源的一部分。继续局部修补只会让两套协议长期共存。

## Decision

本轮采用一次性迁移方案，目标协议如下：

### 1. 任务事实（`issues/`）

- 目标目录：`issues/`
- 命名：`tkNNNN.<state>.<board>.<slug>[.<prio>].md`
- 状态映射：
  - `open -> tdo`
  - `doing -> doi`
  - `done -> dne`
  - `blocked -> bkd`
  - `wontfix -> cand`

说明：
- 现有 `issues/NNNN-<slug>.md` 直接迁成 `tkNNNN...`
- 现有 issue id 保留，不重编号
- 历史已完成 task 默认可不补 `prio`

### 2. 计划 / 研究 / 参考 / 报告（正式文档）

- 新目标目录：`docs/plan/`
- 命名：
  - `plNNNN.<state>.<board>.<slug>.md`
  - `rsNNNN.<state>.<board>.<slug>.md`
  - `rfNNNN.<state>.<board>.<slug>.md`
  - `rpNNNN.<state>.<board>.<slug>.md`

说明：
- `docs/design/` 迁入 `docs/plan/`，统一改为 `pl*`
- `docs/notes/` 迁入 `docs/plan/`，按内容分类为 `rs*` 为主
- 需要长期保留的评审、验收、复盘文档归到 `rp*`
- “同一主题尽量共用同一 id”，没有关联 task 的历史资料再补新 id
- 用户已确认接受 `docs/design/` 与 `docs/notes/` 合并迁到 `docs/plan/`

### 3. 时间线与归档

- `docs/tasks/` 保留为“时间线索引目录”，不再承担正式 task 真相源
- `docs/archive/` 保留归档职责，但清退 vendor/build 噪声文件

## Why

1. 文件名直接带状态，检索时不再依赖先打开 front matter。
2. 同一种文档进入同一种命名，不再混用“日期命名”和“编号命名”。
3. `issues/` 保持 task 真相源，`docs/plan/` 统一承接计划/研究/参考/报告，结构更直。
4. 迁移后可直接用 `ls` / `rg` / `find` 按状态、board、优先级筛选。

## Mapping Rules

### `board` 来源

按以下顺序确定：

1. 现有文件名中的模块词（如 `auth/api/runtime/tools/ghost`）
2. `labels` / `scope` 中的稳定模块词
3. 若仍不明确，先归入 `runtime` 或 `product`，进入人工复核名单

### `slug` 规则

- 只保留主题，不重复 kind/state/board
- 全部转为 `[a-z0-9-]`
- 删除日期、issue 编号、阶段编号等外部信息

### `prio` 规则

- 仅 `tk*` 可带 `p0|p1|p2`
- 只给当前活动或明确标记过优先级的 task 回填
- 历史 done task 不强制补优先级，避免伪精确
- 用户已确认：已完成 task 默认不补 `prio`

### ID 规则

1. `tk` 复用现有 issue id
2. 与某个 task 强绑定的 `pl/rs/rf/rp` 尽量复用同 id
3. 无绑定关系的历史文档统一从 `9000+` 新号段补 id

## Execution Batches

### Batch 0：冻结合同

- 新建 `docs/plan/README.md`
- 改写 `docs/README.md`、`issues/README.md`
- 修改 `scripts/check-doc-sync.ts`，让它校验新命名协议与允许状态

### Batch 1：迁移 `issues/`

- 生成 rename map：`issues/NNNN-*.md -> issues/tkNNNN.*.md`
- 每批 30-50 个文件，`git mv` 后立即更新引用
- 同步把旧 `status:` 文本状态收口为文件名状态槽

### Batch 2：迁移 `docs/design/` 与 `docs/notes/`

- 创建 `docs/plan/`
- `docs/design/*.md -> docs/plan/pl*.md`
- `docs/notes/*.md -> docs/plan/rs*.md`
- 完成后让旧目录只保留 README/迁移说明，最终退出

### Batch 3：收口 `docs/tasks/` 与 `docs/archive/`

- `docs/tasks/` 只保留时间线与派单索引
- 清点 `docs/archive/**/.build/**`、vendor docs、重复 README
- 删除动作走单独清单，必要时由用户手工确认执行
- 用户已确认允许删除 `docs/archive/retired-desktop-bridge/mac/msgcode-desktopctl/.build/checkouts/**`

### Batch 4：验收

- `rg` 清零旧路径引用
- `npm run docs:check`
- 抽查关键入口：根 `README.md`、`docs/README.md`、`issues/README.md`
- 验证状态视图是否可直接按文件名检索

## User Help Needed

你可以重点帮两类事情：

1. 删除确认
   - `docs/archive/retired-desktop-bridge/mac/msgcode-desktopctl/.build/checkouts/**`
   - 其他 build/vendor 产物型文档
2. 模糊字段拍板
   - 少数历史文件的 `board`
   - 是否给历史 task 补 `prio`

## Risks

1. 旧链接断裂：仓库内约 246 处 `issues/NNNN-*.md` 引用需要统一替换。
2. `board` 推断不准：必须先产出 rename map 再人工过一遍。
3. 目录迁移过大：`docs/design`/`docs/notes` -> `docs/plan` 需要分批，不可一把梭。
4. archive 删除不可逆：删除类动作必须单列审批，不与 rename 混批。

## Deliverables

1. 一份全量 rename map
2. 一套新协议检查脚本
3. 新的目录入口与状态视图
4. 删除候选清单
5. 机器生成的人工复核清单（board / prio）

## Acceptance

迁移完成后，用户无需打开正文，只看文件名和目录就能判断：

- 这是什么类型的文档
- 当前处于什么状态
- 属于哪个 board
- 是否高优先级
