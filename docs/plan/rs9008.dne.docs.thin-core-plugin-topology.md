# 薄 Core 与非传统插件体系整理

## 目标

冻结一条统一判断，避免后续把 `msgcode` 做厚：

- `msgcode` 只保留薄 core
- 复杂能力通过“非传统意义的插件”增强
- 页面 / 控制台 / PWA 只是这些插件能力的未来投影，不是插件本体

这里的“插件”不是传统 IDE / npm / VSCode 式插件，
而是：

**一组可挂接在通用任务 harness 之上的能力协议、对象模型与工作方式。**

## 核心结论

`msgcode` 的目标形态应当是：

**薄 Core + 多种能力插件 + 后续可选的展示面。**

### 薄 Core 负责

- run lifecycle
- session 统一语义
- task / blocked / resume
- context policy
- channel routing
- artifact / delivery
- human handoff

### 插件负责

- 给智能体增加新的工作方式
- 给 run/task 增加新的对象模型
- 给复杂任务增加新的治理协议
- 在不增厚 core 的前提下，扩展能力边界

### 展示面负责

- 把插件能力投影给用户看
- 提供可视化、介入、治理、回放
- 不是系统主脑，只是消费 core + plugin 的结果

## 为什么这里要坚持“非传统插件”

因为当前项目的插件，不是：

- 一个小工具入口
- 一个简单脚本
- 一个 UI 扩展按钮

而是：

**一套可选启用的能力协议。**

它们通常包括：

- 对象模型
- 状态机
- 行为约束
- 触发条件
- 输出格式
- 与 core 的边界

所以这里的 plugin，更接近：

- capability pack
- behavior protocol
- task mode
- execution governance layer

而不是传统意义的 package extension。

## 三层结构

### 第一层：薄 Core

永远保持小而稳定。

核心原则：

- 不替插件做决策
- 不替专项执行引擎做专业判断
- 不把高层产品形态提前焊死到内核

### 第二层：插件能力

插件是主扩展位。

#### A. 专项执行引擎插件

例子：

- coding agent harness
- browser capability pack
- vision capability pack
- future local service capability pack

这类插件回答：

**某类专项任务该由谁执行、如何调用、结果如何回到 run/session。**

#### B. 重任务治理插件

例子：

- transparent execution plugin
- heavy-task governance plugin

这类插件回答：

**复杂项目任务如何拆、如何汇报、何时停下、何时请求用户介入。**

它增强的是智能体的“复杂任务治理能力”，不是页面本身。

#### C. 交付/展示投影插件（后续）

例子：

- transparent execution console
- project dashboard
- review surface

这类不是 core，也不是 task intelligence 本体，
而是把插件能力投影给用户的消费层。

#### D. 基础沟通识别插件

例子：

- character-identity skill

这类插件回答：

**在群聊或多角色沟通中，当前发言人是谁、之前见过没有、忘记时该查哪份本地对照表。**

它们依赖 core 暴露的原始身份事实（如 `channel/chatId/senderId`），
但人物别名、角色、备注、自我介绍维护都不进入 core。

## 冻结的架构判断

以后判断一个设计该归到哪里，只问这三个问题：

### 1. 这是 core 吗？

如果它回答的是：

- run 如何统一
- session 如何统一
- context 如何统一

那它属于 core。

### 2. 这是插件吗？

如果它回答的是：

- 智能体新增了一种复杂工作方式
- 新增一套任务治理/对象/状态协议
- 这套能力不是所有任务都必须开启

那它属于插件。

### 3. 这是展示面吗？

如果它回答的是：

- 用户怎么看见这些对象
- 如何可视化、筛选、审批、介入

那它属于 surface / console。

## 冻结的产品判断

`msgcode` 后续会有不少插件，但不是传统意义的插件。

更准确地说：

**它会有越来越多的能力协议插件，而薄 core 只提供统一 runtime substrate。**

## 非目标

- 本文不引入插件平台
- 本文不引入 plugin manager
- 本文不改变当前 Agent Core Phase 2 / Phase 3 排期
- 本文不把 future console 提前推进成当前实现

## 后续约束

以后新增任何“插件”相关设计前，先回答：

1. 这是不是在增强智能体的一种工作方式？
2. 这层能否不进入 core，而作为可选能力协议存在？
3. 这是不是只是某个 future UI 的需求，误被提前塞进 core？

如果第 3 条为“是”，默认不进 core。
