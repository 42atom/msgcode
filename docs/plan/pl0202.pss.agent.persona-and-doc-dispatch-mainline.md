# Persona And Doc Dispatch Mainline

## 问题

当前仓库已经有两块现成能力：

- `/task` + `TaskSupervisor`：能管理显式长任务
- `subagent`：能把任务交给 `codex / claude-code`

但还缺两层正式合同：

1. 子代理到底靠什么“人格”工作
2. 派单、巡检、验收到底以什么为真相源

如果这两点不收口，就会继续出现：

- 不同 agent 需要不同技能包的误解
- 派单状态只存在运行时，不在文档里
- 用户看不懂过程，只能被动等结果
- 任务看板和实际执行状态脱节

## 目标

- 让 `skills` 成为公共能力池，不再按 agent 复制安装
- 让 `persona` 成为角色装配说明书，而不是新的执行层
- 让 `issues/*.md` 成为父子任务派单、验收、交付的正式真相源
- 让主脑负责拆单、派单、巡检、验收、汇报
- 让用户只通过任务文档和最终交付物感知状态

## 非目标

- 不引入新的 orchestration 平台
- 不新增第二套任务状态数据库去覆盖文件名状态
- 不按 client 复制技能目录
- 不为了“更完整”先做多 agent 控制面

## 单一主链

主链固定为：

`用户目标 -> 父任务文档 -> 子任务文档 -> subagent 执行 -> 主脑验收 -> 父任务交付`

角色分工固定为：

- `skills`：表达可用能力
- `persona`：表达工作风格、方法和质量门槛
- `subagent client`：表达执行臂
- `issues/*.md`：表达状态真相源
- `AIDOCS/reports/*` 或 workspace 产物：表达最终交付

## 方案 A：公共技能池 + Persona 装配

### 决策

- 所有子代理默认共享同一套公共 skills
- 不再按 `codex / claude-code / local` 各自复制技能包
- 主脑启动子代理时，显式提供：
  - client
  - persona
  - task card

### Persona 应表达什么

- 适用场景
- 偏好工作流
- 质量门槛
- 禁止动作
- 回传格式

### Persona 不应表达什么

- 安装逻辑
- 独立工具清单
- 第二套状态系统
- 绕过主脑的自主汇报机制

## 方案 B：文档即派单控制面

### 决策

- 用户目标对应一个父任务文档
- 每次正式派单对应一个子任务文档
- 主脑先验收子任务，再推进父任务
- 用户默认只验父任务和最终交付

### 建议关系

- 父任务：用户目标
- 子任务：执行单
- 子任务 `assignee`：`codex / claude-code / local / ...`
- 子任务 `reviewer`：默认 `agent`
- 父任务 `reviewer`：默认 `user`

### 文档绑定建议

子任务正文中至少写清：

- 派单目标
- 执行工作目录
- 预期产物
- 验收标准
- 关联父任务
- 关联 subagent taskId

## 执行顺序

1. 先收口 persona 协议  
   防止后面把“谁执行”和“怎么做”继续写散。

2. 再收口父子任务文档协议  
   让派单行为进入文档真相源。

3. 再补 `/task` 与 `subagent` 绑定  
   让运行时知道如何把状态回写到父子任务。

4. 最后再看外部优秀实践  
   只做参考校正，不在当前阶段抢先平台化。

## 任务拆分

- [ ] `tk0202` 共享技能池与 persona 文档协议
- [ ] `tk0203` doc-first 派单、巡检、验收与交付监督
- [ ] `rs0202` persona / 父任务 / 子任务模板参考

## 验收口径

1. 主脑可把一个用户目标拆成父子任务文档。
2. 子代理启动时可以显式带上 persona，而不是依赖独立技能包。
3. 用户可以只通过 `issues/` 和任务看板看进度，不依赖 tmux 过程。
4. 子任务完成后，主脑必须先验收真实产物，再推进父任务状态。
5. 后续参考外部实践时，不破坏这套 file-first 真相源。
