# PL0265：Appliance Bootstrap Without Brew Or Npm Install

## 结论

`0265` 不该继续讨论“怎么优化 bootstrap 脚本”，而该直接回答：

1. 怎么组装 Core Appliance 包
2. 怎么安装到目标机器
3. 怎么首次生成本地真相源
4. 怎么升级而不回退到 `brew` / `npm install`

一句话：

- 开发者 bootstrap 继续留在仓库
- appliance bootstrap 另起正式主链

## 当前问题

现有 bootstrap 仍然服务开发者环境：

1. `bootstrap/bootstrap.sh`
   - 先找 Homebrew
   - 再跑 `brew bundle`

2. `bootstrap/bootstrap-agent-pack.sh`
   - 继续走 Homebrew 安装增强工具

3. `bin/msgcode`
   - 仍依赖 `tsx` 与源码目录

所以现在的问题不是“脚本写得不够好”，而是：

- 主入口就不是交付入口

## 目标

`0265` 的目标是定义一条新的正式安装主链：

1. Package Assembly
2. Install
3. First-Run Init
4. Upgrade

## 非目标

1. 不直接决定 DMG/PKG 二选一
2. 不实现 Electron 壳
3. 不实现 pack 下载器
4. 不动当前开发者 bootstrap

## Phase 1：Package Assembly

要产出一份 Core Appliance 包，最少包含：

1. 已构建的 `msgcode` 入口
2. Node runtime
3. 核心 JS 与依赖
4. memory 核心依赖
5. `SOUL/ORG` 模板
6. 最小诊断入口

这一层的验收是：

- 包体已经能替代 `bin/msgcode + npm install + tsx`

## Phase 2：Install

安装过程只做三件事：

1. 把 Core Appliance 放到固定安装目录
2. 写入最小启动入口
3. 预置全局配置根骨架

这里补一条硬规则：

- Appliance / App 的基础运行依赖必须明确分成两类：
  1. 随包携带
  2. 预安装软件必须清单
- 凡属于基础运行依赖，不允许继续靠运行时“碰巧存在”。
- install / doctor 必须前置报出缺失项；运行时不再把这类缺失改写成模型能力问题。

安装过程不应做：

1. 现场联网下载
2. 现场跑 `brew`
3. 现场跑 `npm install`
4. 现场编译源码

## Phase 3：First-Run Init

首次启动只生成最少真相源：

1. `~/.config/msgcode/.env`
2. `~/.config/msgcode/souls/default/SOUL.md`
3. `~/.config/msgcode/souls/active.json`
4. `<workspace>/.msgcode/SOUL.md`
5. `<workspace>/.msgcode/ORG.md`
6. `<workspace>/memory/`
7. `<workspace>/AIDOCS/reports/`

`ORG.md` 第一版只允许：

1. 名称
2. 交税地
3. 统一社会信用代码

## Phase 4：Upgrade

升级只允许替换：

1. Core Appliance 运行物
2. 默认模板资源
3. 可升级的 pack

升级不应覆盖：

1. `ORG.md`
2. `SOUL.md`
3. `memory/*.md`
4. `issues/`
5. `AIDOCS/`

## 预安装软件必须清单

如果某项依赖既不随包携带，又是正式主链的基础运行条件，它就必须进入“预安装软件必须清单”。

这份清单的独立真相源是：

- /Users/admin/GitProjects/msgcode/docs/plan/rf0002.rvw.product.appliance-required-preinstall-software.md

这份清单的规则是：

1. 它服务 appliance / App，不服务开发者 bootstrap。
2. 缺任何一项，都应在 install / doctor 阶段 fail-closed。
3. 运行时不再继续猜，不再临时降级，不再把缺失伪装成“AI 无法访问本地设备”。

以当前仓库事实看，托管 Bash 就属于这类依赖：

1. 若将来 App 不自带 Bash runtime，就必须把托管 Bash 列入预安装软件必须清单。
2. 若将来 App 自带 Bash runtime，就应把它从预安装清单移入随包携带内容。

一句话：

- 基础依赖只能有两种命运：随包，或预安装强校验。
- 不允许再留“运行时才发现缺了”的灰区。

按当前仓库事实，已经明确应进入清单的项有两层：

### A. Core Appliance 基础项

1. 托管 Bash
   - 若 App 不自带 Bash runtime，就必须预装
   - 当前正式路径：`/opt/homebrew/bin/bash` 或 `/usr/local/bin/bash`
2. `zstd`
   - 若 App 继续通过当前 `workspace-wpkg-install` 主链安装 `.wpkg`，且不自带 `zstd`，就必须预装

### B. 可选能力包基础项

1. Browser pack
   - `Google Chrome / Chromium`
2. Desktop pack
   - `ghost-os`
3. tmux coding pack
   - `tmux`

这些能力默认不属于 Core Appliance 基础项；只有用户安装对应能力包时，才进入那台机器的预安装检查范围。

## 建议的安装包结构

建议最小结构：

```text
msgcode-appliance/
├── runtime/
│   ├── bin/msgcode
│   ├── node/
│   ├── app/
│   └── deps/
├── templates/
│   ├── SOUL.default.md
│   └── ORG.default.md
├── install/
│   ├── install.sh
│   ├── first-run-init.sh
│   └── doctor.sh
└── packs/
    └── (optional, empty by default)
```

## 与现有 bootstrap 的关系

保留：

1. `bootstrap/bootstrap.sh`
2. `bootstrap/bootstrap-agent-pack.sh`
3. `bootstrap/Brewfile*`

但它们降级为：

- 开发者 bootstrap

不再作为：

- appliance 正式安装主链
