# Work Vitals And Session Continuity Stack

## 问题

msgcode 现在已经逐渐逼近“AI 基础操作系统”的主线，但下一段最容易走歪的地方也很明显：

- 如果先做 `session continuity`，系统会越来越像“会聊天的 agent”
- 如果先做 `work continuity`，系统才会先成为“不会丢活的工作者”

当前仓库已经有这些积木：

- `issues/` 文档协议
- `/task` 与 `TaskSupervisor`
- `subagent`
- `heartbeat`
- `scheduler / jobs`

但它们还没有被收成一条明确的架构栈。

## 结论

下一段主线必须按这个顺序建：

1. `truth continuity`
2. `work continuity`
3. `process continuity`
4. `vitals / self-protection`
5. `session continuity`
6. `persona continuity`

这里面，真正要先落地的只有三刀：

1. `work continuity foundation`
2. `vitals projection`
3. `session continuity on top`

## 轻路径 / 重路径

不是所有请求都该进入 `work continuity`；入口要先分轻路径和重路径。

### `run path`

适用：

- 用户发的是一个具体问题
- 用户发的是一个一次性命令
- 不要求跨 session 续跑
- 不要求派单、巡检、交付闭环

特征：

- 一次一 run
- 不强制创建 task document
- 不强制写 dispatch / checkpoint
- 对齐轻量消息流和即时工具调用

### `task path`

适用：

- 用户发起的是需要多轮推进的工作
- 任务需要跨 session 续跑
- 任务需要派单、巡检、验收或交付
- 任务一旦中断，不允许靠聊天记忆凑恢复

特征：

- 进入 `issues + dispatch + checkpoint`
- 明确恢复协议
- 明确验收与交付

### 区分原则

主判断看用户意图，不要求用户显式选模式。

- 具体问题 / 单次命令 -> `run path`
- 长工作 / 多轮推进 / 需要交付 -> `task path`

这层判断应该放在入口，不该让所有请求都掉进重路径。

## 为什么顺序不能反

### 1. session continuity 不是地基

会话连续性当然重要。

但如果系统只记得：

- 聊过什么
- 说话口气
- transcript 历史

却不记得：

- 还欠什么
- 做到哪
- 从哪继续
- 证据在哪

那它只是一个健谈系统，不是可靠工作者。

### 2. work continuity 才决定信任

真正的信任建立在：

- 任务真相源
- 派单真相源
- artifact 真相源
- verify / log / evidence
- checkpoint / blocked / resume / restart

这些东西如果不硬，session 再顺也只是假连续。

### 3. vitals 是自然长出来的，不该独立造神

`vitals` 不是一个新控制面。

它只是从真实任务、日志、运行时状态中投影出来的一组压力信号，供 heartbeat 做背压和自保。

最像 Unix 的方式不是“发明情绪模块”，而是：

- 像 `/proc` 一样提供视图
- 像 `load average` 一样提供压力
- 像 `signal` 一样触发策略

## 架构栈

## 1. Truth Continuity

这一层只回答一件事：

- 什么是真的

最小真相源：

- `issues/*.md`
- `<workspace>/.msgcode/dispatch/*.json`
- artifact 路径
- verify / run / log 证据

规则：

- 真相源必须可见、可 grep、可迁移
- 不允许隐藏数据库覆盖这些文件

## 2. Work Continuity

这一层只回答一件事：

- 挂了以后还怎么接着干

最小协议：

- 显式恢复指针（主锚点对齐 `checkpoint.nextAction`）
- checkpoint
- blocked / resume
- retry
- parent / child / dispatch 绑定
- restart 后继续点

这一层的目标不是聊天自然。  
而是“不会丢活”。

关键要求：

- 系统不能靠翻 transcript 猜“刚才做到哪了”
- 必须存在显式恢复指针，告诉主脑恢复后的首个执行动作

### 恢复指针协议

主收口直接复用现有 runtime 的 `checkpoint` 语义，不额外发明第三套指针系统。

最小字段：

- `currentPhase`
- `summary`
- `nextAction`
- `updatedAt`

存放位置：

- 父任务 / 主任务：runtime task record 的 `checkpoint`
- 委派子任务：`<workspace>/.msgcode/dispatch/*.json` 镜像最小 `checkpoint`
- 子代理执行臂：`<workspace>/.msgcode/subagents/*.json` 只记录执行臂状态，不升级为真相源

更新时机：

- 每次 durable checkpoint 写盘
- 进入 `blocked`
- `resume` 对齐后
- 进入 `completed / failed`

不这样做：

- 不把恢复指针塞进任务文件 front matter
- 不要求每个 tool event 都强刷一次指针
- 不允许靠 transcript 猜恢复后的首个 action

### Record 边界表

| Record | 路径 | 承载 | 不承载 |
|---|---|---|---|
| task document | `issues/*.md` | 任务身份、文件名状态、验收、交付链接 | 易变运行指针、pane tail、瞬时错误风暴 |
| dispatch record | `<workspace>/.msgcode/dispatch/*.json` | parent/child/client/subagent 绑定、最小 checkpoint、artifact refs、派单恢复信息 | 文件名状态真相、用户验收结论 |
| subagent record | `<workspace>/.msgcode/subagents/*.json` | 执行臂 session/group/status/done marker | 父任务业务状态、最终验收真相 |
| task runtime record | task store 持久化 | supervisor checkpoint、attempt/retry、blocked/recovery context | 用户可见任务索引、派单拓扑 |

### `work capsule` 最小结构

恢复时不让模型自己考古，而是 runtime 先拼一份最小工作胶囊。

最小字段：

- `taskId`
- `phase`
- `checkpoint`
  - `summary`
  - `nextAction`
  - `artifactRefs`
  - `evidenceRefs`
- `activeDispatch`
  - `subtaskIds`
  - `blockedBy`
  - `subagentRefs`
- `nextAction`
  - `type`
  - `params`
  - `evidenceRequired`
- `vitalsSnapshot`（optional）
  - `load`
  - `risk`
  - `policyMode`

目标不是完整回放历史，而是让模型一眼知道：

- 这是什么任务
- 现在做到哪
- 下一步做什么
- 缺什么证据

### 工程约束

- `single-writer`
  - 同一 workspace 的真相文件一次只允许一个写入者
  - 其他 session 默认只读或排队，不直接改真相源
- `atomic write`
  - runtime records 必须以原子方式落盘
- `drift detection`
  - 恢复时比对 task document、dispatch、subagent、runtime checkpoint
  - 一旦发现不一致，进入保守模式，不假装一致

## 3. Process Continuity

这一层回答：

- 具体哪个运行时还活着

最小对象：

- subagent session
- browser / desktop harness
- heartbeat runner
- scheduler / cron wakeup
- resource lease

这一层可以丢。  
但丢了之后必须能从前两层恢复。

## 4. Vitals / Self-Protection

这一层回答：

- 现在该继续顶着跑，还是该降级自保

最小信号：

- `load`
- `stall`
- `risk`
- `headroom`
- `readiness`

派生窗口：

- `explore_window`

最小梯度：

- `defer`
- `degrade`
- `shed`
- `kill`
- `restart`

关键约束：

- `vitals` 是投影，不是真相源
- 背压梯度是 runtime policy，不是任务状态
- 每个信号都必须能回溯到物理事实，像 `loadavg` 一样可验证
- 实现命名优先使用 Unix 风格短词，不优先使用叙事型心理词
- `kill / restart` 必须遵守最小作用域优先

### Global Gate

单 workspace 的 `vitals` 不足以回答整机级“唤醒风暴”。

如果多个 workspace 在同一时刻同时 wake，系统还需要一个极薄的全局资源闸门。

最小原则：

- 不新建中心调度器
- 不把全局 gate 做成第二状态系统
- 只为少数真正共享的高价值资源建 gate

建议最小资源：

- `llm-tokens`
- `browser`
- `desktop`
- `subagent:<client>`

建议最小实现：

- 在 `~/.config/msgcode/runtime/gates/` 或系统 `tmp` 下使用文件锁
- workspace 在消费 wake / 派发高成本动作前先 claim 对应 gate
- claim 失败时不算任务失败，而是进入 `defer`

一句话：

- workspace 级 `vitals` 负责局部自保
- `global gate` 负责跨 workspace 背压

### `readiness / explore_window` 的物理映射

- `readiness`
  - 工具可用性
  - 显式恢复指针是否可读且新鲜
  - verify 缺口是否阻塞
  - 必需资源是否已就绪
- `explore_window`
  - 不是一等信号
  - 只从 `load / stall / risk / headroom` 组合推导
  - 且必须存在待探索候选项

### `vitals` 输出结构

- `signals`
  - `load`
  - `stall`
  - `risk`
  - `headroom`
  - `readiness`
- `derived`
  - `explore_window`
- `reasons`
  - 物理事实来源与解释
- `policy`
  - `mode`
  - `auto`
  - Phase 1 只允许 `normal / defer / degrade`
- `computedAt`

### `vitals` 失败回退

`vitals` 不是地基，所以计算失败时不能拖垮主链。

最小回退策略：

- 进入保守模式
- 不开新坑
- 不派新子任务
- 只做 verify / checkpoint / 收尾
- `truth continuity / work continuity` 仍保持可恢复

## 5. Session Continuity

这一层回答：

- 会话怎么接得自然

最小职责：

- transcript continuity
- memory injection
- compaction / rebuild
- natural conversation flow

关键约束：

- session 必须消费前面的工作真相源
- session 不能反过来成为任务真相源
- session 更像感官层缓存，恢复时优先即时重构必要上下文，而不是同步沉重 transcript 状态

### rebuild 输入 -> 处理 -> 输出

输入：

- `issues/*.md`
- dispatch records
- task checkpoint
- subagent records
- artifact / verify 证据
- 可选 transcript tail / memory summary

处理：

- 先回答：
  - 还欠什么
  - 做到哪
  - 从哪续
  - 证据在哪
- 再选择活跃恢复指针、派单绑定、活动执行臂

输出：

- 一个最小 work capsule
- 供 session bootstrap / prompt injection 使用
- 不要求全文 transcript 回放

## 6. Persona Continuity

这一层回答：

- AI 看起来还是不是同一个长期工作者

最小对象：

- persona docs
- stable work style
- long-term voice / quality bar / decision posture

这一层永远排在最后。  
因为它最像“结果收益”，不是地基层。

## 落地顺序

### Phase 1

先做 `tk0205`：

- dispatch truth
- recovery foundation
- work continuity protocol
- 输出结构化 `work recovery protocol` 供 `tk0206 / tk0207` 消费

### Phase 2

再做 `tk0206`：

- vitals projection
- heartbeat policy
- 自保梯度第一阶段

### Phase 3

最后做 `tk0207`：

- session rebuild
- transcript / memory / compaction
- 在工作地基上补自然会话连续性

## Guardrails

- 不新建第二任务状态系统
- 不新建情绪数据库
- 不新建总控平台
- 不让 session 抢真相源地位
- 不让 vitals 越过任务决策边界
- 不让 kill / restart 越级先作用于整机
- 不允许用 transcript 猜恢复点

## 一句话收口

这段主线真正要做的，不是“让 AI 更像人在聊天”。

而是：

- **先让 AI 成为一个不会丢活的长期工作者**
- **再让它成为一个会自然续上的长期对话者**
