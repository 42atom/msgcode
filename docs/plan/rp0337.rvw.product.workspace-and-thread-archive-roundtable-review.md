# RP0337：工作区与线程归档圆桌审议

Decision:
- Recommendation: go
- Confidence: high
- Scope: 先按“工作区目录移动 + 线程文件移动 + 单独历史页”进入后续拆单

Why:
- `SystemsArchitect` 认为，归档最适合用目录移动表达，不需要额外归档表或状态中心。
- `SkepticalOperator` 认为，`/clear` 与 `archive` 必须强分；一个是清上下文，一个是藏历史。
- `UserAdvocate` 认为，主界面必须只看活跃内容，历史查看应单独成页，否则小白会被旧线程淹没。
- `ExecutionLead` 认为，第一版最稳的是只拆三刀，不提前做搜索、自动归档和 session 搬运。

Risks:
1. 若线程归档同时搬动 session 原文，恢复链会立刻变重。
2. 若主界面仍混入归档内容，archive 就失去意义。
3. 若恢复时自动重命名，会制造新的隐式状态。

Guardrails:
- 前置条件:
  - 工作区 archive 只通过目录移动实现
  - 线程 archive 只通过 `threads/ -> archived-threads/` 移动实现
- 监控指标:
  - 主界面是否仍然只读活跃目录
  - 历史页是否只承担查看与恢复
- 回滚条件:
  - 一旦实现要求新增 archive registry、session archive controller、自动 reconcile，立即回滚到 file-move 基线

Next Actions:
1. 拆 workspace archive root 与 restore 合同
2. 拆 thread archive 与 `/clear` 边界合同
3. 拆历史归档页与 restore 入口合同
