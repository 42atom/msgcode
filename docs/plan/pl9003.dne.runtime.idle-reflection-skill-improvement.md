# Idle Reflection Skill Improvement MVP

## Problem

我们希望吸收 Hermes 的核心精华：让智能体把真实任务里的可复用方法沉淀为更好的 `skill`，而不是每次从零摸索。

但 `msgcode` 不能照搬 Hermes 的重平台路线。当前仓库已经明确要求：

- 系统做薄
- 优先服务 LLM，不替 LLM 做主
- 优先通过 `skill` / prompt / 合同改进能力，而不是新增控制层

如果把“自我改善”直接塞进任务完成热路径，会带来三类确定风险：

1. 打断执行主链，让任务收尾和自我改造混在一起
2. 把一次性的偶然成功过早固化成正式能力
3. 让运行时行为和 repo 真相源发生漂移

因此需要一套更薄的实现：任务在线阶段只留下可反思证据，真正的反思与改进候选生成放到空闲时执行。

## Occam Check

- 不加它，系统具体坏在哪？
  - 模型会持续重复试探相同路径，同类经验无法沉淀；同时我们又没有一条薄的、可审查的自我改善入口，只能继续靠聊天记忆或人工回想。
- 用更少的层能不能解决？
  - 能。只定义一套文件协议和一个空闲时反思流程，不新增后台常驻学习面，不新增多阶段治理系统。
- 这个改动让主链数量变多了还是变少了？
  - 主任务主链数量不变，仍然只有一条执行主链；“反思”被明确降为旁路插件，不进入热路径。

## Decision

采用 **“在线留种子，空闲做反思，显式 apply 生效”** 的最小方案。

核心决策：

1. `skill` 是自我改善的一号落点，但不是运行时直接修改对象
2. 在线阶段只记录 `reflection seed`，不直接改 `SKILL.md`、prompt 或代码
3. 空闲阶段读取 seed，生成 `improvement candidate`
4. 正式能力只允许通过 `candidate -> apply -> replay` 进入 repo 真相源
5. 夜间定时反思是插件式触发，不是系统核心主链
6. `self-management` 与 `self-reflection` 属于同一主题能力族，但运行时必须拆成不同 skill，不合并成一个大 skill

关键权衡：

- 我们牺牲“任务刚做完立刻更新”的即时性，换取更低的误判率和更稳的主链
- 我们不追求 Hermes 式完整自学习闭环，只保留对 `msgcode` 最有价值的经验压缩能力
- 我们把“何时反思”从核心逻辑中拿掉，避免因为调度策略把系统做重

拆分理由：

- `self-management` 处理的是显式配置修改与自我资料维护，风险是误配和越权
- `idle-reflect` 处理的是空闲反思与候选生成，风险是过度总结和错误沉淀
- 两者的触发时机、输入源、输出物、风险模型都不同，合并后会变成职责混杂的大 skill

## Alternatives

### 方案 A：任务完成后立刻反思并直接改 skill

- 优点：反馈快，看起来“更聪明”
- 缺点：最容易把偶然路径固化；会污染执行热路径；和“系统做薄”冲突

### 方案 B：做 Hermes 式更完整的自动学习闭环

- 优点：能力最强，故事也最好讲
- 缺点：需要更多状态源、更多治理边界、更多调度逻辑；对 `msgcode` 来说过重

### 方案 C：在线留种子，夜间空闲反思，显式 apply 生效（推荐）

- 优点：主链最薄，证据更完整，可回放、可回滚
- 缺点：能力沉淀有延迟；第一次实现需要把文件协议写清楚

推荐：方案 C

## Plan

1. 冻结最小文件协议
   - 定义在线阶段写入的种子文件：`<workspace>/.msgcode/improvement/queue.jsonl`
   - 定义空闲反思产物目录：`<workspace>/.msgcode/improvement/inbox/`
   - 定义正式报告目录：`<workspace>/AIDOCS/reports/`
   - 验收：任何一次“值得沉淀”的任务，都能先落为 seed，而不是直接修改 repo 文件

2. 冻结三段式状态流
   - `task -> queue seed -> idle reflection -> candidate -> apply -> replay`
   - 在线阶段职责：只记任务类型、相关 skill、一句话教训、证据路径、成功/失败
   - 空闲阶段职责：只负责归纳重复模式并生成 candidate
   - apply 阶段职责：只在显式触发下修改 repo 真相源
   - 验收：文档中明确每一段谁负责、写什么、绝不允许跨段越权

3. 定义核心与插件边界
   - 核心：
     - `skill` 作为主落点
     - candidate 文件协议
     - `apply + replay` 生效口径
     - `self-management` 与 `idle-reflect` 的分工边界
   - 插件：
     - 夜间 2:12 本地时间触发空闲反思
     - 可选的历史会话补证据
     - 可选的 session recall
   - 验收：任何“调度/回忆/批处理”能力都不得进入执行热路径

4. 定义未来实现挂点，但本 plan 不直接实现
   - prompt / 说明书侧：
     - [prompts/agents-prompt.md](/Users/admin/GitProjects/msgcode/prompts/agents-prompt.md)
     - 未来新增或收紧 `self-management`、`idle-reflect`、`improvement-apply` 三类说明书与触发条件
   - runtime skill 侧：
     - [src/skills/runtime/index.json](/Users/admin/GitProjects/msgcode/src/skills/runtime/index.json)
     - 未来新增 `self-management` runtime skill：只处理显式自我配置/自我资料修改
     - 未来新增 `idle-reflect` runtime skill：只处理空闲反思、candidate 和报告生成
     - 未来可选新增 `improvement-apply` runtime skill：只处理显式 apply 与 replay
   - 执行链侧：
     - [src/agent-backend/tool-loop.ts](/Users/admin/GitProjects/msgcode/src/agent-backend/tool-loop.ts)
     - 未来仅在任务结束时写 seed，不做即时反思
   - 验收：后续实现不能偏离“在线轻记账、离线慢反思”的主线

5. 定义夜间空闲反思的默认口径
   - 默认触发时间：用户当地时间夜间 `2:12`
   - 默认动作：
     - 读取 `queue.jsonl`
     - 若为空则退出
     - 若存在重复或高价值模式，则生成 candidate 和报告
     - 严禁直接修改 `src/skills/runtime/`、`prompts/` 或代码
   - 验收：定时任务只做候选生成，不做正式 apply

## Risks

1. `queue.jsonl` 如果写得过杂，夜间反思会变成噪声放大器；回滚/降级：收紧 seed 字段，只保留最小证据字段
2. 如果空闲反思没有重复门槛，系统会沉淀大量一次性经验；回滚/降级：要求至少重复出现或明确高价值才生成 candidate
3. 如果把夜间调度写进核心执行逻辑，主链会重新变重；回滚/降级：保留调度为外接插件，核心只认文件协议
4. 如果 candidate 可以跳过 replay 直接生效，repo 真相源会被污染；回滚/降级：强制 `apply -> replay -> verified` 状态流，不允许跳步

## Test Plan

后续实现时，至少验证以下场景：

1. 正常路径
   - 任务结束后只产生一条 seed，不产生 skill 改动
2. 空闲反思路径
   - 夜间任务读取 queue 后生成 candidate 与报告，但不直接改 repo
3. 空队列路径
   - queue 缺失或为空时，夜间任务直接退出
4. 越权路径
   - 夜间任务不得直接写 `src/skills/runtime/`、`prompts/` 或代码
5. 生效路径
   - 只有显式 apply 后，才允许进入正式真相源，并要求 replay 通过
