# PL0323：core settings 聚合读合同

## 问题

settings 页的数据现在是分散的：

- `appliance profile`
- `appliance general`
- `appliance capabilities`
- `appliance doctor`

这对 CLI 没问题，对 UI 不够好。

但 `doctor` 和前三段不是一类读面：

- `profile/general/capabilities` = core settings
- `doctor` = probe/诊断

## 单一主链

建议保持：

`UI -> appliance settings (aggregate read) -> 现有三个 core 读面`

不要变成：

- `UI -> 每个 tab 狂起子命令`
- `UI -> 自己拼第二份缓存`
- `UI -> 常驻 RPC`

## 返回结构

顶层：

- `status`
- `warnings`
- `errors`
- `data`

`data` 内分三段：

- `profile`
- `general`
- `capabilities`

`doctor` 继续单独读取。

每段自带：

- `status`
- `warnings`
- `errors`
- `data`

消费规则：

- 先看 JSON envelope
- 再看顶层和分段 `status`
- `warning` 继续展示数据
- `warning` 即使带非零退出码，UI 也先解析 stdout JSON

## 推荐动作

1. 保留现有细分 CLI 不动。
2. 聚合读面只服务 settings 初始化与显式刷新。
3. `doctor` 继续按需独立读取。
4. UI 不做 tab 级 CLI 轮询。
