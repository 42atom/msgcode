# RF0000：Electron Runtime Principles

## 结论

Electron 只是壳，不是主脑。

`msgcode` runtime 继续是唯一引擎。Electron 只负责展示、入口和极薄系统整合。

## 核心定位

长期原则只保留这一句：

> Electron 只是一个聋瞎的仪表盘。它负责画图和转发，不负责算计。

这意味着：

- Renderer 只做渲染
- Main 只做系统整合和拉起 runtime
- 核心业务逻辑不落到 Electron 内部

## 性能铁律

### Main 进程不做重活

Main 进程禁止承载重 I/O、重计算和大对象遍历。

需要重处理时：

- 下沉到底层 `msgcode` runtime
- 或独立 worker

不要把 Main 进程变成新的业务服务器。

### Renderer 不保留大状态真相

前端内存只保留当前视图所需的短时状态。

真相源继续留在文件系统和 runtime：

- thread 文件
- workstate
- 各类注册表

UI 不另造第二状态中心。

### 长列表必须虚拟化

长 thread、长记录、长列表都必须按可见区域渲染。

不允许把整条历史一次性挂满 DOM。

### 重解析要下移

重的 Markdown 解析、结构化转换和大文本加工，默认不留在主渲染线程。

## IPC 原则

IPC 只传薄消息，不传大对象快照。

优先传：

- 脏标记
- 增量事件
- 游标请求

避免传：

- 全量 thread
- 大块实体
- 一次性完整 JSON 快照

一句话：

- 传“该刷新哪里”，不要传“所有内容”。

## 隔离与安全

Electron 必须坚持默认隔离：

- `contextIsolation: true`
- `nodeIntegration: false`

Renderer 不直接碰宿主原生能力。它只能通过受控 preload 桥和主进程对话。

## WPKG 边界

`wpkg` 不是主进程插件机制。

长期原则是：

- `wpkg` 不直接 `require()/import()` 进入主进程
- `wpkg` 只通过 manifest、注册表和窄消息面与宿主协作
- 如果后续需要执行能力，也必须保持隔离

## file-first 原则

Thread、标题、chatId、工作状态都属于 runtime/file-first 范畴。

UI 只读，不发明自己的真相源。

如果 UI 和文件真相冲突，一律以 runtime 落盘结果为准。

## 适用边界

这份文档只约束长期 Electron 架构边界。

它不回答：

- 当前页面长什么样
- 当前布局是几栏
- 当前按钮怎么命名
- 当前新建聊天具体怎么交互

这些属于当前产品与 UI 约束，不属于长期架构原则。
