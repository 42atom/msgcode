# RP0398：Hermes Parser Lessons Learned

## 结论

`parser` 在成熟 agent 里不是隐藏细节，而是显式配置能力。

从 `hermes-agent` 看，正确方向不是“所有模型共享一套 parser 假设”，而是：

- parser 按模型或 server 显式配置
- request contract 与 parse contract 分开
- 输出协议按模型族显式收口
- 错误优先暴露，不靠自动 repair/fallback 掩盖

## 已确认事实

### 1. Hermes 已把 parser 抬成显式配置

在：

- `/Users/admin/GitProjects/GithubDown/hermes-agent/environments/hermes_base_env.py`

存在：

- `tool_call_parser: str`

并且这个值会传给 server manager，用于双向 tool call 翻译。

这说明 `parser` 在它那里已经是运行配置面，不是内部细节。

### 2. Hermes 已踩过真实 parser 坑

在：

- `/Users/admin/GitProjects/GithubDown/hermes-agent/RELEASE_v0.2.0.md`

明确记录了：

- DeepSeek V3 多行 JSON 参数 parser 修复
- tool call repair middleware

这说明 parser 差异不是理论问题，而是现场问题。

## 我们该吸收什么

### 吸收 1：parser 显式配置

对 msgcode，最薄配置面应至少包括两层：

- request contract
  - `tool_protocol`
  - `request_max_output_tokens`
  - `request_stop_sequences`
  - `reasoning_effort_policy`
- parse contract
  - `parser_kind`
  - `completion_signal`
  - `tool_call_shape`

### 吸收 2：按模型族收 contract

不要假设：

- 所有模型都稳定输出同一种 tool call 形状
- 所有模型都能接受同一套 stop/max-output 约束
- 所有模型都用同一种“完成”信号

### 吸收 3：把 parser 问题留在模型层

宿主、plugin、sidecar 不负责解释模型输出。
parser 问题属于：

- model/provider/server
- parser contract
- prompt/output policy

但“输出长度不受控”首先属于 request contract，而不是 parser 自己去补救。

## 我们不吸收什么

### 不吸收 1：repair middleware 主导

`hermes-agent` 的 release 里能看到 repair 思路，但 msgcode 当前 guardrail 更硬：

- 不做自动 repair parser
- 不做 fallback parser
- 不吞 parser 错误
- 不从自由文本里猜 tool call

### 不吸收 2：过重环境抽象

我们不因为 parser 差异，就把：

- provider
- runner
- host
- plugin

重新包成一个大控制面。

## 对 tk0398 的直接启发

`tk0398` 第一刀应该只做：

1. 显式 request contract 配置
2. 显式 parse contract 配置
3. 先覆盖当前真实不稳的模型族

不做：

1. 自动探测
2. 自动修复
3. 平台化 parser host

## Guardrails

- fail fast and loudly
- parser 错误优先暴露
- 没理解协议时，不硬编码假装支持
- 新模型先显式接入，再允许进入主链

## 一句话

先吸收老师踩过的坑，再做显式配置；不要先发明 repair 层，再等线上事故来教育我们。

## 对 msgcode 当前模型族的落点

按当前代码现状，第一版只需要四族：

- `tmux:codex`
  - 只收 parse contract
- `tmux:claude-code`
  - 只收 parse contract
- `agent:openai-compat`
  - 收 request + parse
- `agent:minimax`
  - 收 request + parse

这样已经足够覆盖当前主链，不需要先引入更大的 parser 抽象层。
