# RP0289：Appliance Sites Host Contract Slice

## 结论

`0263` 的第二刀仍然不该去搭 Electron。

更直接的做法是先给壳一份稳定的站点入口 JSON：

- `msgcode appliance sites --workspace <labelOrPath> --json`

它让主门厅和 sidecar 入口分开：

1. `hall` 负责首页基础状态
2. `sites` 负责站点入口列表

## 已落地实现

更新：

- [appliance.ts](/Users/admin/GitProjects/msgcode/src/cli/appliance.ts)

新增命令：

- `msgcode appliance sites --workspace <labelOrPath> --json`

## 真相源

第一版不扫目录，不猜站点，不造注册中心。

只认一份可选文件：

- `<workspace>/.msgcode/sites.json`

规则：

1. 有文件就读
2. 没文件就返回空列表
3. JSON 非法或条目缺字段时给 warning

这样主链仍然很短：

- 文件真相源 -> CLI JSON -> Electron 壳

## 字段

每个站点项当前最小字段：

1. `id`
2. `title`
3. `entry`
4. `kind`
5. `description`（可选）

其中 `kind` 只允许：

1. `sidecar`
2. `external`

## 为什么这样做

这次刻意没有做：

1. 自动发现站点目录
2. HTTP 注册中心
3. preload / IPC gateway
4. 壳内写死入口

原因很简单：

- 站点入口本来就是产品文件面的一部分
- 先把入口显式落到 `.msgcode/sites.json`，后面壳和站点都好演进

2. 缺少 `sites.json` 时返回空列表而不是报错

## 推荐下一步

如果继续推 `0263`，下一刀最值钱的是：

1. 给 `hall` 补 `siteCount`
2. 或给 `sites` 补最小 pack 归属字段

不要现在去碰：

1. Electron 工程
2. 站点视觉稿
3. 大后台路由系统
