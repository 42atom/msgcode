# Plan: Doc Directory Closure And Active Link Cleanup

## Problem

文件名协议已经迁完，但目录协议还没完全收死：

1. `docs/design/`、`docs/notes/` 退役后仍可能被误当正式入口
2. 现役文档正文还残留旧目录引用
3. `docs:check` 还缺“退役目录禁入”的硬门

如果只停在文件名迁移，仓库结构仍会持续往旧目录漏回去。

## Decision

本轮三件事合并做：

1. 目录清退
2. 现役正文旧路径清洗
3. `docs:check` 增加退役目录禁入

目录口径：

- `issues/`：任务真相源
- `docs/plan/`：正式 plan/research/ref/report 主目录
- `docs/tasks/`：时间线/派单索引
- `docs/archive/`：历史归档
- `docs/design/`、`docs/notes/`：退役目录，不再承载现役协议

## Workstreams

### 1. 目录清退

- 方案 A：直接删除 `docs/design/`、`docs/notes/`
- 方案 B：各保留 1 个极短 README，明确“已迁入 `docs/plan/`，禁止新增”

默认推荐：先用方案 B 稳定一个版本，再看是否彻底删除。

### 2. 现役正文旧路径清洗

优先级：

1. 根入口：`README.md`、`CONTRIBUTING.md`、`docs/README.md`、`issues/README.md`
2. 活跃任务：`issues/*.md` 中非终态文件
3. 时间线：`docs/tasks/*.md`
4. 现役正式文档：`docs/plan/*.md`

archive 历史正文不要求本轮全清，只要不污染正式入口。

### 3. 校验收紧

- `docs:check` 增加：
  - 禁止新增 `docs/design/*.md`、`docs/notes/*.md` 正式文档
  - 禁止现役文档引用退役目录
  - 允许 archive/退役 README 保留有限说明

## Acceptance

完成后，仓库应该满足：

- 用户只会被引导到 `docs/plan/`
- 现役文档不再把退役目录当正式路径
- `docs:check` 可以阻止旧目录回流

## Risks

- `docs/tasks/` 历史正文引用较多，可能需要分批替换
- 若直接删除退役目录，外部书签会断，需要先确认是否接受
- `docs:check` 禁得太死，可能误伤 archive/退役说明，要分清现役与历史范围
