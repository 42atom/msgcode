# retire-gmail-readonly-tests-from-release-gate

## Problem

仓库仍把 `gmail-readonly` 当作浏览器域的活跃验收线，导致主回归继续为一个已经不准确的需求背书。当前需要收口的是“发布门槛真相源”，不是为历史需求继续续命。

Issue: 0090

## Occam Check

1. 不加这次收口，系统具体坏在哪？
   - `bun test` 会继续执行已经不代表真实需求的 Gmail 测试，发布门槛持续出现假阻断。
2. 用更少的层能不能解决？
   - 能。直接归档测试与任务单，更新活跃索引，不加兼容层、不改运行时代码。
3. 这个改动让主链数量变多了还是变少了？
   - 变少了。浏览器域回归只保留仍代表当前需求的测试。

## Decision

采用最小可删方案：

1. 将两个 `gmail-readonly` 测试文件从 `test/` 主树移除
2. 将 Gmail 只读验收任务单移到 `docs/archive/retired-gmail-readonly/`，避免继续出现在 `docs/tasks` 活跃集合
3. 更新 `docs/tasks/README.md`
   - 从当前执行窗口里移除 `P5.7-R7B`
   - 从当前任务单列表里移除 `p5-7-r7b-gmail-readonly-acceptance.md`
4. 保留运行时代码与 CLI 子命令，后续若要彻底退役再单独开单

## Alternatives

### 方案 A：保留测试，只把断言改松

- 缺点：仍然把过期需求挂在主回归里，问题没解决。

### 方案 B：直接删除 Gmail 运行时代码和命令

- 缺点：范围扩大，风险上升，也超出“先让发布门槛回到真实需求”的目标。

### 方案 C：仅归档测试与任务单，保留运行时代码（推荐）

- 优点：边界最小，可回滚，能最快恢复发布门槛可信度。
