# Managed Bash Mainline

## 问题

当前主链对外暴露的是 `bash` 工具，但运行时实现还没有把它收口成真正单一的 Bash 合同：

- 提示词和 README 都对外说“用 bash”
- runner 仍使用 `shell: true`
- 在 macOS 上这条链实际会落到 `/bin/sh`
- 系统自带 `/bin/bash` 又是 3.2，和“现代 Bash”不是一回事

结果是：

- 模型很容易继续试探 `sh/bash/zsh`
- “工具名”和“实际 shell”不一致
- 安装链也没有保证主链 Bash 一次到位

## 目标

- 默认安装一份托管现代 Bash（Homebrew `bash`）
- `bash` 工具只执行这份托管 Bash，不再依赖 `shell: true`
- 缺失托管 Bash 时，返回真实物理边界错误，不回退到 `/bin/sh` 或 `zsh`
- 提示词、README、测试与安装合同全部对齐同一口径
- 提供一个不阻塞主链的一键增强工具集安装入口，作为可选 profile

## 非目标

- 不把 `zsh` 升格为正式工具合同
- 不为了 shell 兼容再加 wrapper / adapter / fallback 层
- 不做多套 shell 自动切换
- 不把这件事扩展成通用环境管理平台
- 不把所有推荐工具都塞进默认必装主链

## 决策

### 单一主链

`bash` 工具的正式合同收口为：

- Apple Silicon: `/opt/homebrew/bin/bash`
- Intel macOS: `/usr/local/bin/bash`

两者语义相同，都是 “Homebrew 安装的现代 Bash”。

### 失败策略

- runner 只允许执行上述托管 Bash
- 找不到时直接报缺失依赖
- 不回退 `/bin/sh`
- 不偷偷改走用户登录 shell
- 不默认教模型用 `zsh`

### 安装分层

- 默认层：只包含主链必需能力，当前先收口 `bash`
- 可选增强层：一键安装推荐 agent 工具集，但不改变主链 shell 合同
- 两层都必须有清晰真相源，不允许“README 说可选，实际默认全装”

## 执行顺序

1. 先收口运行时合同  
   让 `bash` 真正变成“托管 Bash”，先消灭名实不符。

2. 再补安装合同  
   让新机默认装上这份 Bash，并能做最小 doctor 验证。

3. 再补可选增强安装包  
   把推荐工具集做成可选 profile，不挤进默认主链。

4. 最后对齐文档与提示词  
   把模型和用户看到的口径写死，避免旧语义回流。

## 任务拆分

- [ ] `tk0198` 运行时：`bash` runner 改为显式托管 Bash，补缺失依赖错误与回归测试
- [ ] `tk0199` 安装链：默认安装 Homebrew Bash，补最小 bootstrap / doctor 合同
- [ ] `tk0201` 配套安装链：提供可选的一键增强 agent 工具集安装入口
- [ ] `tk0200` 文档链：提示词、README、相关说明统一写明 shell 合同

## 验收口径

1. `src/runners/bash-runner.ts` 不再使用 `shell: true` 执行 `bash` 工具。
2. `bash` 工具显式调用托管 Bash，并以 `-lc` 或等价方式执行命令。
3. 缺失托管 Bash 时，返回可诊断错误；不回退到 `/bin/sh`、`/bin/bash` 3.2 或 `zsh`。
4. 安装链默认包含 Homebrew `bash`，并有最小验证步骤证明已安装。
5. 可选增强工具集有单独入口，且不与默认主链混装。
6. 提示词与 README 明确写明：
   - `bash` 工具不是用户登录 shell
   - 默认不假设 `zsh`
   - 默认不假设系统 `/bin/bash` 3.2
7. 至少有一组测试锁住：
   - runner 使用的 shell 路径
   - 托管 Bash 缺失时的错误
   - prompt / docs 关键口径

## 风险

- 现有测试或脚本可能隐式依赖 `shell: true` 的 `/bin/sh` 行为
- 旧环境若未安装 Homebrew Bash，会从“模糊可用”变成“明确失败”
- prompt 与 README 若不同步，容易再次出现工具名与实际合同漂移
- 可选增强包若和默认层边界不清，会再次回到“推荐即默认全装”的混乱状态

## 备注

- 这轮先做最小可删版本：托管 Bash + 明确错误 + 文档对齐。
- 后续如果要加 `doctor` 或 machine bootstrap，也只能围绕这份单一合同补，不新增第二主链。
