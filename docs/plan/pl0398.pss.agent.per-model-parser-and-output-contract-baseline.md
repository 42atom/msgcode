# PL0398：Per-model Parser And Output Contract Baseline

## 结论

不同模型不再假设共享同一 parser 和同一输出约束。

第一刀只做显式配置，并且分清 request 与 parse：

- request contract
  - `model_selector`
  - `tool_protocol`
  - `request_max_output_tokens`
  - `request_stop_sequences`
  - `reasoning_effort_policy`
- parse contract
  - `parser_kind`
  - `completion_signal`
  - `tool_call_shape`

不做自动推断。

建议最小配置键：

- request contract

## 错误语义

第一刀只允许三类显式错误：

1. `MODEL_CONTRACT_MISSING`
- 含义：当前 model 未绑定 request/parse contract
- 动作：直接失败，要求显式接入

2. `PARSER_KIND_MISMATCH`
- 含义：绑定的 `parser_kind` 与实际输出形状不符
- 动作：直接失败，保留原始输出证据

3. `COMPLETION_SIGNAL_MISSING`
- 含义：绑定的 `completion_signal` 没有出现，无法判定完成
- 动作：直接失败，不靠 repair/stable guess 偷补

注意：
- `max_output_tokens / stop / reasoning_effort` 问题不归为 parser 错误，归为 request contract 错误。

## Guardrails

- 不做自动探测
- 不做 fallback parser
- 不做 parser repair middleware
- 不吞掉 parser 错误
- parser 只解析结构化输出，不猜自由文本工具意图
- 让错误尽早暴露
