# RP0345：Main UI Prototype 圆桌审阅

## Decision

- Recommendation: go
- Confidence: medium
- Scope: 保留桌面三栏与工作区->线程树，继续减解释层，不改 thread 主哲学

## Why

- 现在最重要的用户心智已经收住：左边选工作区和线程，中间只做当前线程，右边只做当前线程相关可观测
- “thread 很碎”的主要来源不是 thread 哲学，而是 archive 缺位与 web `chatId` 尚未落地
- 桌面 App 约束已经清楚：不做移动端，不做纵向堆叠响应式，布局比所谓“自适应”更重要

## 共识

### 1. 保留三栏，不做网页式响应式降级

主原型应继续按桌面 App 设计：

- 左栏：工作区 -> 线程树
- 中栏：当前线程聊天
- 右栏：当前线程相关工作状况

不要再引入：

- 小屏时纵向堆叠
- 右栏抽屉化
- 移动端适配心智

### 2. 左栏只做工作区和线程

左栏不再承担：

- 最近访问
- folder 分类
- 主题卡片
- 解释性说明

只保留：

- 工作区
- 线程
- 很轻的人物入口
- 很轻的新建聊天入口

### 3. 中栏是主角

当前线程区必须继续做减法：

- 删除大块解释带
- 不再把设计说明放进聊天区
- 聊天内容用普通正文，不做厚泡泡

### 4. 右栏只做当前线程相关信息

右栏不能再像内部控制台。

第一版只接受：

- 工作状况
- 线程相关的日历/定时任务

不接受：

- 一堆静态卡片
- 和当前线程无关的固定内容
- 设计文案或调试说明

### 5. archive 是下一步真正该解的噪音源

当前左栏工作区很多，噪音大的根因是：

- 冒烟/测试工作区还没归档

所以后续视觉清爽，优先靠 archive，不靠改 thread 哲学。

## Risks

1. 如果让 UI 继续承担更多解释职责，中栏会再次长成“说明书界面”
2. 如果让右栏重新变胖，小白用户会再次失焦
3. 如果没有 archive，真实工作区越来越多后，左栏会继续显得脏

## Guardrails

- 前置条件:
  - UI 继续只读真相源
  - 新建聊天不直接造空 thread
  - thread 标题继续由 runtime 落盘
- 监控指标:
  - 用户能否在 3 秒内看懂“我在哪个工作区、哪条线程”
  - 中栏是否仍以聊天为主
  - 右栏是否只剩当前线程相关信息
- 回滚条件:
  - 任何新增结构如果让中栏再次被说明块挤压，就回退

## Next Actions

1. 继续实现 archive 主线，优先消 workspace 噪音
2. 后续如果继续改原型，只允许继续做减法，不再加新层
3. 等 web `chatId` 和 thread front matter 真正落地后，再判断线程树是否还需要收口
