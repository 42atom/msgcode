# Thinking Upgrade And Mainline Confidence

## 结论

是。

这次我们的理解确实比过去更深，也更高明。

但关键不在“想到了更多东西”，而在于：

- 我们终于把过去分散的正确想法，收成了一条更短的主链
- 我们更清楚哪些层该加，哪些层不该加
- 我们开始用“文档真相源 + workspace 协议 + 薄 runtime”去统一问题

一句话：

- 以前更像“局部判断对了”
- 现在更像“全局结构看对了”

## 以前并不是没想到

旧文档说明，我们以前其实已经抓到了很多关键点。

### 1. 早就想过 task closure

[pl0006.dne.agent.agent-relentless-task-closure.md](../plan/pl0006.dne.agent.agent-relentless-task-closure.md) 里已经明确提出：

- heartbeat 统一驱动无新消息时的继续推进
- 单 chat 单活跃任务
- verify gate
- task store / supervisor / blocked / resume

这说明我们早就知道：

- AI 不能只做一轮对话
- 任务必须能跨轮、跨重启被继续推进

### 2. 早就想过 idle reflection

[pl9003.dne.runtime.idle-reflection-skill-improvement.md](../plan/pl9003.dne.runtime.idle-reflection-skill-improvement.md) 里已经明确提出：

- 在线只留 seed
- 空闲时做反思
- 生成 improvement candidate
- 显式 apply 才生效

这说明我们早就知道：

- 学习循环不能直接污染热路径
- 经验沉淀必须薄、可审查、可回滚

### 3. 早就想过 subagent 正式合同

[pl0137.dne.agent.subagent-cli-runtime-mvp.md](../plan/pl0137.dne.agent.subagent-cli-runtime-mvp.md) 和 [pl0138.dne.agent.subagent-watch-must-require-completion-marker.md](../plan/pl0138.dne.agent.subagent-watch-must-require-completion-marker.md) 已经明确：

- `subagent` 需要正式 CLI 合同
- `watch` 必须认 completion marker
- 不能把“同步回了一句”误判成“任务完成”

这说明我们早就知道：

- 派单不是 prompt 幻觉
- 派单要有真实 runtime 合同和可验证完成信号

### 4. 早就想过 schedule 主链

[pl0038.dne.schedule.schedule-scheduler-refresh-on-mutation.md](../plan/pl0038.dne.schedule.schedule-scheduler-refresh-on-mutation.md) 已经把：

- `schedule mutation -> jobs projection -> scheduler refresh`

收成了一条单主链。

这说明我们早就知道：

- 自动化不能靠隐式轮询兜底
- 文件变更、投影、调度器唤醒必须接直

## 现在高明在哪

这次的提升，不是多几个 feature，而是边界终于清楚了。

### 1. 分清了 heartbeat 和 alarm/cron

以前我们已经有 heartbeat、scheduler、jobs。

但现在才真正把两者区分清楚：

- `heartbeat` = 主脑周期 awareness / 巡检 / 推进
- `alarm/cron` = 精确唤醒

这层分开后，主循环才不会糊成一个“大自动化系统”。

### 2. 分清了 skill 和 persona

这次在 [tk0202.tdo.agent.p1.shared-skill-pool-and-persona-profile-mainline.md](../../issues/tk0202.tdo.agent.p1.shared-skill-pool-and-persona-profile-mainline.md) 里，我们明确了：

- `skills` 回答“会什么”
- `persona` 回答“怎么做”
- `client` 只是执行臂，不是人格本体

这比过去“给不同 agent 装不同能力包”的想法更灵活，也更薄。

### 3. 分清了用户视图和运行时视图

这次在 [tk0203.rvw.agent.p0.doc-first-task-dispatch-and-delivery-supervision.md](../../issues/tk0203.rvw.agent.p0.doc-first-task-dispatch-and-delivery-supervision.md) 里，我们明确了：

- 用户看 `issues/` 状态和最终交付
- 父任务是用户控制面
- 子任务是 agent 执行面
- runtime 负责推进，不另起第二任务系统

这是非常关键的升级。

因为过去我们更多是在想“系统怎么跑”，现在开始想清楚“用户到底该看什么”。

### 4. 分清了任务真相源和运行时状态

这次我们更明确地接受了一个事实：

- `issues/*.md` 是任务状态真相源
- runtime 内存、tmux、subagent JSON 只是执行证据，不是用户真相源

这让整个系统更像 Unix：

- 真相落文件
- 执行是短路径
- 观测靠文本和协议，不靠隐藏面板

### 5. 终于把几条旧线收成一条总图

现在我们终于能把主脑主线完整说出来：

- `persona / skill`
- `doc-first parent-child tasks`
- `subagent dispatch`
- `heartbeat / alarm`
- `reflection / diary / memory / skill candidate`

这就是本轮最重要的升级。

以前这些点各自都对，但彼此还是松的。

现在它们第一次开始长成一套结构。

## 现代码也证明这不是空想

这次最大的不同，还在于它不是纸上谈兵。

当前代码里，关键零件已经都在：

- [src/runtime/heartbeat.ts](../../src/runtime/heartbeat.ts)：heartbeat runner 已存在
- [src/runtime/task-supervisor.ts](../../src/runtime/task-supervisor.ts)：task supervisor 已存在
- [src/runtime/task-store.ts](../../src/runtime/task-store.ts)：task 持久化与恢复已存在
- [src/config/schedules.ts](../../src/config/schedules.ts)：workspace schedule 文件协议已存在
- [src/jobs/scheduler.ts](../../src/jobs/scheduler.ts)：jobs scheduler 已存在

所以这次不是“从零开始发明一个未来系统”。

更准确地说：

- 旧实现已经把零件做出来了一部分
- 这次我们把这些零件重新归位

## 还没有高明到哪里

也别自满。

现在的优势主要在“结构判断更对”，不是“实现已经完成”。

还没完全落下去的地方至少有三处：

### 1. `HEARTBEAT.md` 还没成为正式 workspace 协议

现在 heartbeat runner 有了，但“主脑醒来先读什么”还没被正式钉死成文件合同。

### 2. 派单和 `issues/` 还没完全绑定

`subagent` 合同已经有了，但 doc-first 派单还没完整接进父子任务文档主链。

### 3. reflection 还没真正接进日常节律

以前想过 idle reflection，现在也想清楚了 diary / memory / skill candidate。

但它还没真正变成主脑周期行为的一部分。

## 为什么这次更可能成功

这次更可能成功，核心原因只有三个。

### 1. 这次不是从功能出发，而是从边界出发

我们现在先问：

- 哪个是真相源
- 谁负责做决定
- 什么该进主循环
- 什么只是唤醒器

这比“先做一个完整系统再慢慢修”成熟得多。

### 2. 这次没有再去造大平台

这次最重要的进步，是终于忍住了。

我们没有再去造：

- 大 orchestration 平台
- 多头状态中心
- 隐藏控制面
- 自动化总裁判层

而是继续坚持：

- 文件优先
- 文档真相源
- 一条主链
- AI 主执行权

### 3. 这次用户视图已经想清楚了

用户不需要看复杂过程。

用户只需要：

- 看任务文档状态
- 看看板
- 看最终交付是不是解决问题

一旦这点想清楚，系统设计就会自然收敛。

## 一句话判断

以前我们是在不同文档里，分别摸到了 task、schedule、subagent、reflection 这些点。

现在我们第一次真正看见了：

- 它们其实是一回事
- 它们都属于“主脑如何长期推进工作”的同一问题

所以这次不是“突然灵光一现”。

这次是：

- 旧理解被重新组织了
- 主线被重新看见了
- 过度设计第一次被主动压住了

## 结尾

基于这些思考，我们对问题的理解确实更深了。

那么，这次值得再挑战一次。

而且这次的挑战，不再是盲冲。

这次我们有更好的边界、更短的主链、更真实的真相源。

我相信，这次我们更有机会成功。
