# PL0330：工作区 people 写边界基线

## 结论

People 第一版不要发明 `people.json`。

继续只认两份文件：

- `.msgcode/character-identity/*.csv`
- `.msgcode/people-pending.json`

写入口收成一个最薄命令：

- `msgcode appliance people-save`

它只做一件事：

- 按唯一键 upsert 一条人物记录到 CSV

## 唯一键

第一版统一成：

- `channel + chatId + senderId`

原因：

- 现有 `character-identity` CSV 就是按这组事实存
- 只靠旧 pending 键不够确定该写哪一个 chat CSV

因此，`people-pending.json` 在进入真实写链前，也必须能提供：

- `channel`
- `chatId`
- `senderId`

当前若仍缺 `chatId / senderId`，那只是读取阶段残留债务，不能继续带进写主链。

## 第一版允许写什么

只允许写：

- `alias`
- `notes`

其中：

- `alias` 对应原型里的“统一称谓”
- `notes` 对应原型里的“备注”

第一版刻意不支持：

- `tags`
- 自由编辑 `role`
- 跨 CSV 合并人物

### 为什么不支持 tags

当前没有稳定真相源能承接多 tag。

如果为了原型里的 `开发者, 用户, 提需求` 直接另起文件，会把 people 从轻人物簿做成半套 CRM。

所以第一版应当诚实：

- 原型可保留 tag 作为设计探索
- 真正实现先不落地 tag 写入

## 最薄写入口

建议合同：

```text
msgcode appliance people-save \
  --workspace <labelOrPath> \
  --channel <channel> \
  --chat-id <chatId> \
  --sender-id <senderId> \
  --alias <alias> \
  [--notes <notes>] \
  --json
```

行为：

1. 若对应 CSV 不存在，按现有 skill 约定先创建目录与表头
2. 若记录不存在，追加一行
3. 若记录已存在，只更新：
   - `alias`
   - `notes`
   - `last_seen_at`
4. 保留：
   - `first_seen_at`

这样“存入通讯录”和“编辑人物”都走同一个命令。

## pending 的处理

`people-pending.json` 继续由导入侧维护。

第一版不要让 UI 去删 pending，也不要新增“认领后删除”控制面。

更薄的规则是：

- 读取时按唯一键做差集
- 已经进入 CSV 的 pending 项，在 UI 上自动隐藏

也就是：

- 真相源仍然是一份 pending 文件
- UI 不负责维护它
- runtime 只负责读时过滤

## UI 对应关系

原型动作与真实写链对应：

- 存入通讯录
  - 调 `appliance people-save`
  - 默认把渠道 `displayName` 作为初始 `alias`

- 编辑
  - 调同一个 `appliance people-save`
  - 只更新 `alias/notes`

- tags
  - 第一版从真实实现中去掉

## 明确不做

1. `people.json`
2. UI 直写 CSV
3. 自动推荐匹配
4. 删除 pending 文件中的条目
5. 自由标签系统
6. 全局人物中心

## 一句话

People 第一版就是：

- 读 CSV
- 读 pending
- 用一个 `save` 命令 upsert CSV
- 读时自动隐藏已认领 pending

够用就停。
