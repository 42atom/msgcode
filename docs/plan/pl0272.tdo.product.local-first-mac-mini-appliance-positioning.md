# PL0272：Local-First Mac mini Appliance 定位计划

## 结论

msgcode 当前不应继续被定义成：

- 云平台
- 企业 SaaS 底座
- 多租户通用 Agent 平台

更准确的产品定义应该是：

**msgcode 是一个面向小微机构本地部署使用的私有 Agent 系统，以 Mac mini 一体式交付为主，强调隐私、本地可控和持续扩展。**

一句话：

- 交付的是一台机器里的智能系统
- 卖点是隐私、本地、可控、可持续升级
- 收费方式不是纯账号订阅，而是“基础机 + 能力包 + 服务包”

## 为什么要这样收口

如果继续按“企业平台 / 云上化”去讲，后面会自然长出：

- 多租户
- SSO
- RBAC
- 云数据库
- 托管控制面
- SaaS 计费

这些不是当前产品的真实灵魂。

而用户真正想卖的价值是：

- 数据不出本机
- 运行在客户自有硬件
- 一体式交付
- 可持续按模块增购
- 可持续运维和升级

一旦主叙事漂到云上，这些价值反而会下降。

## 产品定义

### 正确定义

msgcode 的当前目标产品应定义为：

**面向小微机构本地部署使用的私有 Agent 系统 / Agent Appliance**

首选硬件载体：

- `Mac mini`

核心属性：

- local-first
- privacy-first
- offline-capable
- single-tenant by default
- file-first truth
- 可按能力模块扩展

### 不正确的定义

当前阶段不应定义为：

- 多租户企业平台
- 云原生 Agent SaaS
- 组织级通用工作流平台
- 任意企业开箱即用的全渠道底座

## 面向客户的试用口径

### 试用不叫“企业平台试用”

建议改成：

- 本地私有部署试点
- Mac mini 一体式试用
- 单点能力包试用

### 试用形态

先卖的不是“大平台”，而是：

- 单组织
- 单管理员
- 单设备
- 单部署实例
- 1-2 条高价值流程

也就是：

**single-tenant pilot product**

### 当前不承诺

试用阶段不默认承诺：

- 多租户
- SSO
- 复杂 RBAC
- 云托管
- 任意模型自由切换
- 跨平台统一安装体验

## 商业分层

### 1. 基础机

交付内容：

- `Mac mini`
- msgcode 本地运行时
- 最小管理面
- 本地模型与工具基础能力

这层卖的是：

- 私有化
- 本地可控
- 可用的基础 Agent 工作环境

### 2. 能力包

可持续增购项：

- 渠道接入包
- 本地知识处理包
- 文档/报告处理包
- 自动化流程包
- 垂直场景包
- 图查询/上下文增强包

这层卖的是：

- 客户业务扩展
- 按需加装
- 逐步升级

### 3. 服务包

服务内容：

- 初装部署
- 本地模型调优
- 数据接入
- 流程定制
- 运维巡检
- 版本升级
- 故障诊断

这层卖的是：

- 长期服务收入
- 持续粘性
- 非一次性交付

## 首先服务谁

这套产品首先不是为“大而全企业市场”设计的。

第一服务对象更准确应是：

1. 用户自己运营的公司
2. 对隐私敏感的小团队 / 单组织
3. 愿意接受一体式本地交付的客户

这决定了后续优先级应偏向：

- 可控
- 稳定
- 易交付
- 易维护

而不是偏向：

- 多租户规模化
- 公有云分发
- SaaS 自助开通

## 对架构的直接影响

### 优先级上升

后续更该优先的方向：

- 本地离线可运行
- 依赖自包含
- 本机安装/恢复
- 本机健康检查
- 本机日志与诊断
- 本地模型封装
- 本地数据备份与迁移

### 优先级下降

当前不应优先的方向：

- 多租户
- 云控制面
- 公有云数据库
- SaaS 计费系统
- 复杂组织模型
- 统一全平台安装链

### 对现有计划的影响

#### 对 `pl0262`

`pl0262` 不应默认沿“泛企业平台”收敛。

后续更合理的解释是：

- 自包含交付
- appliance 外壳
- 本地模型边界
- 安装链

都应服务 `Mac mini appliance`，而不是先服务云平台。

#### 对 `pl0267`

`pl0267` 继续保持高优先级。

原因：

- 一体机产品也必须先有稳定 verify 闭环
- 没有它，现场维护和持续升级都会变贵

#### 对 `pl0259`

`WORKSTATE` 更适合这种产品形态。

因为本地私有 appliance 的价值之一，就是长期持续工作而不丢上下文姿态。

#### 对 `pl0270`

图驱动上下文路由仍然应保持 sidecar 形态。

但未来它更像：

- appliance 上的增值能力包

而不是云平台基础设施。

## 当前推荐外部说法

建议外部统一这样说：

`msgcode` 是一个本地私有、安装在 Mac mini 上的一体式 Agent 工作系统。  
它优先服务单组织和隐私敏感场景，先以试点和能力包方式交付，而不是先做成公有云 SaaS 平台。

## 当前不该说的话

当前不建议再高频使用：

- 企业智能体底座
- 通用 Agent 平台
- 云原生企业版
- 多租户工作流系统

因为这些说法会把产品拉到错误赛道。

## 最小试用定义

当前试用建议固定为：

- 1 台 Mac mini
- 1 个本地部署实例
- 1 个管理员
- 1-2 条明确流程
- 1 份试用成功标准

成功标准不是“像平台一样完整”，而是：

- 本地可稳定运行
- 隐私边界清晰
- 流程能闭环
- 可升级
- 可运维

## 后续拆单建议

基于这份定位，后续更合理的拆法是：

1. `Mac mini appliance delivery baseline`
2. `local-first pilot trial contract`
3. `capability package catalog`
4. `appliance bootstrap and recovery`
5. `appliance local ops console`

## 非目标

这份定位计划不直接决定：

- Electron / Tauri / 其他 App shell
- MLX 的最终跨平台路线
- 企业权限系统
- 云同步方案

这些都要服从 appliance 定位，而不是反过来定义产品。

## 判断口径

以后遇到新需求，先问一句：

`这个需求是在增强 Mac mini appliance 的本地私有价值，还是在把产品拖向通用云平台？`

如果是后者，默认降权。
