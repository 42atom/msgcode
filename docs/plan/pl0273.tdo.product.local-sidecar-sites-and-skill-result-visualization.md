# PL0273：本地 Sidecar 站点与 Skill 结果可视化计划

## 结论

未来产品不应把所有信息都塞进一个主 UI。

更合理的结构是：

1. 一个主 UI 工作面
2. 多个按主题拆开的本地 sidecar 站点
3. skill 或能力包产出结果文件
4. 站点只负责读取这些结果并做可视化

一句话：

- 主 UI 做当前工作与入口
- skill 负责产出
- 文件负责沉淀
- 本地网站负责展示

这条线当前只做独立规划，不进入现役开发主线。

## 为什么要独立规划

如果把这些能力直接并进主 UI，后面会自然长出：

- 大而全后台
- 重度信息架构
- 高敏数据混放
- 主界面越来越胖

这会和当前产品主线冲突。

当前主线应该继续聚焦：

- 本地私有 appliance
- Feishu-first 现役运行面
- 任务闭环
- verify 主链

而 sidecar 站点属于：

- 后续产品扩展面
- 能力包可视化面
- 本地数据驾驶舱

所以它必须独立规划、延后实现。

## 产品结构

## 1. 主 UI 工作面

主 UI 只承接当前工作与最基础入口：

- 当前场景工作区
- 当前线程
- 工作状况
- 日历 / 定时任务
- 已安装能力包入口
- 本地站点入口

它的职责是：

- 做当前工作面
- 做导航与入口
- 做最小状态总览

它不应承担：

- 全部经营数据展示
- 大量结果列表
- 高敏个人画像
- 每个场景的详细操作后台

## 2. 本地 Sidecar 站点

每个站点只负责一个关注面。

例如：

- 财务站
- 税务站
- 组织站
- 小红书运营站
- 销售线索站
- 报告与证据站

它们的共同特征是：

- 本地运行
- 独立入口
- 只读优先
- 以文件真相源或本地索引为底

## 3. Skill 与站点的关系

不应让站点直接驱动 skill。

默认主链应是：

`skill / ability pack -> 文件真相源 -> sidecar 网站`

也就是：

- skill 负责采集、处理、生成结果
- 结果先落文件
- 网站读取文件并做展示

这样做的好处：

- UI 不绑死 skill 执行逻辑
- skill 可以独立升级
- 站点可以独立换壳
- 真相源仍然留在文件层

## 可视化主链

未来每个 sidecar 站点都应该先回答 3 个问题：

1. 看什么
2. 读哪些文件
3. 这些文件由谁写

如果这三件事说不清，就不该先做 UI。

## 关注面分层

## 第一层：机构基础面

适合放在主 UI 或最基础的 sidecar 里：

- 公司名称
- 主体信息
- 机构简介
- 基础联系人
- 当前运行状态

这是“门厅层”。

## 第二层：机构经营面

适合做成主题 sidecar：

- 财务
- 税务
- 组织
- 风险
- 内容运营
- 销售线索

这层优先服务“机构本身”。

## 第三层：高敏个人面

例如：

- 老板个人财务风险建议
- 投资建议
- 身体健康情况
- 工作节律提醒

这层不应默认并入主 UI，也不应和机构面混在一起。

原因：

- 隐私级别更高
- 风险更高
- 责任边界更敏感

更合理的方式是：

- 独立高敏扩展包
- 独立 sidecar 页面
- 独立权限与入口

## 推荐的首批站点

如果以后要试做，建议按轻重排顺序：

1. 组织站
2. 财税站
3. 小红书运营站
4. 销售线索站
5. 高敏个人站

原因：

- 组织站最稳，直接服务飞书人物识别和机构理解
- 财税站价值大，但要更注意数据口径
- 小红书运营站最容易形成“skill 产出 -> 文件 -> 列表/日志可视化”闭环
- 高敏个人站最后做，避免过早碰高风险建议

## 示例：小红书运营站

适合的主线是：

- `xiaohongshu` 相关 skill 负责产出
- 结果落在本地文件
- 站点展示：
  - 已发内容列表
  - 待发队列
  - 每日运营日志
  - 复盘记录
  - 数据摘要

它不是“大后台”，而是一个围绕单一场景的本地结果面板。

## 包形态建议

## 基础包

建议只包含：

- 主 UI 工作面
- 基础设置与我的资料
- 站点入口
- 最小结果浏览

已安装能力包在 UI 上应至少分两类：

1. 默认（内置）
2. 用户安装

## 机构扩展包

可以按主题增购：

- 组织包
- 财务包
- 税务包
- 内容运营包
- 销售包

## 高敏扩展包

单独定义，不与机构基础包混售：

- 个人财务风险包
- 投资建议包
- 健康与节律包

## 数据与协议原则

这条线必须继续服从 msgcode 当前总原则：

1. 文件优先
2. 主链做薄
3. UI 不抢 skill 的执行权
4. 建议必须能回到证据

这意味着：

- 站点默认先做只读
- 站点展示的数据必须有来源
- 高风险建议必须带时间、依据和边界提示

## 与当前主线的关系

这条线明确不进入当前主线顺序。

也就是说：

- 不影响 `pl0267`
- 不影响 `pl0259`
- 不影响 `pl0270`
- 不抢当前 appliance 主线带宽

它当前只是：

- 一条独立产品规划线
- 未来 UI 与能力包扩展的准备工作

## 后续拆单建议

如果以后要继续拆，建议按下面方式独立推进：

1. `local organization cockpit sidecar`
2. `local finance and tax cockpit`
3. `local social operation log site`
4. `high-sensitivity owner advisory site`
5. `sidecar site result file protocol`

## 非目标

这份计划当前不决定：

- 用什么前端框架
- 主 UI 的最终技术栈
- sidecar 站点是否统一壳
- 权限系统的最终实现
- 任何具体页面视觉设计

## 判断口径

以后判断一个新页面该不该进主 UI，可以只问两件事：

1. 这是机构门厅信息，还是专题关注面？
2. 它是基础信息，还是某个 skill/能力包的结果可视化？

如果属于“专题关注面”或“结果可视化”，默认优先做 sidecar 站点，不进主 UI。
