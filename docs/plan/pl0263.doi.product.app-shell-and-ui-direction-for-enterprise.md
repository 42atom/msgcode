# PL0263：App Shell And UI Direction For Appliance

## 结论

`0263` 直接收成这句话：

- Web-first UI
- Electron 薄壳
- `msgcode` runtime 继续做主脑
- sidecar 站点继续独立

不再把“原生 / Electron / 纯浏览器”当成三选一讨论。

## 为什么需要 App

当前 CLI / daemon 主链能跑，但不适合正式 appliance 交付。

真正缺的是这几件事：

1. 启动入口
2. 授权入口
3. runtime 存活状态
4. 本地站点统一入口
5. 最小诊断入口

也就是说：

- App 的价值不是替代 `msgcode`
- App 的价值是把 appliance 变成产品

## 为什么选 Electron 薄壳

当前约束已经很清楚：

1. UI 寿命优先
2. 本地网站和 sidecar 站点会持续增长
3. 未来希望给 Linux / Windows 保留 UI 兼容可能
4. 壳的主要职责是授权、启动和入口，不是业务主脑

在这个前提下：

### 不选重原生

原因：

1. UI 复用面太差
2. 不适合未来多个 Web sidecar
3. 更容易把壳做成新的中心

### 不选纯浏览器入口

原因：

1. 产品感不足
2. 授权、启动、升级入口太散
3. 不适合作为 appliance 正式交付壳

### 选 Electron 薄壳

原因：

1. 最适合承载本地 Web UI
2. 最适合做启动器和授权宿主
3. 最适合给 sidecar 站点统一入口
4. 最适合给未来 UI 跨平台留后路

## 三层结构

## 1. Electron 壳

只负责：

1. 启动 bundled runtime
2. 显示授权状态
3. 显示 runtime 状态
4. 打开主门厅
5. 打开 sidecar 站点
6. 提供最小诊断入口

不负责：

1. task 执行
2. memory / workstate
3. skill 主逻辑
4. 文件真相源

## 2. msgcode Runtime

继续负责：

1. 会话
2. 调度
3. skill
4. tool bus
5. 文件真相源
6. 日志与证据

## 3. Web UI

继续负责：

1. 主门厅
2. 站点入口
3. 结果可视化
4. 局部只读管理面

## 最小 UI 面

第一版不做“大后台”，只做门厅。

主门厅只允许这几块：

1. 机构卡片
   - 名称
   - 交税地
   - 统一社会信用代码

2. runtime 状态
   - 是否在线
   - 当前版本
   - 最近心跳/健康状态

3. 能力包入口
   - 已安装 pack
   - 未安装 pack 入口

4. 站点入口
   - 财税站
   - 组织站
   - 运营站
   - 其他 sidecar

5. 诊断入口
   - 日志
   - doctor
   - 基本证据面

不进第一版主门厅的：

1. 大量任务列表
2. 复杂聊天主界面
3. 全量经营数据后台
4. 高敏个人面

## 与 sidecar 的关系

主 UI 不是大而全后台。

正式关系是：

- 主 UI 做门厅
- sidecar 站点做专题面
- `msgcode` 继续产出文件真相源

也就是：

`msgcode runtime -> 文件 -> sidecar 站点 -> Electron 门厅`

## 与 `0264` 的关系

`0263` 必须服从 `0264`：

1. core appliance 不默认承诺本地模型
2. local model pack 是可选项
3. Apple media pack 是 mac-only 可选项

所以 UI 第一版不应该做：

1. 重本地模型控制台
2. 模型安装编排面
3. 复杂 provider 切换面

## Deferred

这些都明确延后：

1. Electron 工程脚手架
2. React/Vue 技术选型
3. DMG 更新系统
4. 精细视觉设计稿
5. 大量页面 IA

## 验收

1. 说明为什么需要 App，而不是继续纯 CLI
2. 明确 Electron 是薄壳，不是第二主脑
3. 明确主门厅最小 UI 面
4. 明确主 UI 与 sidecar 的分工
