# PL0263：App Shell And UI Direction For Appliance

## 结论

`0263` 直接收成这句话：

- Web-first UI
- Electron 薄壳
- `msgcode` runtime 继续做主脑
- sidecar 站点继续独立

不再把“原生 / Electron / 纯浏览器”当成三选一讨论。

## 为什么需要 App

当前 CLI / daemon 主链能跑，但不适合正式 appliance 交付。

真正缺的是这几件事：

1. 启动入口
2. 授权入口
3. runtime 存活状态
4. 本地站点统一入口
5. 最小诊断入口

也就是说：

- App 的价值不是替代 `msgcode`
- App 的价值是把 appliance 变成产品

## 为什么选 Electron 薄壳

当前约束已经很清楚：

1. UI 寿命优先
2. 本地网站和 sidecar 站点会持续增长
3. 未来希望给 Linux / Windows 保留 UI 兼容可能
4. 壳的主要职责是授权、启动和入口，不是业务主脑

在这个前提下：

### 不选重原生

原因：

1. UI 复用面太差
2. 不适合未来多个 Web sidecar
3. 更容易把壳做成新的中心

### 不选纯浏览器入口

原因：

1. 产品感不足
2. 授权、启动、升级入口太散
3. 不适合作为 appliance 正式交付壳

### 选 Electron 薄壳

原因：

1. 最适合承载本地 Web UI
2. 最适合做启动器和授权宿主
3. 最适合给 sidecar 站点统一入口
4. 最适合给未来 UI 跨平台留后路

## 三层结构

## 1. Electron 壳

只负责：

1. 启动 bundled runtime
2. 显示授权状态
3. 显示 runtime 状态
4. 打开主门厅
5. 打开 sidecar 站点
6. 提供最小诊断入口

不负责：

1. task 执行
2. memory / workstate
3. skill 主逻辑
4. 文件真相源

## 2. msgcode Runtime

继续负责：

1. 会话
2. 调度
3. skill
4. tool bus
5. 文件真相源
6. 日志与证据

## 3. Web UI

继续负责：

1. 主门厅
2. 站点入口
3. 结果可视化
4. 局部只读管理面

## 最小 UI 面

原型收口后，`0263` 不再把主 UI 定义成“纯门厅”。

第一版更准确的结构是：

1. Main UI
2. Settings
3. Sidecar 站点

### 1. Main UI

主 UI 只服务当前场景工作区里的实际工作，不承担全局配置。

当前冻结范围：

1. 场景工作区与线程
2. 当前线程聊天区
3. 工作状况
4. 日历 / 定时任务
5. 次级入口：
   - 基座
   - 邻居

补充冻结约束：

1. 小屏时右侧可观测面必须降成抽屉或可折叠面
2. 右侧内容必须对当前线程上下文敏感，不能长期固定同一套静态卡片
3. 邻居消息必须有全局未读提示，不能藏在独立页面里无人知晓
4. 主 UI 输入框只做瘦终端，不得自建第二真相源
5. 左侧导航必须支持“最近/未读”优先视图，不能只剩折叠树

这里的用户可见一级对象统一叫：

- 线程

不要在 UI 上再并列暴露：

- session
- thread
- channel

其中 `channel` 只是线程属性，不是用户一级概念。

### 2. Settings

Settings 单独成页，不和主工作面混放。

当前冻结范围：

1. 我的资料
   - 我的称呼
   - 智能体全局行为准则（SOUL）
   - 组织信息

2. 通用
   - 默认场景工作区
   - 多语言
   - 日志路径
   - 启动方式
   - 账号连接

3. 智能体能力配置
   - 大脑模型
   - 视觉模型
   - 语音输出
   - 听觉识别
   - 画图模型
   - 视频模型
   - 音乐模型
   - 联网搜索

4. 诊断

### 3. People 页

People 不再属于全局设置。

它属于：

- 当前场景工作区的独立页面

职责：

1. 标识本工作区已知人物
2. 承接待关联渠道人物
3. 维护统一称谓 / tags / 备注

不把它做成全局人物系统。

### 不进第一版

1. 大而全经营后台
2. 多层项目管理面
3. 高敏个人画像面
4. thread / session 双词并存
5. 把 People 混进 Settings
6. UI 自己维护消息队列、乐观气泡或本地聊天真相

## 与 sidecar 的关系

主 UI 不是大而全后台。

正式关系是：

- Main UI 做当前工作
- sidecar 站点做专题面
- `msgcode` 继续产出文件真相源

也就是：

`msgcode runtime -> 文件 -> sidecar 站点 / Main UI`

## 输入框边界

Main UI 里的 Composer 只负责投递输入。

它不负责：

1. 先行上屏用户气泡
2. 本地维护消息队列
3. 本地重试逻辑
4. 生成第二份聊天状态

正确主链应是：

`用户输入 -> runtime -> 写入底层会话文件 -> UI 被动回显`

## 与 `0264` 的关系

`0263` 必须服从 `0264`：

1. core appliance 不默认承诺本地模型
2. local model pack 是可选项
3. Apple media pack 是 mac-only 可选项

所以 UI 第一版不应该做：

1. 重本地模型控制台
2. 模型安装编排面
3. 复杂 provider 切换面

## Deferred

这些都明确延后：

1. Electron 工程脚手架
2. React/Vue 技术选型
3. DMG 更新系统
4. 精细视觉设计稿
5. 大量页面 IA

## 验收

1. 说明为什么需要 App，而不是继续纯 CLI
2. 明确 Electron 是薄壳，不是第二主脑
3. 明确主门厅最小 UI 面
4. 明确主 UI 与 sidecar 的分工
