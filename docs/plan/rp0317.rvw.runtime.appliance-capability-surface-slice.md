# RP0317：appliance capabilities 读面实现切片

## 结论

这次没有做 provider 中心。

只做了一条最薄读面：

- `msgcode appliance capabilities --workspace <...> --json`

并且只认现有硬事实：

1. workspace config 里的 runtime/provider/model 槽位
2. 本地后端注册表
3. Qwen TTS 路径
4. Whisper 模型目录

## 为什么这么收

因为设置页里“智能体能力配置”的 8 个入口，并不是 8 个都已经有稳定真相源。

当前真正能站住的只有：

- 大脑模型
- 视觉模型
- 语音输出
- 听觉识别

而这些也不是都来自同一个中心。

如果为了让 UI 好看，提前把：

- 画图
- 视频
- 音乐
- 联网搜索

都做成“已配置”，那就是在造假。

## 这次的来源

### 1. 大脑模型

来源：

- `runtime.kind`
- `agent.provider`
- 当前 backend lane
- 当前 lane 的 text 模型槽位

输出：

- `configured`
- `source`
- `model`
- `note`

### 2. 视觉模型

来源：

- workspace 显式 `model.<lane>.vision`
- 若当前是 local lane，允许回退到本地后端默认视觉模型

### 3. 语音输出

来源：

- workspace 显式 `model.<lane>.tts`
- 否则只认物理存在的 Qwen TTS 路径

这里故意不根据空想默认值判定“已配置”。

### 4. 听觉识别

来源：

- Whisper 模型目录是否物理存在

### 5. 其余能力位

当前没有统一真相源：

- 画图
- 视频
- 音乐
- 联网搜索

所以本轮全部保持空态。

## 这次刻意不做

1. 不做 provider 实体
2. 不做 API key/endpoint 管理
3. 不做连接测试
4. 不做 image/video/music/search 的假状态
5. 不做 settings 写回

## 当前边界

这条切片只回答：

- 当前这个工作区，哪些能力位已经有硬事实
- 哪些没有

它不是未来完整设置中心，也不承诺现在就能编辑这些能力。
