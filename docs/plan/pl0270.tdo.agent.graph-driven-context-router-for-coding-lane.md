# PL0270：Graph-Driven Context Router for Coding Lane

## Context

当前 coding lane 已经开始收口到：

- `spec / issue / dispatch` 真相源
- `test / types / e2e / verify` 的确定性反馈基础设施

但还有一个明显缺口：

- Agent 在真正读代码前，仍然容易靠 `rg` 和经验在全仓里自己猜“该看什么”

如果仓库继续变大，这会带来三个问题：

1. 读文件范围过大
2. 无关上下文干扰高
3. 改完后很难稳定反推影响范围与相关测试

所以这条线要解决的不是“知识图谱平台”，而是：

**给 coding agent 增加一个图查询驱动的上下文路由器。**

一句话：

> 输入任务 / diff / 错误栈，输出最值得读、最值得改、最值得验证的最小文件集合。

## 产品定义

第一版产品定义不叫：

- 知识图谱平台
- 代码数据库
- 图查询服务

第一版产品定义叫：

**本地 TypeScript 代码上下文路由 CLI**

它只负责三件事：

1. 缩小候选文件范围
2. 解释为什么选这些文件
3. 推断最值得跑的验证集合

## Goal / Non-Goals

### Goal

- 在 Agent 读代码前，先通过结构关系缩小上下文
- 在 Agent 修改后，能反查影响范围和相关测试
- 以本地 CLI 形态接入现有 shell-first 主链
- 让输出天然适合 Agent 消费

### Non-Goals

- 不做大而全知识图谱平台
- 不做 Web UI
- 不做云服务
- 不先做多语言通用支持
- 不先做向量检索优先
- 不把它做成新的 runtime 控制面

## 设计判断

### 1. 图只是实现细节，不是产品本体

真正的产品不是 graph。

真正的产品是：

- `context`
- `impact`
- `tests`
- `verify`

也就是：

- 图存在是为了让 Agent 少读错文件
- 不是为了做一个“看起来很完整”的 graph system

### 1.5 第一阶段先以提示词/skill 优先路由接入

第一阶段不把这条线做成主链硬依赖。

更稳的接法是：

- 先作为系统提示词与 skill 合同里的优先路由器
- 遇到代码任务、报错、diff、review 时，优先尝试 `msgcode graph context/impact`
- 若不可用或结果不够好，自然退回现有主链：
  - `rg`
  - `read_file`
  - `bash`
  - `test / types / e2e / verify`

这样做的目的：

- 不抢 Agent 决策权
- 不把 graph 变成新的控制层
- 先验证它是否真的提升上下文选择质量

### 2. 上下文选择必须发生在读文件之前

推荐执行链：

```text
task / stack / diff
-> context router
-> minimal read set
-> agent read / plan / edit
-> impact
-> selected tests / verify
```

不推荐：

```text
task
-> agent 全仓搜索
-> 自己猜该读什么
```

### 3. 与当前主线的关系

这条线不是独立体系。

它应该服务当前主线：

- 前半段接 `spec / issue / prompt`
- 后半段接 [pl0267.tdo.agent.deterministic-feedback-harness-for-coding-lane.md](/Users/admin/GitProjects/msgcode/docs/plan/pl0267.tdo.agent.deterministic-feedback-harness-for-coding-lane.md)

也就是：

- `context router` 解决“先读什么”
- `feedback harness` 解决“改完后怎么验证”

## 第一版能力边界

第一版只做 TypeScript/JavaScript 仓库。

第一版最小能力：

1. `build`
   - 全量建索引
2. `update`
   - 增量更新指定文件
3. `impact`
   - 反推影响范围
4. `context --json`
   - 输出最小上下文集合

第一版明确不做：

- 图可视化
- MCP server
- daemon
- watch
- embedding first
- 跨仓库共享索引
- 完整通用 symbol graph 平台

## 输出合同

这条线如果要真正服务 Agent，输出必须是机器可消费的，而不是只做人类文本。

`context --json` 第一版至少要返回：

- `files`
- `reasons`
- `readOrder`
- `tests`
- `verify`

推荐形态：

```json
{
  "files": [
    {
      "path": "src/auth/service.ts",
      "score": 0.96,
      "reasons": ["in stack", "defines target symbol"]
    }
  ],
  "readOrder": ["src/routes/login.ts", "src/auth/service.ts"],
  "tests": ["test/auth/login.test.ts"],
  "verify": [
    "bun test test/auth/login.test.ts",
    "./node_modules/.bin/tsc --noEmit"
  ]
}
```

## 技术路线

### P0 实现策略

- 先做 TS 专用
- 先做 CLI
- 先做 SQLite
- 先做 file graph + 局部 symbol graph

推荐：

- `ts-morph`
- `SQLite`
- `commander` 或 `cac`

### 图粒度

第一版优先：

- file edges
- import / export
- re-export
- test links
- 局部 symbol / direct call

不追求：

- 全量精确调用图
- 所有动态场景都覆盖

原因很简单：

- TS 动态 import、DI、字符串路由会很快把系统做厚
- 我们只需要够用的 prior，不需要伪装成真理

## 评分原则

图查询结果不能直接原样返回，必须做排序。

强信号优先：

- changed file
- in stack
- defines target symbol
- direct caller/callee
- related test

中信号：

- same module
- entry file
- 二跳依赖

惩罚项：

- 高频公共 util
- 与当前任务关系弱但连接度高的基础设施文件

目标不是返回更多，而是：

**返回更少，但足够完成任务。**

## 与仓库原则的兼容性

这条线必须遵守仓库级约束：

1. 不抢 Agent 决策权
   - 只做 prior，不做裁决
2. 不新增厚 orchestrator
   - 只做 sidecar index + CLI
3. 不创造第二状态真相源
   - 索引是派生物，可重建
4. 优先 shell-first
   - Agent 通过 CLI 调它，不强行引入新运行面

一句话：

- 它应该是 sidecar，不是第二 runtime

补充一条产品收口原则：

- 如果后续 benchmark 和真实任务都证明这条路线有效，且输出合同稳定、回退需求显著下降，再考虑把它从 sidecar 能力逐步收进 `msgcode` 主能力面。
- 在那之前，它先作为可退回、可重建、可关闭的增强器存在。

## 验收方式

这条线后续如果真做，不能只看“能不能建图”，要看它是否真的提升 Agent。

最小 benchmark 口径应比较两组：

1. `rg + agent`
2. `context router + agent`

关注：

- 首轮命中关键文件数
- 总读取文件数
- token 消耗
- 首轮修改成功率
- 相关测试命中率
- 平均修复轮数

没有这组评估，就不能证明它值得进入主线。

## 分阶段路线

### Phase 1

只做：

- `build`
- `update`
- `impact`
- `context --json`

目标：

- 先证明“上下文缩小”真的有价值

### Phase 2

增强：

- 测试映射
- 更稳定的 symbol 解析
- `diff` 输入支持
- explain/debug 输出

### Phase 3

再考虑：

- `watch`
- `mcp`
- `serve`

前提：

- Phase 1/2 已经证明收益成立

## 后续拆解建议

这份 plan 后续若要拆 task，建议顺序是：

1. P0 CLI 基线
2. context 输出合同
3. impact -> tests/verify 接线
4. benchmark 与收益验证

也就是先证明：

- 它能稳定提升上下文选择质量

再决定要不要继续往 `watch / mcp / serve` 推。

这不是“做知识图谱平台”的计划。

这是一条：

**给 coding lane 增加 graph-driven context router 的 sidecar CLI 计划。**
