# PL0293：Main UI 线程主视图数据合同

## 结论

主界面每个区块都必须直接落到文件真相源。

不允许：

- 前端自维护消息队列
- 前端自维护工作状况
- 为了 UI 新建控制层

一句话：

- 左边读线程索引
- 中间读当前线程
- 右边读工作状况和日历

## 最小映射

### 左栏：场景工作区与线程

第一版直接复用：

- `<workspace>/.msgcode/threads/*.md`

读取规则：

1. 当前工作区名直接取工作区目录名
2. 线程列表来自 `threads/*.md`
3. `Recent` 视图按最近修改时间或最近 turn 时间排序
4. 线程副信息先只显示：
   - 标题
   - 来源属性（从 `chatId` 推出 `feishu` / `web` 等）

不单独发明：

- `thread-index.json`

除非现有 `threads/*.md` 证明不足。

### 中间：当前线程聊天

第一版直接读：

- 当前选中的 `threads/<name>.md`

读取方式：

1. 解析 front matter
2. 解析 `## Turn` / `### User` / `### Assistant`
3. 渲染为只读聊天流

这比现在就强推 `sessions/*.jsonl` 更稳，因为：

- `threads/*.md` 已经是线程级文件
- `sessions/*.jsonl` 当前仍偏渠道原始记录

Composer 仍只做投递，不先行上屏。

### 右栏：工作状况

第一版只读两类真相：

1. 当前线程自己的最近工作近况
2. `<workspace>/.msgcode/status.log` 的最近几条

读取优先级：

1. 若 `status.log` 存在，读最近几条
2. 若不存在，只显示当前线程的最近状态
3. 若两者都没有，就显示空态

不允许 UI 自己拼一段“聪明摘要”。

### 右栏：日历 / 定时任务

第一版直接读：

- `<workspace>/.msgcode/schedules/*.json`

只展示最小字段：

- 时间
- 消息
- 启用状态

不发明新的日历聚合层。

### 邻居未读

这块当前缺正式真相源。

所以 `0293` 先钉两条：

1. 若没有稳定 mailbox 文件，不显示未读 badge
2. 等 `0291` 明确 mailbox/health 文件合同后再接

不要为了先画红点去发明本地缓存。

### 人物入口

第一版只展示入口和人数。

人数若当前没有正式人物表文件，就降级为：

- 不显示数字
或
- 显示 `0`

不要猜。

## 读取原则

主界面只读：

- 当前工作区
- 当前线程
- 最近几条共享近况

不做：

- 全工作区全文扫描
- 多线程并行读全量聊天
- 自动跨线程合并原文

## 缺口

当前真正还缺的不是 UI，而是三份最小真相：

1. `status.log`
2. 人物表
3. 邻居 mailbox/health

所以 `0293` 的职责不是补造数据，而是把没有真相源的地方明确降级。

2. People 的工作区文件真相源
