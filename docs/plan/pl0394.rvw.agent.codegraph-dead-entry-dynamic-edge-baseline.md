## Current Evidence

当前 `dead-entry` 共 43 个。前排样本集中在：

- `src/routes/cmd-info.ts`
- `src/routes/cmd-model.ts`
- `src/routes/cmd-schedule.ts`
- `src/routes/cmd-task-impl.ts`
- `src/routes/store.ts`
- `src/routes/workspace-resolver.ts`

这批样本说明当前误报主要有两类：

1. 类型声明被算成入口噪声
2. 动态分发入口没有回接到静态图

同时，execution-flow 已经列出一批真实动态入口族：

- `src/cli.ts` 下的 `command:*`
- `src/commands.ts` / `src/runtime/singleton.ts` 的 `SIGINT|SIGTERM|SIGUSR2`
- `src/electron/main.ts` 的 `activate|ready-to-show|window-all-closed`
- `src/output/*.ts`、`src/tmux/session.ts`、`src/runners/*` 的 `data|close|error` 事件回调

## Thin Follow-up Order

最薄补图顺序固定为：

1. CLI / slash command 注册与分发
2. Electron 生命周期与 bridge 启动
3. scheduler / heartbeat / wake / signal 事件
4. output reader / tmux / runner 事件回调

原因：

1. 这些入口族已经有真实 entry 证据
2. 它们直接解释当前 `dead-entry` 噪声
3. 先补它们能最快降低误报，不需要先做更大的 graph 平台化
