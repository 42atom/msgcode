# PL0292：工作区共享 status.log 基线

## 结论

每个工作区保留一份：

- `<workspace>/.msgcode/status.log`

它不是聊天历史，不是 `summary`，也不是 `WORKSTATE`。

它只是一本追加式的工作近况簿。

一句话：

- 原始聊天各在线程里
- 共享近况只进 `status.log`

## 它解决什么

它解决的是：

- 网页线程不知道飞书线程最近已确认了什么
- branch 有了阶段性结果，但主线程不该去读它的整段原文
- UI 想展示“工作状况”，又不该直接拼聊天历史

它不解决：

- 原始会话连续性
- 自动上下文同步
- 多线程自动协调
- 实时监控

## 与现有文件的边界

### `summary`

`summary` 服务记忆连续性和压缩恢复。

它回答：

- 之前发生了什么

### `WORKSTATE`

`WORKSTATE` 服务当前任务的恢复姿态。

它回答：

- 下一窗口该怎么接上

### `status.log`

`status.log` 服务同一工作区内的其他线程。

它回答：

- 当前已经定下来了什么

## 文件位置

最小落点：

- `<workspace>/.msgcode/status.log`

要求：

- append-only
- 线程共享
- 不改旧记录

展示时若需要“新的在前”，由 UI 倒序读；文件本体保持自然追加顺序。

## 记录内容

第一版只允许两类：

1. 已确认决定
2. 已确认状态变化

不要先单列更多类型。

如果一条待办或阻塞真的值得共享，也应该并入状态变化表达。

## 记录格式

固定为一行一条：

```text
时间 | 线程名 | 类型 | 一句话内容 | 原文引用
```

示例：

```text
2026-03-20T15:04:00+08:00 | feishu-main | decision | 先查退款，不动预算 | .msgcode/sessions/feishu-main/messages.jsonl#182
2026-03-20T15:12:00+08:00 | refund-sub | state | 等待财务复核 | .msgcode/sessions/refund-sub/messages.jsonl#27
```

约束：

- 只能单行
- 不复制聊天原文
- 不写长摘要
- 不写未确认猜测
- 必须带原文引用
- 若内容里出现 `|`，统一转成全角 `｜`

## 写入合同

模型不直接写原始文本行。

后续如果进入实现，必须通过单一 append 合同入盘：

- 模型只产出结构化意图
- 写入器统一做单行格式化
- 写入器统一负责原子追加

这样做的目的只有一个：

- 不让 `status.log` 被模型输出漂移污染

## 去重规则

第一版必须做最小去重。

写入前，只检查最近几条：

- 同一线程
- 同一类型
- 同一句内容

如果已经存在，就不重复追加。

这样做的目的不是“聪明合并”，只是防止同一句结论在短时间里被刷屏。

第一版明确不做：

- 语义去重
- 同义句归并
- 模型级重写压缩

## 读取原则

第一版默认不要频繁读。

只在这两种场景读最近几条：

1. 线程启动 / 恢复时
2. 当前线程明确需要了解其他线程近况时

读取方式：

- 只读 tail
- 例如最近 10 到 20 条

不要：

- 全文件扫描
- 定时轮询
- 每句都读

`status.log` 不是实时监控面。

所以 UI 必须诚实表达：

- 显示最近一条状态时间
- 不暗示“系统正在实时上报”

## 写入原则

第一版不做系统 hook，不做定时器，不做 UI 按钮。

只通过 prompt 约束当前线程里的 Agent：

- 仅当当前线程形成了阶段性结论，且这个结论值得其他线程知道时，才允许追加 1 条
- 这里的“阶段性结论”只允许两种：
  - 已确认决定
  - 已确认状态变化
- 没有命中，就什么都不做

也就是：

- 默认不写
- 命中才写
- 每次阶段性收尾最多一条

## UI 消费口径

主界面里的“工作状况 / Work Status”只读：

- 当前线程自己的工作状况
- `status.log` 最近几条共享近况

它不直接读别的线程原文。

UI 第一版只需把 `status.log` 理解成：

- 最近工作近况
- 最近已确认共识

不要把它渲染成“聊天流”。

用户可见词统一用：

- 线程

不要在这条线里再并列引入：

- session
- channel

如果内部还要保留更细的工作线，统一改叫：

- branch
- 子线

## Prompt 口径

第一版只需要一条很短的约束：

- 你知道当前工作区存在一份共享状态文件 `<workspace>/.msgcode/status.log`
- 默认不要读取全文件
- 仅在需要了解其他线程近况时读最近几条
- 仅当当前线程形成阶段性结论时，才追加 1 条

## 圆桌收口

本轮圆桌审议后的新增硬约束：

1. `status.log` 不能退化成第二份聊天摘要
2. 记录必须单行，便于 tail 和解析
3. 必须做最近几条去重
4. UI 继续只读，不承担写入裁决

## 轮转与归档

`status.log` 是 append-only，但不意味着永远只保留一个无限增长的文件。

第一版只先把格式和边界钉住。

后续进入实现时，应补：

- 按日期或体积轮转
- 老文件归档

这件事属于实现阶段，不在当前设计里展开。

## 第一版不做

- 自动线程切换 hook
- 5 分钟定时写
- 关键词扫描
- 中间协调器
- 线程互相读全量 `jsonl`
- UI 手工“记一笔”

## 推荐下一步

如果进入实现，顺序应该是：

1. 先补 prompt 合同
2. 再补最小文件读写 CLI 或 shell 合同
3. 最后才把 UI 的“工作状况”接到这份文件

不要先写自动同步机制。
