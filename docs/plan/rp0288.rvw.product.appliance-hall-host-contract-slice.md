# RP0288：Appliance Hall Host Contract Slice

## 结论

`0263` 的第一刀不该是先搭 Electron。

更薄也更对的做法是：

1. 先让 runtime 自己输出一份稳定 JSON
2. Electron 壳只负责消费这份 JSON

所以这次落的是：

- `msgcode appliance hall --workspace <labelOrPath> --json`

它是第一版门厅 host contract。

## 已落地实现

新增：

- [appliance.ts](/Users/admin/GitProjects/msgcode/src/cli/appliance.ts)

并接进：

- [cli.ts](/Users/admin/GitProjects/msgcode/src/cli.ts)

当前合同：

1. 输入
   - `--workspace <labelOrPath>`
2. 输出
   - 单份 JSON Envelope
3. 主要字段
   - `workspacePath`
   - `org`
   - `runtime`
   - `packs`
   - `sites`

其中：

### org

直接读：

- `<workspace>/.msgcode/ORG.md`

第一版只暴露三项：

1. 名称
2. 交税地
3. 统一社会信用代码

### runtime

不新造状态层，直接复用现有 probe 结果，压成：

1. `summary.status`
2. `summary.warnings`
3. `summary.errors`
4. `categories[]`
5. `appVersion`
6. `configPath`
7. `logPath`

### packs / sites

第一版先返回空数组。

这是刻意留白，不是缺实现。
因为 pack 和 sidecar 站点还没有稳定真相源，不该现在假造控制面。

## 为什么这刀是对的

这次没有做：

1. HTTP host
2. 本地守护 API
3. Electron preload / IPC 网关
4. 第二套 runtime 状态服务

所以主链仍然很短：

- `msgcode CLI -> JSON -> Electron 壳`

这比“先搭桌面工程再反推 host API”更稳。

2. 输出是可解析 JSON

4. `runtime.categories`、`packs`、`sites` 字段都存在

## 推荐下一步

如果继续推 `0263`，下一刀最值钱的是：

1. 再补一个 `msgcode appliance sites --workspace ... --json`

不要现在去碰：

1. Electron 工程脚手架
2. UI 组件库
3. preload IPC
