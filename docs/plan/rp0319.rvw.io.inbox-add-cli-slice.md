# RP0319：inbox add CLI 切片

## 结论

这轮只补了一条最小投递主链：

- `msgcode inbox add`
- `<workspace>/.msgcode/inbox/*.new.<transport>.*.md`

没有做：

- inbox 消费器
- Web 会话执行主链
- UI 自己上屏

## 本轮实现

新增：

- `src/cli/inbox.ts`
- `test/p7-tk0319-inbox-add-cli.test.ts`

扩展：

- `src/cli.ts`
- `src/cli/help.ts`

## 落盘主链

当前只做这些动作：

1. 解析 `workspace/chatId/text`
2. 生成或接收 `messageId`
3. 复用 `createInboxRequest()`
4. 写入 `.msgcode/inbox/rqNNNN.new.<transport>.<slug>.md`
5. 返回 envelope

## 关键收刀

这轮刻意收了三刀：

1. 不消费  
只落 `new` 文件，不假装已经进入处理主链。

2. 不猜当前线程  
`--chat-id` 保持必填，不偷偷从别处推断。

3. 不造 Web 控制面  
还是复用现成 inbox 文件真相。

## 结果

现在主界面 composer 至少已经有一条正式、可审计、file-first 的投递面。

后续如果要继续接通：

- runtime 消费
- 线程回写
- UI 被动渲染

都应该沿这条文件主链往前走，不另起第二套输入系统。
