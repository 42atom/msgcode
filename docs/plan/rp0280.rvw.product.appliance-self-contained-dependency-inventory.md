# RP0280：Appliance 自包含交付依赖清单

## 结论

当前 msgcode 距离“本地 appliance 自包含、不现场下载”还差 5 个硬缺口：

1. 运行入口仍是源码态 `tsx`
2. `bash` 合同仍绑定 Homebrew 路径
3. 浏览器 / 桌面能力仍依赖外部安装物
4. 部分能力仍依赖平台原生 addon 或外部二进制
5. 交付基线还没有把“核心包”和“能力包”彻底分开

一句话：

- 现在的仓库能跑
- 但还不是可交付 appliance
- 主要问题不是功能不够，而是依赖还没收口成产品边界

再补一条交付铁律：

- 对 appliance / App 而言，基础运行依赖不能继续是“建议安装”。
- 它们必须被明确收成：
  1. 随包携带
  2. 预安装软件必须清单
- install / doctor 必须前置发现缺口；运行时不再碰运气。

## 盘点口径

本清单只回答：

1. 当前哪些依赖阻止“现场不下载”
2. 哪些依赖应该进 appliance 核心包
3. 哪些依赖应该降为可选能力包

不回答：

1. 具体用 Electron 还是别的壳
2. 具体怎么签名、打包、更新
3. 具体怎么做 MLX 跨平台

## 现状事实

### 1. CLI 入口仍是源码态

证据：

- [bin/msgcode](/Users/admin/GitProjects/msgcode/bin/msgcode)

当前 `msgcode` 入口不是编译产物，而是：

- 优先调用本地 `node_modules/.bin/tsx`
- 找不到再回退 `npx tsx src/cli.ts`

这意味着当前正式入口默认要求：

- Node.js
- `tsx`
- `src/*.ts`
- `node_modules`

结论：

- 这条链天然不符合 appliance 自包含交付
- 第一版 appliance 必须先把 CLI 收成可直接运行的已构建产物

### 2. 默认安装链仍依赖 Homebrew 和 npm

证据：

- [README.md](/Users/admin/GitProjects/msgcode/README.md)
- [bootstrap/Brewfile](/Users/admin/GitProjects/msgcode/bootstrap/Brewfile)
- [bootstrap/Brewfile.agent](/Users/admin/GitProjects/msgcode/bootstrap/Brewfile.agent)

当前默认安装步骤是：

- `brew bundle --file bootstrap/Brewfile`
- `npm install`

可选 agent 工具集还要求：

- `tmux`
- `uv`
- `bun`
- `rg`
- `fd`
- `jq`
- `fzf`
- `bat`
- `eza`

结论：

- 当前安装链显然是开发者环境，不是 appliance 交付链
- appliance 现场如果还要先装 Homebrew，再装 npm 依赖，就不算自包含

### 3. bash 合同仍绑定 Homebrew Bash

证据：

- [README.md](/Users/admin/GitProjects/msgcode/README.md)
- [bootstrap/doctor-managed-bash.sh](/Users/admin/GitProjects/msgcode/bootstrap/doctor-managed-bash.sh)

当前正式 `bash` 合同只认：

- `/opt/homebrew/bin/bash`
- `/usr/local/bin/bash`

缺失时直接提示：

- `brew install bash`

结论：

- 这也是 appliance 的硬阻塞
- 要么把托管 Bash 一起带进包
- 要么把运行时合同改成不再强绑定 Homebrew Bash 路径

### 4. 浏览器能力不自包含

证据：

- [README.md](/Users/admin/GitProjects/msgcode/README.md)
- [src/runners/browser-patchright.ts](/Users/admin/GitProjects/msgcode/src/runners/browser-patchright.ts)

当前正式浏览器主链是：

- `patchright`
- `connectOverCDP`
- 共享工作 Chrome root

也就是说，当前至少依赖：

- `patchright` npm 依赖
- 本机 Chrome/Chromium
- 可用的 remote debugging 进程

结论：

- 浏览器能力不该默认算进第一版 appliance 核心包
- 更合理的是作为可选能力包处理

### 5. 桌面能力不自包含

证据：

- [README.md](/Users/admin/GitProjects/msgcode/README.md)
- [src/runners/ghost-mcp-contract.ts](/Users/admin/GitProjects/msgcode/src/runners/ghost-mcp-contract.ts)

当前桌面能力面明确依赖：

- `ghost-os`
- `brew install ghostwright/ghost-os/ghost-os`

结论：

- `ghost_*` 当前也不应算进第一版 appliance 核心包
- 它应被视作可选桌面能力包

### 6. memory 核心含平台依赖

证据：

- [package.json](/Users/admin/GitProjects/msgcode/package.json)
- [docs/protocol/MEMORY.md](/Users/admin/GitProjects/msgcode/docs/protocol/MEMORY.md)

当前 memory 核心依赖：

- `sqlite-vec`
- `node:sqlite`

这类路径意味着：

- 需要考虑原生模块或平台兼容
- 打包时不能继续假设“现场 npm install 一下就好”

结论：

- memory 仍应保留在 appliance 核心包
- 但必须被明确列入“需要预打包和平台验证”的核心依赖

### 7. Feishu 不是安装阻塞，但仍是运行外部依赖

证据：

- [README.md](/Users/admin/GitProjects/msgcode/README.md)
- [package.json](/Users/admin/GitProjects/msgcode/package.json)

当前现役消息主链仍是：

- `feishu`
- `FEISHU_APP_ID`
- `FEISHU_APP_SECRET`

结论：

- 这不阻止“离线安装”
- 但阻止“完全内网运行”
- 所以 `离线安装版` 和 `完全内网版` 必须分开定义

## 当前阻塞项分类

## A. 必须先解决，才配叫 appliance 自包含交付

1. 源码态 `tsx` CLI 入口
2. `npm install` 依赖
3. Homebrew Bash 路径绑定
4. 核心依赖的预打包问题（尤其 `sqlite-vec`）

这里的第 3 条需要更明确：

- 若将来 App 不自带托管 Bash，就必须把 Bash 记入“预安装软件必须清单”。
- 若将来 App 自带托管 Bash，就应从预安装清单移除。
- 无论二选一哪条，都不能再维持“运行时才发现机器没有 Bash”的灰区。

## B. 不该放进第一版核心包，应该降级成能力包

1. `ghost-os`
2. Patchright + Chrome/Chromium
3. `tmux`
4. `uv`
5. `bun`
6. 各类终端增强工具（`rg/fd/jq/fzf/bat/eza`）

## C. 不阻止离线安装，但阻止完全内网运行

1. Feishu 通道
2. 云模型 API

## 建议的第一版基线

## 第一版 Core Appliance

只承诺：

1. 预打包的 msgcode runtime
2. 预打包的 Node 运行时与核心依赖
3. 文件系统 / issue / memory / 调度主链
4. 最小机构卡片
5. install / doctor 能前置校验所有基础依赖缺口

最小机构卡片只允许：

1. 名称
2. 交税地
3. 统一社会信用代码

不要先长成复杂组织模型。

## 第一版 Optional Packs

单独处理：

1. Feishu pack
2. Browser pack
3. Desktop pack
4. Tmux coding pack

## 预安装软件必须清单口径

这份清单的独立真相源是：

- /Users/admin/GitProjects/msgcode/docs/plan/rf0002.rvw.product.appliance-required-preinstall-software.md

这份清单只收“基础运行依赖”，不收普通增强工具。

应该进入清单的典型项：

1. 不随包携带、但正式主链硬依赖的 shell/runtime
2. 不随包携带、但正式主链硬依赖的系统组件

不该进入清单的项：

1. 只服务开发者体验的工具
2. 可选能力包的增强组件
3. 可以缺席而不影响 Core Appliance 启动的东西

一句话：

- 预安装软件必须清单是产品边界，不是开发者愿望单。

## 当前已明确应列入清单的项

按当前仓库事实，已经可以明确列出的只有这些：

### Core Appliance 基础项

1. 托管 Bash
   - 触发条件：App 没有自带 Bash runtime
   - 当前正式合同路径：
     - `/opt/homebrew/bin/bash`
     - `/usr/local/bin/bash`
2. `zstd`
   - 触发条件：App 仍沿用当前 `.wpkg -> tar.zst` 安装主链，且未自带解压 runtime

### 可选能力包基础项

1. Browser pack
   - `Google Chrome / Chromium`
2. Desktop pack
   - `ghost-os`
3. tmux coding pack
   - `tmux`

## 当前明确不进入清单的项

这些要么应该随包，要么只是开发者增强，不应混入“必须预装”：

1. `Node.js`
2. `tsx`
3. `npm`
4. `uv`
5. `bun`
6. `rg`
7. `fd`
8. `jq`
9. `fzf`
10. `bat`
11. `eza`

## 暂不写死的项

这些线路还没完成产品边界收口，现在不该抢先写成“必须预装”：

1. `LM Studio / OMLX`
2. `mlx-whisper` 及其模型资产
3. 其他本地模型服务

原因不是它们不重要，而是 `0264` 还在决定它们未来属于：

1. Core Appliance
2. 可选能力包
3. 完全不承诺

这样做的好处是：

- 核心交付线先变简单
- 不把所有平台依赖一次性绑死
- 更符合“基础机 + 能力包”的产品口径

## 两个必须分开的交付定义

### 1. 离线安装版

要求：

- 现场安装不下载
- 所有核心运行文件已随包携带

不要求：

- 完全不连外部 API
- 完全不依赖 Feishu / 云模型

### 2. 完全内网版

额外要求：

- 不依赖云模型
- 不依赖 Feishu 公网链路
- 本地模型 / 本地消息链 / 本地更新链都要另定

这不是 `0262` 第一刀该承诺的范围。
