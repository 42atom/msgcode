# RP0281：Core Appliance 核心包边界

## 结论

第一版 appliance 必须做减法。

Core Appliance 只保留 6 类能力：

1. 预打包 runtime
2. 文件真相源主链
3. `SOUL + ORG + memory` 认知主链
4. 调度与任务主链
5. 最小 CLI/诊断面
6. 本地日志与证据面

这 4 类东西不进核心包，全部降成 Optional Pack：

1. Feishu
2. Browser
3. Desktop
4. Tmux coding

一句话：

- 核心包先保证“单机单机构能稳定跑”
- 外围能力全部后挂
- 不把第一版 appliance 做成开发者全家桶

## 第一版 Core Appliance 定义

### 目标

第一版 Core Appliance 只承诺：

1. 单机
2. 单组织
3. 单管理员优先
4. 本地文件真相源
5. 本地任务/记忆/证据可持续
6. 现场安装不下载

### 不承诺

第一版不默认承诺：

1. Feishu 一开机就能用
2. 浏览器自动化
3. 桌面自动化
4. tmux coding lane
5. 完全内网运行
6. 本地模型默认内置

## Core Appliance 必含内容

## 1. 预打包 msgcode runtime

必须包含：

1. 已构建的 `msgcode` CLI 入口
2. 已打包的 Node 运行时
3. 运行所需核心依赖

目标是消灭这条现状依赖：

- `tsx`
- `src/*.ts`
- `npm install`

结论：

- 第一版 Core Appliance 不允许再以源码态启动

## 2. 文件真相源主链

必须包含这些目录/协议：

1. `issues/`
2. `.msgcode/`
3. `memory/`
4. `AIDOCS/`

原因：

- 这是 msgcode 最值钱的主链
- appliance 可以没有浏览器，但不能没有文件真相源

## 3. 认知主链

第一版必须包含：

1. `SOUL.md`
2. `ORG.md`
3. `memory/*.md`
4. `summary.md`
5. `WORKSTATE` 读取能力

也就是：

- Agent 要知道自己是谁
- 要知道服务的是哪个机构
- 要能持续恢复工作态

## 4. 调度主链

第一版必须保留：

1. task / heartbeat / dispatch 主链
2. context rebuild 主链
3. verify evidence 主链

原因：

- appliance 的价值不是静态知识库
- 而是这台机能持续接活、推进、恢复

## 5. 最小 CLI / 诊断面

第一版至少要保留：

1. 启动
2. 健康检查
3. 基本状态查看
4. memory index/search
5. 日志定位入口

原因：

- 现场交付没有最小诊断面，后续运维成本会爆炸

## 6. 本地日志与证据面

必须保留：

1. `~/.config/msgcode/log`
2. `AIDOCS/reports`
3. `.msgcode/evidence/*`

原因：

- 这条线直接服务排障
- 也服务后续人工接管

## Optional Packs

## 1. Feishu Pack

内容：

1. Feishu transport
2. Feishu 凭据配置
3. Feishu live smoke 基座

为什么不进核心包：

- 它阻止“完全内网运行”
- 也不是所有 appliance 场景都需要

## 2. Browser Pack

内容：

1. Patchright
2. Chrome/Chromium
3. `connectOverCDP` 主链

为什么不进核心包：

- 外部依赖重
- 浏览器本体大
- 第一版先不把它绑死

## 3. Desktop Pack

内容：

1. `ghost-os`
2. `ghost_*` 主链

为什么不进核心包：

- 外部安装物重
- 权限和诊断成本高

## 4. Tmux Coding Pack

内容：

1. `tmux`
2. coding lane 相关工具集
3. 可选 agent toolset

为什么不进核心包：

- 它更像开发/重任务能力包
- 不是 appliance 第一版必须能力

## 两种交付定义

## 1. 离线安装版

要求：

1. 现场安装不下载
2. Core Appliance 能本地启动
3. 机构卡片可本地写入
4. 不要求所有能力都离线可用

允许：

1. 后续再接 Feishu
2. 后续再接云模型
3. 某些 pack 后装

## 2. 完全内网版

额外要求：

1. 不依赖 Feishu 公网
2. 不依赖云模型 API
3. Browser/Desktop pack 也要各自有内网交付口径

结论：

- 完全内网版不是第一版 Core Appliance 的默认承诺
- 它是后续更高阶交付档位

## 最小机构卡片

## 文件落点

第一版建议直接落：

- `<workspace>/.msgcode/ORG.md`

原因：

1. 跟当前 workspace 真相源主链一致
2. 人类可直接读写
3. 不需要先发明新的全局组织数据库

## 字段

第一版只允许 3 个字段：

1. 名称
2. 交税地
3. 统一社会信用代码

建议模板：

```md
# 机构信息

- 名称：<机构名称>
- 交税地：<省/市/区或税务归属地>
- 统一社会信用代码：<18位代码>
```

## 不允许先加的字段

第一版禁止默认加入：

1. 组织树
2. 员工名册
3. 审批流
4. 财务策略
5. 投资偏好
6. 健康信息
7. 复杂权限模型

原因：

- 小机构最怕复杂
- 机构身份第一步只需要“知道服务的是谁”

## 一句话结尾

第一版 appliance 最重要的不是“什么都能做”，而是：

- 核心包足够小
- 机构身份足够简单
- 真相源足够稳
- 外围能力能以后再挂
