# PL0395：Conversation IO Channel Baseline

## 结论

先把语音收成“会话通道”，不做统一音频平台。

产品层只保留：

- `input_channel`
- `output_channel`

实现层明确分两族：

1. 本机音频设备族
2. 远程桥接族

其中 `HomePod` 归远程桥接族，最终落为 `homepod-bridge` pack。

外部能力默认先走薄 service，不先做平台。service 只负责健康检查、状态、启动/停止与最薄请求转发，不负责再造控制面。

## UI Position

- `main.html`：顶栏右侧只显示 `语音外放设备：...`
- `settings.html`：`general` 视图承载完整音频设置
- 首页只做感知，不做控制台

## 最小通道真相

建议先收为：

- `input_channel`
  - `system-mic`
  - `homepod-bridge`
  - `none`

- `output_channel`
  - `system-default`
  - `audio-device:<device-id>`
  - `homepod-bridge`
  - `none`

## 顺序

1. 先做 settings/CLI/runtime 共用的 channel 真相
2. 先做本机音频主链
3. 再做 HomePod bridge pack

`wpkg` pack 可以带一小段边车运行时代码，比如本机 HTTP bridge、health check 和启动脚本；但不能带第二套 runtime 世界，也不兼容 `openclaw` 插件体系。我们只吸收桥接原理，后端直接接 `msgcode`。

## 非目标

- 不做 openclaw 兼容层
- 不做统一语音平台
- 不让 HomePod 成为 core 硬依赖
