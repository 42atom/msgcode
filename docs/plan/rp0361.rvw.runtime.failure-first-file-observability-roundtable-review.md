# RP0361：failure-first file observability 圆桌审议

## 参会角色

- SystemsArchitect
- SkepticalOperator
- OperationsSRE
- DevilsAdvocate
- ExecutionLead

## Decision

- Recommendation: pivot
- Confidence: high
- Scope: `0361` 收口成母线审议报告，不再承载具体实现；实现切到 `0362/0363/0364`

## Why

- `0361` 现在同时覆盖失败证据、append-only 遥测、强状态图腾三块边界，继续挂在一张单上会自然长厚。
- 现有真相源已经把前情拆开了：`0269` 管 verify 证据写回，`0292/0294` 管共享近况与 append/tail 合同，`0204` 才触到极少数跨重启强状态。
- 最省成本的推进顺序就是：先证据，再账本，后图腾。这个顺序也最不容易偷渡第二状态系统。

## Risks

1. `0363` 若写重，会和 `0292/0294` 的 `status.log` 近况簿撞边，长成第二状态系统。
2. `0364` 若不坚持默认空 allowlist，会把普通业务状态误升格成图腾。
3. `0362` 若不限制为高价值失败现场，会退化成全量 dump。

## Guardrails

- `0362` 只收难复现、跨边界、高价值失败现场，优先复用 `.msgcode/evidence/`。
- `0363` 只做单一 append-only 账本，不拆多本，不做 server，不做 dashboard。
- `0364` 默认空 allowlist，只允许极少数跨重启、人工可介入强状态。
- 三条子线都不改现有 `status.log` 合同，不新增控制面。

## Next Actions

1. 先立 `0362`，把失败证据快照边界、命名、体积和保留窗口钉死。
2. 再立 `0363`，把 append-only 轻量遥测账本收成单一本合同。
3. 最后立 `0364`，只审强状态图腾 allowlist，且默认空。
