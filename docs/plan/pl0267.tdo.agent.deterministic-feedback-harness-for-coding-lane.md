# PL0267：Coding Lane 确定性反馈基础设施

## Context

当前仓库已经有这些能力：

- `bun test`
- `tsc --noEmit`
- browser / live smoke
- `verify` 命令
- `issues/` 与 evidence 真相源

但这些能力还没有形成 coding lane 的默认反馈内环。

现在要吸收的有益部分是：

- 让 Agent 在代码任务里默认进入“改代码 -> 立即验证 -> 看真实错误 -> 再修”的快反馈循环

不能吸收的部分是：

- 新增一个系统替 Agent 自动无限修到过的平台

## Goal / Non-Goals

### Goal

- 为 coding lane 定义正式的 deterministic feedback harness
- 让验证能力从“可用”变成“默认合同”
- 把失败证据写回标准位置，缩短下一轮修复路径

### Non-Goals

- 不做新的 finish supervisor
- 不做系统自动无限重试
- 不把整仓都改造成自动修复平台

## Suggested Order

1. `tk0267`
2. `tk0268`
3. `tk0269`

原因：

- 不先写合同，runtime 侧很容易做厚
- 不先定义验证包，所谓“快反馈”就会继续漂在任务上下文里
- 证据回写要在验证包边界明确后再收

## Acceptance Criteria

1. coding lane 的快反馈不再只是习惯，而是正式合同
2. 验证包范围明确：`test / types / e2e / custom verify`
3. 证据回写只做记录，不替 Agent 裁决
