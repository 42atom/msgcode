# PL0321：Web composer inbox 消费基线

## 问题

现在只有：

- file-first 投递

还没有：

- file-first 消费

如果继续硬写，很容易出现三条歪路：

1. 新起 Web transport
2. 新起 inbox daemon
3. UI 自己维护聊天状态

## 目标

只回答一件事：

- Web composer 的输入，最小应该怎么进入现有处理主链

## 约束

- 不新增平台层
- 不新增第二真相源
- 不破坏现有 Feishu 主链
- 能复用就复用

## 当前候选

### 方案 A：复用 listener 主链

- inbox 只是原始输入面
- 消费时仍构造 `InboundMessage`
- 交给现有 `listener.handleMessage()`

问题：

- `listener` 本质是通道适配层，不是中立执行核心
- 里面带 `routeByChatId(...allowDefaultFallback=false)`、`saveCurrentSessionContext(transport=feishu)`、`sendClient` 等 Feishu 假设
- 为了 Web 去改它，容易污染现有 Feishu 主链

### 方案 B：复用 RuntimeRouterHandler 主链

- inbox 继续是原始输入面
- 消费器只负责把 inbox 请求还原成最小 `InboundMessage`
- 再直接调用 `RuntimeRouterHandler.handle()`
- 输出继续落在现有 `.msgcode/sessions/*.jsonl` 和 `.msgcode/threads/*.md`

优点：

- 复用真正的工作主链
- 不需要新 transport
- 不需要 daemon
- UI 仍然只是 dumb terminal

风险：

- 需要一个最薄 inbox 请求读取器
- 需要定义 triage 时机
- 第一版最好只承接自然语言，不抢 slash/route 控制面

## 当前结论

- 不复用 listener
- 复用 `RuntimeRouterHandler.handle()` 这条核心执行路径
- 下一刀只开 one-shot consumer，不开 daemon，不扩 transport，不做 optimistic UI

## 下一步

1. 保持 file-first：`msgcode inbox add` 继续只落文件
2. 下一刀只实现 one-shot web consumer
3. 读取 inbox 请求 -> 调 `RuntimeRouterHandler.handle()` -> 结果由现有 session/thread 真相源承接
4. 不做 daemon，不做新 transport，不做 UI 自己上屏
