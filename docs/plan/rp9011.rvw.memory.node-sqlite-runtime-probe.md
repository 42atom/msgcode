# Node SQLite 运行时探针

## 结论

今天这轮本机探针结论是：

1. `node:sqlite` 在当前环境可用，但仍带 `ExperimentalWarning`
2. `node:sqlite` 可以和 `sqlite-vec` 一起工作
3. 若要加载扩展，必须在建库时传 `{ allowExtension: true }`
4. `busy_timeout`、文件库上的 `journal_mode=WAL`、`RETURNING`、只读打开都可工作
5. 因此它**具备替换 `better-sqlite3` 的潜力**，但不建议在没有适配层和回归验证前直接切主链

一句话：这不是“不可行”，而是“值得做一张薄迁移单”。

## 运行环境

- Node: `v25.2.0`
- 探针发生时仓库仍使用：
  - `src/memory/store.ts` -> `better-sqlite3 + sqlite-vec`
  - `src/cli/todo.ts` -> 只在历史 `todo.db` 导入路径上使用 `better-sqlite3`
- 后续执行结果：
  - 已新增 `src/deps/sqlite.ts`
  - `memory/store.ts` 与 `cli/todo.ts` 已切到 `node:sqlite`
  - `better-sqlite3` 依赖已删除

## 探针事实

### 1. `node:sqlite` 已存在同步 API

命令：

```bash
node -e "const s=require('node:sqlite'); console.log(Object.keys(s));"
```

结果：

- 导出 `DatabaseSync`
- 导出 `StatementSync`
- Node 同时打印 `ExperimentalWarning: SQLite is an experimental feature`

### 2. `DatabaseSync` 自带扩展加载能力

命令：

```bash
node -e "const {DatabaseSync}=require('node:sqlite'); const db=new DatabaseSync(':memory:'); console.log(typeof db.loadExtension, Object.getOwnPropertyNames(Object.getPrototypeOf(db)).sort().join('\n')); db.close();"
```

结果：

- `db.loadExtension` 存在
- 同时存在 `enableLoadExtension`

### 3. 默认不能直接加载 `sqlite-vec`

命令：

```bash
node -e "const sqliteVec=require('sqlite-vec'); const {DatabaseSync}=require('node:sqlite'); const db=new DatabaseSync(':memory:'); try { sqliteVec.load(db); } catch (e) { console.error(e.message); process.exitCode=1; } finally { db.close(); }"
```

结果：

- 报错：`extension loading is not allowed`

### 4. 事后补开扩展也不行

命令：

```bash
node -e "const sqliteVec=require('sqlite-vec'); const {DatabaseSync}=require('node:sqlite'); const db=new DatabaseSync(':memory:'); try { db.enableLoadExtension(true); sqliteVec.load(db); } catch (e) { console.error(e.message); process.exitCode=1; } finally { db.close(); }"
```

结果：

- 报错：`Cannot enable extension loading because it was disabled at database creation.`

### 5. 建库时允许扩展后，`sqlite-vec` 可正常工作

命令：

```bash
node -e "const sqliteVec=require('sqlite-vec'); const {DatabaseSync}=require('node:sqlite'); const db=new DatabaseSync(':memory:', { allowExtension: true }); try { sqliteVec.load(db); console.log(db.prepare('select vec_version() as version').get()); } finally { db.close(); }"
```

结果：

- 成功加载
- `vec_version()` 返回 `v0.1.7-alpha.2`

### 6. 现主链依赖的基础 SQL 能力可用

探针覆盖：

- `prepare().run()`
- `prepare().get()`
- `prepare().all()`
- `INSERT ... RETURNING`
- `ON CONFLICT ... RETURNING`
- `readOnly: true`
- `PRAGMA busy_timeout`
- 文件库上的 `PRAGMA journal_mode=WAL`

结果：

- 都能跑通

## 对当前仓库的含义

对 [store.ts](/Users/admin/GitProjects/msgcode/src/memory/store.ts) 来说，最大的变化不是 SQL 语法，而是连接初始化口径：

1. 现在是 `new BetterSqlite3(path)`
2. 若换 `node:sqlite`，memory 主链至少要改成 `new DatabaseSync(path, { allowExtension: true })`
3. `busy_timeout` / `journal_mode` 不能继续走 `.pragma()`；要改成 `exec("pragma ...")` 或查询式读取

对 [todo.ts](/Users/admin/GitProjects/msgcode/src/cli/todo.ts) 来说，历史 `todo.db` 导入路径也可改成 `DatabaseSync(dbPath, { readOnly: true })`。

## 仍未确认的边界

迁移前还需要补三类事实：

1. `sqlite-vec` 在真实文件库、真实 schema 初始化、真实向量写入/检索链路上是否完全兼容，不只是一条 `vec_version()`
2. `node:sqlite` 的结果对象、blob/buffer、错误对象、锁竞争语义，是否会让现有测试或日志口径漂移
3. 既然 `node:sqlite` 仍是 experimental，仓库是否愿意接受这条平台依赖边界

## 建议

建议走一张很薄的迁移单，不直接在本报告里动主链：

1. 先加一个 `sqlite runtime adapter`
2. 让 `memory/store.ts` 和历史 `todo.db` 导入都走同一适配层
3. 先跑 Node-only 的 memory/todo 验证
4. 全绿后再删除 `better-sqlite3` 依赖
