# PL0321：Web composer inbox 消费基线

## 问题

现在只有：

- file-first 投递

还没有：

- file-first 消费

如果继续硬写，很容易出现三条歪路：

1. 新起 Web transport
2. 新起 inbox daemon
3. UI 自己维护聊天状态

## 目标

只回答一件事：

- Web composer 的输入，最小应该怎么进入现有处理主链

## 约束

- 不新增平台层
- 不新增第二真相源
- 不破坏现有 Feishu 主链
- 能复用就复用

## 当前候选

### 方案 A：复用 listener 主链

- inbox 只是原始输入面
- 消费时仍构造 `InboundMessage`
- 交给现有 `listener.handleMessage()`

优点：

- 主链最短
- 原有线程/dispatch/evidence 能复用

风险：

- 现有 listener 带 Feishu 假设
- `runtime.current_transport` 现在只接受 `feishu`
- route 也按 chatId 路由

### 方案 B：新起 Web 消费器

- UI 写 inbox
- 一个新 runner 专门扫 inbox 再处理

优点：

- 不碰现有 listener

风险：

- 直接长出第二主链
- 不符合当前原则

## 当前倾向

- 只接受方案 A
- 但要继续收薄：
  - 不先扩 transport 枚举
  - 不先写 daemon
  - 先证明现有 listener 哪些点真挡路

## 下一步

1. 先做 listener / route / workspace-config 的最小复用盘点
2. 再决定是否开实现单
3. 若挡路点超过 2-3 个，就停下，不强上
