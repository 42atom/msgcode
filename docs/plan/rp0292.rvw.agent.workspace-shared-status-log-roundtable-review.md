# RP0292：status.log 圆桌审议

## 参会角色

- SystemsArchitect
- SkepticalOperator
- UserAdvocate
- OperationsSRE

## Decision

- Recommendation: go
- Confidence: medium
- Scope: 保留 `status.log`，但必须进一步收紧为“单行、去重、只记阶段性结论”的共享近况簿

## Why

- 现有系统里，`summary` 偏记忆连续性，`WORKSTATE` 偏恢复姿态，缺一份线程间共享的近况面。
- `status.log` 作为 append-only 文本文件，符合当前仓库“文件即真相、文本优先、少加层”的主线。
- 只要它不复制聊天原文、不自动轮询、不变成第二份会话历史，就能以极低复杂度解决跨线程近况共享问题。

## Risks

1. 如果写入触发过宽，它会退化成第二份聊天摘要。
2. 如果格式不够硬，后续 UI 和 Agent 都会解析不稳。
3. 如果没有去重规则，几条线程会把同一结论反复写爆。
4. 如果 UI 把它展示成实时面，用户会误判系统“在线上报”。

## Guardrails

- 前置条件:
  - 只允许记录“已确认决定”和“已确认状态变化”
  - 一条记录必须单行
  - 必须带原文引用
  - 后续实现必须通过单一 append 合同写入
- 监控指标:
  - 单线程重复记录率
  - 最近 20 条中相同内容占比
- 回滚条件:
  - 如果实际试跑后它开始承载长摘要或重复记录，应回退为仅人工/显式线程阶段收尾时写入

## Next Actions

1. 在 `0292` 中增加单行约束与去重规则
2. 把“本轮形成”改成“阶段性结论/状态变化”这种更硬的写入条件
3. 保持 UI 只读 tail，不把它渲染成聊天流
4. 明确它不是实时监控面
