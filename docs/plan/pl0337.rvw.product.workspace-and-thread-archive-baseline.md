# PL0337：工作区与线程归档基线

## 结论

归档只做一件事：

- 把不再活跃的内容从当前主界面移走
- 但不删除，不改写，不装成 `/clear`

第一版按两层收：

1. 总工作区归档
2. 工作区内线程归档

都坚持文件移动，不加数据库，不加归档中心。

## 为什么需要 archive

主界面以后会越来越密：

- 工作区会越来越多
- 每个工作区下的线程也会越来越多

如果所有历史内容都继续挤在活跃面，用户会越来越难找到当前真正活跃的东西。

但这不意味着要删除历史。

更稳的做法是：

- 活跃面只看当前活跃内容
- 历史内容移进 archive
- 需要时从 archive 查看或恢复

这也是 thread 线条感变碎时，第一优先要做的事。

也就是说：

- 先用 archive 把测试/冒烟/旧历史从活跃面移走
- 再去判断 thread 本身的边界是否有问题

## archive 与 /clear 的边界

这条边界必须钉死。

`/clear` 的含义是：

- 清掉短期上下文
- 开启新线程
- 原线程仍然存在

`archive` 的含义是：

- 从活跃列表移走
- 文件进入 archive 目录
- 历史仍然可看、可恢复

一句话：

- `/clear` 是继续工作前的清场
- `archive` 是把旧东西收进柜子

## 总工作区归档

### 目的

把整整个工作区从当前工作集移走，但保留完整目录。

### 文件真相源

工作区根下新建一层隐藏目录：

- `/Users/admin/msgcode-workspaces/.archive/`

规则：

- 活跃工作区仍在 `/Users/admin/msgcode-workspaces/<workspace>/`
- 归档工作区移动到 `/Users/admin/msgcode-workspaces/.archive/<workspace>/`

UI 文案可以叫：

- 项目归档

但文件系统仍保持简单 ASCII 路径。

### 恢复

恢复就是一次反向移动：

- `.archive/<workspace>` → 根目录 `<workspace>`

第一版若遇到同名冲突：

- 直接失败
- 提示用户先处理现有工作区

不自动重命名。

## 工作区内线程归档

### 目的

把不再活跃的线程从当前工作区线程列表移走，但保留历史。

### 文件真相源

每个工作区内新增一层：

- `<workspace>/.msgcode/archived-threads/`

规则：

- 活跃线程只读 `<workspace>/.msgcode/threads/`
- 归档线程只读 `<workspace>/.msgcode/archived-threads/`

归档动作就是：

- 把线程 markdown 文件从 `threads/` 移到 `archived-threads/`

### 为什么不动 session jsonl

第一版先不移动 session 原文。

理由很简单：

- 线程 markdown 已经有 `chatId`
- 只要 archive 页面能通过 `chatId` 找回 session 原文，就够了
- 不必为了“更完整”再设计一层 session archive

这更薄。

### 恢复

恢复就是反向移动：

- `archived-threads/<thread>.md` → `threads/<thread>.md`

同样：

- 同名冲突直接失败
- 不偷偷改名

## 主界面口径

主界面永远只看活跃内容：

- 左边工作区树只读根目录活跃工作区
- 左边线程列表只读当前工作区 `threads/`

也就是说：

- 归档工作区不出现在主工作区树
- 归档线程不出现在当前线程列表

这正是 archive 的价值。

## 历史归档页

第一版单独一页就够，不混进主工作面。

它至少有两块：

1. 项目归档
   - 显示 `.archive/` 下的工作区
   - 支持查看和恢复

2. 线程归档
   - 显示当前工作区 `archived-threads/`
   - 支持查看和恢复

这一页不做复杂筛选，不做搜索平台。

只做：

- 看
- 恢复

## 恢复动作

恢复仍然坚持 file-first：

- Workspace restore = 目录移动
- Thread restore = 文件移动

不做：

- 额外同步器
- 状态数据库
- 隐式 reconcile

## UI 入口

第一版建议：

- 主界面不直接显示 archive 内容
- 提供单独的“历史归档”入口页

这样能保持主工作面干净。

## 不做什么

第一版明确不做：

- 自动归档
- 自动恢复
- 热冷分层策略
- 归档搜索
- 跨工作区历史聚合
- 归档后的自动摘要重写

## 圆桌收口

圆桌收住的三条：

1. archive 优先用目录移动表达，不用状态旗标表达
2. 线程 archive 不碰 session 原文，避免做重
3. 历史归档页单独存在，不污染主工作面

## 推荐拆解

这条线拆三刀就够：

1. 总工作区 archive root 与 restore 合同
2. 线程 archive 与 `/clear` 边界合同
3. 历史归档页与 restore 入口合同
