# RF0001：Electron Current UI Constraints

## 结论

这份文档只记录当前这版 UI 的阶段性约束。

它不是永久架构原则。

`ui-protype` 是用户侧 UX 原型真相。

正式 UI 设计与实现可以另起炉灶，落到 `src` 里做；但必须持续和 `ui-protype` 对比，确认结构不漂。原型不能被实现切片静默改写或替代。

## 当前桌面 UI 约束

当前桌面 UI 继续按桌面产品处理，不按移动端网页处理。

当前约束包括：

- 主工作面保持桌面布局，不往移动端塌缩
- 长列表按桌面工作流处理
- 默认单主窗口优先
- thread 继续是用户一级对象
- 新建聊天不直接创建空 thread

## thread / new chat 边界

当前这条线的真相仍然在：

- [pl0341.pss.product.web-thread-and-new-chat-boundary-baseline.md](/Users/admin/GitProjects/msgcode/docs/plan/pl0341.pss.product.web-thread-and-new-chat-boundary-baseline.md)
- [tk0341.pss.product.p1.web-thread-and-new-chat-boundary-baseline.md](/Users/admin/GitProjects/msgcode/issues/tk0341.pss.product.p1.web-thread-and-new-chat-boundary-baseline.md)

这里不再重复发明规则，只承认当前方向：

- web thread 必须有稳定 `chatId`
- 标题回到 runtime/front matter
- 新建聊天不抢 thread 创建权

## 窗口与布局

当前产品阶段默认：

- 主窗口优先
- 多窗口不是默认主路径
- 页面布局服务桌面工作流，不做移动端妥协式折叠

这些都是当前产品约束，不应上升成永久宪法。

## 使用方式

如果在做：

- 当前 UI 原型
- 当前主界面布局
- 当前 thread / new chat 交互

优先看这份参考和相关 `pl0341 / tk0341~0344`。

同时默认遵守一条双轨约束：

- `ui-protype` 继续承载用户侧结构真相
- 正式实现单独推进，但要持续对照原型
- 实现切片不能把原型当成可静默覆盖的草稿

如果在做：

- Electron 主进程职责
- IPC
- 隔离
- file-first

优先看 [rf0000.rvw.product.electron-runtime-principles.md](/Users/admin/GitProjects/msgcode/docs/plan/rf0000.rvw.product.electron-runtime-principles.md)。
