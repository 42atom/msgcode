# PL0262：本地 Appliance 自包含、App/UI 与交付计划

## Context

当前主线已经能跑，但正式交付口径仍偏开发者环境：

- 安装依赖 `brew` / `npm install`
- 运行依赖源码态 `tsx`
- shell 合同绑定 Homebrew Bash
- 浏览器、desktop、模型后端都还带明显的机器前提

当前产品定义已经收口为：

- 小微机构本地部署
- `Mac mini` appliance
- 隐私优先
- 单机单组织优先

所以这条线不该再用“企业平台”心智去拆，而应该回答 appliance 交付的四个现实问题：

1. 如何自包含交付，不现场下载
2. 如何先定义机构身份最小面
3. 如何承接 App/UI 外壳
4. 如何处理本地模型与安装链边界

这已经不是单点修 bug，必须先拆成几条边界清楚的任务。

配套调研报告见：

- [rp0262](/Users/admin/GitProjects/msgcode/docs/plan/rp0262.rvw.product.enterprise-app-capability-layering-and-ip-boundary.md)
- [rp0280](/Users/admin/GitProjects/msgcode/docs/plan/rp0280.rvw.product.appliance-self-contained-dependency-inventory.md)
- [rp0281](/Users/admin/GitProjects/msgcode/docs/plan/rp0281.rvw.product.core-appliance-package-boundary.md)
- [rp0282](/Users/admin/GitProjects/msgcode/docs/plan/rp0282.rvw.product.appliance-delivery-manifest-v1.md)

## 结论

这条线先拆成四刀：

1. 先定义 appliance “自包含、不下载”的基线与验收
2. 再定 App shell / UI 形态，不先默认 Electron 就是答案
3. 再定本地模型后端与 MLX/mac 的边界
4. 最后收安装链，让 appliance 现场不再依赖 `brew` 与 `npm install`

## Goal / Non-Goals

### Goal

- 为本地 appliance 形成一条清晰推进路线
- 把“交付形态”“App/UI”“模型后端”“安装链”拆开
- 避免后续把产品、运行时、模型和打包问题揉成一团

附加约束：

- 机构身份第一版只允许 3 个字段：
  - 名称
  - 交税地
  - 统一社会信用代码

### Non-Goals

- 不在本单里直接选定 Electron
- 不在本单里直接实现打包系统
- 不在本单里直接重写模型后端
- 不在本单里先做复杂机构对象模型

2. `tk0264`
3. `tk0263`
4. `tk0265`

原因：

- 不先定义“appliance 到底要求什么”，后面所有技术选择都会漂
- 不先定模型后端边界，跨平台 App/UI 方案就是空谈
- 不先定 App shell 方向，安装链打包目标不明确

1. 四张子任务边界清楚，不互相抢职责
2. 后续可以按这四张单逐张推进，而不是开一张“大而全平台版”
3. plan 明确推荐顺序与非目标
