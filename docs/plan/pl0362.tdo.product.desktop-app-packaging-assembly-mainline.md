# PL0362：Desktop App Packaging Assembly Mainline

## 结论

现在不缺大方向，缺的是一条很薄的装配主线。

这条线只回答一件事：

- 怎么把现有 `CLI/runtime/UI surfaces` 装成 `build artifact -> app -> dmg`

一句话：

- 现有主脑不变
- 现有 UI 面不重做
- 只补装配顺序

## 为什么现有文件不能承载这个职责

### `tk0262`

它回答的是 appliance 自包含交付基线：

- 什么叫正式交付
- 机构最小面是什么
- 能力边界是什么

它不回答 Desktop App 要怎么装配出 app 和 dmg。

### `tk0263`

它回答的是 App shell / UI 方向：

- 为什么需要 App
- 为什么是 Electron 薄壳
- Main UI 最小面是什么

它不回答 packaging assembly 的阶段顺序和产物合同。

### `tk0265`

它回答的是 bootstrap：

- 怎么摆脱 `brew`
- 怎么摆脱 `npm install`
- install / first-run / upgrade 怎么分

它不回答桌面宿主壳如何把这些现有运行物组装成 Desktop App。

### `pl0274`

它是离线规划，重点是：

- Electron 做宿主
- runtime 做主脑
- Web 做界面

它没有收成现役执行主线，也没有把 `artifact / app / dmg` 装配顺序收成正式计划。

### `rf0000` / `rf0001`

这两份是参考：

- 一个约束长期 Electron 原则
- 一个约束当前 UI 边界

参考不是执行单，不能替代 packaging 主线。

## 输入与输出

## 输入

这条线只承认现有输入：

1. `msgcode` CLI/runtime 运行物
2. 已定义的 host/UI surfaces
3. `tk0265` 所定义的 bootstrap/runtime 产物

其中 `tk0371` 这一刀先把输入面收窄到：

1. compiled runtime + launcher
2. appliance bootstrap 脚本与 manifest 合同

它先不吞宿主壳源码树，也不吞 dmg 分发链。

## 输出

这条线只承认 3 个输出：

1. 可复用的 build artifact
2. 本地 Desktop App bundle
3. `dmg` 分发物

## 不新增的东西

默认不新增：

1. 第二主脑
2. 前置控制面
3. 大而全交付平台
4. 授权中心
5. 自动更新系统

## 主线顺序

最小顺序固定为：

1. 收输入面
   - 明确 CLI/runtime/UI surfaces 哪些已经存在
2. build artifact assembly
   - 对应 [tk0371.rvw.product.p1.build-artifact-assembly.md](/Users/admin/GitProjects/msgcode/issues/tk0371.rvw.product.p1.build-artifact-assembly.md)
   - 明确哪些运行物进入 `bundle-root`
   - 明确排除 `src/**`、开发态安装链与宿主/dmg 职责
   - 明确下游只能消费 artifact，不回退到源码安装链
3. app bundle host assembly
   - 对应 [tk0372.rvw.product.p1.app-bundle-host-assembly.md](/Users/admin/GitProjects/msgcode/issues/tk0372.rvw.product.p1.app-bundle-host-assembly.md)
   - 明确 app bundle 只消费 `tk0371` 产出的 `bundle-root`
   - 明确 Electron 只做宿主装配：放置 host 二进制、声明启动入口、把本地 Web/UI 打开到 bundled runtime
   - 明确 app bundle 完成面的最小验收只到“本地可启动的 Desktop App”，不前推回 artifact，也不后推进 dmg 分发
   - 明确排除源码树直读、授权中心、自动更新、签名、公证和跨平台安装器
4. dmg packaging / signing / notarization chain
   - 对应 [tk0373.rvw.product.p1.dmg-packaging-signing-notarization-chain.md](/Users/admin/GitProjects/msgcode/issues/tk0373.rvw.product.p1.dmg-packaging-signing-notarization-chain.md)
   - 明确输入只承认 `tk0372` 产出的 `.app` bundle，不回读源码树、artifact 组装细节或运行时构建链
   - 明确顺序固定为：先把 `.app` 包进 `dmg` 分发容器，再对正式分发物执行签名与 notarization 交付链
   - 明确 `dmg` 只是一层外发容器，不承担宿主职责、不重定义 app 结构、不创造新的产品母线
   - 明确排除自动更新、App Store、跨平台安装器和额外 artifact 规则

这里的关键约束是：

- `dmg` 不创造新职责
- `app` 不创造新主脑
- `artifact` 不回退成源码安装链

对应 `tk0372`，宿主切口再收紧为 4 句：

1. 输入只认 `bundle-root`
2. 宿主只做装配和启动，不重打包 runtime 合同
3. 输出只认本地 `.app` bundle 与稳定启动面
4. `dmg` 只是在这份 `.app` 外面再包一层分发外壳

## 子任务切法

这条母线只拆 3 张最薄子任务：

1. `tk0371`
   - 只回答 build artifact 怎么收
2. `tk0372`
   - 只回答 app bundle 宿主怎么装
3. `tk0373`
   - 只回答 dmg 打包、签名、公证链怎么收

不再额外拆：

1. 授权
2. 自动更新
3. 跨平台安装器
4. 新的总控 plan

## 边界

这条线只负责装配。

它不负责：

1. 重写 CLI
2. 重写 runtime
3. 设计新 UI
4. 设计授权协议
5. 设计更新协议
6. 宣称全平台都成立

## 验收口径

这条 plan 完成时，至少应能回答：

1. 当前 packaging assembly 的输入面到底有哪些。
2. `tk0371 -> tk0372 -> tk0373` 各自的职责是什么。
3. 为什么这是一条新主线，但又不是新母线。
4. 后续实现切片应围绕哪 3 个产物推进，而不是横向再散开。

## 与主线索引的关系

`pl0211` 应把当前活线切到这条 packaging assembly 主线。

原因很简单：

- observability 这批薄切片已经收口
- 现在更阻塞交付的是 app / artifact / dmg 装配主线

## 一句话

现在该推进的不是再谈 Electron 对不对，而是把已经收口的 CLI/runtime/UI surfaces 装成可交付的 Desktop App 主线。
