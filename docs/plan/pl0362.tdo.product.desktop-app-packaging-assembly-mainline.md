# PL0362：Desktop App Packaging Assembly Mainline

## 结论

现在不缺大方向，缺的是一条很薄的装配主线。

这条线只回答一件事：

- 怎么把现有 `CLI/runtime/UI surfaces` 装成 `build artifact -> app -> dmg`

一句话：

- 现有主脑不变
- 现有 UI 面不重做
- 只补装配顺序

## 为什么现有文件不能承载这个职责

### `tk0262`

它回答的是 appliance 自包含交付基线：

- 什么叫正式交付
- 机构最小面是什么
- 能力边界是什么

它不回答 Desktop App 要怎么装配出 app 和 dmg。

### `tk0263`

它回答的是 App shell / UI 方向：

- 为什么需要 App
- 为什么是 Electron 薄壳
- Main UI 最小面是什么

它不回答 packaging assembly 的阶段顺序和产物合同。

### `tk0265`

它回答的是 bootstrap：

- 怎么摆脱 `brew`
- 怎么摆脱 `npm install`
- install / first-run / upgrade 怎么分

它不回答桌面宿主壳如何把这些现有运行物组装成 Desktop App。

### `pl0274`

它是离线规划，重点是：

- Electron 做宿主
- runtime 做主脑
- Web 做界面

它没有收成现役执行主线，也没有把 `artifact / app / dmg` 装配顺序收成正式计划。

### `rf0000` / `rf0001`

这两份是参考：

- 一个约束长期 Electron 原则
- 一个约束当前 UI 边界

参考不是执行单，不能替代 packaging 主线。

## 输入与输出

## 输入

这条线只承认现有输入：

1. `msgcode` CLI/runtime 运行物
2. 已定义的 host/UI surfaces
3. `tk0265` 所定义的 bootstrap/runtime 产物

## 输出

这条线只承认 3 个输出：

1. 可复用的 build artifact
2. 本地 Desktop App bundle
3. `dmg` 分发物

## 不新增的东西

默认不新增：

1. 第二主脑
2. 前置控制面
3. 大而全交付平台
4. 授权中心
5. 自动更新系统

## 主线顺序

最小顺序固定为：

1. 收输入面
   - 明确 CLI/runtime/UI surfaces 哪些已经存在
2. 装 build artifact
   - 明确哪些运行物进入可打包产物
3. 装 app bundle
   - 明确 Electron 宿主只如何承接这些产物
4. 装 dmg
   - 明确正式分发物只是在 app 外再包一层交付介质

这里的关键约束是：

- `dmg` 不创造新职责
- `app` 不创造新主脑
- `artifact` 不回退成源码安装链

## 边界

这条线只负责装配。

它不负责：

1. 重写 CLI
2. 重写 runtime
3. 设计新 UI
4. 设计授权协议
5. 设计更新协议
6. 宣称全平台都成立

## 验收口径

这条 plan 完成时，至少应能回答：

1. 当前 packaging assembly 的输入面到底有哪些。
2. `build artifact -> app -> dmg` 各自的职责是什么。
3. 为什么这是一条新主线，但又不是新母线。
4. 后续实现切片应围绕哪 3 个产物推进，而不是横向再散开。

## 与主线索引的关系

`pl0211` 应把当前活线切到这条 packaging assembly 主线。

原因很简单：

- observability 这批薄切片已经收口
- 现在更阻塞交付的是 app / artifact / dmg 装配主线

## 一句话

现在该推进的不是再谈 Electron 对不对，而是把已经收口的 CLI/runtime/UI surfaces 装成可交付的 Desktop App 主线。
