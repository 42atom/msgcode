# RP0308：status-log CLI 与 prompt 圆桌复核

## 结论

这刀可以保留。

保留：

- 单一 `status-log` CLI
- `add/tail` 两个最小动作
- prompt 只给读写口径

不要再长：

- hook
- timer
- 自动催写
- UI 乐观状态

## 圆桌意见

### 命令边界

不要把这组命令塞进 `appliance`。

原因：

- `status.log` 是工作区主链，不是壳合同
- 主脑自己也要用

所以独立顶层 `status-log` 更干净。

### 写入方式

不要让 prompt 自己拼原始行。

CLI 已经足够：

- 接收结构化字段
- 统一清洗
- 统一格式化
- 调 runtime append

### 读取方式

prompt 有了 `tail` 就够了。

不要再加：

- `head`
- `list`
- `grep`
- `watch`

### Prompt 约束

要写清楚：

- 它是共享近况，不是聊天历史
- 默认只读最近几条
- 只有已确认决定或状态变化才写

不要写“很重要时”“感觉需要时”这种虚词。

## 本轮吸收

1. 只保留 `add/tail`
2. `status-log` 保持顶层命令，不塞 `appliance`
3. prompt 明确禁止手写原始日志行

## 结果

`0307` 现在仍然符合这条边界：

- 真相仍在文件
- 读写仍然很薄
- 决策仍在模型
- 系统不抢线程编排权
