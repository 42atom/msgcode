# PL0294：status.log 追加与 tail 读取合同

## 结论

`status.log` 以后只能通过一个很薄的 append 合同写入。

不要让模型直接写原始行。  
不要让 UI 扫全文件做二次总结。

一句话：

- 写：append 一条结构化状态
- 读：tail 最近几条

## append 输入

最小输入字段固定为：

1. `thread`
2. `kind`
3. `summary`
4. `refPath`
5. `refLine`

其中：

- `thread`
  - 当前线程标识
- `kind`
  - `decision` 或 `state`
- `summary`
  - 一句话
- `refPath`
  - 工作区相对路径
- `refLine`
  - 原文行号

不接受模型直接提供完整格式化文本。

## 落盘格式

落盘器统一格式化成：

```text
时间 | 线程名 | 类型 | 一句话内容 | 相对路径#L行号
```

约束：

1. 单行
2. 所有换行去掉
3. 内容里的 `|` 转成全角 `｜`
4. 时间由写入器生成，不由模型瞎写
5. 必须通过单一 runtime writer 原子追加

## 追加边界

第一版不做：

1. 定时写
2. 每轮强制写
3. 切线程 hook
4. 空闲 heartbeat 催写

只有当上层明确要“追加一条状态”时，才走 append 合同。

这件事继续由 prompt 约束模型判断，不由 runtime 替它做主。
writer 只负责清洗、格式化、追加，不替日志做去重判断。

## 读取合同

UI 第一版只允许：

- tail 最近 10 条

不允许：

1. 扫全文件
2. 把 status.log 当时间线聊天流
3. 前端自己再生成第二层摘要

## UI 消费

“工作状况 / Work Status” 只读：

1. 当前线程自己的工作状况真相源
2. `status.log` 最近几条共享状态

展示要求：

1. 必须显示最后更新时间
2. 不得暗示实时监控
3. 若文件不存在，直接空态

## 运行时边界

`status.log` 不是：

1. session 共享总线
2. 实时监控流
3. 聊天历史副本
4. 自动协调器

它只是：

- 一本工作区共享的最近工作状况簿

## 下一步

真正进实现时，只拆一刀：

1. 单一 runtime writer：`appendWorkspaceStatus(...)`
2. UI tail 读取器

不要再长更多层。
