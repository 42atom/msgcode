# Heartbeat Alarm And Reflection Mainline

## 问题

当前 `msgcode` 已经有很多零件：

- `scheduler / jobs`
- `/task / TaskSupervisor`
- `subagent`
- `issues/` 文档协议
- 任务看板 HTML

但它还缺一个最关键的东西：

- 主脑什么时候自己醒来
- 醒来先看什么
- 看完后怎么推进
- 完成后怎么反思与回收

没有这条主循环，前面的能力都会停留在“用户推一下，系统动一下”。

## 目标

- 让每个 workspace 都能有周期性的主脑 heartbeat
- 让精确定时 alarm/cron 成为 heartbeat 的唤醒器，而不是第二大脑
- 让任务巡检、subagent 监督、交付推进、反思回收进入一条主链
- 让状态继续以文件为真相源

## 非目标

- 不做重型 agent orchestration 平台
- 不引入隐藏状态数据库覆盖文档状态
- 不把反思系统先做成复杂评分/训练框架
- 不为了“更智能”破坏单一主链

## 设计判断

### 第一性原则

主脑节律应该拆成三段，而不是一段大循环：

1. `heartbeat`：周期意识巡检
2. `alarm/cron`：精确唤醒
3. `reflection`：任务完成后的经验回收

三者不能混。

### 为什么不能混

- 把 cron 和 heartbeat 混在一起，系统会失去“周期 awareness”和“精确定时”的边界
- 把 reflection 混进普通任务执行，系统会不断噪声化
- 把状态放进隐藏 runtime，用户就失去对任务的文档级可见性

## 方案

## 1. Heartbeat

heartbeat 是主脑主会话里的周期性巡检，不是精确定时器。

物理实现口径先收成最薄版本：

- 第一版仍由 msgcode 现有 heartbeat runner 托管
- workspace 不各自起常驻独立进程
- 先通过 `wake record + heartbeat tick` 避免“50 个 workspace 各自盲轮询”
- 真正有到期 wake 或 runnable work 时才进入更重的扫描与执行

关于横向伸缩性，默认原则是：

- 不做每个 workspace 都高频 polling
- 不先上中心调度器
- 后续若文件触发确有价值，可把 `fsevents` 作为唤醒优化，而不是新的真相层或唯一触发器

一句话：

- heartbeat 先是统一节律
- wake record 才是每个 workspace 的到点合同
- `fsevents` 以后可以做优化，不做地基

它每次醒来优先读取：

- `<workspace>/.msgcode/HEARTBEAT.md`
- `issues/` 中与当前 workspace 关联的父任务 / 子任务
- `<workspace>/.msgcode/dispatch/*.json`
- 当前 workspace 的 subagent 状态

heartbeat 要做的最小动作：

1. 看未完成任务
2. 看 subagent 有没有卡住
3. 看产物是否已齐
4. 能推进就推进状态
5. 无事则静默 ack

建议静默响应口径：

- `HEARTBEAT_OK`

### `HEARTBEAT.md` 与真相源边界

`HEARTBEAT.md` 不是第二状态系统。

它只承载：

- 主脑本轮关注点
- 简短巡检 checklist
- 局部提醒与思考草稿

它不承载：

- 任务状态真相
- 子任务是否完成
- 派单是否成功
- 最终验收结论

硬规则：

- `issues/*.md` 文件名状态仍是任务状态真相源
- `.msgcode/wakeups/*.json` 与 `.msgcode/dispatch/*.json` 是运行合同真相源
- 若 `HEARTBEAT.md` 与以上文件冲突，一律以前者之外的真相源为准

一句话：

- `HEARTBEAT.md` 是主脑草稿
- `issues/ + wakeups/ + dispatch/` 才是硬状态

## 2. Alarm / Cron

alarm/cron 只负责“什么时候叫醒”，不负责“醒来后怎么办”。

适用场景：

- 半小时后复查某张子任务单
- 明早 9 点写日报
- 每天下午 6 点做一轮工作区巡检
- 子代理超时后延迟复查

文件真相源建议：

- `<workspace>/.msgcode/schedules/*.json`

原则：

- cron 不做思考
- cron 只投递唤醒
- 唤醒后的判断仍由 heartbeat 或显式任务流程完成

## 3. Reflection

reflection 不独立成第四个系统。

先作为两类时机下的固定 checklist：

- 子任务 / 父任务完成时
- 每日收口 heartbeat 时

最小回收动作：

1. 写简短日记
2. 写 memory candidate
3. 写 skill candidate

文件落点建议：

- 日记：`AIDOCS/reports/daily/<date>.md`
- memory candidate：先写在日记或任务证据里，后续再入正式 memory
- skill candidate：先写在日记或任务证据里，后续再转 skill 单

### Reflection 的价值抽取

reflection 不能退化成“摘要堆填区”。

最小口径：

- `memory candidate` 不是自动入 memory
- `skill candidate` 不是自动改 `SKILL.md`
- candidate 先作为待审核改进项落盘

建议增加一个极薄动作：

- `reflection review`

它不是新系统，只是下一次 heartbeat 醒来时的优先 checklist：

1. 有没有待审核 candidate
2. 哪些只该归档
3. 哪些应升级成正式 task / skill 单 / memory 合并动作

一句话：

- 反思先产出 candidate
- 下一轮 heartbeat 负责决定是否吸收

## 与现有主链的结合

### 与 `tk0202`

- persona 文档定义“怎么做”
- heartbeat 负责“什么时候回看和改进”

### 与 `tk0203`

- doc-first 派单定义“做到哪”
- heartbeat 负责“是否该继续派单 / 催单 / 验收 / 关闭”

### 与 `scheduler / jobs`

- 复用现有 workspace schedule 投影链
- 但要收口到 heartbeat / alarm 语义，不再只停在 generic jobs

### 与 `subagent`

- subagent 仍是执行臂
- heartbeat 负责监督和回收，不给 subagent 升格成主脑

## 文件真相源建议

建议最少只新增或正式化这几处：

- `<workspace>/.msgcode/HEARTBEAT.md`
- `<workspace>/.msgcode/schedules/*.json`
- `<workspace>/.msgcode/dispatch/*.json`
- `AIDOCS/reports/daily/*.md`

继续复用：

- `issues/*.md`

## 执行顺序

1. 先定义 heartbeat 文档协议  
   先明确主脑醒来读什么。

2. 再定义 alarm/cron 唤醒协议  
   让定时器只做唤醒，不抢主脑决策。

3. 再把 `/task + subagent + issues` 接进 heartbeat  
   让巡检能真正推进任务。

4. 再补 reflection review 口径  
   让 candidate 不只落盘，还能进入下一轮审核。

5. 最后补 reflection 落点  
   让系统开始积累长期价值。

## 实现分期

### Phase 1: Heartbeat 协议模板

先把 heartbeat 读写合同钉死，不先扩成复杂执行框架。

最小交付：

- `<workspace>/.msgcode/HEARTBEAT.md` 模板
- 固定读取顺序：
  - `HEARTBEAT.md`
  - `wakeups/records/*.json`
  - `issues/*.md`
  - `dispatch/*.json`
  - `subagents/*.json`
- 固定输出语义：
  - 有事就推进
  - 无事就 `HEARTBEAT_OK`

验收：

- heartbeat 输入面固定
- `HEARTBEAT.md` 只做草稿，不承载硬状态

### Phase 2: Alarm 文件协议

在已有 wake 主链上，把“什么时候叫醒”正式文件化。

最小交付：

- `<workspace>/.msgcode/schedules/*.json` 协议
- 与 `wake job / wake record` 的映射口径
- missed wake 的 catch-up 规则

验收：

- alarm 只负责唤醒，不负责决策
- schedule truth 与 wake truth 不冲突

### Phase 3: Reflection Review

最后补反思的最小吸收动作，不让 candidate 只堆日志。

最小交付：

- 日记落点模板
- candidate 命名与落点约定
- 下一轮 heartbeat 的 `reflection review` checklist

验收：

- candidate 不自动升级
- heartbeat 能优先审核 candidate 去向

## 验收口径

1. 存在工作区级 heartbeat 协议与最小模板。
2. 存在工作区级 alarm/cron 文件协议。
3. heartbeat 能基于文件真相源判断“有事 / 无事”。
4. 无事时能静默，不制造噪音。
5. `HEARTBEAT.md` 的定位被明确限定为草稿/提示，不与 `issues/` 和 `.msgcode/wakeups/` 争夺真相源地位。
6. 子任务或父任务完成后，存在最小反思回收步骤，且 candidate 不会直接变成无审核的正式 memory/skill。
7. 这条主循环能成为 `tk0202 + tk0203` 的驱动器。

### 经理型主脑补充口径

后续对 `tk0204` 的验收，不再只看“heartbeat 是否存在”，而要看主脑是否已经表现出经理工作方式。

1. 默认优先派单：
   - 面对可委派任务时，主脑默认优先走 `dispatch + subagent`
   - 只有明确的小任务，才允许主脑亲自执行
2. 完成必须有证据：
   - `completed` 不能只靠子代理口头回报
   - 必须绑定 artifact、evidence 或明确验收结果
   - 无证据不算完成
3. 文档状态必须真实推进：
   - 主脑不能只写 `.msgcode/dispatch/*.json`
   - 子任务完成后，必须推进对应 `issues/*.md` 状态
   - 必要时继续推进父任务
4. 用户沟通必须忠诚：
   - 遇到阻塞、失败、证据不足、需求变化时，主脑必须及时对用户说真话
   - 不能把唤醒词、任务原文或猜测性成功当正式交付
5. 测试要证明“会管理”，不只“会派一次”：
   - 读任务
   - 选 persona
   - 发 dispatch
   - 子代理执行
   - 主脑验收
   - 回写文档
   - 继续推进下一单或如实汇报阻塞

## 竞争判断

我们要超越他们，不是靠更厚，而是靠更短：

- 比 `pi-mono` 更强的任务真相源
- 比 `openclaw` 更薄的控制面
- 比 `hermes-agent` 更真、更文件化的学习回收

一句话目标：

- 做出一个 **最像人、又最像 Unix 的主脑循环**。
