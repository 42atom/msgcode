# Goal

只吸收两条：

1. 仓库级原则
2. I/O 侧 inbox 薄单

不做 dispatch / wake / memory 文件名重构。

## Principle

- 新增的人类可读、可巡检、带生命周期的文件面，优先沿用 `issues/` 命名哲学
- 机器协议和运行时缓存保持稳定 ID + JSON
- 巡检体验优先通过只读派生视图补，不回头改协议主链

## Explicit Non-Goals

- 不把 `.msgcode/dispatch/*.json` 改成文件名状态编码
- 不把 `.msgcode/wakeups/*.json` 改成文件名状态编码
- 不把 `memory/*.md` 拆成多套 `dc/ev/fl/kn` 主链
- 不做一次性全仓文件名重构

## Why

- `issues/` 这套命名哲学适合人类任务真相面
- `dispatch / wake / subagent` 更像机器协议面，主需求是稳定读写，不是 `ls`
- 观测性可以通过 `.msgcode/STATUS` 这类只读投影补齐

## Next

- 开 inbox 薄单：
  - 原始用户请求落盘
  - 最小状态流转：`new -> triaged`
  - 不碰 dispatch / wake / memory 主链

## Final Decision

- dispatch / wake / memory 文件名重构：不做
- inbox 文件名索引模式：值得单独做

## Acceptance

1. 这份计划明确写清：dispatch/wake/memory 文件名重构不做。
2. 这份计划明确写清：inbox 文件名索引模式值得单独做。
3. 已关联一张 I/O inbox 薄任务单，而不是开大重构。
