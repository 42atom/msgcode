# PL0274：Electron 薄壳与 Bundled msgcode Runtime 计划

## 结论

如果 `msgcode appliance` 要尽快形成可见人的客户端形态，同时为未来 UI 层跨平台保留可能，当前最合理的方向是：

1. Electron 薄壳
2. bundled `msgcode` runtime
3. 本地 Web UI
4. 按主题拆开的 sidecar 站点

一句话：

- Electron 做宿主
- `msgcode` 做主脑
- Web 做界面

这条线当前只做独立规划，不进入现役开发主线。

## 为什么现在更适合 Electron

当前产品已经收口为：

- `Mac mini` appliance
- 本地私有部署
- DMG 分发
- 本地 UI 为主
- 将来可能保留 Linux / Windows 的 UI 兼容可能

在这个前提下，Electron 的价值不在“跨平台桌面应用”这个大词，而在 4 个更现实的点：

1. 让本地 Web UI 有一个稳定的桌面宿主
2. 把授权、启动和更新装进一个产品壳
3. 让未来多个 sidecar 站点有统一入口
4. 给未来 UI 层跨平台保留后路

## 为什么不走“重原生壳”

纯原生壳的优势是产品感强，但当前有两个明显弱点：

1. UI 寿命更受系统和技术栈影响
2. 不利于后续多个 Web sidecar 站点复用

而当前产品更像：

- 本地 appliance
- 多个结果面板
- 多个专题站点
- `msgcode` runtime 为核心

这更适合：

- Web-first UI
- 极薄宿主壳

## 三层结构

## 1. Electron 薄壳

Electron 只承接宿主职责：

- 首次安装引导
- appliance 授权
- 启动/停止 `msgcode`
- 本地健康检查入口
- 打开本地主 UI
- 打开 sidecar 站点
- 更新入口
- 最小托盘/菜单栏能力

Electron 不应承担：

- 任务执行
- 记忆真相
- dispatch 真相
- skill 逻辑
- 工作态逻辑

## 2. Bundled msgcode Runtime

`msgcode` 继续作为主脑存在。

它负责：

- 会话
- 调度
- skills
- tool bus
- 文件真相源
- 任务执行
- 日志和证据

原则上：

- Electron 不替代 `msgcode`
- Electron 不复制 `msgcode`
- Electron 只托管和启动 `msgcode`

## 3. 本地 Web UI

本地 Web UI 负责：

- 门厅首页
- 状态总览
- 能力包入口
- 结果可视化
- sidecar 站点

这意味着：

- 主 UI 是 Web
- 未来财税站、组织站、运营站也继续是 Web
- Electron 只是把这些本地 Web 面打包进产品

## 薄壳该做什么

## 必须做

1. 授权入口
2. appliance 身份识别
3. 启动 bundled runtime
4. 监控 runtime 是否存活
5. 打开本地主 UI
6. 打开 sidecar 站点
7. 提供升级入口

## 可以做，但应保持克制

1. 菜单栏入口
2. 托盘状态
3. 最小日志查看
4. 一键诊断

## 不该做

1. 重业务逻辑
2. 任务编排
3. memory / workstate 真相管理
4. skill 执行中心
5. 大而全管理后台

一句话：

Electron 只做宿主和门厅，不做第二主脑。

## 授权模型

当前壳的核心职责之一是授权。

建议最小授权面只包含：

- `appliance_id`
- `org_id`
- `license`
- enabled capability packs
- service term / expiry

授权应绑定：

- 这台机器
- 这个机构
- 这份能力包合同

而不应绑定：

- 任务状态
- memory 内容
- runtime 内部流程

## 分发口径

这条线默认服务：

- `DMG` 分发
- Developer ID
- notarization
- 自有更新源

不以 App Store 为目标。

这更符合当前产品形态：

- 本地私有
- 自带 runtime
- 可持续能力包增购
- appliance 交付

## 与 sidecar 站点的关系

这份计划必须和 `PL0273` 配套理解。

关系应该是：

- 主 UI 门厅由 Electron 承载
- 财税、组织、运营等专题页面由本地 Web sidecar 承接
- Electron 统一提供入口
- `msgcode` 仍负责生产结果文件

也就是：

`msgcode runtime -> 文件真相源 -> 本地 Web sidecar -> Electron 入口`

## 与跨平台的真实关系

Electron 的意义是：

- 为 UI 层跨平台保留可能

不是：

- 今天就让整个 `msgcode appliance` 已跨平台

必须明确区分：

1. UI 跨平台
2. runtime 跨平台
3. 本地自动化能力跨平台

当前能保留的是第 1 条，不应误说成三条都成立。

## 最小见人版本

如果以后要做最小可展示版本，建议目标是：

1. 一个 Electron 门厅壳
2. 内置 `msgcode` runtime
3. 一个本地主 UI 首页
4. 至少 1 个 sidecar 站点入口
5. 可见的授权状态
6. 可见的运行状态

这已经足够形成“产品感”。

## 推荐拆单顺序

如果以后真要推进，建议按下面顺序独立拆：

1. `appliance license and identity contract`
2. `bundled msgcode runtime bootstrap`
3. `electron shell host contract`
4. `local web hall entry`
5. `dmg signing notarization and update chain`

不要一开始就直接做：

- 大而全客户端
- 所有 sidecar 站点
- 重视觉系统

## 与当前主线的关系

这条线明确不进入当前主线顺序。

也就是说：

- 不影响 `pl0267`
- 不影响 `pl0259`
- 不影响 `pl0270`
- 不抢当前 appliance 主线 runtime 带宽

它当前只是：

- 一条独立产品封装线
- 为“拿出去见人”的客户端壳做准备

## 非目标

这份计划不直接决定：

- Electron 具体技术栈
- React/Svelte/Vue 的最终选择
- sidecar 站点的视觉设计
- Linux / Windows 版 runtime 的实现

## 判断口径

以后判断某个功能该不该进 Electron，可以只问三件事：

1. 这是宿主职责，还是 `msgcode` 主脑职责？
2. 这是门厅入口，还是专题站点？
3. 这是授权/启动/更新问题，还是业务逻辑问题？

如果属于业务逻辑，默认不进 Electron。
