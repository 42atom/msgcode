# PL0296：工作区人物真相源合同

## 结论

第一版只做工作区级人物簿。

不要：

- 全局人物中心
- 自动匹配推荐
- 多工作区人物复用

先只把这三样钉住：

- 人物真相源
- 待关联渠道人物
- 主界面人物数量

## 人物真相源

第一版不新起 `people.json`。

直接复用现有 skill 已定义的：

- `<workspace>/.msgcode/character-identity/<channel>-<chat-token>.csv`

CSV 继续使用既有表头：

- `channel`
- `chat_id`
- `sender_id`
- `alias`
- `role`
- `notes`
- `first_seen_at`
- `last_seen_at`

UI 读取时再映射成当前原型需要的卡片字段：

- `title` <- `alias` 或 `sender_id`
- `tags` <- `role`
- `note` <- `notes`
- `channels[0]` <- `channel + chat_id + sender_id`

也就是：

- 真相源继续是 CSV
- UI 只做只读映射

第一版不做跨 CSV 归并。

## 待关联渠道人物

建议文件：

- `<workspace>/.msgcode/people-pending.json`

只收最小字段：

1. `channel`
2. `chatId`
3. `senderId`
4. `username`
5. `displayName`
6. `seenAt`

这就够支撑：

- 待关联列表
- 选择真实人物
- 新建人物
- 忽略

并且固定：

- `channel + chatId + senderId` 是第一版唯一关联键
- `people-pending.json` 由导入侧维护，UI 只读

## 主界面人物入口

主界面只读：

- `.msgcode/character-identity/*.csv`

人数规则：

- 目录存在：统计有效记录数
- 文件不存在：不显示数字

不要把 `pending` 算进去。

## people 页面读取

左侧通讯录：

- 读并映射 `character-identity/*.csv`

右侧待关联：

- 读 `people-pending.json`

空态规则：

- 两个文件都不存在：显示空态 + 获取渠道人物入口
- 只有 `pending`：左侧空，右侧展示待关联
- 只有 `people`：右侧空，左侧展示通讯录

## 明确留后

这张单不解决：

1. 飞书成员拉取
2. 自动匹配
3. 全局人物底册
4. 跨工作区共享人物
5. 跨 CSV 合并成同一真实人物

如果以后继续拆，下一刀应该是：

- `people pending ingestion from feishu`

不是先把人物系统做重。
