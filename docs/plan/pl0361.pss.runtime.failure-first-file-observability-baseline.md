# failure-first file observability baseline

## 结论

`0361` 只保留母线，不再承载具体实现。

本轮正式拆成三张 review 单：

1. `0362`：失败证据快照边界
2. `0363`：轻量遥测账本合同
3. `0364`：强状态图腾 allowlist

顺序固定：

1. 先 `0362`
2. 再 `0363`
3. 最后 `0364`

## 为什么现有文件不能继续混着承载

- `tk0269` 只覆盖最后一次 verify 证据写回，范围不包括 API/Feishu 失败快照、轻量遥测和状态图腾。
- `tk0292/tk0294` 只回答 `status.log` 的共享近况合同，不等于遥测账本。
- `tk0204` 只触碰 heartbeat 主循环中的强状态语义，不该顺手长成状态旗帜系统。
- 若继续把三条线都挂在 `0361`，母线就会重新变成总口号。

## 目标

- 把 `0361` 收成单纯的母线与顺序说明。
- 用一份圆桌报告把边界和 guardrails 钉死。
- 把后续实现切到独立子单，避免互相偷职责。

## 非目标

- 不在本单写代码。
- 不在母线上继续细化实现字段。
- 不新增 server、dashboard、metrics 平台。
- 不改 `status.log` 现有 append / tail 合同。

## 母线保留的共享约束

### 1. file-first

- 能落文件的先落文件。
- 但只有高价值事实才允许持久化。

### 2. failure-first

- 默认先保失败证据，再谈统计。
- 不能因为想“更完整”就做全量 dump。

### 3. append-only

- 共享账本与状态近况都只能追加，不倒灌业务真相。
- UI 和模型默认不读全量账本。

### 4. strong-state default empty

- 强状态图腾默认空 allowlist。
- 只有证明“跨重启 + 不可从现有真相推导 + 人工必须直接介入”时，才允许入列。

## 实施顺序

### 1. `0362`

- 先收失败证据快照边界。
- 原因：排障价值最高，且能直接复用 `.msgcode/evidence/`。

### 2. `0363`

- 再收轻量遥测账本合同。
- 原因：账本最容易长成第二状态系统，必须在 `0362` 之后收薄。

### 3. `0364`

- 最后审强状态图腾 allowlist。
- 原因：这块最容易偷渡第二状态系统，应该默认空。

## 验收口径

1. `0361` 本身不再背实现，只保留母线、顺序和共享约束。
2. `0362/0363/0364` 的 scope / non-goals / acceptance 彼此不重叠。
3. 任何文件化 observability 都不替代 `issues/`、`status.log` 或运行时主链真相源。
