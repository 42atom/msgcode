# PL0401：workspace 身份与路径失效重定位基线

## 结论

最薄主链不是 git 跟踪，也不是目录监控。

最薄主链是：

1. route / thread 继续主用绝对路径
2. workspace 额外拥有稳定 `workspaceUid`
3. 路径失效时，按 `workspaceUid` 在 `WORKSPACE_ROOT` 下做轻量找回
4. 找不回就明确报失联

## 为什么不用 git

- git 解决的是内容版本，不是目录身份
- 非 git 目录、父目录改名、整体搬家，都不能稳靠 git 兜住
- 线程恢复问题本质是文件系统定位，不是版本控制

## 单一主链

### workspace

新增最小身份真相：

`<workspace>/.msgcode/workspace.json`

最小字段：

- `workspaceUid`
- `createdAt`
- `label`

### route

`routes.json` 继续保存：

- `workspacePath`
- `label`

并新增：

- `workspaceUid`

### thread

thread front matter 保留：

- `workspacePath`

并新增：

- `workspaceUid`

## 恢复流程

```text
读取 route
  -> workspacePath 存在：直接使用
  -> workspacePath 不存在：
       按 workspaceUid 扫 WORKSPACE_ROOT 下的 workspace.json
         -> 找到：回写新 workspacePath
         -> 找不到：标记失联并明确提示
```

## 代码接线

### `src/routes/cmd-bind.ts`

- `/bind` 时确保 workspace identity 文件存在
- 读取或创建 `workspaceUid`
- 写入 route

### `src/routes/store.ts`

- `RouteEntry` 增加 `workspaceUid`
- 去掉“路径失效直接删 active route”
- 改成“先 resolve，再决定是否失联”

### `src/runtime/thread-store.ts`

- front matter 增加 `workspaceUid`
- 解析旧 thread 时允许缺省，保持兼容

### 可选小模块

若要隔离职责，只允许新增一个很小的运行时模块：

`src/runtime/workspace-locator.ts`

只做两件事：

- `ensureWorkspaceIdentity(workspacePath)`
- `resolveWorkspacePathByUid(workspaceUid)`

## 边界

- 不做全盘搜索
- 不做 inode/FSEvents 常驻跟踪
- 不静默改绑到别的目录
- 不因为路径暂时失效就删 route

## Verify
