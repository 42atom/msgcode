# Work Vitals Session BDD Design

## 问题

`tk0205 / tk0206 / tk0207` 都属于基础设施主线。

这类单最大的风险不是“做不出来”，而是：

- 做出来了，但行为口径漂了
- 任务真相源和运行时投影混了
- session continuity 提前抢地基
- vitals 变成玄学叙事，不是可测系统语义

所以这段主线必须先做 BDD 设计，再做实现。

## BDD 目标

这组测试要锁住三件事：

1. `work continuity` 真能在 session 丢失后继续推进
2. `vitals` 真的是从事实投影出来的自保信号
3. `session continuity` 真的是建立在工作地基之上的上层收益

一句话：

- 先锁行为，再写实现。

## 作用域边界

这组 BDD 只锁 `task path`。

- `run path` 仍是一问一跑的轻路径
- 只有进入多轮推进、跨 session 续跑、派单巡检的任务，才进入这组测试范围

## 测试分层

### 1. 仓库内 BDD 锁

这是第一层，必须先做。

目标：

- 锁协议
- 锁状态边界
- 锁恢复行为
- 锁策略梯度

形态：

- `bun:test`
- 纯 TS 行为测试
- 临时 workspace + 临时 `.msgcode/` 目录
- 文件真相源 + runtime 调用混合验证

### 2. 仓库内集成恢复测试

这是第二层，也必须做。

目标：

- 模拟 session 丢失 / restart
- 从磁盘真相源重建工作面
- 验证 heartbeat / task supervisor 续跑

形态：

- TaskSupervisor + HeartbeatRunner + dispatch files
- 不依赖真实外部通道

### 3. 后续 live 验收

这是第三层，本轮不先写实现，但要在设计里预留。

目标：

- 验证真实执行臂、真实浏览器、真实 restart 路径

形态：

- 串行 smoke
- 不进入本轮仓库内测试主链

## 总原则

### 1. 锁行为，不锁文案

测试要锁：

- 状态边界
- 文件落点
- 恢复结果
- 策略选择

不要锁：

- 解释性字符串
- 长文案
- prompt 原句

### 2. 真相源优先

测试必须持续证明：

- `issues/` 和 workspace 文件是事实
- `vitals` 是投影
- `session` 是缓存 / 恢复辅助

### 3. 先锁 Phase 1，再锁 Phase 2

对于 `vitals`：

- 第一阶段只锁 `defer / degrade`
- 不先锁自动 `kill / restart`

对于 `session continuity`：

- 第一阶段只锁“从工作地基重建”
- 不先锁高级记忆召回和对话自然度

## 场景矩阵

## A. `tk0205` Work Continuity Foundation

### A1. 父任务 / 子任务 / dispatch 共同定义恢复面

Given:

- workspace 下存在父任务文档
- 存在至少一个子任务文档
- 存在对应 `dispatch/*.json`
- 存在 subagent record 或 artifact path

When:

- 当前 session 丢失
- 主脑重新启动并读取 workspace 真相源

Then:

- 系统能判断父任务未完成
- 系统能判断哪些子任务仍在进行 / blocked / 待验收
- 系统能给出下一步恢复点
- 不依赖 transcript 才能知道“还欠什么”

### A1b. 轻问题不强行进入 task path

Given:

- 用户发来的是单次问题或一次性命令
- 不要求派单、巡检、验收或跨 session 续跑

When:

- 系统在入口判断处理路径

Then:

- 请求进入 `run path`
- 不强制创建 task document
- 不强制写 dispatch / checkpoint

### A2. checkpoint 在 restart 后继续生效

Given:

- task supervisor 已为某任务写出 checkpoint
- 已写出显式恢复指针（主锚点对齐 `checkpoint.nextAction`）
- 任务当前处于 `pending` 或 `blocked`

When:

- 进程重启
- supervisor 从持久化状态恢复

Then:

- checkpoint 仍可读
- 恢复指针仍可读
- 恢复指针来源可定位：
  - 主任务来自 task runtime record 的 `checkpoint`
  - 委派任务可从 dispatch record 读到镜像 `checkpoint`
- `resume` 后续跑基于 checkpoint，而不是从空白任务重新开始
- 不需要翻 transcript 或长日志去猜“恢复后第一步该做什么”
- 恢复指针只在 durable checkpoint 写盘与状态边界时更新，不要求每个 tool event 都写盘

### A3. dispatch record 不是第二状态系统

Given:

- 子任务文件名状态与 dispatch record 不一致

When:

- 系统重建工作面

Then:

- 任务状态以文件名状态槽位为准
- dispatch record 仅作为派单与恢复辅助
- 系统记录 drift，但不让 dispatch 抢真相源
- dispatch record 只承载：
  - parent / child / client / subagent 绑定
  - 最小 checkpoint
  - artifact / verify refs
- dispatch record 不承载：
  - 文件名状态真相
  - 用户验收结论

### A4. 没有 session 也不能丢活

Given:

- transcript 缺失或 session id 已变化
- 但 `issues/`、dispatch、artifact、verify 证据仍在

When:

- 主脑恢复

Then:

- 系统仍可重建工作面
- 不会把任务误判为完成

### A5. work capsule 先于模型恢复

Given:

- 存在 task / dispatch / subagent / artifact / verify 事实
- transcript 不完整或完全缺失

When:

- runtime 组装恢复输入

Then:

- runtime 先产出最小 `work capsule`
- 模型消费的是胶囊，不是自己遍历全部文件考古
- 胶囊至少包含 taskId、phase、checkpoint、nextAction

## B. `tk0206` Vitals Projection

### B1. `load` 来自未完成与逾期，而不是主观写值

Given:

- 存在多个 overdue child tasks
- 存在 stale review 或长时间未收口父任务

When:

- 计算 vitals

Then:

- `load` 升高
- vitals 输出带理由字段
- 不需要手写 `load=7`

### B1b. `readiness / headroom / explore_window` 也必须来自物理事实

Given:

- context budget 剩余空间下降
- 近期 tool 连续失败
- 恢复指针缺失或已过旧
- 必需工具缺失或 verify 缺口未补齐
- 当前 `load` 低且存在待探索候选

When:

- 计算 vitals

Then:

- `readiness / headroom / explore_window` 的变化都能追溯到客观输入
- 不允许出现“只有解释词，没有事实来源”的信号

### B2. `stall` 来自错误风暴和资源冲突

Given:

- 同类错误连续出现
- 存在 resource contention 或工具缺失

When:

- 计算 vitals

Then:

- `stall` 升高
- heartbeat 倾向 `unblock_only` 或排障优先

### B3. `risk` 高时不许继续蛮干

Given:

- 当前任务包含高风险动作
- verify 证据缺失或危险操作即将发生

When:

- heartbeat 读取 vitals

Then:

- 策略转为 `verify_or_confirm_first`
- 不继续打开新动作链

### B4. `headroom` 低时优先整理和收尾

Given:

- 长会话未收口
- 上下文切换频繁
- 碎片任务堆积

When:

- heartbeat 读取 vitals

Then:

- 策略转为 `checkpoint_then_reflect`
- 不优先开新探索

### B5. Phase 1 不自动 kill

Given:

- `load / stall` 已高且 `headroom` 已低

When:

- 第一阶段 heartbeat 运行

Then:

- 只允许进入 `defer / degrade`
- 不自动执行 `kill / restart`
- vitals 输出里必须带 `policy` 字段，而不是只靠注释约定

### B6. vitals 文件是投影，不可人工持久化篡改

Given:

- 手工改写 `<workspace>/.msgcode/runtime/vitals.json`

When:

- 系统重新计算 vitals

Then:

- 人工写入值会被覆盖
- 最终行为仍以真实任务/日志/运行时事实计算

### B7. vitals 失败时进入保守模式

Given:

- vitals 计算过程报错或输入不完整

When:

- heartbeat 请求当前策略

Then:

- 系统进入保守模式
- 不开新坑
- 不派新子任务
- 只做 verify / checkpoint / 收尾

## C. `tk0207` Session Continuity On Work Foundation

### C1. rebuild 先读工作地基，再读 transcript

Given:

- 存在 work truth files
- transcript 也存在

When:

- session rebuild

Then:

- 系统先回答：
  - 还欠什么
  - 做到哪
  - 从哪续
  - 证据在哪
- transcript 只作为辅助连续性，不抢主导
- rebuild 的输入至少包括：
  - `issues/*.md`
  - dispatch records
  - task checkpoint
  - subagent records
  - artifact / verify 证据
- rebuild 的输出是最小 work capsule，而不是全文 transcript 回放

### C1b. 不完整 transcript 不阻塞恢复

Given:

- transcript 缺失、截断或 compaction 后被压缩
- 但 work truth files 仍完整

When:

- session rebuild

Then:

- 系统仍可恢复必要工作面
- transcript 不完整不会阻塞“继续做事”

### C2. transcript 说“完成了”但证据不齐，不算完成

Given:

- transcript 中有“已完成”类文本
- 但任务文件仍非终态
- artifact / verify 证据不足

When:

- session rebuild

Then:

- 系统仍将任务视为未完成
- 进入补证据或继续执行路径

### C3. compaction 不能替代工作真相源

Given:

- session 发生 compaction
- memory flush 已写出简要记忆

When:

- 主脑恢复工作

Then:

- 仍优先读取 `issues/`、dispatch、artifact、verify 证据
- memory 只辅助，不替代 task truth

### C4. session continuity 只是上层收益

Given:

- work continuity 已成立

When:

- transcript / memory / rebuild 都正常

Then:

- 系统在不丢活的前提下获得更自然的对话续接
- 而不是把自然续接当成恢复依据

## 推荐测试文件挂点

建议先起三组仓库内测试锁：

- `test/*work-continuity-foundation*.test.ts`
- `test/*vitals-self-protection*.test.ts`
- `test/*session-continuity-foundation*.test.ts`

命名最终可按仓库当前 `p5-7-r*` 规则落。

## 推荐测试夹具

### 1. 临时 workspace 夹具

至少能快速生成：

- `issues/*.md`
- `.msgcode/dispatch/*.json`
- `.msgcode/schedules/*.json`
- `.msgcode/subagents/*.json`
- artifact 假文件

### 2. restart 夹具

通过：

- 先写盘
- 再销毁 in-memory 对象
- 再重新 `createTaskSupervisor()` / `HeartbeatRunner`

来模拟 restart

### 3. vitals 夹具

只构造最少事实输入：

- overdue
- retry streak
- resource contention
- verify gap

避免把 vitals 测试写成大而全集成。

## 不该先测什么

- 不先测 fancy memory 检索
- 不先测复杂人格切换
- 不先测真实外部通道 delivery
- 不先测大型多 workspace 并发矩阵

这些都不是第一阶段地基。

## 通过标准

这组 BDD 设计如果落得对，后续实现应满足：

1. 没有 session，也不丢活
2. 有 session，只会更顺，不会更假
3. `vitals` 真能影响 heartbeat 策略
4. `vitals` 不会变成第二真相源
5. 背压与自保是 runtime policy，不是文档状态

## 一句话收口

这组测试的真正目的，不是证明系统“很聪明”。

而是证明：

- **它先是一个不会丢活的工作系统**
- **然后才是一个会自然续聊的会话系统**
