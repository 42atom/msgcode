# PL0290：WPKG 可安装能力包基线

## 结论

第一版 `wpkg` 不需要是复杂新格式。

先直接收成：

- 一个带 `manifest.json` 的 `tar.zst` 包

它的职责也只做三件事：

1. 给主界面增加 pack / site 入口
2. 给 agent 增加 skill 发现面
3. 给未来授权与商业留下稳定字段

一句话：

- 先做可安装能力包
- 不先做市场

## 物理形态

第一版 `wpkg` 默认就是 `tar.zst` 归档，只是后缀和语义是：

- `*.wpkg`

解压后最小目录结构建议为：

```text
company-finance.wpkg
├── manifest.json
├── web/
├── skills/
└── assets/
```

当前不强制要求：

1. 自定义二进制格式
2. postinstall 脚本
3. 远程依赖下载
4. 本地常驻服务

## 为什么从 zip 改到 tar.zst

这次不再把 `wpkg` 定义成 zip，原因是：

1. `wpkg` 本质上是目录树，不是面向随手双击解压的通用附件。
2. `tar.zst` 更适合这种 `manifest / web / skills / assets` 的内容包。
3. 切换只影响包壳，不影响主链：
   - `manifest`
   - 安装后注册表
   - 主界面消费
   - agent 发现

所以这是一次物理格式收紧，不是能力面扩张。

## Manifest 最小字段

第一版只定义这些字段：

1. `id`
2. `name`
3. `version`
4. `author`
5. `sites`
6. `skills`
7. `requires`
8. `commercial`
9. `licenseType`

其中：

- `commercial`
  - 只是给未来商业留位
- `licenseType`
  - 只是给未来授权留位

当前不进入任何支付实现。

## 三条硬边界

`wpkg` 后续实现应继续遵守三条硬边界。

### 1. 死板 manifest 准入

包能提供什么、需要什么，必须先写在 `manifest.json` 里。

主脑只认显式声明，不猜目录，不靠运行时试探，不给未声明能力偷偷开口子。

一句话：

- 没写进 manifest，就等于没有。

### 2. 运行隔离

后续如果 pack 需要执行能力，不应直接进主进程内存空间。

正确方向是：

- pack 保持隔离
- 崩溃不拖垮主脑
- 主脑只记录日志和结果

这里借的是 Chrome 扩展的隔离纪律，不是去做完整插件平台。

### 3. 窄消息面

pack 和主脑之间应优先通过很窄的消息面协作：

- 文件真相源
- 注册表
- stdio JSON
- 其他等价的薄协议

不应让 `wpkg` 长成任意宿主 API 调用面。

一句话：

- pack 交换声明、结果和消息，不直接拿宿主执行权。

## 哪些内容允许进包

第一版允许进包的内容，只收这四类：

1. `manifest.json`
2. `web/`
3. `skills/`
4. `assets/`

换句话说，当前 `wpkg` 允许装的是：

- sidecar 页面
- 简单 SaaS 前端
- skill 说明书
- 静态资源与模板

## 哪些核心能力不该直接进包

第一版不应直接进包的核心能力：

1. 任意执行脚本
2. 任意后台常驻进程
3. 直接进入主进程的运行时代码
4. 任意宿主 API
5. 需要宿主偷偷补依赖的运行时
6. 若必须保护知识产权的核心算法/规则/授权判断，不应只靠明文 pack 交付

这条边界的意思不是“永远不做”，而是：

- 第一版 `wpkg` 先做能力包
- 不先做插件执行平台

## 安装后真相源

安装 `wpkg` 后，不直接让主界面扫包目录。

统一写两份注册表：

1. `<workspace>/.msgcode/packs.json`
2. `<workspace>/.msgcode/sites.json`

### packs.json

主界面用来展示：

- 内置包
- 用户安装包
- 包版本
- 包作者
- 是否启用

### sites.json

主界面用来展示：

- 可打开的基座 / sidecar 站点入口

这意味着：

- 主界面只读注册表
- 不直接解析每个包的内部目录

## Agent 发现口径

第一版也不要让 agent 直接扫包目录。

安装时应把 skill 发现面显式写入注册表或索引：

1. pack 内的 skill 路径
2. pack 的说明信息
3. pack 的能力范围

主链应是：

`wpkg -> 解压 -> 注册 packs/sites/skills -> 主界面 / agent 读取`

而不是：

`主界面或 agent 运行时自己去猜目录`

## 与主界面的关系

主界面只做三件事：

1. 显示已安装 packs
2. 显示 packs 带来的站点入口
3. 显示 pack 的基本状态

主界面不负责：

1. 承载包的业务建模
2. 执行包内 skill
3. 解释包内部资源结构

## 与基座业务的关系

`wpkg` 提供的是业务能力单元，不是 UI 主体。

所以必须坚持：

- 包的业务设计与主界面解耦
- 主界面只做安装、入口和可见性
- 业务内容仍在 pack 自带的 Web / skill / assets 里

## 第一版明确不支持

1. 包市场
2. 支付
3. 分成
4. 云端包下载
5. 任意安装脚本
6. 任意二进制注入
7. 插件直接 `require()/import()` 进入主进程
8. 任意宿主 API 暴露
