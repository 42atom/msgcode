# RP0285：App Shell UI Surface Matrix v1

## 结论

v1 直接选：

- Electron 薄壳
- 本地 Web 门厅
- sidecar 站点

不选：

- 重原生壳
- 纯浏览器交付

## 对比矩阵

| 方案 | 优点 | 致命问题 | 结论 |
| --- | --- | --- | --- |
| 纯浏览器入口 | 最薄，开发快 | 缺产品壳，授权/启动/升级分散，见人感弱 | 不作为正式 appliance 主交付 |
| 重原生壳 | 质感强 | UI 寿命差，Web sidecar 复用差，未来跨平台差 | 不适合当前路线 |
| Electron 薄壳 | 最适合装本地 Web、启动 runtime、统一入口 | 体积更重 | 当前最合适 |

## 为什么不是“继续只有 CLI”

CLI 适合开发，不适合交付。

appliance 需要一个可见入口，至少承接：

1. 机构身份
2. 授权状态
3. runtime 状态
4. 站点入口
5. 诊断入口

没有这层，产品永远像“装好的一套脚本”。

## 为什么不是“先做大后台”

因为那会立刻犯三个错：

1. 主 UI 变胖
2. sidecar 站点失去意义
3. 壳开始吞 runtime 职责

第一版必须做减法，但不再等于“只有门厅”。

## v1 UI 冻结范围

### Main UI 必须有

1. 场景工作区与线程
2. 当前线程聊天
3. 工作状况
4. 日历 / 定时任务
5. 基座与邻居的次级入口

并补三条约束：

1. Main UI 只面向桌面窗口，不做移动端或纵向堆叠响应式
2. 左侧以“工作区 -> 线程”树为主，不发明额外 folder 分类
3. 右侧内容必须随当前线程切换
4. 邻居必须有全局未读提示
5. 输入框只能做哑终端，不得自建消息真相源

### Settings 必须单独存在

1. 我的资料
2. 通用
3. 智能体能力配置
4. 诊断

### 不做

1. 机构卡片当首页主角
2. “项目/session/thread/channel” 混用
3. 大经营后台
4. 高敏个人画像
5. 把 People 混进 Settings
6. 由前端自己先画聊天气泡

## 人物与工作区边界

第一版不做全局人物中心。

People 应属于：

1. 当前场景工作区
2. 作为独立页面存在
3. 只维护统一称谓、tags、备注和渠道身份

## 与 sidecar 站点的冻结关系

Main UI 只给入口，不吞主题站点业务。

专题数据继续去：

1. 财税站
2. 组织站
3. 运营站
4. 其他场景站

这条边界如果不先钉死，后面 UI 一定会重新长成巨型后台。
