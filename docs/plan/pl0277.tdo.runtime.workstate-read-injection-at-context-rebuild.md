# PL0277：WORKSTATE 读回接线

## 结论

下一步不做 graph/router。

先做一条更薄、更直接的主链：

- `taskId`
- `WORKSTATE` 文件
- `context-policy`
- 模型上下文

## 为什么现在做这一步

`0276` 的三份样本已经证明：

- summary 更擅长保留“发生了什么”
- `WORKSTATE` 更擅长保留“现在该怎么接”

那下一步最高价值动作就是：

- 让已有 task 续跑主链，在恢复时先看到 `WORKSTATE`

## 最小实现

### 1. 读取边界

只在这类链路读：

- `source = task`
- `source = heartbeat`

并且必须同时满足：

- `workspacePath` 存在
- `taskId` 存在

message 链不读。

### 2. 文件定位

固定读取：

- `<workspace>/.msgcode/workstates/<taskId>.md`

找不到文件就跳过，不报错，不 fallback。

### 3. 注入顺序

模型上下文顺序固定为：

1. `WORKSTATE`
2. `summary`
3. `window`
4. 当前 prompt

原因：

- `WORKSTATE` 是恢复骨架
- `summary` 是事实压缩面

### 4. 职责边界

`WORKSTATE` 只补“恢复姿态”，不能覆盖：

- issue 真相
- checkpoint 真相
- dispatch 真相

系统也不理解其语义，不做裁决，只负责原样读回。

## 非目标

不做：

1. message 链自动猜测 taskId
2. `WORKSTATE` 自动写盘
3. graph/router
4. 新 manager / selector / policy 层

## 验收口径

1. `assembleAgentContext()` 能在有 `taskId` 时读回 `WORKSTATE`
2. dialog/tool-loop 两条入口都能看到它
3. 回归锁覆盖：
   - task 链可注入
   - message 链不误注入
   - 顺序在 summary 前
