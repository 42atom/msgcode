# PL0258：MSA 文档级上下文编码收编计划

## 结论

MSA 对 msgcode 的真正启发，不是去实现模型层 `document-wise RoPE`，而是把它翻译成我们自己的
“document-wise context assembly”：

- 每份文件保留自己的局部坐标
- 当前轮上下文保留独立的全局顺序
- 检索、拼装、生成三段分离
- 不把多份文件压成一条匿名长文本

这和 msgcode 当前“Everything is a file”的方向一致，但我们现在还没有把这套结构表达得足够彻底。

## MSA 官方可确认的设计原语

以下内容来自 MSA 官方 GitHub README，可公开确认：

1. `document-wise RoPE`
   - 每个文档的位置从 0 重置，避免 train-short / infer-long 的位置漂移。

2. `global RoPE (active context)`
   - 当前激活上下文仍保留全局顺序，用来表达 `background -> query -> generation`。

3. `three-stage inference`
   - offline global memory encoding
   - online routing & context assembly
   - sparse generation

4. `Top-k document selection`
   - 不是把全部记忆直接灌给模型，而是先路由，再拼装当前需要的稀疏上下文。

5. `document decoupling`
   - 多文档不是一条天然连续文本；文档边界应被结构本身表达出来，而不是只靠分隔符。

## 对 msgcode 的映射

### 我们已经像的地方

1. 真相源本来就是分文件的
   - `SOUL.md`
   - `memory/*.md`
   - `issues/*.md`
   - `.msgcode/sessions/*/summary.md`
   - 附件与报告文件

2. 当前主链已经不是单纯聊天窗口
   - `src/runtime/context-policy.ts` 负责 `summary / window / checkpoint / soul`
   - `src/listener.ts` 负责 `attachment / derived evidence / memory inject`

3. 已经有“先检索/整理，再回答”的雏形
   - memory search
   - attachment derived evidence
   - artifact roster

### 我们还不像的地方

1. 文档内部坐标不够稳定
   - `memory` 注入当前主要是 `[记忆] path:start-end + snippet`
   - 其他块如 `summary`、`attachment`、`artifact roster` 仍偏文本拼贴，不是统一文档块协议

2. 全局顺序表达不够显式
   - 现在是 `summary / window / prompt / attachment / memory` 多路拼接
   - 但没有一套稳定的“背景块 -> 当前请求 -> 输出目标”排序合同

3. 文档边界表达不一致
   - `SOUL`、`summary`、`memory`、`attachment` 各自格式不同
   - 模型收到的是若干异构文本块，不是统一的“文件级上下文块”

4. 检索和拼装还没有明确分层
   - 当前 `listener.ts` 同时做附件整理、记忆注入、内容拼接
   - `context-policy.ts` 只覆盖 summary/window/checkpoint/soul，尚未统一 memory/attachment

## 收编原则

### 原则 A：把 MSA 翻译成文件级协议，而不是模型级机制

我们不实现 RoPE；我们实现它在 runtime 层的等价物：

- `document-wise local anchors`
- `global assembly order`
- `top-k context blocks`

### 原则 B：局部顺序与全局顺序分开

每个文件块都保留自己的局部坐标：

- 文件路径
- 文档类型
- 文档日期/版本
- 段落或行号范围
- chunk 序号

当前轮再单独表达全局顺序：

- background
- memory
- session
- current request
- execution target

### 原则 C：不新增大平台，只收口现有注入主链

不做新的 memory service / context platform / orchestrator。
优先在现有两条主链就地收口：

- `src/runtime/context-policy.ts`
- `src/listener.ts`

## 目标产物

我们建议为 msgcode 定义一套最小“文档级上下文块”合同。

示意字段如下：

- `doc_kind`
  - soul | memory | summary | window | issue | attachment | artifact | checkpoint
- `doc_id`
  - 稳定标识，不依赖本轮拼装位置
- `source_path`
  - 真相源路径
- `local_anchor`
  - 行号范围、段落号、chunk 序号等
- `doc_time`
  - 文档自身时间
- `global_slot`
  - 在当前轮上下文中的角色：background / current / target / derived
- `trust`
  - truth / derived / cache
- `content`
  - 实际注入文本

重点是：局部坐标和全局槽位同时存在，但不混在一起。

## 分阶段计划

### Phase 1：定义上下文块协议

目标：

- 产出一份 `docs/protocol/CONTEXT_BLOCKS.md` 或等价文档
- 明确 `doc_kind / local_anchor / global_slot / trust` 四个核心概念

范围：

- 统一术语
- 统一块头格式
- 定义最小排序规则

非范围：

- 不改检索算法
- 不改 memory index 存储

验收：

- 仓库内有一份明确协议，可用于后续实现和 review

### Phase 2：收口注入面为统一块协议

目标：

- 让 `SOUL / summary / window / memory / attachment` 输出统一块头

优先修改：

- `src/runtime/context-policy.ts`
- `src/listener.ts`

做法：

- 先统一格式，不改召回逻辑
- 把现有异构文本拼接改为“文件块列表”

非范围：

- 不新增独立 manager

验收：

- 同一轮 prompt 中，主要上下文来源都能看到一致的块结构

### Phase 3：把 memory / attachment 纳入统一排序主链

目标：

- 不再让 `memory` 和 `attachment` 在 `listener.ts` 里各自临时拼文本
- 交给统一的上下文组装顺序处理

建议顺序：

1. soul
2. checkpoint
3. summary
4. recent window
5. retrieved memory blocks
6. attachment / derived blocks
7. current request

非范围：

- 不做自动多跳检索编排

验收：

- 全局顺序固定，日志和调试输出可观测

### Phase 4：让检索返回“块”，不是“裸 snippet”

目标：

- `memory search` 与附件派生结果返回统一上下文块元数据

优先修改：

- `src/memory/store.ts`
- `src/listener.ts`
- 必要时 `docs/protocol/MEMORY.md`

做法：

- 保留现有索引
- 只增加返回结构和格式化协议

验收：

- 记忆注入不再只是 `[记忆] path:start-end`
- 能明确区分文档身份、局部锚点、全局槽位

### Phase 5：补观测与 smoke

目标：

- 让我们能看见“本轮到底拼了哪些文档块”

建议补充：

- context roster 日志
- 每轮注入块数量与字符数
- 不同 `doc_kind` 的占比

验收：

- 日志能回答：
  - 这轮用了哪些文档
  - 每个文档块在什么全局槽位
  - 哪些是 truth，哪些是 derived

## 最小可删版本

如果只做最小一刀，我建议只做：

1. 定义统一上下文块协议
2. 收口 `SOUL / summary / window / memory / attachment` 的块头格式
3. 固定全局排序

先不要做：

- 新检索器
- 新索引
- 多轮 interleave
- 稀疏调度平台

## 风险

1. 容易把“文件级协议”做成新的平台层
   - 规避：只允许 helper / formatter / assembler，禁止新 manager

2. 容易把 MSA 的模型术语硬搬到 runtime
   - 规避：只借鉴结构，不假装实现 RoPE

3. 容易让 prompt 变长
   - 规避：统一块协议后，仍保留 top-k 和 budget 裁剪

## 建议的后续任务拆分

1. `context-blocks protocol`
2. `listener attachment/memory context block unification`
3. `runtime context-policy global slot ordering`
4. `memory search result block metadata`
5. `context roster logging and smoke`
