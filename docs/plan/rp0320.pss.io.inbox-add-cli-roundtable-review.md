# RP0320：inbox add CLI 圆桌审查

Decision:
- Recommendation: go
- Confidence: medium
- Scope: 只接受 inbox add 作为原始输入投递面，不顺手补消费器或 Web transport

Why:
- 现有 `.msgcode/inbox/` 已经是仓库里的 file-first 真相源，继续复用最省。
- `--chat-id` 必填，避免 CLI 偷猜当前线程。
- 只落 `new` 文件，保持边界诚实，不假装全链路已打通。

Risks:
1) 主界面若直接调用它，用户发完消息后暂时看不到后续响应。
2) 后续若消费器落地不克制，容易再长出第二套输入编排层。
3) 若 UI 自己先画气泡，会立刻破坏单一真相源。

Guardrails:
- 前置条件: UI 只能把它当投递面，不能当聊天真相面。
- 监控指标: inbox 文件是否稳定落盘；help-docs 合同是否仍可见。
- 回滚条件: 若后续被迫为它补大量 UI 状态和后台轮询，则停下重审。

Next Actions:
1) 保持 `msgcode inbox add` 只落文件。
2) 下一刀若继续，只补“谁消费 inbox”的最小主链。
3) UI 一律被动渲染 runtime 回写，不做 optimistic bubble。

Evidence:
- Docs: tk0234, tk0319, rp0319
- Code: `src/cli/inbox.ts`, `src/runtime/inbox-store.ts`
- Tests: `bun test test/p7-tk0319-inbox-add-cli.test.ts`
