# PL0295：wpkg 注册表与主界面 packs 可见面

## 结论

`wpkg` 进入实施前，先不要碰安装器。

先只把这条主链钉死：

- `wpkg`
- 解压到固定目录
- 写 `packs.json`
- `appliance` 读取 `packs`
- 主界面显示“默认内置 / 用户安装”

一句话：

- 先让 pack 被看见
- 再让 pack 被安装

## 当前缺口

`WPKG.md` 已经定义：

- `packs.json`
- `sites.json`

但当前运行时只落了半条：

- `appliance sites`

还没落：

- `appliance packs`
- `hall.packs`

所以主界面虽然已经有“已安装能力包”的设计，但还没有真正文件合同。

## 安装目录约定

第一版先固定：

- `<workspace>/.msgcode/packs/<pack-id>/`

里面放解压后的包内容。

为什么放这里：

- 仍属于工作区
- 不需要全局包中心
- 主界面和 agent 都能在当前工作区内找到注册结果

不做：

- 全局 pack 共享目录
- 独立数据库
- 远程 registry

## packs.json 最小字段

第一版每个 pack 只收这些：

1. `id`
2. `name`
3. `version`
4. `author`
5. `enabled`
6. `installType`
7. `sourcePath`

其中：

- `installType`
  - `builtin` 或 `user`

这一个字段就足够支撑 UI 分组，不先加更多状态。

## appliance packs / hall 读取

第一版主界面不直接扫 pack 目录。

只读：

- `<workspace>/.msgcode/packs.json`

`appliance` 侧只需要两种最小消费：

1. `appliance hall`
   - 返回 `packs`
2. 可选 `appliance packs`
   - 单独输出 pack 列表 JSON

如果只保留一个入口，也优先先把 `hall.packs` 补实，不急着多开命令。

## UI 分组口径

主界面只做两组：

1. 默认内置
2. 用户安装

分组规则只看：

- `installType`

不要再引入：

- 来源平台
- 付费状态
- 作者组织树

这些都留后。

## 明确留后

这张单明确不解决：

1. pack 安装器
2. pack 升级
3. pack 卸载
4. agent skill 发现
5. 商业授权

真正的下一刀，如果继续拆，应是：

- `wpkg skill registry and agent discovery`

而不是现在就把所有事情揉进 `0295`。
