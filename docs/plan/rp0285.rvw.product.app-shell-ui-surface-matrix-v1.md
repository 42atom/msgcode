# RP0285：App Shell UI Surface Matrix v1

## 结论

v1 直接选：

- Electron 薄壳
- 本地 Web 门厅
- sidecar 站点

不选：

- 重原生壳
- 纯浏览器交付

## 对比矩阵

| 方案 | 优点 | 致命问题 | 结论 |
| --- | --- | --- | --- |
| 纯浏览器入口 | 最薄，开发快 | 缺产品壳，授权/启动/升级分散，见人感弱 | 不作为正式 appliance 主交付 |
| 重原生壳 | 质感强 | UI 寿命差，Web sidecar 复用差，未来跨平台差 | 不适合当前路线 |
| Electron 薄壳 | 最适合装本地 Web、启动 runtime、统一入口 | 体积更重 | 当前最合适 |

## 为什么不是“继续只有 CLI”

CLI 适合开发，不适合交付。

appliance 需要一个可见入口，至少承接：

1. 机构身份
2. 授权状态
3. runtime 状态
4. 站点入口
5. 诊断入口

没有这层，产品永远像“装好的一套脚本”。

## 为什么不是“先做大后台”

因为那会立刻犯三个错：

1. 主 UI 变胖
2. sidecar 站点失去意义
3. 壳开始吞 runtime 职责

第一版必须把 UI 收成门厅。

## v1 主门厅冻结范围

### 必须有

1. 机构卡片
2. runtime 状态
3. pack 入口
4. sidecar 站点入口
5. 最小诊断入口

### 可以有，但要克制

1. 最近提醒
2. 最近一条风险提示
3. 当前版本与更新提示

### 不做

1. 大聊天工作台
2. 大任务面板
3. 经营数据总后台
4. 高敏个人画像

## 机构卡片冻结范围

第一版仍然只保留 3 个字段：

1. 名称
2. 交税地
3. 统一社会信用代码

不扩字段，不做复杂机构模型。

## 与 sidecar 站点的冻结关系

主门厅只给入口，不吞主题站点。

专题数据继续去：

1. 财税站
2. 组织站
3. 运营站
4. 其他场景站

这条边界如果不先钉死，后面 UI 一定会重新长成巨型后台。

## 推荐下一步

1. 用 `0263` 把这条方向正式推到 `doi`
2. 后续如果真要实现，再单开：
   - Electron host contract
   - local hall IA
   - sidecar site contract
