# PL0400：Message timeout background task promotion

## 结论

这刀做的是后台化，不是默认子代理。

最薄主链：

1. `message` 前台最多 180 秒
2. 超时后 tool-loop 返回 `continuable`
3. listener 把当前请求落成 runtime task
4. 飞书立刻回执“已转后台继续”
5. heartbeat 立即补踢一轮
6. task 完成后主动回原 chat

## 为什么不用默认 subagent

- 事故根因是前台 message run 生命周期太长，不是缺少第二个代理人格
- `task` 已经是现成后台主链：有持久化、heartbeat、终态通知
- `subagent` 是另一条 tmux 执行臂，适合明确委派，不适合作为所有长任务的默认落点

换句话说：

- `task` 解决“别堵前台”
- `subagent` 解决“要不要换执行臂”

先解前者，才是奥卡姆最短路径。

## 结构

### message

- 负责即时对话
- 负责快速回执
- 不再背长生命周期工具回合

### task

- 负责后台长活
- 负责持久化、heartbeat 续跑、终态回帖
- 默认仍由主脑执行，不自动切 tmux

### subagent

- 负责明确委派给 `codex|claude-code`
- 走 tmux 执行臂
- 状态落到 `.msgcode/subagents/*.json`
- 不是本次事故修复的默认路径

## 任务号

用户面新增三位任务号：

- `001`
- `002`
- `003`

内部 UUID 继续保留，不暴露给日常聊天。

命令口径：

- `/task status`
- `/task status 7`
- `/task cancel 7`
- `/task resume 7`

其中 `status` 在无活跃任务时回最近任务，解决“忘了任务还活不活着”的手机输入问题。
