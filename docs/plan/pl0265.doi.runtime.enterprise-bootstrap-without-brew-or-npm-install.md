# PL0265：Appliance Bootstrap Without Brew Or Npm Install

## 结论

`0265` 不该继续讨论“怎么优化 bootstrap 脚本”，而该直接回答：

1. 怎么组装 Core Appliance 包
2. 怎么安装到目标机器
3. 怎么首次生成本地真相源
4. 怎么升级而不回退到 `brew` / `npm install`

一句话：

- 开发者 bootstrap 继续留在仓库
- appliance bootstrap 另起正式主链

## 当前问题

现有 bootstrap 仍然服务开发者环境：

1. `bootstrap/bootstrap.sh`
   - 先找 Homebrew
   - 再跑 `brew bundle`

2. `bootstrap/bootstrap-agent-pack.sh`
   - 继续走 Homebrew 安装增强工具

3. `bin/msgcode`
   - 仍依赖 `tsx` 与源码目录

所以现在的问题不是“脚本写得不够好”，而是：

- 主入口就不是交付入口

## 目标

`0265` 的目标是定义一条新的正式安装主链：

1. Package Assembly
2. Install
3. First-Run Init
4. Upgrade

## 非目标

1. 不直接决定 DMG/PKG 二选一
2. 不实现 Electron 壳
3. 不实现 pack 下载器
4. 不动当前开发者 bootstrap

## Phase 1：Package Assembly

要产出一份 Core Appliance 包，最少包含：

1. 已构建的 `msgcode` 入口
2. Node runtime
3. 核心 JS 与依赖
4. memory 核心依赖
5. `SOUL/ORG` 模板
6. 最小诊断入口

这一层的验收是：

- 包体已经能替代 `bin/msgcode + npm install + tsx`

## Phase 2：Install

安装过程只做三件事：

1. 把 Core Appliance 放到固定安装目录
2. 写入最小启动入口
3. 预置全局配置根骨架

安装过程不应做：

1. 现场联网下载
2. 现场跑 `brew`
3. 现场跑 `npm install`
4. 现场编译源码

## Phase 3：First-Run Init

首次启动只生成最少真相源：

1. `~/.config/msgcode/.env`
2. `~/.config/msgcode/souls/default/SOUL.md`
3. `~/.config/msgcode/souls/active.json`
4. `<workspace>/.msgcode/SOUL.md`
5. `<workspace>/.msgcode/ORG.md`
6. `<workspace>/memory/`
7. `<workspace>/AIDOCS/reports/`

`ORG.md` 第一版只允许：

1. 名称
2. 交税地
3. 统一社会信用代码

## Phase 4：Upgrade

升级只允许替换：

1. Core Appliance 运行物
2. 默认模板资源
3. 可升级的 pack

升级不应覆盖：

1. `ORG.md`
2. `SOUL.md`
3. `memory/*.md`
4. `issues/`
5. `AIDOCS/`

## 建议的安装包结构

建议最小结构：

```text
msgcode-appliance/
├── runtime/
│   ├── bin/msgcode
│   ├── node/
│   ├── app/
│   └── deps/
├── templates/
│   ├── SOUL.default.md
│   └── ORG.default.md
├── install/
│   ├── install.sh
│   ├── first-run-init.sh
│   └── doctor.sh
└── packs/
    └── (optional, empty by default)
```

## 与现有 bootstrap 的关系

保留：

1. `bootstrap/bootstrap.sh`
2. `bootstrap/bootstrap-agent-pack.sh`
3. `bootstrap/Brewfile*`

但它们降级为：

- 开发者 bootstrap

不再作为：

- appliance 正式安装主链

## 验收口径

`0265` 完成时，至少要能回答：

1. appliance 包的最小目录结构是什么
2. 安装过程是否完全不需要 `brew` / `npm install`
3. 首次启动会生成哪些文件
4. 升级时哪些文件绝不能被覆盖

## 推荐下一刀

`0265` 的下一步不是写打包脚本，而是先出一份：

- `bootstrap route v1`

也就是把现有开发者 bootstrap 与 appliance bootstrap 的差异逐项列出来。
