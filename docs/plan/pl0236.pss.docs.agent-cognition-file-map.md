# Problem

msgcode 已经有不少和 Agent 认知相关的文件真相源，但它们分散在不同层级：

- 根 README 有一部分
- protocol 文档各说一段
- issues / plan 里又各补一段

结果是边界虽然存在，整体感知仍不够直观。缺的不是新控制层，而是一张统一的文件结构表，而且最好就放在 README。

## Occam Check

- 不加新层，能不能解决？
  - 能。先把现有文件面盘清楚，形成一张总表。
- 这张表是不是在替主链做决定？
  - 不是。它只是把已经存在的主链摊开给人和模型看。
- 先补文档而不是先补文件，是否更接近单一主链？
  - 是。先证明缺口，再决定要不要补结构。

## Decision

采用“先文档化、后补缺口”的最小方案：

1. 盘点 Agent 认知相关文件面
2. 先在 README 输出统一结构表，但只放面向人类的文件面
3. 区分：
   - 人类巡检面
   - 机器协议面
   - 正式真相源
   - 派生视图 / 缓存 / 证据
4. 若发现缺口，只提出最小补充建议

## Candidate Coverage

结构表至少覆盖：

- `~/.config/msgcode/souls/*`
- `<workspace>/.msgcode/SOUL.md`
- `<workspace>/memory/*.md`
- `<workspace>/.msgcode/sessions/*`
- `<workspace>/.msgcode/inbox/*`
- `<workspace>/.msgcode/dispatch/*`
- `<workspace>/.msgcode/reflection/*`
- `issues/*`
- `AIDOCS/*`

## Output Shape

建议输出至少包含一张表，字段可为：

- 文件面
- 典型路径
- 作用
- 真相级别
- 生命周期
- 谁读
- 谁写
- 是否面向人类巡检

## Plan

1. 读取现有 README / protocol / issue 主线
2. 汇总现有 Agent 认知文件面
3. 判断哪些是“认识你”的认知面，哪些只是运行时执行面
4. 在 README 产出统一结构表
5. 补一句总判断：
   - Agent 到底靠哪些文件认识用户、记住工作、延续任务
6. 仅在必要时补最小后续建议

## Risks

- 风险：把执行文件和认知文件混成一锅
  - 回滚：强制区分“认知面 / 执行面 / 证据面”
- 风险：为了画表而偷开新层
  - 回滚：只记现状和最小缺口，不先发明新目录

## Acceptance

1. README 里存在一张统一结构表。
2. README 这张表不混入 `.json/.jsonl/.ndjson` 机器协议文件。
3. 能回答“Agent 怎么认识你”这个问题。
4. 能回答“哪些文件是正式真相源，哪些只是运行时派生物”。
5. 后续 README / 架构说明 / 对外文章都可以引用这张表，而不用再东拼西凑。
