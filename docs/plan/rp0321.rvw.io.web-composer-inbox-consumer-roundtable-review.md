# RP0321：Web composer inbox 消费圆桌审查

Decision:
- Recommendation: hold
- Confidence: medium
- Scope: 先停在设计盘点，不直接进实现

Why:
- `msgcode inbox add` 已经把原始输入面做薄了，下一刀会直接碰 `listener / routes / runtime.current_transport`。
- 现有 listener 主链带明确的 Feishu 假设，直接硬接很容易把“复用主链”做成“污染主链”。
- 新起 Web 消费器或 daemon 会立刻违反当前原则。

Risks:
1) 为 UI 体验着急，临时长出第二消息主链。
2) 为了让 Web 看起来能聊，开始偷偷扩 transport 和 route 语义。
3) UI 先画气泡，再倒逼 runtime 跟上，形成第二真相源。

Guardrails:
- 前置条件: 先盘点 listener 真正挡路点，再决定是否动代码。
- 监控指标: 任何方案都必须继续以文件为唯一输入真相。
- 回滚条件: 一旦方案需要 daemon / queue / optimistic bubble，立即停。

Next Actions:
1. 保持 `0321` 先做盘点，不直接实现。
2. 若后续继续，只能尝试复用 listener 主链。
3. 明确 UI 继续是 dumb terminal，不管理聊天真相。

Evidence:
- Docs: tk0319, tk0321, tk0234
- Code: `src/runtime/inbox-store.ts`, `src/listener.ts`, `src/routes/store.ts`
- Tests: `bun test test/p7-tk0319-inbox-add-cli.test.ts`
