# RP0330：people 写边界圆桌审阅

Decision:
- Recommendation: go
- Confidence: high
- Scope: 只推进 `alias/notes` 写入与 pending 过滤；不推进 tags 和全局人物中心

Why:
- 现有 `character-identity` CSV 已经给出稳定键和稳定文件落点，继续沿用最薄
- 原型里的“统一称谓 / 备注”能直接映射到 `alias / notes`
- `tags` 当前没有真相源，硬上只会逼出 `people.json` 或第二套 CRM

Risks:
1. 继续用不完整 pending 键，会写不准目标 CSV
2. 若让 UI 直接删 pending，会长出第二控制面
3. 若把 `role` 当多 tag 用，迟早会污染现有 skill 语义

Guardrails:
- 前置条件: 唯一键统一成 `channel + chatId + senderId`
- 前置条件: 第一版写入口只允许 `alias / notes`
- 监控指标: 不出现 `people.json`、不出现 UI 直写 CSV、pending 已认领项可自动隐藏
- 回滚条件: 若实现逼迫 runtime 新起人物中心，立即退回 CSV + pending + save 单命令

Next Actions:
1. 先补 `people-pending` 的键合同
2. 再实现 `appliance people save`
3. 最后在读面补“已认领 pending 隐藏”
