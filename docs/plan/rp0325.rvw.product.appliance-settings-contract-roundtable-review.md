# RP0325：settings 合同圆桌复核

Decision:
- Recommendation: pivot
- Confidence: medium
- Scope: 保留 0322 和 0323；0324 并回 0323；doctor 不进入首屏聚合

Why:
- `SystemsArchitect`：`0323` 可以保留，但只能是进程内薄聚合；`0322` 只准 profile 可写；`0324` 单拆会让同一条读链再长一层。
- `SkepticalOperator`：真正的风险不是缺功能，而是三张单一起长成 settings 子系统。尤其聚合层不能补默认值、不能加缓存、不能掩盖文件级真相。
- `UserAdvocate`：用户真正关心的是 warning 别被吞、设置页别卡死、只读项别假装可编辑。把 `doctor` 从首屏聚合里拿出去更诚实。
- `ExecutionLead`：当前顺序应当是先 0323，再视是否出现保存按钮决定 0322；`0324` 并入 0323 更简洁。

Risks:
1) `0323` 若做成 settings 专用 envelope 平台，会形成第二读模型。
2) `0322` 若做成“万能保存器”，会掩盖跨文件真相与文件级失败。
3) UI 若只看 shell exit 或只看顶层成功，warning 仍然会被吞。

Guardrails:
- `0323` 只聚合 core settings：profile/general/capabilities；doctor 继续独立按需读取。
- `0323` 只能做薄 fan-in，不准加缓存、默认值填充、RPC、守护进程。
- `0322` 当前只允许 profile 可写；其他 tab 明确只读。
- UI 裁决只看 envelope JSON 的顶层/分段 `status`，不是 shell 文本猜测。
- 只读 tab 必须表现成只读，不给保存假象。

Next Actions:
1. 先推进 `0323`，定义 `appliance settings` 的 core aggregate 读面。
2. `0322` 保留，但只在设置页真的要露保存按钮时再进实现。
3. 不再单独推进 `0324`，其规则继续留在 `0323` 里。
