# PL0402：Provider Failure Evidence Handoff

## 结论

这条线只做失败事实回流，不做恢复。

第一刀要解决的是：

- provider 失败事实如何落盘
- 下一轮模型如何看到最近一次相关失败

不解决的是：

- 自动 retry
- fallback
- repair
- 错误分流控制

## 最小证据结构

```json
{
  "provider": "minimax",
  "model": "MiniMax-M2.7",
  "upstreamStatus": 500,
  "errorType": "api_error",
  "message": "unknown error, 520 (1000)",
  "requestId": "0612c9e5d11c990422624f6c3f0ade2d",
  "runId": "02bce2b2-f0e2-4df9-a7d7-483295b7dad8",
  "occurredAt": "2026-03-25T08:03:51.349+08:00"
}
```

## 落盘边界

- 证据归 run / thread 关联面所有
- 第一刀优先复用现有 run/thread 真相，不新造全局告警中心
- 如果现有 run evidence 已能承载，就优先就地收口

## 回流边界

- 只回流最近一次相关 provider 失败
- 只回流只读事实
- 不附带系统解释性长文
- 不把失败变成系统替模型做结论

建议最小回流文案形状：

```text
最近一次 provider 调用失败：
- provider=minimax
- model=MiniMax-M2.7
- upstream_status=500
- message=unknown error, 520 (1000)
- request_id=0612c9e5d11c990422624f6c3f0ade2d
```

## 接线顺序

1. 找到 provider 失败统一出口
2. 落最小证据
3. 在下一轮上下文装配时读取最近一次相关失败
4. 仅在存在失败证据时注入

## Guardrails

- 不新增 retry
- 不新增 fallback
- 不新增 repair middleware
- 不吞原始 provider 错误
- 不引入新的错误控制中心
- 只做最近一次失败的 handoff，不做历史长列表注入

## Review Checklist

1. 有没有把 evidence handoff 偷偷做成 retry 入口？
2. 有没有把 provider 原始错误改写得过头？
3. 有没有把多个失败历史一股脑塞进模型上下文？
4. 有没有因为这个特性新增 manager / supervisor / controller？
5. 有没有破坏现有 fail-fast 行为？
