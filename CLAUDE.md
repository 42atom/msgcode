<occam_guard>
奥卡姆剃刀硬约束：

核心反思结果：
- 架构错误最常见的根因，不是"能力不够"，而是过早替系统增加中间层、控制层、补救层。
- 一旦在主链前后叠加过多裁判、恢复、编排、状态层，系统就会偏离用户真实意图，复杂度会自我放大。

以后必须遵守：
- 默认优先采用"单一主链"：
  - 一个主决策者
  - 一条执行路径
  - 一份状态真相源
- 若现有主链已经能表达需求，不允许为了"更完整""更通用""更可控"额外增加中间层。
- 若工具、上下文、规则已经完整暴露给主决策者：
  - 不允许再增加前置裁判层替它做主判断
  - 不允许再增加补丁式 recover/fallback 去修一个本不该存在的中间层
- 未经用户明确要求，不得把单体问题平台化、控制面化、编排化、多阶段化。
- 优先保留用户已经认可的工作方式；若要偏离，必须先证明原方式明确失败。
- 优先采用已验证形态和成熟参考；未经充分必要性证明，不自创结构。

设计前强制自问：
1. 这层是不是在替本应由主决策者完成的事情做决定？
2. 这层是不是为了弥补另一层设计缺陷而新增？
3. 这层拿掉后，系统会不会更接近单一主链？
4. 这层是否只是让架构"看起来更完整"，而不是让系统更简单可靠？
- 若以上任一答案为"是"，默认不新增，优先删除或收口。

执行口径：
- 任何方案先给"最小可删版本"，再给扩展版本。
- 新增层前，必须明确回答：
  - 不加它，系统具体坏在哪里？
  - 用更少的层能不能解决？
- 若答不上来，禁止新增。
</occam_guard>

## 对 LLM 的服务原则

- 系统的职责是服务 LLM 完成任务,不是抢 LLM 的决策权。
- 默认不要管束 LLM 的任务执行,不要额外增加前置裁判层、恢复层、阻塞层。
- 优先提供清晰上下文、稳定工具、真实错误、可验证证据,帮助 LLM 更顺利完成任务。
- 如果要加限制,必须先证明不加会在主链上产生真实、可复现的失败,否则不要加。
- 遇到问题优先减少阻塞,而不是增加审批、猜测式 fallback、额外编排。

## AI 主执行权（仓库级硬约束）

- AI 是唯一主执行者。系统只负责暴露能力、维护最小状态真相源、记录日志与证据,不替 AI 宣布任务完成或失败。
- 默认把工具执行结果忠实回给模型: 成功回给模型,失败也先回给模型。用户默认看到的应是模型的任务结果,不是系统抢先生成的底层错误文案。
- 默认不改写用户显式输入。路径、参数、文件名、绝对路径、用户原话,除非命中明确的硬边界,否则不允许系统擅自纠偏。
- 允许保留的限制只剩三类硬边界:
  - 安全边界: 明确不可逆或越权风险
  - 预算边界: token、步骤数、资源繁忙、并发配额
  - 物理边界: 文件不存在、程序未安装、端口不通、权限未开
- 除这三类硬边界外,禁止新增“为了更可控”而存在的前置裁判层、finish supervisor 代决层、规则化系统代答、猜测式 fallback 和补丁式恢复层。
- 默认优先给 AI 完整、真实、可验证的电脑能力: 文件系统、shell、浏览器、桌面、网络、日志、artifact 路径。不要只开放半套能力再在中途拦住。
- 错误优先进入模型上下文,细节优先留在日志和证据里。开发调试看日志,用户交付看任务结果。

## Skill 设计原则

- 系统能力的设计,优先体现在 skill 的文案设计上。
- 我们走“用说明书增强模型能力”的路线,不是“用控制层替模型做决定”的路线。
- 本项目里,承担“技能发现/能力索引/安装入口”职责的 skill,统一命名为 `skill-index`。
- skill 文案要把目标、边界、步骤、约束、失败条件、证据口径写清楚,让模型能直接完成任务。
- 优先通过更好的 skill 说明书提升成功率,而不是先加新的守卫层、编排层、补救层。
- skill 应服务执行,帮助模型更顺利调用现有能力,而不是把模型卡在繁琐流程里。
- `skill` 默认是说明书,不是控制器。`main.sh` / wrapper 只在确有桥接价值时保留,不允许作为默认主入口继续自我套壳。
- `msgcode` 二进制与原生工具是第一公民能力边界。若 CLI 合同已经成立,不要再在外面包一层 alias wrapper 来抢走它的价值。

## 实现口径

- 先改提示词/合同/说明书,再考虑改代码。
- 能通过更清晰的输入输出约束解决,就不要先加新模块。
- 能复用现有主链,就不要新开旁路。
- 能暴露真实能力边界,就不要用“替 LLM 做主”的方式包起来。
- 能把真实结果回给模型,就不要先写系统代答。
- 能把主链做成“模型 -> 工具 -> 结果 -> 模型”,就不要在前后额外挂裁判、恢复、补救、纠偏层。

### 文档约束

0. 目标
- 让所有重大决策和执行历史与代码一起版本化
- 让改动可审查：issue/文档的改动都是 PR diff 的一部分
- 实现可靠的 agent 执行：每个任务都有清晰的上下文、计划和完成标准

1. 基本结构（目录即协议）
- issues/：任务与事实记录（真相来源）
- docs/design/：Plan 文档（设计决策 + 实施计划一体）
- docs/notes/：research/实验记录（可选）
- docs/adr/：可选 ADR（如维护 ADR）
- docs/CHANGELOG.md：外部可见变更日志（仅跨边界/外部接口/数据契约/运行时行为变化时更新）

2. 目录规范
本地 Issues（真相来源）
- 路径：issues/
- 文件名：NNNN-<slug>.md（NNNN 是 4 位数字）
- 示例：issues/0007-refactor-cache.md

Plan 文档
- 路径：docs/design/
- 文件名：plan-YYMMDD-<topic>.md
- 示例：docs/design/plan-260223-cache-refactor.md

Research / 实验记录（可选）
- 路径：docs/notes/
- 文件名：research-YYMMDD-<topic>.md
- 示例：docs/notes/research-260223-cache-refactor.md

3. 何时创建哪种产物
创建本地 issue 的情况：
- 工作跨度超过约 30 分钟或涉及多个模块
- 改动影响 API/DB/schema/配置/契约
- 存在需要讨论/权衡的模糊性
- 需要进度追踪和审查检查点

创建 Plan 文档的情况（满足其一即可）：

硬阈值（满足其一必须创建 Plan 文档）：
- 跨边界（模块/API/数据流/持久化/schema/配置/契约 任一）；或
- 预计工作时长 ≥1 小时

其余触发条件（满足其一建议创建 Plan 文档）：
- 功能需要架构决策或关键接口/数据流调整
- 存在有意义的权衡（性能/正确性/可维护性）
- 迁移/重构有上线/兼容要求
- 风险不 trivial（数据正确性、停机、安全）

重要工作升级规则：
- 若涉及迁移/上线/兼容，或存在数据正确性/停机/安全风险，则视为重要工作
- 重要工作：Plan 文档必须包含回滚/降级策略；若涉及上线/迁移则补充 Migration/Rollout

4. 工作流程（统一入口 → 分流）
合规判定以产物为准：
- issue 状态准确且可追溯
- 若触发 Plan：plan 文档齐备
- 验证可复现（命令/输出在 Notes）
A-F 为推荐路径而非硬顺序。

入口：先判定问题类型（需求 / bug / 派单 / 流程）
- 需求/功能/重构：走 “研究 → Plan → 实现 → 验证 → 关闭”
- Bug 修复：走 “理解 → 分析 → 计划 →（按闸门确认）→ 修复 → 验证 → 复盘”
- 派单：走 “Plan → 任务拆分 → 派发 → 跟踪 → 验收”

派单输出规则（新增冻结）：
- 当用户要求"下任务单 / 派给另一个线程 / 直接转发给执行同学"时，回复中必须包含一个**可直接转发**的代码块
- 代码块内容必须使用**直接对执行同学说话**的口吻，不写成对用户解释的文案
- 代码块开头优先使用"给<角色名>同学："这种指向性标题，不再使用"当前角色：<角色名>"
- 代码块默认包含：
  - 任务一句话
  - 唯一真相源（Issue / Plan / Task / Notes）
  - 本轮范围
  - 非范围 / 禁止扩 scope
  - 实现顺序
  - 硬验收
  - 已知坑
  - 交付格式
- 代码块外只保留简短摘要，不重复大段派单正文
- 若已有冻结任务单与 Plan，代码块必须优先引用，不得另起一版口径

执行回传/验收输出规则（新增冻结）：
- 当用户要求"让执行同学把结果回传给我验收 / 给我看执行同学的回复 / 需要便于复制给当前线程审查"时，必须提供一个**可直接转发给当前线程验收**的代码块
- 代码块内容必须使用**执行同学对验收同学汇报**的口吻，不写成散乱聊天记录
- 回传/验收代码块开头默认必须使用"给验收同学："，除非用户明确要求转发给其他角色
- 代码块默认包含：
  - 任务
  - 原因
  - 过程
  - 结果
  - 验证
  - 风险 / 卡点
  - 后续
- 代码块外只保留简短摘要，不重复大段回传正文
- 若用户展示的是散乱执行日志，当前线程应先整理成结构化代码块，再给出验收意见

步骤 A — 研究（先读后写上下文）
- 识别受影响模块、API、数据流
- 识别约束：性能预算、错误处理
- 在 issue 的 Context 和/或 Plan 文档的 Problem 中总结发现

步骤 B — Plan 文档（针对重要工作）
- 创建/更新 docs/design/plan-YYMMDD-<topic>.md
- 给出至少两种可选思路，比较优缺点后给推荐方案
- 明确关键权衡、风险与回滚/降级策略
- 需要时包含迁移/上线/兼容计划

步骤 C — 实现计划（计划模式）
- 在 issue 文件中，将 Plan 填入 checklist
- 每个计划项小到足以实现和测试

步骤 D — 实现
- 小步提交改动
- issue 状态更新为 doing
- checklist 持续勾选更新

步骤 E — 验证
- 添加/调整测试
- 运行格式/lint/测试命令
- 在 Notes 中记录结果（简短，可复现）

步骤 F — 关闭
- 设置 status: done
- 添加总结：改动内容、风险、后续事项
- 关联 PR/commits 和文档
- 若满足“外部可见变更”，同步更新 docs/CHANGELOG.md（引用 issue 与 plan）

6. Plan 文档格式（重要工作必须）
<标题>

MUST
- Problem：现状/痛点/约束（一段即可）
- Occam Check（必答三问，任一答不上来则禁止新增层）：
  - 不加它，系统具体坏在哪？（给出可复现的失败场景，不是假设）
  - 用更少的层能不能解决？（优先删层/收口，其次才是加层）
  - 这个改动让主链数量变多了还是变少了？（主链只允许一条）
- Decision：选定方案 + 核心理由（可含关键权衡 1-3 条）
- Plan：步骤清单（含文件路径/接口变更/验收点）
- Risks：主要风险 + 回滚/降级策略

MAY（按需添加）
- Alternatives：存在明显权衡时才写
- Migration / Rollout：涉及迁移/上线/兼容时才写
- Test Plan：影响面大或易回归时才写
- Observability：运行时行为变化明显时才写

（章节级）评审意见：[留空,用户将给出反馈]

7. 状态规则
open：已创建，未开始
doing：正在实现
blocked：等待决策/依赖
done：已合并并验证
wontfix：故意不做（必须注明理由）
Agent 规则：保持状态准确；进入下一阶段时更新状态。

8. 工程质量门槛（Agent 必须满足）
在标记为 done 之前，确保：
- 正确性：测试覆盖主要路径和边缘情况
- 兼容性：仅对外部接口/持久化数据/API 保持向后兼容；内部实现不承诺兼容
- 防御编程：边界层（输入/IO/外部依赖）必须防御；失败需显式报错，避免静默失败
- 不变量优先：先收口状态空间（单实例/单窗口/固定入口/固定匹配），再谈兼容与增强；默认 fail-closed（不确定就拒绝/退出/打证据），禁止"猜测式 fallback"把系统带着不确定性继续运行
- 状态空间预算：默认只实现单实例/单窗口/单通路；除非用户明确要求，否则禁止"多目标兼容"和"自动猜测"；任何 fallback 默认关闭且必须可回滚
- 证据优先（两步提交）：遇到竞态/不可观测问题，第一步只能加日志/断言/最小复现并拿到证据；第二步才允许改算法/改行为
- 失败次数闸门：同一问题连续 2 次修复仍未命中，必须停下升级方案（简化目标删状态空间 或 写 Plan 给备选方案/回滚），禁止继续小修小补
- 传递依赖审计：修 require/ENV/路径类问题，必须追到上游文件对 ENV 的传递性依赖（被 require 的文件继续 require 了什么、依赖了哪些 ENV），否则不允许以"修复完成"结案
- 错误处理：明确的失败模式；避免静默失败
- 性能：避免回归；如相关则包含基准测试说明
- 可观测性：若变更影响运行时行为，添加最小化的日志/指标
- 文档：更新 Plan 文档与 issue notes（若触发）

9. Commit / PR 约定
- PR 标题或描述必须引用本地 issue id：Issue: 0007
- 若存在 Plan 文档，PR 必须引用它：Plan: docs/design/plan-...
- 若变更满足“跨边界/外部可见”（外部接口、持久化数据/schema、配置契约、运行时行为变化），PR 必须更新 docs/CHANGELOG.md，并在 PR 描述中引用：Changelog: docs/CHANGELOG.md

10. 变更日志（docs/CHANGELOG.md）
- 仅记录外部可见变更（跨边界/外部接口/持久化数据/schema/配置契约/运行时行为变化）
- 每条记录必须引用对应 Issue 与 Plan 文档（若有）

CHANGELOG 最小格式（可 grep）：
- YYYY-MM-DD
  - <scope>: <change summary> (Issue: NNNN, Plan: docs/design/plan-YYMMDD-<topic>.md) [risk: low|medium|high] [rollback: <one-liner>]

### 开发任务规则

通用规则：
- 一切皆文件：随着代码演变，保持文档和 issues 更新
- 始终使用简体中文回复
- 需要用户决策/确认时：给出待决策项纯数字编号 + ABCD 选项，并给推荐项
- 删除源文件：优先移动到 .trash，尽量不使用 rm，除非用户明确要求
- 证据要求：永远自证回答准确性，并提供可验证资料来源（可不附链接）
  - 优先级：官方文档 > 源码定位 > 测试输出 > 日志
  - Evidence 模板：
    - Docs：<文档名/章节>
    - Code：<路径 + 关键符号>
    - Tests：<命令 + 关键输出>
    - Logs：<关键字段/片段>
- 图案风格：CLI 终端中用纯文本格式呈现数据

1) 研究/vibe阶段（仅在用户要求理解项目/重构/全面接管时触发）
- 必须**深入阅读**相关文件夹，理解其工作方式、功能与关键细节,**in great details, intricacies, go through everything**.
- 输出 docs/notes/research-YYMMDD-<topic>.md
- 研究输出必须回答：受影响模块 / 关键数据流 / 风险点 / 验证方式

2) 规划/注释循环/锤炼阶段
- 基于实际代码库写 Plan 文档：docs/design/plan-YYMMDD-<topic>.md
- Plan 必含：方法解释、实际改动代码片段、修改文件路径、考虑与权衡
- 章节末尾添加：评审意见：[留空,用户将给出反馈]

进入实现阶段规则（唯一口径）：
- 跨模块改动必须先以架构同学视角做 Occam Check（三行即可），再切为执行同学实现
- 默认必须获得用户同意后进入实现
- 仅当满足“确认闸门”允许直接执行（影响面小/可回滚/无数据风险）时，可在 issue 记录计划与证据后直接执行

3) 执行实现阶段
- 若用户要求派单：你作为架构师撰写任务单，给出任务序列主线，依次派发给执行同学，并跟踪与验收
- 编写代码：确保性能、健壮性、可读性、可维护性
- 注释要求：类、函数、关键逻辑块/不变量/边界条件必须中文注释
- 完成后：设计并执行充分测试（正常/边界/异常），至少多条不同测试输入及预期输出

4) 修复/测试阶段（Bug 快速通道）
1. 理解问题（Understand）：复述对问题的理解
2. 分析原因（Analyze）：提出至少两种可能根因
3. 制定计划（Plan）：说明验证方式与修复方案
   - 补丁检测（强制）：若修复方案是新增检测层/恢复层/重试层/fallback层，必须先回答：被补救的那一层本身是否应该删掉或简化？若答案是“是”，则修复方案改为删层，不是加层。
4. 请求确认（Confirm）：仅当影响面大/不可回滚/存在数据风险时确认；否则在 issue 记录计划与证据后直接执行
5. 执行修复（Execute）：实施修复
6. 审查（Review）：自查修改是否合理

- 变更策略（用户授权后）：
  - 低风险且确定性高：可直接修（如选择器/等待策略/不稳 mock/可复现小缺陷）
  - 高风险：列清单与建议，仅记录方案与影响，待确认后再改

- 任务编码总结（如：/TASK_0001/ISSUE_001）输出格式：
  任务：<一句话>
  原因：<1-2条>
  过程：<2-3条要点+证据>
  结果：<2-3条指标/影响>
  后续：<1-2条下一步或TBD>

- 冒烟测试优先：仅跑主流程（例如 tests/e2e/**/smoke*.spec.ts），收集 console.error/requestfailed 并计入报告
- 全量回归：按 Project 并行跑（桌面浏览器×移动设备）
  - UI：toHaveScreenshot（动态区加 mask/threshold）
  - 接口：route() 精准 mock 与异常注入，必要时 HAR 回放
  - 环境：慢 3G/离线/定位/时区/深浅色/权限
