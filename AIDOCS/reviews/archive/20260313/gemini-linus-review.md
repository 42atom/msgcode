# Linus 视角的架构审阅：为什么你的 `src/` 有 5.4 万行？

我直接看了各目录和 Top 20 最大文件的统计结果。直说吧，**这个代码库的结构已经严重偏离了你宣称的“系统一定要做薄”和“Unix 哲学”**。

如果 Linus 看了这 5.4 万行的 `src/`，他大概会把你这几个目录和文件拎出来骂：

## 1. CLI 层是个巨大的焦油坑 (8,890 行)
**现状：**
`src/cli` 是最大的目录，多达 8890 行！像 `file.ts` (1229 行)、`memory.ts` (965 行)、`schedule.ts` (848 行)、`browser.ts` (814 行)。
**Linus 会说：**
> "CLI 只是一个 UI 壳！它应该薄得像纸一样！它凭什么占掉整个代码库将近 20% 的体积？"

**架构问题：**
核心的业务逻辑（比如文件操作、记忆存储、任务调度）被**直接硬编码**在 CLI 解析器里了。Unix 哲学是核心逻辑作为纯粹的函数/库（Library），CLI 只是把 `process.argv` 翻译下然后调用库。你的这些 CLI 文件不仅在解析参数，还在搞核心的事务处理。建议把真逻辑抽到核心层，CLI 降维到只有一两百行。

## 2. 三个万行级的 God Objects：违反“单一职责”
### 💣 `src/agent-backend/tool-loop.ts` (2396 行)
**现状：**
一个文件 2.4K 行。
**Linus 会说：**
> "你在写什么？一战的重型轰炸机吗？一个循环调度器怎么可能需要两千多行代码？里面塞了多少判断、临时状态、脏代码和特化逻辑？"

**架构问题：**
它不仅仅是一个 "Loop"，它同时承载了 prompt 装配、上下文压实、工具路由、解析、重试、错误兜底乃至部分具体的工具副作用逻辑。这是最典型、最危险的“控制流全聚合”。这东西必须拆！

### 💣 `src/tools/bus.ts` (1695 行)
**现状：**
Bus（总线）的核心理念是：我只管分发，我不管细节。
**架构问题：**
1695 行的总线说明它并不是个“总线”，而是个**怪兽大杂烩注册表**。大量的工具胶水层、权限校验与参数转换被写死在里面了，甚至连某些特化工具（如 DesktopSessionPool 碎片）可能也在里面。

### 💣 `src/listener.ts` (1032 行) & `src/handlers.ts` (925 行)
**架构问题：**
两个加起来接近 2K 行的网络边缘层/事件监听层。同理，监听器只应该负责接线和序列化，业务逻辑漏到底层了。

## 3. 多重控制面的幽灵散落 (控制流重复)
**现状：**
- `src/routes/` 4301 行
- `src/tmux/` 2585 行
- 早前退役但仍然残留极多脚手架的路由逻辑

**架构问题：**
你号称“不新增中间控制面”，但实际上，`tmux`、`routes`、`cli`、`listener` 这四个平面各自都在做**指令解析和路由分发**。系统里至少有 3 种进入主逻辑的入口。这违背了“主链做直、做薄”的理念。你需要一个像 Unix Pipe 一样的统一入口（stdin 协议），外部端（tmux / feishu webhook / CLI stdin）全部翻译成标准协议流，交给唯一的核心管道。这样你的 `src/routes` 和 `src/tmux` 都可以被砍掉 70%。

## 🔪 建议的开刀路径（做薄的具体抓手）

如果你真的想践行 **"一件事只由一个模块做"** 和 **"系统做薄"**，你需要启动三个“瘦身战役”：

1. **CLI 剥离手术**：把 `src/cli/*.ts` 里面涉及核心事务的逻辑抽到领域库，确保每个 CLI 文件只有单纯的 `commander`/`yargs` 壳和一条直接导向 domain 的调用。这块能砍 3K - 4K 行。
2. **肢解 `tool-loop.ts` 巨兽**：把 2.4K 行的文件按关注点撕开：Prompt 装载器（已抽了一部分在 prompt.ts，继续）、循环状态机本身（纯粹的 State Machine）、Provider 适配器。核心的 Loop 大概只需要 300 行。
3. **拍扁 Bus 和 Handlers**：把总线降级为真总线（纯反射/注册制映射），避免手写重复胶水层。

你目前的架构，比起 `Unix 极简工具集`，确实更像一个长满藤蔓的 `Enterprise J2EE Monolith`。要敢于用斧头砍。
