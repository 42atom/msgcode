## 1. 结论

本项目未来的开发方式，必须从“传统软件思维”切换到“智能体软件思维”。

一句话定义：

**用 LLM 处理 LLM 的问题，用系统处理系统的问题。**

这不是偏好，而是架构总原则。

系统不再负责：
- 猜测用户意图
- 替 LLM 决定下一步
- 用前置裁判、协议阻断、次数限制去卡住主链

系统只负责：
- 暴露 skills、路径、状态、工具
- 执行动作
- 返回真实结果
- 记录证据

LLM 负责：
- 理解任务
- 读取 skill
- 选择策略
- 组合工具
- 持续循环
- 在结束前自我审查

---

## 2. 为什么要这样做

过去这段时间已经证明，很多问题并不是“模型不会”，而是“系统在拦模型”。

典型错误包括：

- `no-tool` 提前结束
- `MODEL_PROTOCOL_FAILED` 把主链打死
- 只给半套工具面
- 让系统替 LLM 做“这是不是执行型请求”的判断
- 用硬编码步骤数或工具调用次数卡住自己
- 把不完整能力包装成“系统内建能力”，误导 LLM

这些错误的共同根因是：

**系统越界去做了本应由 LLM 完成的上下文判断与策略选择。**

因此，本指南的目标不是让系统“更聪明”，而是让系统：

- 更薄
- 更稳
- 更少抢权
- 更容易被 LLM 正确使用

---

## 3. 基本原则

### 3.1 铁律

1. 系统只能支持 LLM，不能替 LLM 做主。
2. 系统只负责给工具、给 skill、给路径、给状态、给真实结果。
3. 严禁系统基于消息内容判断“这轮应不应该调用工具”。
4. 严禁系统用 `no-tool`、协议失败、硬编码步骤数等方式把主链提前打死。
5. 遇到困难，不判死，继续循环。
6. Skill 是能力说明书，不是提示词装饰物。
7. 优先让正确路径变成最自然、最容易的路径，而不是去惩罚错误路径。
8. 任何“为了更可控”而新增的裁判层、恢复层、补丁层，默认都应被怀疑。

### 3.2 奥卡姆剃刀

任何设计前先回答三问：

1. 不加这层，系统具体坏在哪里？
2. 用更少的层能不能解决？
3. 这层加进去后，主链是变少了还是变多了？

答不上来，就不要加。

## 4. 系统与 LLM 的职责边界

### 4.1 系统负责什么

系统负责确定性、执行性、状态一致性问题：

- tool bridge
- CLI / wrapper / shell 执行
- 文件、目录、状态真相源
- workspace / route / chat / job / artifact / download 路径
- transport / scheduler / browser / storage 执行内核
- 错误码、失败语义、日志、证据

系统解决的是：

**能不能做、做到哪里、当前状态是什么。**

### 4.2 LLM 负责什么

LLM 负责上下文理解和策略选择问题：

- 当前要读哪个 skill
- 先用 read_file 还是先用 bash
- schedule 应该用 cron 还是一次性方案
- 浏览器任务先 snapshot 还是先 action
- 现在证据是否足够宣布完成
- 失败后下一步如何调整

LLM 解决的是：

**应该怎么做、先做什么、是否做完。**

### 4.3 禁止越界

禁止系统去做这些事情：

- 判断“这句话像不像执行型请求”
- 判断“这轮需不需要工具”
- 判断“先读哪个 skill”
- 判断“下一步必须调哪个工具”
- 判断“工具调用到第几步就该停”

这些都属于 LLM 的工作。

## 5. Skills 的正式定位

### 5.1 Skill 是能力说明书

Skill 不应被写成散乱提示词，也不应被写成系统内部备忘录。

Skill 的正式定位是：

**赋予 LLM 能力的说明书。**

一个好的 skill 应回答：

- 这是什么能力
- 什么时候用
- 允许使用哪些底层工具
- 唯一入口是什么
- 正确命令模板是什么
- 常见错误是什么
- 如何验证成功
- 失败后怎么排查

### 5.2 系统对 skill 的职责

系统只做三件事：

1. 告诉 LLM skills 目录在哪里
2. 告诉 LLM `index.json` 在哪里
3. 暴露读取和执行它所需的底层工具

系统不做：

- 指定某一轮必须读哪个 skill
- 指定必须先读完 skill 才能继续
- 根据业务类型替 skill 设计固定流程

### 5.3 skill 文案标准

未来所有关键 skill 应尽量按以下结构书写：

1. `name`
2. `description`
3. 能力
4. 何时使用
5. 唯一入口
6. 核心规则
7. 最短正确示例
8. 最常见错误示例
9. 验证方式
10. 排障方式

Alma 的 skill 和 `skill-creator` 给我们的启发是：

**skill 必须写到让 LLM 读完就能自然照着做，而不是读完还要猜。**

## 6. 薄编排原则

### 6.1 什么叫“聪明但克制的编排”

系统不是不能编排，而是只能做**薄编排**。

薄编排的定义：

- 帮 LLM 把路铺平
- 不替 LLM 决定方向
- 不把系统变成流程裁判

例如：

- 指向 `skills/index.json`
- 提供 wrapper 入口
- 统一参数格式
- 给 supervisor 一个唯一结束口

这些属于薄编排。

### 6.2 什么叫“厚编排”

以下都属于不该要的厚编排：

- 消息意图识别器
- no-tool 降级器
- protocol failed 判死器
- 关键词规则路由
- “如果是创建就……如果是删除就……”的大量流程树

厚编排的根本问题是：

**系统开始代替 LLM 决策。**

### 6.3 `/loop` 给我们的启发

`/loop` 这类设计最值得学的不是“做一个 slash command”，而是：

- 用户入口非常薄
- 自然语言理解交给 LLM
- 底层调度内核很硬很小
- 结构化动作与调度状态由系统负责

因此，对本项目的正确借鉴是：

- 入口要薄
- skill 要清楚
- wrapper 要稳
- CLI / scheduler / browser 等内核要小而确定

而不是：

- 再造一层新的通用编排平台

## 7. 多 Agent 角色模型

### 7.1 方向

未来系统应当围绕多个 LLM 角色展开，而不是围绕越来越复杂的单一控制器展开。

推荐最小角色集合：

1. 主执行 Agent
2. 完工监督员 Agent
3. 评审 Agent
4. 历史 Agent

### 7.2 角色关系

这不是传统审批流。

正确关系是：

- 主执行 Agent 负责做事
- 监督员负责判断能否结束
- 评审员负责发现缺陷与风险
- 历史员负责提醒重复犯错

它们共享：

- 同一套 skills
- 同一套路径真相源
- 同一套工具面
- 同一份状态证据

### 7.3 配置原则

第一阶段不要过度设计多模型平台。

推荐起步方式：

- 监督员默认复用主模型
- 只换 prompt 和温度
- 只在“准备结束”时调用

也就是说：

**先把机制跑通，再谈模型分工优化。**

---

## 8. 最小监督闭环

### 8.1 定位

监督员不是过程控制官。

它的唯一职责是：

**结束前再看一眼。**

### 8.2 最小闭环

主 Agent 准备结束时：

1. 调用同模型 supervisor
2. supervisor 看当前已有上下文
3. 只返回：
   - `PASS`
   - `CONTINUE: <原因>`
4. 如果连续 3 次 `CONTINUE`
   - 停止循环
   - 返回阻塞原因

### 8.3 注意事项

监督员不应：

- 参与每一步执行
- 成为第二个主 Agent
- 成为规则引擎
- 成为新的控制平台

监督员只负责：

**防止“没做完就停”或“没做却说做了”。**

## 9. 开发方法论

### 9.1 开发一个智能体能力时，优先级顺序

正确顺序：

1. 先把 skill 写清楚
2. 再把 wrapper / CLI / 状态真相源写稳
3. 再做最小运行时接线
4. 最后才考虑监督与评审

错误顺序：

1. 先加意图识别
2. 先加流程控制
3. 先加各种 fallback
4. 先限制 LLM

### 9.2 出问题时怎么排

遇到问题先问：

1. 是能力不存在？
2. 还是 skill 没教清楚？
3. 还是 wrapper/CLI 参数合同没写稳？
4. 还是系统在中途拦路？

不要先问：

- “模型是不是不稳定”
- “要不要再加一层判断”

### 9.3 首选修复顺序

一旦能力主链已经跑通，后续优先修：

1. skill 文案
2. wrapper 透明兼容
3. 参数合同
4. 状态一致性

最后才是：

5. 新架构
6. 新控制层

## 10. 反模式清单

以下都是高风险反模式，后续开发默认禁止：

### 10.1 用系统替 LLM 做消息意图识别

例如：

- 这是“创建型请求”
- 这是“删除型请求”
- 这是“执行型请求”

这类逻辑看似聪明，实际上很脆。

### 10.2 用半套能力面诱导执行

例如：

- 给 `read_file`
- 不给 `bash`
- 让 LLM 读完 skill 之后再把它拦住

这会导致主链被拆断。

### 10.3 用 `no-tool` / `MODEL_PROTOCOL_FAILED` 提前结束

这种逻辑最大的坏处不是严格，而是：

**把一个本该继续推进的任务假装成已经结束。**

### 10.4 把不完整能力伪装成通用能力

例如：

- 只有 `cron`
- 却告诉 LLM “你有通用定时任务能力”

这会诱导错误实现。

### 10.5 为弥补旧设计缺陷继续叠补丁层

例如：

- recover
- fallback
- retry
- parser
- detector
- bypass

如果这些层是在弥补另一层错误设计，优先删旧层，不要叠新层。

## 11. 未来演进路线

### 第一阶段：主链收口

目标：

- skills 成为能力说明书
- wrapper/CLI 成为稳定入口
- 路由和执行主链不再拦路

### 第二阶段：多 Agent 角色化

目标：

- 主执行 Agent
- 监督员
- 评审员
- 历史员

都共享统一真相源和统一工具面。

### 第三阶段：能力卡标准化

目标：

- 所有关键 skill 统一格式
- 统一能力说明书标准
- 统一验证与排障格式

### 第四阶段：Agent 协作系统化

目标：

- 角色之间形成互动
- 人只做编排
- 系统只做支撑
- Agent 自己完成更多判断、验证和纠偏

---

## 12. 最终工作口号

以后开发智能体系统，统一遵循这三句话：

1. **系统只负责指路，不负责拦路。**
2. **用 LLM 处理 LLM 的问题，用系统处理系统的问题。**
3. **先把 skill 写对，再谈架构。**

这就是我们未来的开发方式。
