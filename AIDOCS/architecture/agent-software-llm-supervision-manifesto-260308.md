# msgcode Agent 软件架构宣言（LLM 监督员与多 Agent 角色）

## 1. 结论

msgcode 的目标不是继续长成一个更复杂的传统软件控制系统，而是长成一个：

- 薄系统内核
- 强 LLM 能力层
- 多 Agent 角色协作
- 单一状态真相源

的软件系统。

一句话：

**用 LLM 处理 LLM 的问题，用系统处理系统的问题。**

---

## 2. 为什么要转向 Agent 软件思路

过去的很多问题已经证明，传统软件式的控制思路会持续制造复杂度：

- 用消息意图识别去判断 LLM 该不该调用工具
- 用 no-tool / protocol failed 提前终止执行
- 用半套工具面把 LLM 引到一半再拦住
- 用 shell/正则/if-else 替 LLM 做上下文判断

这些做法的共同问题是：

**系统在替 LLM 做本该由 LLM 自己完成的判断。**

结果不是更稳，而是：

- 主链被打断
- 假成功增多
- 修复越来越补丁化
- 架构越来越像传统流程软件，而不像 Agent 软件

---

## 3. 总体设计原则

### 3.1 铁律

1. 系统只负责提供能力、路径、状态和真实结果。
2. LLM 负责理解任务、读取 skill、选择策略、调用工具、继续循环。
3. 严禁系统基于当前消息内容替 LLM 判断“该不该调用工具”。
4. 严禁系统用前置裁判层、中途协议层、次数限制层把 LLM 提前打死。
5. 遇到困难，不判死，继续循环。
6. Skill 是能力说明书，不是提示词装饰品。
7. 系统只负责指向 skills 目录和必要环境路径，不负责替 LLM 预设执行流程。

### 3.2 系统边界

系统应该做：

- 暴露 runtime skills 目录
- 暴露当前 workspace / config / downloads / artifacts / browser root 等路径
- 暴露底层工具
- 维护状态真相源
- 执行工具
- 返回真实成功或真实失败
- 记录日志与证据

系统不应该做：

- 猜测当前请求是不是“执行型请求”
- 规定模型先读哪个 skill
- 规定模型接下来必须调用哪个工具
- 用 no-tool 直接结束一轮本该继续的执行
- 用人为配额或硬编码步骤数卡死主链

---

## 4. LLM 处理 LLM 的问题

以下问题优先交给 LLM，而不是交给系统硬编码：

- 当前任务应该读哪个 skill
- 读完 skill 后下一步用什么工具
- schedule 应该用 cron / at / launchd / 其他 bash 方案
- 浏览器任务应该先 snapshot 还是先 action
- 当前是否已经“证据足够，可以宣布完成”
- 失败后下一步如何调整

原因很简单：

这些都属于**上下文理解、策略选择、语言判断、任务推进**，本质就是 LLM 擅长的问题。

---

## 5. 系统处理系统的问题

以下问题必须继续由系统负责：

- tool bridge
- workspace / route / session / job 等状态真相源
- 文件、下载物、artifacts 目录规则
- scheduler、browser、transport 等执行内核
- 错误码和失败语义
- 日志、审计、可观测性
- 配置和权限边界

一句话：

**系统负责可执行性，LLM 负责可决策性。**

---

## 6. 多 Agent 角色模型

msgcode 未来不应该只有一个“主模型”，而应该是一个由多个 LLM 角色组成的协作系统。

### 6.1 角色划分

建议的最小角色集合：

1. 主执行 Agent
   - 负责理解用户任务
   - 读取 skill
   - 调用工具
   - 持续推进

2. 完工监督员 Agent
   - 只在“准备结束”时介入
   - 检查证据是否足够
   - 判断是 `pass` 还是 `continue`
   - 不直接写代码，不直接调用业务工具

3. 评审 Agent
   - 发现风险、回归、状态不一致
   - 重点查“没做却说做了”

4. 历史 Agent
   - 对照历史 issue、plan、log、commit
   - 约束重复犯错

### 6.2 角色之间的关系

不是主从审批流，而是协作流：

- 主执行 Agent 负责做事
- 监督员 Agent 负责检查是否能收工
- 评审 Agent 负责发现缺陷
- 历史 Agent 负责提醒历史代价

这不是传统软件里的“固定审批节点”，而是：

**多 LLM 角色围绕同一真相源和同一工具面进行协作。**

---

## 7. LLM 监督员计划

### 7.1 定位

监督员不是过程控制官，不是一路拦截器。

正确定位：

**监督员 = 完工证据审查器**

它只在主 Agent 准备结束时检查：

1. 是否真的改到了目标
2. 是否跑了最小必要验证
3. 是否存在“没做却说做了”
4. 证据是否足够支撑“已完成”

### 7.2 监督员输出

监督员只输出结构化判断：

```json
{
  "decision": "continue",
  "reason": "改了 schedule 投影逻辑，但没有看到 runs.jsonl 的触发证据",
  "required_actions": [
    "创建一个每分钟任务",
    "等待下一分钟",
    "确认 runs.jsonl 出现新记录"
  ]
}
```

或：

```json
{
  "decision": "pass",
  "reason": "代码修改、jobs.json 状态、runs.jsonl 触发记录均已齐备"
}
```

### 7.3 监督员不该做的事

- 不参与每一步执行
- 不替主 Agent 选工具
- 不做新的流程裁判层
- 不制造新的阻断链

---

## 8. Skills 的正式定位

### 8.1 Skills 是能力说明书

Skill 的本质不是“给系统看的提示词附录”，而是：

**赋予 LLM 的能力说明书**

一个好的 skill 应该回答：

- 何时使用
- 可以使用哪些底层工具
- 命令怎么写
- 输入/输出是什么
- 错误写法是什么
- 正确示例是什么

### 8.2 系统对 skill 的唯一职责

系统只需要告诉 LLM：

- runtime skills 目录在哪里
- `index.json` 在哪里
- 当前工作目录在哪里
- 当前 config 在哪里

剩下的由 LLM 自己决定：

- 读哪个 skill
- 怎么用 skill
- 用 bash 还是其他工具执行

### 8.3 不再做的事

- 不再把 scheduler/browser 等能力强注入成固定流程
- 不再暗示“必须先读某个特定 skill”
- 不再把表达不完整的能力伪装成系统内建万能能力

---

## 9. 工具调用主链应该如何演进

### 9.1 要抄的方向

参考 OpenClaw，正确主链应该是：

1. 默认给 LLM 完整底层工具面
2. 工具过滤只来自配置/权限边界
3. skills 注入给 LLM
4. 重点做参数归一化
5. 不做消息级意图识别
6. 不做 `route=no-tool -> MODEL_PROTOCOL_FAILED` 收尾

### 9.2 不该再保留的结构

- allowNoTool 作为默认收尾
- looksLikeExecutionRequest / looksLikeCompletionResponse 这类消息裁判器
- preferredToolName / unexpectedToolNames 这类流程级强约束
- “最多调用 3 次工具”这类传统软件式控制文案

---

## 10. 演进路线

### Phase 1：拆拦路器

目标：

- 去掉 no-tool 假收尾
- 去掉 protocol failed 早杀
- 去掉消息级意图识别
- 让 LLM 稳定进入真实工具链

### Phase 2：稳状态真相源

目标：

- schedule -> jobs -> scheduler 刷新闭环
- workspace / route / thread / task / event queue 的单一真相源
- 消灭“写一边、读另一边”的断裂

### Phase 3：多 Agent 角色化

目标：

- 主执行 Agent
- 完工监督员 Agent
- 评审 Agent
- 历史 Agent

都围绕同一状态真相源和同一工具面工作。

### Phase 4：能力扩张

目标：

- 更完整的 scheduler skill
- 更完整的 browser skill
- 更强的记忆召回和长循环能力

注意：

能力扩张必须发生在“系统不再拦路”之后，而不是之前。

---

## 11. 最终架构宣言

msgcode 的未来方向不是：

- 更复杂的控制面
- 更多前置裁判
- 更多固定流程
- 更多系统替 LLM 做主

而是：

- 更薄的执行内核
- 更强的 LLM 能力层
- 更清晰的 state source
- 更稳定的 tool bridge
- 更成熟的多 Agent 协作

一句话定版：

**msgcode 是 Agent 软件，不是传统流程软件。**

因此：

- 默认支持 Agent 自主决策
- 系统只负责能力、状态、路径、日志和真实结果
- 多 Agent 角色围绕同一真相源协作
- 用 LLM 处理 LLM 的问题，用系统处理系统的问题
